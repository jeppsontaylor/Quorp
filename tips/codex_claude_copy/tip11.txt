## First correction: вҖңRust TUIвҖқ is not the cause of pixelated text

I would not rule out Rust TUI technology. Codex CLIвҖҷs official docs describe it as an open-source coding agent built in Rust, and its public `codex-rs/tui/Cargo.toml` includes `crossterm`, `ratatui`, `syntect`, `diffy`, `unicode-width`, `unicode-segmentation`, and other terminal-rendering dependencies. ([OpenAI Developers][1])

The sharpness is mostly determined by the **terminal emulator, font, HiDPI scaling, ANSI/truecolor output, glyph choice, and redraw strategy**, not by Rust itself. Terminal apps emit cells, Unicode glyphs, and escape sequences; the terminal emulator renders the actual pixels. ANSI SGR sequences support truecolor foreground/background color using RGB escape sequences such as 24-bit `38;2;r;g;b` and `48;2;r;g;b`, and Crossterm exposes RGB colors for modern terminals. ([Wikipedia][2])

My recommendation: build a **terminal-native Rust app** with a custom renderer on top of Crossterm, then optionally use Ratatui for structured views once the visual language is locked. Ratatui itself is a fast, lightweight Rust library for rich terminal UIs, so the problem is not вҖңRatatui vs. not RatatuiвҖқ; the problem is whether we design a beautiful terminal surface instead of a generic boxy dashboard. ([Ratatui][3])

# QUORP technical stack recommendation

## 1. Rendering strategy

Use a **direct terminal renderer** first:

```text
Rust app
  в””в”Җ custom quorp-renderer
       в”ңв”Җ Crossterm raw mode
       в”ңв”Җ alternate-screen rendering
       в”ңв”Җ truecolor palette engine
       в”ңв”Җ Unicode block/braille/sextant renderer
       в”ңв”Җ dirty-region redraw queue
       в””в”Җ optional Ratatui widgets for complex panels
```

The key is to make the terminal feel cinematic without becoming visually noisy. QUORP should look like a dark, glossy, orange-lit coding creature woke up inside the shell.

Use:

| Area                         | Recommended stack                                              |
| ---------------------------- | -------------------------------------------------------------- |
| Terminal control             | `crossterm`                                                    |
| Optional structured UI       | `ratatui`                                                      |
| Text width correctness       | `unicode-width`, `unicode-segmentation`                        |
| ANSI/color support detection | `supports-color`, `anstyle`, `owo-colors` or Crossterm colors  |
| Syntax highlighting          | `syntect`, `two-face`, or Tree-sitter highlighting             |
| Markdown rendering           | `pulldown-cmark` or `comrak`                                   |
| Diff rendering               | `similar`, `diffy`, `git2`/`gix`                               |
| Image-to-terminal text       | `image` + Unicode half-blocks `в–Җ`, `в–„`, `в–Ҳ`                    |
| Async runtime                | `tokio`                                                        |
| Streaming agent output       | `reqwest`, `eventsource-stream`, `tokio-tungstenite` if needed |
| Logging/tracing              | `tracing`, `tracing-subscriber`, `color-eyre`                  |
| Config                       | `serde`, `toml`, `directories`, `figment`                      |
| Fuzzy search                 | `nucleo`, `skim`, or custom matcher                            |
| Workspace walking            | `ignore`, `walkdir`, `globset`                                 |
| Code search                  | `ripgrep` subprocess or Rust `grep` crates                     |
| PTY commands                 | `portable-pty` for interactive processes                       |
| Non-interactive commands     | `tokio::process`                                               |
| Packaging                    | `cargo-dist`, Homebrew formula, npm shim, GitHub releases      |

For the вҖңbrilliantвҖқ look, do **not** rely on terminal image protocols as the default. Kitty and iTerm2 both support inline image mechanisms, but those are terminal-specific; KittyвҖҷs graphics protocol uses a placeholder Unicode cell for images, and iTerm2 has its own inline image protocol. Those are useful as optional premium paths, but QUORPвҖҷs default should use text/Unicode/ANSI so it works broadly. ([Kovid Goyal's Software Projects][4])

Recommended terminals for the most beautiful experience: **WezTerm, Kitty, Ghostty, iTerm2**. WezTerm is cross-platform and implemented in Rust, Kitty is GPU-based and feature-rich, and Ghostty uses native UI plus GPU acceleration. ([WezTerm][5])

## Tool trait

Design every tool as a typed Rust object:

```rust
#[async_trait::async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &'static str;
    fn description(&self) -> &'static str;
    fn json_schema(&self) -> serde_json::Value;
    fn risk(&self, input: &serde_json::Value) -> ToolRisk;

    async fn run(
        &self,
        ctx: ToolContext,
        input: serde_json::Value,
    ) -> anyhow::Result<ToolResult>;
}
```

Risk classes:

```rust
pub enum ToolRisk {
    ReadOnly,
    WorkspaceWrite,
    ShellRead,
    ShellMutate,
    Network,
    SecretsSensitive,
    Destructive,
}
```

Approval policy:

```text
read-only          auto-run
workspace-write    show diff first
shell-read         auto-run if allowlisted
shell-mutate       ask
network            ask unless trusted
secrets-sensitive  block or ask with warning
destructive        never auto-run
```

## Subagents

Use subagents, but restrict tools:

```text
planner       no writes, no shell mutate
researcher    read/search only
coder         patch tools, no destructive shell
reviewer      read/search/test only

```

Claude CodeвҖҷs subagent docs describe allowing or denying tool access per subagent, which is exactly the right model for QUORP. ([Claude API Docs][8])

## Reasoning display

Do not show raw private chain-of-thought. Show **structured reasoning summaries**:

```text
Current hypothesis
  The failing test is caused by the parser accepting empty tool names.

Evidence
  src/parser.rs:81 handles missing name as Ok("")
  tests/parser.rs:42 expects Err(MissingName)

Next action
  Patch parser guard, run parser tests only.
```

Modes:

```text
/fast      short plan, minimal context
/balanced  normal mode
/deep      more planning, more verification, reviewer subagent
```

# Security model

MCP and tool integrations are powerful, but they widen the attack surface. MCP is intended to connect models to external tools/data, and both Claude Code and Codex support MCP-style integrations. ([Claude API Docs][9])

QUORP should have:

```text
1. command allowlist/denylist
2. secret redaction before model calls
3. network policy
4. workspace-root jail
5. read/write path restrictions
6. approval gate before mutations
7. git checkpoint before edits
8. diff preview before applying
9. post-apply verification
10. audit log
```

For shell execution:

```text
macOS/Linux:
  - simple mode: tokio::process
  - interactive mode: portable-pty
  - stronger sandbox: bubblewrap/nsjail/container

Windows:
  - PowerShell support
  - ConPTY for interactive sessions
  - Job Objects for process containment
```

# Final stack choice

I would build QUORP as:

```text
Rust terminal-native app
  Crossterm direct renderer
  Optional Ratatui for heavier panels
  Tokio async core
  Structured tool router
  Typed approval gate
  Git checkpoint/diff engine
  MCP bridge
  Skills/subagents
  Truecolor theme system
  Unicode half-block mascot renderer
```

That gives you the Claude/Codex-style speed and terminal-native feel, while letting QUORP own a much more distinctive visual identity: **charcoal mascot, orange glow, rare magical glyphs, sharp truecolor text, and beautiful red/green diff backgrounds**.
