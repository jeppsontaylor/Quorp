That gives you the Claude/Codex-like behavior: fast streaming, brilliant cards, approval prompts, tool-call traces, green/red diff backgrounds, animated status, rare emojis, and a sharp terminal experience that feels alive instead of “boxed TUI app.”

# 1. The product direction

Claude Code is officially described as an agentic coding tool that can read a codebase, edit files, run commands, and work in the terminal, IDE, desktop app, and browser. Codex CLI is officially described as OpenAI’s local terminal coding agent that can read, change, and run code in the selected directory, and it is open source and built in Rust. Those two facts strongly suggest the correct QUORP target: not a static terminal dashboard, but an **agentic coding console** that streams reasoning summaries, tool calls, diffs, tests, approvals, and final summaries as high-quality terminal events. ([Claude][1])

The mistake would be building “a TUI.” The right move is building a **terminal-native rendering system** that happens to use Rust.

The UI should feel like this:

```text
🦇 QUORP v0.1.0-neon
workspace: ~/repo/infra        mode: workspace-write        model: high-reasoning

🧭 plan
  1. inspect router and patch pipeline
  2. add guarded edit mode
  3. run targeted tests
  4. summarize changed files

🪛 read src/patch/router.rs
  812 lines · 19.4 KB · 4.2 ms

🪡 patch src/patch/router.rs  +18 -6
  risk: medium
  revert: available
  verifier: cargo test -p quorp-patch

🧪 cargo test -p quorp-patch
  running · 12.4s · 41 tests discovered
```

This should be **append-first**, **card-based**, **animated**, **truecolor**, and **streaming**.

# 2. Why “Rust TUI text looks pixelated” is probably the wrong diagnosis

Rust does not render terminal text. The terminal emulator renders the glyphs. A Rust TUI can look bad when the terminal font, fallback emoji font, DPI scaling, ligature settings, or box-drawing glyphs are poor.

So the stack should optimize for terminal emulators that already render text beautifully:

| Target terminal | Why it matters                                                                                                                       |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **WezTerm**     | GPU-accelerated, truecolor, ligatures, color emoji, font fallback, hyperlinks, panes, tabs. ([WezTerm][2])                           |
| **Kitty**       | GPU/SIMD rendering, threaded rendering, images, animations, ligatures, emoji, variable fonts. ([Kovid Goyal's Software Projects][3]) |
| **iTerm2**      | Common macOS target with rich color and image capabilities.                                                                          |
| **Ghostty**     | Excellent modern terminal target, especially for sharp text and GPU-native feel.                                                     |
| **Alacritty**   | Great raw speed, but less visually rich than WezTerm/Kitty.                                                                          |

The QUORP terminal should detect capabilities:

```text
truecolor
OSC 8 hyperlinks
emoji width behavior
Kitty graphics protocol
iTerm image protocol
Sixel
mouse support
bracketed paste
terminal size
font fallback quality
```

Then render in one of these modes:

```rust
enum RenderProfile {
    RichKitty,
    RichWezTerm,
    RichITerm,
    TrueColorUnicode,
    Ansi256,
    PlainNoColor,
}
```

For the mascot and startup screen, the safest beautiful approach is **truecolor Unicode half-block art**. Optional rich mode can use Kitty’s graphics protocol, which is designed to let terminal programs render raster graphics at pixel positions, but the fallback should always be pure text/ANSI. ([Kovid Goyal's Software Projects][4])

# 3. Recommended QUORP architecture

```text
quorp/
  crates/
    quorp-core/          session model, event log, orchestration
    quorp-agent/         model loop, planning, tool-use, reviewers
    quorp-term/          direct ANSI renderer, cards, diffs, splash
    quorp-tools/         read/write/shell/git/LSP/MCP tools
    quorp-patch/         patch planning, apply/revert, diff render
    quorp-index/         repo map, symbols, semantic search
    quorp-lsp/           LSP client integration
    quorp-sandbox/       approvals, policy, path guards
    quorp-protocol/      event protocol for terminal/desktop/web
    quorp-daemon/        local background service
    quorp-cli/           CLI entrypoint
    quorp-desktop/       optional GPUI or Tauri shell
```

The critical architectural rule:

> **The UI is only a projection of an event stream.**

Every agent action emits typed events:

```rust
enum QuorpEvent {
    SessionStarted(SessionInfo),
    UserMessage(UserMessage),
    ReasoningSummary(ReasoningSummary),
    ToolStarted(ToolCall),
    ToolDelta(ToolDelta),
    ToolFinished(ToolResult),
    ApprovalRequested(ApprovalRequest),
    ApprovalResolved(ApprovalDecision),
    PatchProposed(PatchSet),
    PatchApplied(PatchSetId),
    PatchRejected(PatchSetId),
    CommandStarted(CommandRun),
    CommandOutput(CommandOutputChunk),
    CommandFinished(CommandResult),
    TestSummary(TestSummary),
    AgentDone(FinalSummary),
}
```

The terminal, desktop app, app-server, and logs all subscribe to the same event stream.

This gives you:

```text
terminal UI
desktop UI
headless mode
CI mode
record/replay
session restore
remote control
audit logs
```

# 4. Core Rust stack

## Runtime and infrastructure

| Area          | Rust crates / tools                       | Purpose                                               |
| ------------- | ----------------------------------------- | ----------------------------------------------------- |
| Async runtime | `tokio`, `futures`                        | model streams, tool calls, PTYs, file watching        |
| Errors        | `thiserror`, `anyhow`                     | internal + user-facing diagnostics                    |
| Serialization | `serde`, `serde_json`, `toml`, `schemars` | tool schemas, config, event logs                      |
| Logging       | `tracing`, `tracing-subscriber`           | trace every tool call and render event                |
| CLI           | `clap`                                    | `quorp`, `quorp exec`, `quorp splash`, `quorp doctor` |
| Config        | `figment` or custom layered TOML          | global/project/team policy                            |
| IDs           | `uuid`, `nanoid`, `time`                  | session IDs, patch IDs, tool IDs                      |
| Concurrency   | `dashmap`, `parking_lot`, `arc-swap`      | shared caches and state                               |

## Terminal renderer

| Area                 | Rust crates                             | Purpose                                                                 |
| -------------------- | --------------------------------------- | ----------------------------------------------------------------------- |
| Terminal control     | `crossterm`                             | cursor, raw mode, clear, keyboard, mouse                                |
| Escape parsing       | `termwiz`                               | parse/encode escape sequences and terminal capabilities. ([Docs.rs][5]) |
| PTY                  | `portable-pty`                          | run shells/commands in pseudo-terminals                                 |
| Unicode width        | `unicode-width`, `unicode-segmentation` | correct emoji, CJK, grapheme alignment                                  |
| Styling              | `anstyle`, `owo-colors`                 | truecolor ANSI composition                                              |
| ANSI cleanup         | `strip-ansi-escapes`                    | measuring visible widths                                                |
| Optional TUI widgets | `ratatui`                               | only for tables/popups, not the core aesthetic                          |

Do **not** start with a Ratatui-first layout. Ratatui is useful for terminal widgets, but QUORP should be a custom renderer that writes direct ANSI spans and tracks dirty regions. Ratatui is documented as a Rust library for terminal user interfaces; that is valuable, but your desired look is closer to a **streaming terminal design system** than a classic widget grid. ([Docs.rs][6])

# 7. Diff and patch engine

The diff UI is one of the most important emotional features. You specifically like the green/red diff backgrounds, so QUORP should make this a signature feature.

Difftastic is a good design reference because it compares files based on syntax rather than raw lines, producing diffs that are easier for humans to read. ([Difftastic][8])

Recommended stack:

```text
similar        line + word diffs
imara-diff     fast diff core
diffy          patch application
difftastic     optional structural diff inspiration / external integration
tree-sitter    syntax-aware hunk boundaries
```

Diff display style:

```text
🪡 patch src/router.rs  +18 -6
risk: medium     revert: pch_01JZ...     verifier: cargo test -p quorp-router

╭─ hunk 2/5 ─ route_tool_call() ─────────────────────────────╮
│                                                            │
│  141 │ match tool.kind {                                   │
│  142 │     ToolKind::Shell => {                            │
│ -143 │         self.run_shell(cmd).await                   │
│ +143 │         self.run_shell_guarded(cmd, policy).await   │
│ +144 │             .context("guarded shell failed")?       │
│  145 │     }                                               │
│                                                            │
╰────────────────────────────────────────────────────────────╯
```

Implementation details:

```rust
struct PatchSet {
    id: PatchSetId,
    files: Vec<FilePatch>,
    risk: Risk,
    rationale: String,
    revert_patch: Option<String>,
    verification: Option<VerificationPlan>,
}

struct FilePatch {
    path: PathBuf,
    hunks: Vec<Hunk>,
    before_hash: String,
    after_hash: String,
}

struct Hunk {
    old_start: usize,
    old_lines: usize,
    new_start: usize,
    new_lines: usize,
    lines: Vec<DiffLine>,
}
```

Patch UX must include:

```text
preview

reject
edit patch
explain patch
test patch
revert patch
open in editor
copy unified diff
```

Never silently mutate files without a visible patch event.

# 16. Splash implementation strategy

The important trick is this:

> Convert the mascot PNG into ANSI truecolor Unicode half-blocks.

Each terminal cell uses `▀`:

```text
foreground color = top pixel
background color = bottom pixel
glyph = upper half block
```

So one terminal row represents two image rows. This is pure text, but it looks much more detailed than ASCII.

Core conversion idea:

```rust
for y in (0..height).step_by(2) {
    for x in 0..width {
        let top = image.get_pixel(x, y);
        let bottom = image.get_pixel(x, y + 1);

        print!(
            "\x1b[38;2;{};{};{}m\x1b[48;2;{};{};{}m▀\x1b[0m",
            top.r, top.g, top.b,
            bottom.r, bottom.g, bottom.b
        );
    }
}
```

That is the “not Rust TUI” path: direct terminal rendering, direct ANSI, no widget framework required.
