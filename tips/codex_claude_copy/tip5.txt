# Executive direction

I agree with your instinct: **do not build the main experience as a classic full-screen Rust TUI**. Build QUORP as a **terminal-native agent IDE** with a custom ANSI renderer, scrollback-friendly transcript, rich inline cards, high-contrast truecolor styling, beautiful diffs, and a very fast Rust agent core.

Important nuance: **Codex CLI itself is officially described as open source and built in Rust**, and it runs locally in the terminal where it can read, change, and run code in the selected directory. That means Rust is absolutely viable for this product class; the thing to avoid is not Rust, it is a blocky, over-redrawn, widget-grid TUI aesthetic. Claude Code is documented as an agentic coding tool that reads codebases, edits files, runs commands, and works in the terminal, IDE, desktop app, and browser. ([OpenAI Developers][1])

The QUORP product should feel like this:

```text
🜂 QUORP is not a dashboard.
🪡 QUORP is a cinematic terminal-native coding agent.
🧿 It streams, patches, reasons publicly, asks for approval, and glows.
🪩 It has a mascot, a brand, and a visual language.
```

# 1. Why your Rust TUI text felt pixelated

The pixelated feel usually comes from the **terminal emulator, font, glyph choices, box-drawing density, and overuse of cell graphics**, not from Rust itself. A Rust TUI library draws characters into terminal cells; it does not rasterize the font. So the QUORP rule should be:

> Use Rust for speed, but let the terminal emulator render crisp text. Avoid a full-screen grid as the primary interface.

For the main interface, use **direct ANSI/VT rendering** or a light Crossterm wrapper. Crossterm supports terminal styling and RGB colors on terminals that support them, which is exactly what QUORP needs for high-contrast truecolor cards and diff backgrounds. ([Docs.rs][2])

Use Ratatui only for narrow, temporary panels: command palette, patch picker, file tree, or permissions modal. Ratatui is excellent, but it is immediate-mode and redraws the UI every frame; that can feel like a “TUI app” if used as the whole product surface. ([Ratatui][3])

# 2. Recommended QUORP architecture

```text
┌─────────────────────────────────────────────────────────────┐
│                    QUORP Terminal Client                    │
│  ANSI transcript renderer · prompt input · diff cards       │
│  status cards · approval cards · splash screen · themes     │
└───────────────────────┬─────────────────────────────────────┘
                        │ JSON-RPC / JSONL / local socket
┌───────────────────────▼─────────────────────────────────────┐
│                    QUORP Agent Server                       │
│  sessions · model loop · tool broker · policy engine        │
│  memory · checkpoints · telemetry · event stream            │
└───────┬────────────┬─────────────┬──────────────┬───────────┘
        │            │             │              │
        ▼            ▼             ▼              ▼
   File tools    Shell/PTY      Git/diff       LSP/index
   read/write    sandbox        patch flow     symbols/diagnostics
```

This is the same architectural spirit as modern coding agents: a **client layer** that displays state beautifully, and a **long-lived agent runtime** underneath. OpenAI’s Codex App Server docs describe bidirectional JSON-RPC-style communication over stdio JSONL by default, with WebSocket listed as experimental; that is the exact kind of split QUORP should adopt. ([OpenAI Developers][4])

The payoff: terminal client now, GPUI desktop later, VS Code extension later, web client later, all talking to the same QUORP agent server.

# 3. The core Rust stack

| Area                        | Recommended stack                                                      | Why                                                                                                                               |
| --------------------------- | ---------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| CLI entry                   | `clap`, `serde`, `toml`, `directories`, `color-eyre`                   | Fast command parsing, config, error reports.                                                                                      |
| Async runtime               | `tokio`, `tracing`, `tracing-subscriber`                               | Model streaming, tool calls, shell output, LSP, file watchers.                                                                    |
| Terminal output             | **custom ANSI renderer**, optional `crossterm`                         | Crisp transcript rendering, truecolor, cursor control, no full-screen TUI dependency.                                             |
| Focused panels              | `ratatui` only for patch picker, palette, file tree                    | Great widgets, but not the main UI aesthetic.                                                                                     |
| Text buffer                 | `ropey`                                                                | Fast Rust text rope for editor-grade manipulation. Ropey describes itself as a fast and robust text rope for Rust. ([Docs.rs][5]) |
| Syntax parsing              | `tree-sitter`                                                          | Incremental parsing and concrete syntax trees for code-aware highlighting and structural diffs. ([Tree-sitter][6])                |
| Language intelligence       | `lsp-types`, custom LSP client                                         | Diagnostics, definitions, references, hover, code actions.                                                                        |
| Diff engine                 | `imara-diff`, `similar`, custom renderer, optional `difftastic` bridge | Beautiful inline diffs with token highlights and hunk approval.                                                                   |
| Git                         | `git` subprocess first, then `gix`/gitoxide                            | Compatibility first, Rust-native later.                                                                                           |
| Shell/PTY                   | `portable-pty`, `tokio-process`, ring-buffer logs                      | Run commands, stream output, background tasks.                                                                                    |
| Search                      | `ignore`, `ripgrep`, `tantivy` optional                                | Respect `.gitignore`, fast text search, optional semantic/local index.                                                            |
| Fuzzy UI                    | `nucleo` or `skim` integration                                         | Fast command palette, file picker, symbol picker.                                                                                 |
| Agent protocol              | JSON-RPC 2.0-style messages over stdio/Unix socket/WebSocket           | Clean multi-client architecture.                                                                                                  |
| Tool ecosystem              | MCP client/server support                                              | MCP exposes tools, resources, and prompts to AI clients. ([Model Context Protocol][7])                                            |
| Native client later         | GPUI or Tauri                                                          | GPUI is a Rust UI framework from the creators of Zed; Tauri is faster if you want web UI with Rust backend. ([Gpui][8])           |
| Native text rendering later | COSMIC Text + wgpu                                                     | COSMIC Text handles shaping, font discovery, font fallback, layout, rasterization, and editing. ([Docs.rs][9])                    |

# 4. What QUORP should look like

The main interface should be **transcript-first**, not dashboard-first.

```text
✦ you
Add JSON output to quorp status and include test coverage.

🧭 plan
1. Inspect the CLI command.
2. Find the status renderer.
3. Add --json.
4. Patch tests.
5. Run focused test suite.

🛰 scan
src/cli.rs        3 matches
src/status.rs     8 matches

🪡 patch  src/status.rs  +42 -3

🧪 test
cargo test -p quorp status_json ... ok
```

That is the feel: **fast, crisp, readable, high signal**.

Not this:

```text
┌────────┬───────────────┬─────────────┬─────────────┐
│ PANEL  │ PANEL         │ PANEL       │ PANEL       │
│ PANEL  │ PANEL         │ PANEL       │ PANEL       │
└────────┴───────────────┴─────────────┴─────────────┘
```

The latter screams “TUI dashboard.” QUORP should scream “agentic coding instrument.”

# 8. Tool calling design

OpenAI’s tool-calling flow is: give the model tool definitions, receive a tool call, execute it in your app, send tool output back, and continue until final response or more tool calls. That maps cleanly into a QUORP `ToolBroker`. ([OpenAI Developers][10])

## Internal event model

```rust
enum QuorpEvent {
    SessionStarted { id: SessionId, repo: PathBuf },
    UserPrompt { text: String },

    PublicPlanUpdated { steps: Vec<String> },
    ModelTextDelta { text: String },

    ToolProposed { id: ToolCallId, name: String, risk: Risk },
    ApprovalRequested { id: ToolCallId, reason: String },
    ToolStarted { id: ToolCallId },
    ToolOutput { id: ToolCallId, chunk: String },
    ToolFinished { id: ToolCallId, status: ToolStatus },

    PatchPreviewed { id: PatchId, stats: PatchStats },
    PatchHunkAccepted { id: PatchId, hunk: usize },

    CommandStarted { id: ToolCallId, command: String },

    CommandExited { id: ToolCallId, code: i32 },

    CheckpointCreated { id: CheckpointId },
    SummaryReady { text: String },
}
```

# 10. Permission model

Claude Code’s docs describe hooks as shell commands, HTTP endpoints, or prompts that execute at specific lifecycle points, with events such as pre-tool use, permission requests, post-tool use, and session lifecycle events. QUORP should have a similar hookable lifecycle, but with your own policy model and visual language. ([Claude API Docs][11])

## Risk tiers

| Tier  | Examples                                                     | Behavior                       |
| ----- | ------------------------------------------------------------ | ------------------------------ |
| Green | Read files, search repo, git status                          | Auto-run                       |
| Blue  | Run tests, format, typecheck                                 | Auto-run inside repo           |
| Amber | Edit files, apply patch, package install                     | Preview + ask                  |
| Red   | Delete files, network upload, deploy, force push             | Always ask                     |
| Black | Credential scraping, `.env` reads, destructive root commands | Block unless explicit override |

## Approval card

```text
🪬 approval needed
Command wants to modify package files:

  npm install zod

Reason:
  The generated parser needs runtime validation and package-lock.json will change.

[enter] allow once   [a] always for repo   [n] deny   [v] view details
```

## Hook config

```toml
[hooks.pre_tool_use]
shell.exec = ["quorp-policy check-command"]

[hooks.post_patch]
run = ["cargo fmt", "cargo test --workspace"]

[hooks.session_start]
run = ["quorp context hydrate"]

[hooks.permission_request]
run = ["quorp audit pending-tool"]
```

# 11. Diff experience

This should be QUORP’s signature feature.

```text
🪡 patch  src/status.rs  +42 -3
╭────────────────────────────────────────────────────────────╮
│ @@ impl StatusOutput                                      │
│  88 │ let status = self.collect_status()?;                │
│  89 │                                                     │
│ +90 │ if opts.json {                                     │
│ +91 │     println!("{}", serde_json::to_string(&status)?);│
│ +92 │     return Ok(());                                 │
│ +93 │ }                                                  │
│  94 │ self.render_human(status)                          │
╰────────────────────────────────────────────────────────────╯
```

Must-have diff controls:

```text
[a] accept hunk
[r] reject hunk
[e] edit hunk
[x] explain hunk
[s] split hunk
[u] undo last patch
[c] checkpoint
```

Visual rules:

```text
Green background = new code
Red background   = removed code
Blue background  = hunk header
Yellow background = changed token
Violet border    = selected hunk
Orange badge     = patch in progress
```

Implementation plan:

```text
1. Generate unified diff from proposed patch.
2. Parse into files/hunks/lines.
3. Add token-level diff for changed lines.
4. Add tree-sitter semantic labels when available.
5. Render compact inline card in transcript.
6. Offer full patch review mode for wide terminals.
7. Store patch as reversible transaction.
```
