## Recommendation

Build the desktop app as a **Tauri 2 app in `/apps/desktop`**, with a **React + TypeScript + Vite frontend** and a **Rust-owned backend bridge** that reuses Quorp’s existing runtime, session, provider, permissions, sandbox, patch, verify, render, and benchmark crates.

The desktop app should **not** replace the CLI. It should become a second first-class shell over the same Rust agent runtime:

```text
Quorp CLI
  crates/quorp
  crates/quorp_cli
        │
        ▼
Shared Rust agent/session/runtime crates
  quorp_agent_core
  quorp_session
  quorp_provider
  quorp_core
  quorp_permissions
  quorp_sandbox
  quorp_patch_vm
  quorp_verify
  quorp_context
  quorp_memory
        ▲
        │
Quorp Desktop
  apps/desktop              ← Tauri app, web UI, macOS bundle/DMG config
  crates/quorp_desktop_bridge ← typed Rust bridge between UI and runtime
```

I could not complete a local `git clone` from this sandbox because GitHub DNS resolution failed, so I inspected the public GitHub tree/raw files and current official docs instead. The repo already strongly supports this plan: Quorp is explicitly a Rust **TUI-first text editor and agentic interface**, the CLI/session/runtime pieces are split into reusable crates, the workspace already contains agent, session, provider, context, memory, patch, verify, permissions, sandbox, render, term, slash, and benchmark crates, and the runtime already emits rich structured events that are suitable for a desktop event stream. ([GitHub][1])

Tauri is the right packaging layer because it gives you a Rust backend, any HTML/CSS/JS frontend, native WebView rendering, small binaries relative to Electron, built-in `.app`/`.dmg` bundling, and signing/notarization support. Tauri also avoids running a localhost web server for normal app UI by using a native WebView protocol, which is better for Quorp’s security posture. ([Tauri][2])

# 1. Ideal repo location

Put the desktop app here:

```text
/apps/desktop
```

Do **not** put the Tauri app directly under `crates/`.

Reason: the desktop app is not just a Rust library. It contains frontend assets, package manager files, native icons, macOS entitlements, Tauri config, app menus, updater config, and bundle metadata. The reusable Rust logic should live in crates; the native app shell should live under `apps`.

Add one new bridge crate:

```text
/crates/quorp_desktop_bridge
```

This crate should expose desktop-safe DTOs, session controllers, event mapping, app state, and project/session persistence. It should not contain web UI code.

# 2. Product direction: Codex-style mission control, not a generic editor

The right desktop product is a **mission-control app for autonomous coding**, not a full VS Code clone on day one.

OpenAI’s Codex app direction is instructive: project sidebar, active thread, review pane, multiple threads, worktrees, terminal/actions, artifacts, and Git workflows. ([OpenAI Developers][4]) Quorp should use a similar high-level layout, but differentiate with stronger Rust-native proof, sandbox, replay, verification, task DAG, and benchmark surfaces.

The app should open to this shape:

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│ macOS titlebar / Quorp status strip                                        │
│ Project: Quorp  Branch: main  Model: qwen/...  Sandbox: TmpCopy  Tokens 43% │
├───────────────┬─────────────────────────────────────────────┬───────────────┤
│ LEFT SIDEBAR  │ CENTER AGENT THREAD                         │ RIGHT PANEL   │
│               │                                             │               │
│ + Add Folder  │  Agent/tool/diff/validation stream           │ Diff          │
│ Projects      │                                             │ Tasks         │
│   Quorp       │  ┌ Tool call: cargo test                     │ Proof         │
│   Other repo  │  │ passed, 12.3s                             │ Files         │
│               │  └──────────────────────────────             │ Settings      │
│ Threads       │                                             │               │
│   Fix parser  │  Bright red/green diff cards                 │ Selected diff │
│   Bench run   │  Recovery / checkpoint / proof cards         │ or task DAG   │
│               │                                             │               │
│ Trust state   │                                             │               │
├───────────────┴─────────────────────────────────────────────┴───────────────┤
│ Bottom composer                                                             │
│ [prompt textarea.........................................................]  │
│ Plan/Act  Permission: Ask/FullAuto/FullPermissions  Sandbox: TmpCopy/Host   │
│ Model picker  Provider picker  Attach files  Send / Stop / Resume           │
├─────────────────────────────────────────────────────────────────────────────┤
│ Status: running cargo test | checkpoint saved | queue ok | context 43%       │
└─────────────────────────────────────────────────────────────────────────────┘
```

This keeps the agent stream central, puts project/session navigation on the left, and makes diff/proof/tasks always available on the right. The bottom composer should behave like a professional command console, not like a chat toy.

# 3. Frontend stack

Use this stack for the first production desktop app:

```text
Tauri 2
React 19
TypeScript
Vite
Zustand
@tanstack/react-virtual
CodeMirror 6 for focused code views
Custom virtualized diff renderer
CSS variables + minimal Tailwind or plain CSS modules
```

My specific recommendation:

| Layer                     | Choice                                   | Why                                                                                 |
| ------------------------- | ---------------------------------------- | ----------------------------------------------------------------------------------- |
| Native shell              | **Tauri 2**                              | Rust backend, `.app`/`.dmg`, signing/notarization, small footprint, native WebView. |
| Frontend                  | **React + TypeScript + Vite**            | Fastest path to a polished app; mature ecosystem; excellent component model.        |
| State                     | **Zustand**                              | Tiny, fast, no boilerplate, good for normalized event stores.                       |
| Transcript virtualization | **@tanstack/react-virtual**              | Required for huge agent logs and diffs.                                             |
| Code editor/viewer        | **CodeMirror 6**                         | Much lighter than Monaco; good enough for diff/file previews.                       |
| Terminal pane             | **xterm.js only when needed**            | Use for interactive shell; do not render the whole agent transcript through xterm.  |
| Styling                   | **CSS variables + no runtime CSS-in-JS** | Fast, memory-conscious, easy high-contrast/no-color support.                        |
| IPC                       | **Tauri commands + events/channels**     | No local WebSocket server for v1.                                                   |

Do **not** use WebSockets inside the local desktop app for v1. Tauri already provides command invocation for request/response and frontend event/channel delivery for streaming Rust-to-UI data. Tauri’s own docs distinguish commands for frontend-to-Rust calls and events/channels for Rust-to-frontend communication; they also note channels are better for streaming data than ordinary events. ([Tauri][5])

Use WebSockets only later for one of these cases:

```text
1. Remote Quorp daemon
2. Browser-based UI
3. Team/shared run viewer
4. Headless benchmark dashboard
5. Multi-machine agent farm
```

For the local macOS app, avoid a localhost server. Keep the frontend constrained to Tauri commands and capabilities.

# 4. “100% Rust” interpretation

Quorp’s **behavior, security, persistence, process execution, sandboxing, context management, diffing, verification, and runtime control should remain 100% Rust-owned**.

The visual frontend should still be React/TypeScript for the first usable DMG.

A pure Rust GUI using Slint, egui, Dioxus, or Leptos is possible, but I would not choose it for the first polished Mac app. The fastest route to a Codex-quality desktop interface is Tauri + React/TypeScript with the Rust side acting as the sole authority for actions. That gives you the best balance of polish, performance, packaging, and implementation speed.

The security boundary should be:

```text
React/TypeScript frontend:
  - displays state
  - sends typed user intents
  - never executes shell
  - never directly reads arbitrary files
  - never decides permission policy

Rust backend:
  - owns project trust
  - owns permissions
  - owns sandboxing
  - owns process execution
  - owns file reads/writes
  - owns patch/diff/proof generation
  - owns runtime/session state
  - owns audit logs
```

# 5. Rust backend wiring

Quorp already has the important pieces:

* `quorp_core` has mode, permission, sandbox, provider, validation, raw artifact, proof receipt, and event types. ([GitHub][3])
* `quorp_agent_core::runtime` has `AgentRunRequest`, budgets, completion policy, watchdogs, stop reasons, outcomes, checkpointing, and a rich runtime event model. ([GitHub][6])
* The runtime already has a fanout event system with named subscribers like memory writer, rule forge observer, renderer, proof recorder, and benchmark recorder. That is exactly the seam the desktop app should subscribe to. ([GitHub][6])
* The terminal layer already has slash commands for plan, act, auto, permissions, sandbox, status, compact, tasks, checkpoint, rollback, diff, verify, model, provider, memory, rules, and more. Those should become both desktop command-palette actions and menu items. ([GitHub][7])
* The existing render/session code already models task rows, command cards, validation cards, diff previews, and scrollback-native rendering. The desktop should reuse the same semantic concepts, but render them as structured DOM rather than ANSI. ([GitHub][8])

The bridge should look like this conceptually:

```text
React UI
  invoke("start_agent_session", StartAgentSessionRequest)
  listen("agent://event", UiRuntimeEvent)
        │
        ▼
apps/desktop/src-tauri
  Tauri commands
  Tauri app state
  menu handlers
  event pump
        │
        ▼
crates/quorp_desktop_bridge
  DesktopSessionController
  DTO conversion
  RuntimeEvent → UiRuntimeEvent mapping
  project registry
  settings persistence
        │
        ▼
existing Quorp crates
  quorp_agent_core
  quorp_session
  quorp_provider
  quorp_tools
  quorp_permissions
  quorp_sandbox
  quorp_patch_vm
  quorp_verify
```

The desktop app should not parse terminal text. It should consume structured runtime events.

# 6. New bridge crate

Create:

```text
crates/quorp_desktop_bridge
```

Its job:

```text
1. Define frontend-safe DTOs.
2. Convert existing Quorp core/runtime types into UI events.
3. Own desktop session handles.
4. Own recent project/session registry.
5. Provide a stable API to the Tauri command layer.
6. Hide runtime internals from the frontend.
7. Prevent the frontend from requesting unsafe raw operations.
```

# 8. Tauri command examples

Project opening:

```rust
// apps/desktop/src-tauri/src/commands.rs

use tauri::{AppHandle, State};
use quorp_desktop_bridge::dto::ProjectDto;
use crate::state::DesktopState;

type CommandResult<T> = Result<T, String>;

#[tauri::command]
pub async fn list_recent_projects(
    state: State<'_, DesktopState>,
) -> CommandResult<Vec<ProjectDto>> {
    state
        .projects()
        .list_recent()
        .await
        .map_err(|err| err.to_string())
}

#[tauri::command]
pub async fn open_project(
    app: AppHandle,
    state: State<'_, DesktopState>,
    path: String,
) -> CommandResult<ProjectDto> {
    let project = state
        .projects()
        .open_project(path.into())
        .await
        .map_err(|err| err.to_string())?;

    app.emit("project://opened", &project)
        .map_err(|err| err.to_string())?;

    Ok(project)
}
```

Starting an agent session:

```rust
use quorp_desktop_bridge::dto::{AgentStatusDto, StartAgentSessionRequest};

#[tauri::command]
pub async fn start_agent_session(
    app: AppHandle,
    state: State<'_, DesktopState>,
    req: StartAgentSessionRequest,
) -> CommandResult<AgentStatusDto> {
    let session = state
        .sessions()
        .start(req)
        .await
        .map_err(|err| err.to_string())?;

    let session_id = session.session_id().to_owned();
    let subscription = session.subscribe_runtime_events();

    crate::events::spawn_agent_event_pump(
        app.clone(),
        session_id.clone(),
        subscription,
    );

    state
        .sessions()
        .status(&session_id)
        .await
        .map_err(|err| err.to_string())
}
```

Event pump:

```rust
// apps/desktop/src-tauri/src/events.rs

use tauri::{AppHandle, Emitter};
use quorp_desktop_bridge::{
    event_map::UiRuntimeEvent,
    session_controller::RuntimeEventSubscription,
};

pub fn spawn_agent_event_pump(
    app: AppHandle,
    session_id: String,
    mut sub: RuntimeEventSubscription,
) {
    tauri::async_runtime::spawn(async move {
        let mut seq: u64 = 0;

        while let Some(runtime_event) = sub.next().await {
            seq += 1;

            let ui_event = match UiRuntimeEvent::from_runtime(
                &session_id,
                seq,
                runtime_event,
            ) {
                Ok(event) => event,
                Err(err) => UiRuntimeEvent::internal_mapping_error(
                    &session_id,
                    seq,
                    err,
                ),
            };

            // For high-volume streams, batch before emitting or use Tauri channels.
            let _ = app.emit("agent://event", ui_event);
        }
    });
}
```

For high-throughput command output, batch aggressively:

```rust
const MAX_BATCH_BYTES: usize = 8 * 1024;
const MAX_BATCH_DELAY_MS: u64 = 50;
```

Emit shell output chunks every 50 ms or 8 KB, whichever comes first. Do not emit one frontend event per byte or per tiny line.

# 9. Frontend API wrapper

```ts
// apps/desktop/src/api/tauri.ts

import { invoke } from "@tauri-apps/api/core";
import { listen, type UnlistenFn } from "@tauri-apps/api/event";
import type {
  AgentStatus,
  Project,
  StartAgentSessionRequest,
  UiRuntimeEvent,
} from "./types";

export const quorpApi = {
  listRecentProjects(): Promise<Project[]> {
    return invoke("list_recent_projects");
  },

  openProject(path: string): Promise<Project> {
    return invoke("open_project", { path });
  },

  startAgentSession(req: StartAgentSessionRequest): Promise<AgentStatus> {
    return invoke("start_agent_session", { req });
  },

  sendUserMessage(sessionId: string, text: string): Promise<void> {
    return invoke("send_user_message", { sessionId, text });
  },

  cancelSession(sessionId: string): Promise<void> {
    return invoke("cancel_session", { sessionId });
  },

  setPermissionMode(sessionId: string, permissionMode: string): Promise<AgentStatus> {
    return invoke("set_permission_mode", { sessionId, permissionMode });
  },

  setSandboxMode(sessionId: string, sandboxMode: string): Promise<AgentStatus> {
    return invoke("set_sandbox_mode", { sessionId, sandboxMode });
  },

  onAgentEvent(cb: (event: UiRuntimeEvent) => void): Promise<UnlistenFn> {
    return listen<UiRuntimeEvent>("agent://event", (event) => {
      cb(event.payload);
    });
  },
};
```

# 10. Frontend store

Keep the event store normalized and bounded.

```ts
// apps/desktop/src/state/eventStore.ts

import { create } from "zustand";
import type { UiRuntimeEvent } from "../api/types";

const MAX_EVENTS_IN_MEMORY = 20_000;

interface EventState {
  eventsBySession: Record<string, UiRuntimeEvent[]>;
  latestBySession: Record<string, UiRuntimeEvent | undefined>;

  ingest(event: UiRuntimeEvent): void;
  clearSession(sessionId: string): void;
}

export const useEventStore = create<EventState>((set) => ({
  eventsBySession: {},
  latestBySession: {},

  ingest(event) {
    set((state) => {
      const existing = state.eventsBySession[event.sessionId] ?? [];
      const next = [...existing, event];

      if (next.length > MAX_EVENTS_IN_MEMORY) {
        next.splice(0, next.length - MAX_EVENTS_IN_MEMORY);
      }

      return {
        eventsBySession: {
          ...state.eventsBySession,
          [event.sessionId]: next,
        },
        latestBySession: {
          ...state.latestBySession,
          [event.sessionId]: event,
        },
      };
    });
  },

  clearSession(sessionId) {
    set((state) => {
      const eventsBySession = { ...state.eventsBySession };
      const latestBySession = { ...state.latestBySession };
      delete eventsBySession[sessionId];
      delete latestBySession[sessionId];
      return { eventsBySession, latestBySession };
    });
  },
}));
```

The full event history should live on disk as Quorp session artifacts. The frontend should keep enough in memory for smooth UX, not unbounded logs.

# 19. Security and permissions in the desktop app

The frontend must never be able to say “run this shell string on host” directly.

All dangerous actions go through Rust policy checks:

```text
Frontend intent
  "run verification"
      ↓
Rust policy
  Is project trusted?
  What sandbox?
  What permission mode?
  What command category?
  Is network needed?
  Is dependency install needed?
  Are secrets protected?
      ↓
Runtime/tool execution
      ↓
Ledger/proof/audit event
```

Default for newly opened repos:

```text
PermissionMode: Ask
SandboxMode: TmpCopy
Network: Off or Ask
MCP: Disabled until configured
Browser: Disabled until configured
Dependency installs: Ask
Generated executables: Ask
Project rules/config: Untrusted until project trusted
```

Only after the user trusts a project should Quorp read project-local Quorp rules/settings as policy inputs. Even then, repo-authored rules should never be allowed to escalate permissions.

Claude Code’s hooks system is powerful but also security-sensitive: hooks can block tool use and are configured across scopes; hook inputs include session, tool, cwd, and permission mode, and certain hook exits can block actions. ([Claude][13]) Quorp should expose similar hook/rule visibility, but avoid letting untrusted repos silently install active hooks.

Tauri capabilities should be tight. Do not grant broad filesystem or shell plugin permissions to the frontend. Tauri’s plugin permission system is specifically designed to tailor or disable plugin permissions. ([Tauri][14])

# 24. Important first implementation shortcut

There are two ways to wire the first running app:

## Preferred: embedded runtime

The Tauri backend directly calls Quorp runtime/session crates.

```text
Tauri command
  → quorp_desktop_bridge
  → quorp_session / quorp_agent_core
  → runtime events
  → Tauri event/channel
  → React UI
```

This is the correct architecture.

## Temporary: CLI sidecar

The Tauri backend spawns the existing CLI and tails JSONL artifacts.

```text
Tauri command
  → spawn quorp CLI
  → read events.jsonl / stdout JSON
  → map to UI events
```

This can get a demo running quickly, but it should be explicitly temporary. It is harder to stop/resume, harder to show live permissions, harder to route typed events, and easier to break when CLI output changes.

Recommended compromise:

```text
PR 1:
  Create desktop UI shell and bridge trait.
  Implement FakeReplaySessionController using existing benchmark events.jsonl.

PR 2:
  Implement CliSidecarSessionController only if embedded runtime takes longer.

PR 3:
  Replace sidecar with EmbeddedRuntimeSessionController.
```

This gives you a usable UI immediately while keeping the architecture honest.

# 27. Permission prompt UX

Permission prompts should be structured, fast, and auditable.

```text
┌ Permission required ───────────────────────────────────────────────────────┐
│ Tool: shell.exec                                                           │
│ Command: cargo install cargo-nextest                                       │
│ Reason: dependency/tool install                                            │
│ Sandbox: TmpCopy                                                           │
│ Network: required                                                          │
│ Risk: modifies developer environment if run on host                        │
│                                                                            │
│ [Deny] [Allow once] [Allow for this session] [Always allow in this project]│
└────────────────────────────────────────────────────────────────────────────┘
```

Policy should record:

```text
decision
scope
who decided
timestamp
project root
command/tool
risk classification
sandbox mode
permission mode
receipt path
```

Never silently downgrade from sandbox to host. Never let repo rules grant full permissions.

# 29. Memory-conscious rendering rules

Performance rules for the web UI:

```text
1. Virtualize all transcript, diff, file, and task lists.
2. Keep only a bounded event window in memory.
3. Persist full event history on disk through Rust/session artifacts.
4. Batch shell output events.
5. Avoid Monaco in v1.
6. Avoid giant UI libraries.
7. Avoid runtime CSS-in-JS.
8. Avoid rendering ANSI streams as the primary transcript.
9. Precompute diff chunks in Rust.
10. Use stable event IDs and memoized event cards.
```

Virtualized transcript:

```tsx
import { useVirtualizer } from "@tanstack/react-virtual";
import { useMemo, useRef } from "react";
import type { UiRuntimeEvent } from "../../api/types";

export function AgentEventStreamView({ events }: { events: UiRuntimeEvent[] }) {
  const parentRef = useRef<HTMLDivElement | null>(null);

  const rowVirtualizer = useVirtualizer({
    count: events.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 92,
    overscan: 12,
  });

  const items = rowVirtualizer.getVirtualItems();

  return (
    <div ref={parentRef} className="event-stream">
      <div
        className="event-stream-inner"
        style={{
          height: `${rowVirtualizer.getTotalSize()}px`,
          position: "relative",
        }}
      >
        {items.map((virtualRow) => {
          const event = events[virtualRow.index];

          return (
            <div
              key={event.id}
              data-index={virtualRow.index}
              ref={rowVirtualizer.measureElement}
              className="event-row"
              style={{
                position: "absolute",
                transform: `translateY(${virtualRow.start}px)`,
                width: "100%",
              }}
            >
              <EventCard event={event} />
            </div>
          );
        })}
      </div>
    </div>
  );
}
```

Virtualized diff:

```tsx
interface DiffLine {
  id: string;
  oldLine?: number;
  newLine?: number;
  kind: "add" | "del" | "ctx" | "hunk";
  text: string;
}

export function VirtualDiff({ lines }: { lines: DiffLine[] }) {
  const parentRef = useRef<HTMLDivElement | null>(null);

  const virtualizer = useVirtualizer({
    count: lines.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 20,
    overscan: 40,
  });

  return (
    <div ref={parentRef} className="virtual-diff">
      <div style={{ height: virtualizer.getTotalSize(), position: "relative" }}>
        {virtualizer.getVirtualItems().map((row) => {
          const line = lines[row.index];

          return (
            <div
              key={line.id}
              className={`diff-line ${line.kind}`}
              style={{
                position: "absolute",
                transform: `translateY(${row.start}px)`,
                width: "100%",
              }}
            >
              <span className="diff-gutter">
                {line.oldLine ?? ""} {line.newLine ?? ""}
              </span>
              <span className="diff-text">{line.text}</span>
            </div>
          );
        })}
      </div>
    </div>
  );
}
```

# 34. What would make it feel better than Codex/Claude Code

Quorp’s desktop advantage should be **visibility and proof**.

The app should make these things impossible to miss:

```text
1. Current permission mode
2. Current sandbox mode
3. Current trust state
4. Current context pressure
5. Current active task
6. Current validation status
7. Last safe checkpoint
8. Exact files changed
9. Exact commands run
10. Exact proof receipts
11. Exact reason a permission was blocked
12. Exact reason a recovery/repair turn happened
```

Claude Code has strong hooks, permissions, checkpoints, subagents, and background tasks. ([Claude][13]) Quorp can exceed that in the desktop by showing every one of these as a structured, replayable, proof-backed UI object.

Codex exposes status, permissions, planning, agents, and resume/fork commands. ([OpenAI Developers][18]) Quorp should make those persistent panels, not just commands.

# 35. Key implementation risks

## Risk 1: frontend becomes a privileged shell

Mitigation:

```text
No shell plugin exposed to frontend.
No broad fs plugin exposed to frontend.
All actions go through Rust commands.
All commands enforce policy.
All mutating actions emit ledger/proof events.
```

## Risk 2: event stream overwhelms WebView

Mitigation:

```text
Batch shell output.
Virtualize transcript.
Cap frontend event memory.
Persist raw logs on disk.
Surface subscriber backpressure.
Use Tauri channels for high-rate streams.
```

## Risk 3: embedded runtime API is not clean enough yet

Mitigation:

```text
Start with bridge trait.
Support replay controller first.
Use CLI sidecar only as temporary fallback.
Move quickly to embedded runtime.
```

## Risk 4: macOS Finder PATH breaks commands

Mitigation:

```text
Desktop doctor.
PATH repair.
Explicit toolchain discovery.
Visible warning in /status and top strip.
```

## Risk 5: signing/notarization slows testing

Mitigation:

```text
Local dev: pnpm tauri dev.
Local DMG: ad-hoc signingIdentity "-".
Public release: Developer ID + notarization later.
```
