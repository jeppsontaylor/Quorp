## Bottom line

Build the desktop app as a **Tauri 2 app at `apps/desktop`**, backed by a new shared Rust service crate that reuses Quorp’s existing CLI/session/agent runtime instead of duplicating logic.

The most important architectural move is this:

```text
Current:
  crates/quorp -> quorp_cli::dispatch() -> huge CLI/TUI/headless command surface

Proposed:
  crates/quorp_cli          -> thin CLI wrapper
  crates/quorp_desktop_api  -> shared app/session/run orchestration
  apps/desktop/src-tauri    -> Tauri shell + native menus + IPC commands
  apps/desktop/src          -> React/TypeScript mission-control UI
```

Do **not** put the desktop app inside `crates/quorp_cli`. The repo already treats `crates/quorp` as a thin binary wrapper into `quorp_cli`, and `quorp_cli` is already a large terminal command dispatcher with run/proof/benchmark/session paths. The desktop app should sit beside the Rust workspace as an app, while pulling reusable orchestration into Rust crates. The current README and support docs confirm Quorp is a Rust terminal-first app with extracted CLI/session crates, Patch VM receipts, verification reports, bounded runtime event fanout, sandbox modes, and benchmark/proof surfaces; they also explicitly note packaging/signing is not yet represented. ([GitHub][1])

I could not clone through the shell in this environment because GitHub DNS resolution failed, but I inspected the repo through GitHub/raw pages and the current file tree. The plan below is based on the actual workspace layout, entrypoints, CLI dispatch, runtime fields, and Tauri 2’s current docs.

Why this is the right location:

1. `crates/quorp` is intentionally tiny and currently just dispatches to `quorp_cli`; keep it that way.
2. `quorp_cli` already owns many subcommands including session, run, replay, proof, diagnostics, agent, benchmark, scan, index, and permissions; turning it into a desktop backend would make it harder to test and maintain. ([GitHub][2])
3. Quorp already has extracted `quorp_session`, `quorp_agent_core`, `quorp_render`, `quorp_term`, `quorp_slash`, `quorp_permissions`, `quorp_sandbox`, `quorp_verify`, and `quorp_benchmark` crates. The desktop app should compose these, not fork them.
4. Tauri’s normal project shape is an app folder containing frontend files plus a Rust `src-tauri` package, and Tauri’s bundler can create `.app` and `.dmg` outputs. ([GitHub][3])

# 2. Frontend technology stack

Use:

```text
Tauri 2
React + TypeScript
Vite
Zustand or Jotai for UI state
TanStack Virtual for event/diff virtualization
CodeMirror 6 only when an editor pane is opened
xterm.js only when a real terminal pane is opened
CSS variables + plain CSS modules or vanilla CSS
```

I would **not** use WebSockets for the local desktop app. Use Tauri commands and Tauri events:

```text
Frontend -> Rust:
  invoke("start_agent_run", payload)
  invoke("add_workspace", payload)
  invoke("set_permission_mode", payload)

Rust -> Frontend:
  app.emit("quorp://run-event", event_batch)
  app.emit("quorp://permission-request", prompt)
  app.emit("quorp://status", status_snapshot)
```

Tauri’s IPC model already supports command-style request/response and app events; local WebSockets would add a server, port allocation, authentication, reconnect logic, and another failure mode without improving the first desktop release. WebSockets become useful only if Quorp later exposes a remote daemon/browser UI. ([Tauri][4])

Use React rather than a pure Rust GUI for this specific product because the desired app is a dense, polished, web-like agent IDE: virtual scrolling, diff cards, command palette, settings panes, inspector drawers, animated permission prompts, and fast component iteration. The Rust backend remains the source of truth. Node/Vite is a build-time tool; the shipped app uses the macOS WebView and Rust backend, not Electron. Tauri apps use the platform webview and a Rust backend, which is why this route is much lighter than shipping a full browser runtime. ([GitHub][3])

For “very fast and memory conscious,” the key decisions are:

```text
Do:
  - batch runtime events before emitting to the frontend
  - virtualize the event timeline and diff viewer
  - keep full transcripts/events on disk, not in the DOM
  - lazy-load CodeMirror and xterm.js
  - keep frontend state normalized and small
  - expose only high-level Rust commands to the UI
  - use CSS containment/content-visibility for long panes
  - avoid Monaco in the first version

Avoid:
  - per-token IPC events
  - rendering entire JSONL transcripts into React state
  - giving the frontend broad filesystem/shell permissions
  - local HTTP/WebSocket server for the MVP
  - duplicating CLI orchestration in TypeScript
```

## Refactor needed

The CLI currently does too much app orchestration directly: it parses commands, sets up provider/model/sandbox, creates sandbox roots, launches headless runs, snapshots artifacts, handles retry/eval/proof paths, and dispatches many subcommands. The desktop app should not call CLI command functions or spawn a terminal binary unless used as a temporary bridge. Move the reusable run orchestration into `quorp_session` or the new `quorp_desktop_api` crate. The existing CLI can then call the same service. ([GitHub][2])

Recommended new shared service:

```text
crates/quorp_session/src/quorp/run_orchestrator.rs
```

Responsibilities:

```rust
pub struct RunStartRequest {
    pub workspace_root: PathBuf,
    pub goal: String,
    pub model: Option<String>,
    pub provider: ProviderSelection,
    pub permission_mode: PermissionMode,
    pub sandbox_mode: SandboxMode,
    pub autonomy_profile: AutonomyProfile,
    pub max_iterations: Option<usize>,
    pub max_seconds: Option<u64>,
    pub resume_from: Option<PathBuf>,
}

pub struct RunController {
    pub run_id: RunId,
    pub cancel: CancellationToken,
    pub event_rx: tokio::sync::broadcast::Receiver<RuntimeEvent>,
}

pub async fn start_agent_run(
    request: RunStartRequest,
    sink: Arc<dyn RuntimeEventSink>,
) -> anyhow::Result<RunController>;
```

This gives both the CLI and desktop app a shared, testable control path.

```rust
    state: tauri::State<'_, DesktopState>,

#[tauri::command]
async fn set_permission_mode(
    state: tauri::State<'_, DesktopState>,
    workspace_id: String,
    mode: PermissionModeDto,
) -> Result<PermissionStateDto, CommandError>;
```

## Rust-to-frontend event names

```text
quorp://run/event-batch
quorp://run/status
quorp://run/completed
quorp://run/error
quorp://permission/request
quorp://permission/resolved
quorp://workspace/index-progress
quorp://diff/updated
quorp://proof/updated
quorp://context/pressure
quorp://rules/candidate
```

Do not emit every token one at a time. Emit batches:

```rust
#[derive(Debug, Clone, Serialize)]
pub struct UiEventBatch {
    pub run_id: String,
    pub sequence_start: u64,
    pub sequence_end: u64,
    pub events: Vec<UiRunEvent>,
}
```

Batching policy:

```text
Flush when:
  - 16 ms elapsed, or
  - 128 events buffered, or
  - permission prompt appears, or
  - run stops/fails/completes
```

```rust
use quorp_desktop_api::{

pub struct DesktopState {

impl DesktopState {

            inner: Arc::new(RwLock::new(DesktopAppState {
```

## `apps/desktop/src-tauri/src/commands.rs`

```rust
use tauri::{AppHandle, Emitter, State};

use crate::state::DesktopState;
use quorp_desktop_api::{
    AddWorkspaceRequest,
    CommandError,
    PermissionModeDto,
    ResumeRunRequest,
    RunSummary,
    StartRunRequest,
    WorkspaceSummary,
};

#[tauri::command]
pub async fn add_workspace(
    state: State<'_, DesktopState>,
    request: AddWorkspaceRequest,
) -> Result<WorkspaceSummary, CommandError> {
    let mut app_state = state.inner.write().await;
    app_state.add_workspace(request).await
}

#[tauri::command]
pub async fn list_workspaces(
    state: State<'_, DesktopState>,
) -> Result<Vec<WorkspaceSummary>, CommandError> {
    let app_state = state.inner.read().await;
    Ok(app_state.workspaces.list())
}

#[tauri::command]
pub async fn start_agent_run(
    app: AppHandle,
    state: State<'_, DesktopState>,
    request: StartRunRequest,
) -> Result<RunSummary, CommandError> {
    let event_sink = quorp_desktop_api::TauriEventSink::new(app.clone());

    let mut app_state = state.inner.write().await;
    let run = app_state
        .runs
        .start(request, Box::new(event_sink))
        .await?;

    app.emit("quorp://run/status", &run.status_snapshot)
```

## Event bridge

The existing runtime already has a bounded runtime event fanout with subscribers for memory writer, rule forge observer, renderer, proof recorder, and benchmark recorder. That is exactly the right foundation for a desktop event stream; add a desktop subscriber instead of scraping logs when possible. ([GitHub][7])

```rust
use std::sync::Arc;
use tauri::{AppHandle, Emitter};

pub struct TauriEventSink {
    app: AppHandle,
}

impl TauriEventSink {
    pub fn new(app: AppHandle) -> Self {
        Self { app }
    }

    pub fn emit_batch(&self, batch: UiEventBatch) {
        let _ = self.app.emit("quorp://run/event-batch", batch);
    }
}
```

If `run_headless_agent` does not currently accept a runtime sink, add this to the shared run request:

```rust
pub struct SharedRunOptions {
    pub request: AgentRunRequest,
    pub event_sink: Option<Arc<dyn RuntimeEventSink>>,
    pub artifact_dir: PathBuf,
}
```

Then both CLI and desktop can subscribe to the same event stream.

For a very quick first build, there is an acceptable temporary compromise:

```text
MVP bridge:
  Desktop starts run through Rust.
  Existing headless runner writes events.jsonl.
  Desktop tails events.jsonl and renders it.

Better final path:
  Desktop receives RuntimeEvent batches directly from the runtime fanout.
```

The direct event path is faster and cleaner. The JSONL-tail bridge is useful only to get a clickable app on your Mac quickly.

```ts
import type { UiEventBatch, UiRunEvent } from "./types";

export interface TimelineState {
  byId: Map<string, TimelineItem>;
  order: string[];
  activeRunId?: string;
  contextPressure?: number;
  pendingPermissionRequestId?: string;
}

export function reduceEventBatch(
  state: TimelineState,
  batch: UiEventBatch,
): TimelineState {
  let next = state;

  for (const event of batch.events) {
    next = reduceRunEvent(next, batch.runId, event);
  }

  return next;
}

function reduceRunEvent(
  state: TimelineState,
  runId: string,
  event: UiRunEvent,
): TimelineState {
  switch (event.kind) {
    case "assistant_message":
      return appendTimelineItem(state, {
        type: "assistant",
        runId,
        text: event.text,
      });

    case "tool_started":
      return appendTimelineItem(state, {
        type: "tool",
        runId,
        title: event.title,
        status: "running",
      });

    case "diff_updated":
      return appendTimelineItem(state, {
        type: "diff",
        runId,
        files: event.files,
      });

    case "context_pressure":
      return {
        ...state,
        contextPressure: event.pressure,
      };

    case "permission_requested":
      return {
        ...appendTimelineItem(state, {
          type: "permission",
          runId,
          requestId: event.request_id,
          action: event.action,
          risk: event.risk,
          explanation: event.explanation,
        }),
        pendingPermissionRequestId: event.request_id,
      };

    default:
      return appendTimelineItem(state, {
        type: "event",
        runId,
        event,
      });
  }
}
```

# 9. Workspace/folder picker

Use the Tauri dialog plugin for selecting folders, then pass the selected path into a Rust command that canonicalizes, validates, and registers the workspace. Tauri’s dialog docs note selected paths can affect asset/filesystem scope, but for this app the safer design is to let the Rust backend own all filesystem access and not give the frontend broad direct filesystem powers. ([Tauri][9])

Frontend:

```ts
import { open } from "@tauri-apps/plugin-dialog";
import { invoke } from "@tauri-apps/api/core";

export async function addFolderFromDialog() {
  const selected = await open({
    directory: true,
    multiple: false,
    title: "Add Folder to Quorp",
  });

  if (typeof selected !== "string") return;

  return invoke("add_workspace", {
    request: {
      path: selected,
    },
  });
}
```

Rust:

```rust
#[derive(Debug, serde::Deserialize)]
pub struct AddWorkspaceRequest {
    pub path: String,
}

pub async fn add_workspace(
    request: AddWorkspaceRequest,
) -> Result<WorkspaceSummary, CommandError> {
    let path = std::path::PathBuf::from(request.path);
    let canonical = path.canonicalize()?;

    if !canonical.is_dir() {
        return Err(CommandError::invalid_input("selected path is not a directory"));
    }

    let trust = WorkspaceTrust::Untrusted;
    let summary = WorkspaceSummary::from_path(canonical, trust).await?;

    // Persist under user app data, not inside the repo.
    WorkspaceRegistry::default_path()
        .save_workspace(&summary)
        .await?;

    Ok(summary)
}
```

Important: do **not** automatically load repo-authored desktop settings or permissions from an untrusted repository. Existing Quorp has `.rules` guidance in the repo, but desktop trust state should be explicit. The first time a folder is added, mark it:

```text
Untrusted
  - no host writes
  - no network
  - no dependency installs
  - repo-authored config/rules shown as proposed, not active

Trusted
  - user has explicitly approved this workspace
  - host writes can be allowed if selected
  - project-local safe workflow rules may become active
```

# 10. Permissions and sandbox UX

Quorp already has permission primitives and sandbox modes: `read-only`, `ask`, `accept-edits`, `auto-safe`, `yolo-sandbox`, plus host/tmp-copy/git-worktree sandbox support. The desktop app should surface those as first-class controls, not bury them in settings. ([GitHub][10])

Recommended default:

```text
Permission mode: Ask
Sandbox: tmp-copy
Network: Disabled
Dependency installs: Ask
MCP: Disabled
```

Dangerous modes:

```text
Host sandbox:
  show yellow warning

Trusted Host / full permission:
  show red warning
  require explicit workspace trust
  require typed confirmation for first use
  persist per workspace
  allow immediate downgrade

Yolo sandbox:
  allowed only with tmp-copy or git-worktree
  never direct host execution
```

Permission prompt design:

```text
Quorp wants to run:

  cargo test --all

Risk:
  reads repo
  executes local build scripts
  may compile generated code

Workspace:
  ~/src/foo

Sandbox:
  tmp-copy

[Allow once] [Allow for this run] [Deny] [Open details]
```

Patch prompt:

```text
Quorp wants to modify 3 files:

  crates/foo/src/lib.rs
  crates/foo/tests/parser.rs
  Cargo.toml

Patch VM:
  hash preconditions: ok
  rollback available: yes

[Apply patch] [Preview diff] [Deny]
```

The frontend should never directly execute shell commands. Even Tauri’s shell plugin should not be enabled for the UI by default. If a sidecar is used temporarily, run it from Rust with a tightly scoped command path, not from arbitrary frontend shell calls. Tauri supports a granular capabilities system for constraining what frontend windows can access; use that aggressively. ([Tauri][11])

# 12. DMG build pipeline

## Fast local testing, unsigned

On your Mac:

```bash
# from repo root
corepack enable

cd apps/desktop
pnpm install

# dev mode
pnpm tauri dev
```

Build an unsigned local app/DMG:

```bash
cd apps/desktop
pnpm tauri build --bundles app,dmg
```

The outputs are normally under a Tauri target bundle directory such as:

```text
apps/desktop/src-tauri/target/release/bundle/macos/Quorp.app
apps/desktop/src-tauri/target/release/bundle/dmg/Quorp_0.1.0_aarch64.dmg
```

If the workspace uses a shared Cargo target directory, the output may appear under the workspace `target/release/bundle/...`.

Unsigned local builds are fine for immediate testing on your own Mac. They are not fine for distributing to users because Gatekeeper will complain, especially when downloaded from the internet. Apple recommends notarizing macOS software distributed outside the Mac App Store so users can trust Developer ID-signed software checked by Apple. ([Tauri][13])

```text
1. Build frontend
2. Compile Rust desktop app
3. Bundle .app
4. Sign .app with Developer ID Application certificate
5. Create .dmg
6. Sign .dmg
7. Submit for notarization
8. Staple notarization ticket
9. Upload to GitHub Releases / website
```

Tauri supports macOS signing/notarization workflows, including App Store Connect API credential environment variables. ([Tauri][14])

# 16. Sidecar option: use only as a temporary bridge

Tauri supports sidecars through `bundle.externalBin`, but I would not make sidecar execution the permanent architecture. ([Tauri][15])

Sidecar is useful for a **one-week prototype**:

```text
Desktop UI
  -> Rust Tauri command
    -> spawn bundled quorp CLI sidecar
      -> quorp run ...
      -> write events.jsonl
    -> desktop tails events.jsonl
```

Problems with permanent sidecar design:

```text
- harder cancellation
- harder permission prompts
- duplicate process memory
- less precise event streaming
- more packaging complexity
- worse UX around errors
- harder to share in-memory indexes/context
```

Permanent design should be:

```text
Desktop UI
  -> Tauri Rust command
    -> quorp_desktop_api
      -> quorp_session run_orchestrator
        -> quorp_agent_core runtime
```

# 17. Security defaults

Use Tauri capabilities to keep frontend privileges narrow. Tauri 2’s capability model is designed to grant/deny specific permissions per window/webview. ([Tauri][11])

Recommended first-release security policy:

```text
Frontend:
  - can open folder dialog
  - can call typed Quorp commands
  - can open/reveal known paths through backend approval
  - cannot run arbitrary shell
  - cannot write arbitrary files directly
  - cannot read arbitrary files directly

Backend:
  - canonicalizes all paths
  - checks workspace registration
  - applies permission mode
  - routes writes through Patch VM
  - records mutating actions
  - enforces sandbox policy
  - protects secrets
```

For untrusted repos:

```text
- default tmp-copy sandbox
- repo-authored config/rules are proposals, not active policy
- no automatic network
- no automatic dependency install
- no host writes without explicit trust
- no MCP/browser without explicit approval
```

For local API keys:

```text
- do not store secrets in repo
- do not expose keys to frontend logs
- prefer OS keychain through a small Rust service or vetted Tauri plugin
- redact secrets from event timeline
```

# 19. The exact “best quick path”

To get something usable quickly without compromising the long-term architecture:

```text
1. Create apps/desktop Tauri app.
2. Build the UI shell immediately.
3. Add folder picker and workspace sidebar.
4. Add composer with permission/sandbox picker.
5. Start a Quorp run from Rust.
6. Initially tail events.jsonl if direct runtime event wiring is not ready.
7. Replace tailing with RuntimeEvent fanout subscription.
8. Move CLI run orchestration into shared Rust service.
9. Add unsigned DMG target.
10. Add signing/notarization later.
```

This gets you a Mac app fast, while keeping the end-state clean.

# 20. My strongest recommendation

Use **Tauri 2 + React/TypeScript + Rust shared orchestration**.

Use this layout:

```text
apps/desktop                 # desktop product
apps/desktop/src-tauri       # Tauri native shell
crates/quorp_desktop_api     # typed desktop backend API
crates/quorp_session/...     # shared run orchestrator
```

Use this runtime wiring:

```text
React composer
  -> Tauri invoke(start_agent_run)
    -> quorp_desktop_api::RunService
      -> quorp_session::run_orchestrator
        -> quorp_agent_core::runtime
          -> RuntimeEventFanout
            -> Tauri event batches
              -> React virtualized timeline
```

Use this first local build command:

```bash
cd apps/desktop
pnpm install
pnpm tauri dev
pnpm tauri build --bundles app,dmg
```

Use this product layout:

```text
left:    workspaces / sessions / tasks / rules
center:  virtualized agent timeline with tool calls and red/green diffs
right:   status / permissions / context / tasks / proof inspector
bottom:  chat composer with model + permission + sandbox controls
top:     workspace/model/context/trust/status mission bar
```

This gives Quorp a serious desktop surface without weakening the terminal CLI. It also keeps the product Rust-native where it matters: runtime, tools, permissions, sandboxing, patching, proof, verification, session artifacts, and packaging.
