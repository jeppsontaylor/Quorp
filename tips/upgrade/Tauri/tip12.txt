I would build the desktop app as **Tauri 2 + React/TypeScript + Rust-linked Quorp core**, with the app living at:

```text
/apps/quorp-desktop
/crates/quorp_desktop_core
```

The big architectural decision is: **do not make the desktop app a terminal emulator that shells out to `quorp run` as the main runtime path**. Use Quorp’s Rust crates directly, stream semantic runtime events into a graphical timeline, and keep the CLI as a first-class sibling. Quorp already has the right foundation: the CLI is split into `quorp_cli`, session/runtime helpers are split into `quorp_session`, the runtime emits typed `RuntimeEvent`s, and the repo explicitly notes that release packaging/signing is still missing.  

One note: I could not use a direct `git clone` inside the execution container because GitHub DNS resolution failed, so I inspected the repository through the GitHub connector and current public Tauri/OpenAI docs instead.

│   ├── quorp_slash               # existing slash command registry
│   └── quorp_desktop_core        # NEW: typed desktop adapter
└── apps/
    └── quorp-desktop             # NEW: Tauri app
        ├── package.json
        ├── index.html
        ├── vite.config.ts
        ├── tsconfig.json

This keeps the current terminal binary intact. The existing `crates/quorp/src/main.rs` is already just a thin CLI dispatch wrapper:

```rust
fn main() {
    quorp_cli::dispatch();
}
```

That is excellent for a dual CLI + desktop product because the CLI binary does not own the real architecture. 

The root workspace currently includes only crate-style members under `crates/`, with `crates/quorp` as the default member. The desktop Rust crate should be added explicitly as a workspace member, while the web frontend remains under `apps/`.

## 2. Why Tauri 2 is still the right choice

Use **Tauri 2**. It is the best fit for Quorp’s actual codebase because Quorp is already Rust-heavy, already has session/runtime crates, and needs a real macOS `.app` / `.dmg` quickly.

Tauri can produce macOS app bundles and DMGs; its docs show `cargo tauri build --bundles dmg`, and explain that DMGs are the common distribution wrapper for macOS apps outside the App Store. ([Tauri][1])

For this product, the frontend should be a web UI, not a pure Rust GUI. The runtime, sandbox, permissions, verification, context, memory, and ledger stay in Rust. The graphical app shell should use the web stack because Quorp needs a sophisticated IDE-like UI: virtualized timelines, diff views, command palette, settings panels, model controls, keyboard navigation, and fast iteration.

### My stack recommendation

```text
Tauri 2
React + TypeScript
Vite
CodeMirror 6 for editor/composer/diff-adjacent text
@tanstack/react-virtual for event/diff virtualization
Zustand or a tiny custom external store for UI state
Rust/Tauri commands for control plane
Tauri events or IPC channels for runtime event stream
No WebSocket for local desktop v1
```

Do **not** use WebSockets for the first desktop app. Tauri already gives you a Rust ↔ frontend command/event bridge, and Tauri permissions/capabilities can scope what the frontend may call. Tauri’s permissions model grants commands to windows/webviews through capabilities and can scope access; that is exactly the right trust boundary for a local coding-agent app. ([Tauri][2])

Use WebSockets only later if Quorp gains a separate remote/headless daemon.

## 3. Why `/apps/quorp-desktop` is the ideal location

Put the Tauri app here:

```text
/apps/quorp-desktop
```

Not here:

```text
/crates/quorp_desktop
```

Reason: a Tauri app is not just a Rust crate. It contains `package.json`, Vite config, frontend source, app icons, macOS bundle config, Tauri capabilities, entitlements, and DMG settings. It is an application, not a library.

Add a small Rust adapter crate here:

```text
/crates/quorp_desktop_core
```

This crate should contain no web UI and no Tauri-specific window behavior. It should expose the **desktop runtime API**:

```rust
pub struct DesktopRunRequest { ... }
pub struct DesktopRunHandle { ... }
pub enum DesktopEvent { ... }

pub struct DesktopRunManager { ... }

impl DesktopRunManager {
    pub fn add_workspace(...);
    pub fn start_run(...);
    pub fn cancel_run(...);
    pub fn resume_run(...);
    pub fn read_ledger_page(...);
    pub fn summarize_run(...);
    pub fn load_settings(...);
    pub fn update_permission_mode(...);
}
```

Then `apps/quorp-desktop/src-tauri` becomes a thin Tauri host that converts Tauri commands into `quorp_desktop_core` calls.

This follows the repo’s existing direction: Quorp already split CLI and session into separate crates, and `AGENT_SUPPORT.md` identifies `quorp_cli`, `quorp_session`, `quorp_agent_core`, `quorp_term`, and `quorp_render` as primary surfaces.

## 4. The key refactor needed before the desktop app feels clean

There is one important code smell to fix early:

`crates/quorp_session/src/quorp/agent_runner.rs` currently imports TUI-specific chat and command bridge types:

```rust
use crate::quorp::tui::chat_service::{ChatServiceMessage, ChatServiceRole, StreamRequest};
use crate::quorp::tui::command_bridge::CommandBridgeRequest;
```

That same file also defines the provider client, recording client, command-bridge executor, event sinks, usage summary, and routing summary. 

For the desktop app, extract the native host pieces out of the TUI namespace.

### Change this

```text
crates/quorp_session/src/quorp/tui/chat_service.rs
crates/quorp_session/src/quorp/tui/command_bridge.rs
```

### Into this

```text
crates/quorp_session/src/quorp/provider_host.rs
crates/quorp_session/src/quorp/tool_host.rs
crates/quorp_session/src/quorp/runtime_host.rs
```

Then both CLI/TUI and desktop can use the same host services.

The desktop should **not** know about terminal rendering, crossterm, or TUI internals. It should know about semantic events:

```rust
RuntimeEvent::RunStarted
RuntimeEvent::ModelRequestStarted
RuntimeEvent::ModelRequestFinished
RuntimeEvent::ToolCallStarted
RuntimeEvent::ToolCallFinished
RuntimeEvent::ValidationStarted
RuntimeEvent::ValidationFinished
RuntimeEvent::PolicyDenied
RuntimeEvent::RecoveryTurnQueued
RuntimeEvent::CheckpointSaved
RuntimeEvent::RunFinished
```

The runtime already exposes these typed events and a bounded `RuntimeEventFanout` with named subscribers and backpressure reporting. That is perfect for a desktop event stream.

### `crates/quorp_desktop_core/src/quorp_desktop_core.rs`

Starter shape:

```rust
use std::collections::BTreeMap;
use std::path::PathBuf;
use std::sync::{Arc, Mutex};
use std::sync::atomic::{AtomicBool, Ordering};

use anyhow::Context as _;
use quorp_agent_core::{
    AgentRuntimeStatus, RuntimeEvent, StopReason, SubscriberCursor,
};
use quorp_core::{PermissionMode, RunMode, SandboxMode};
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DesktopProject {
    pub id: String,
    pub name: String,
    pub root: PathBuf,
    pub trusted: bool,
    pub permission_mode: PermissionMode,
    pub sandbox_mode: SandboxMode,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DesktopRunRequest {
    pub workspace_root: PathBuf,
    pub prompt: String,
    pub run_mode: RunMode,
    pub permission_mode: PermissionMode,
    pub sandbox_mode: SandboxMode,
    pub model_id: Option<String>,
    pub base_url_override: Option<String>,
    pub max_steps: usize,
    pub max_seconds: Option<u64>,
    pub max_total_tokens: Option<u64>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DesktopRunSummary {
    pub run_id: String,
    pub run_dir: PathBuf,
    pub workspace_root: PathBuf,
    pub status: AgentRuntimeStatus,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "kind", rename_all = "snake_case")]
pub enum DesktopEvent {
    Runtime {
        run_id: String,
        event: RuntimeEvent,
    },
    LedgerCursor {
        run_id: String,
        cursor: SubscriberCursor,
    },
    RunArtifact {
        run_id: String,
        name: String,
        path: PathBuf,
    },
    Error {
        run_id: Option<String>,
        message: String,
    },
}

pub trait DesktopEventSink: Send + Sync + 'static {
    fn emit(&self, event: DesktopEvent);
}

#[derive(Debug)]
pub struct DesktopRunHandle {
    pub run_id: String,
    pub run_dir: PathBuf,
    cancel: Arc<AtomicBool>,
}

impl DesktopRunHandle {
    pub fn cancel(&self) {
        self.cancel.store(true, Ordering::Relaxed);
    }
}

#[derive(Default)]
pub struct DesktopRunManager {
    runs: Mutex<BTreeMap<String, DesktopRunHandle>>,
}

impl DesktopRunManager {
    pub fn new() -> Self {
        Self::default()
    }

    pub fn list_projects(&self) -> anyhow::Result<Vec<DesktopProject>> {
        // First PR: load from ~/.quorp/desktop/projects.json.
        // Later: move to quorp_storage SQLite if project metadata grows.
        Ok(Vec::new())
    }

    pub fn add_project(&self, root: PathBuf) -> anyhow::Result<DesktopProject> {
        let canonical = std::fs::canonicalize(&root)
            .with_context(|| format!("failed to canonicalize {}", root.display()))?;

        let loaded = quorp_config::load_settings(&canonical)
            .with_context(|| format!("failed to load settings for {}", canonical.display()))?;

        let name = canonical
            .file_name()
            .and_then(|value| value.to_str())
            .unwrap_or("workspace")
            .to_string();

        Ok(DesktopProject {
            id: loaded.trust.project_id,
            name,
            root: canonical,
            trusted: loaded.trust.trusted,
            permission_mode: loaded.settings.permissions.mode,
            sandbox_mode: loaded.settings.sandbox.mode,
        })
    }

    pub fn cancel_run(&self, run_id: &str) -> anyhow::Result<()> {
        let runs = self.runs.lock()
            .map_err(|_| anyhow::anyhow!("desktop run manager poisoned"))?;

        let Some(handle) = runs.get(run_id) else {
            anyhow::bail!("run `{run_id}` not found");
        };

        handle.cancel();
        Ok(())
    }
}
```

This crate should grow into the stable desktop API. The Tauri layer should stay tiny.

### `apps/quorp-desktop/src-tauri/src/quorp_desktop.rs`

```rust
use std::path::PathBuf;
use std::sync::Arc;

use quorp_desktop_core::{
    DesktopEvent, DesktopProject, DesktopRunManager, DesktopRunRequest, DesktopRunSummary,
};
use serde::Serialize;
use tauri::{AppHandle, Manager, State};

#[derive(Debug, thiserror::Error)]
pub enum CommandError {
    #[error("{0}")]
    Message(String),
}

impl From<anyhow::Error> for CommandError {
    fn from(error: anyhow::Error) -> Self {
        Self::Message(format!("{error:#}"))
    }
}

impl Serialize for CommandError {
    fn serialize<S>(&self, serializer: S) -> Result<S::Ok, S::Error>
    where
        S: serde::Serializer,
    {
        serializer.serialize_str(&self.to_string())
    }
}

pub struct DesktopState {
    pub runs: Arc<DesktopRunManager>,
}

#[tauri::command]
fn list_projects(state: State<'_, DesktopState>) -> Result<Vec<DesktopProject>, CommandError> {
    Ok(state.runs.list_projects()?)
}

#[tauri::command]
fn add_project(
    state: State<'_, DesktopState>,
    path: PathBuf,
) -> Result<DesktopProject, CommandError> {
    Ok(state.runs.add_project(path)?)
}

#[tauri::command]
async fn start_run(
    app: AppHandle,
    state: State<'_, DesktopState>,
    request: DesktopRunRequest,
) -> Result<DesktopRunSummary, CommandError> {
    let runs = state.runs.clone();

    // PR 1 can return a stub summary.
    // PR 2 should call into quorp_session/quorp_agent_core directly.
    let summary = DesktopRunSummary {
        run_id: format!("run-{}", chrono::Utc::now().timestamp_millis()),
        run_dir: std::env::temp_dir().join("quorp-desktop-placeholder-run"),
        workspace_root: request.workspace_root.clone(),
        status: quorp_agent_core::AgentRuntimeStatus::Idle,
    };

    let event = DesktopEvent::Runtime {
        run_id: summary.run_id.clone(),
        event: quorp_agent_core::RuntimeEvent::RunStarted {
            goal: request.prompt,
            model_id: request.model_id.unwrap_or_else(|| "default".to_string()),
        },
    };

    app.emit("quorp://runtime-event", event)
        .map_err(|error| CommandError::Message(format!("failed to emit desktop event: {error}")))?;

    let _ = runs;
    Ok(summary)
}

#[tauri::command]
fn cancel_run(
    state: State<'_, DesktopState>,
    run_id: String,
) -> Result<(), CommandError> {
    Ok(state.runs.cancel_run(&run_id)?)
}

pub fn run() {
    quorp_log::init();
    quorp_tracing::init();

    tauri::Builder::default()
        .manage(DesktopState {
            runs: Arc::new(DesktopRunManager::new()),
        })
        .plugin(tauri_plugin_dialog::init())
        .plugin(tauri_plugin_opener::init())
        .plugin(tauri_plugin_store::Builder::new().build())
        .plugin(tauri_plugin_window_state::Builder::new().build())
        .invoke_handler(tauri::generate_handler![
            list_projects,
            add_project,
            start_run,
            cancel_run,
        ])
        .run(tauri::generate_context!())
        .expect("failed to run Quorp desktop app");
}
```

For PR 2, replace the stub event with a real runtime task. The important part is that frontend state receives **semantic runtime events**, not raw terminal text.

## 9. Frontend app layout

The product should feel like **mission control for autonomous coding**, not a decorated terminal.

OpenAI’s current Codex desktop/app UX publicly emphasizes projects linked to folders, a left sidebar for history/projects, a main workspace/thread/chat area, and agents running in sandboxed environments with approvals for elevated actions. ([OpenAI][3])

Quorp should match that structure, then beat it by making the runtime ledger, proof, permissions, context pressure, recovery state, and diffs first-class.

## 13. Runtime event reducer

### `src/stores/runsStore.ts`

```ts
import { create } from "zustand";
import type { DesktopEvent, RuntimeEvent } from "../bindings/types";

export interface TimelineItem {
  id: string;
  runId: string;
  kind: string;
  timestamp: number;
  payload: RuntimeEvent;
}

interface RunState {
  activeRunId: string | null;
  timeline: TimelineItem[];
  appendDesktopEvent: (event: DesktopEvent) => void;
  clearRun: (runId: string) => void;
}

function eventKind(event: RuntimeEvent): string {
  return String(event.event ?? event.kind ?? "unknown");
}

export const useRunsStore = create<RunState>((set) => ({
  activeRunId: null,
  timeline: [],

  appendDesktopEvent: (event) => {
    if (event.kind !== "runtime") {
      return;
    }

    const item: TimelineItem = {
      id: `${event.run_id}:${Date.now()}:${Math.random().toString(16).slice(2)}`,
      runId: event.run_id,
      kind: eventKind(event.event),
      timestamp: Date.now(),
      payload: event.event
    };

    set((state) => ({
      activeRunId: event.run_id,
      timeline: appendBounded(state.timeline, item, 5000)
    }));
  },

  clearRun: (runId) => {
    set((state) => ({
      timeline: state.timeline.filter((item) => item.runId !== runId)
    }));
  }
}));

function appendBounded<T>(items: T[], item: T, max: number): T[] {
  if (items.length < max) {
    return [...items, item];
  }

  return [...items.slice(items.length - max + 1), item];
}
```

The UI should keep only a bounded in-memory event window. The canonical long-term record should be the Rust run ledger. Quorp already has hash-chained run ledger events and subscriber cursors, which are ideal for replay/resume and desktop history.

3. Show approval modal for `PromptUser`.
4. Persist session allowlist.
5. Never allow full-auto on host.

Acceptance:

* Untrusted project cannot elevate via project settings.
* Full-auto host is blocked.
* Network defaults off.
* Approval decisions are logged.

## 25. Suggested `justfile` commands

## 26. What not to do

Do **not**:

1. Put the Tauri app inside `crates/quorp`.
2. Replace the CLI with the desktop app.
3. Build the UI around xterm.js as the main runtime view.
4. Use WebSockets for local app v1.
5. Enable broad Tauri filesystem or shell permissions.
6. Let project config elevate permissions without trust.
7. Store provider API keys in frontend state.
8. Render giant diffs or logs without virtualization.
9. Treat DMG signing/notarization as required for internal testing.
10. Ship full-auto host execution as a default.

## 27. The fastest usable path

Build in this order:

```text
Day 1:
  Tauri app opens.
  Add Folder works.
  Static Codex-style layout works.
  Settings/trust/sandbox display works.

Day 2-3:
  Start a real Quorp run.
  Stream RuntimeEvent cards.
  Show tool calls, validation, run finished.
  Write run artifacts.

Day 4-5:
  Diff inspector.
  Permission picker.
  Sandbox panel.
  Local .app and ad-hoc DMG.

Week 2:
  Approval modal.
  Proof/replay viewer.
  Context/status panel.
  Rules/memory panels.
  Signed/notarized release pipeline.
```

The first meaningful milestone is not “DMG exists.” The first meaningful milestone is:

```text
Open Quorp.app
Add a folder
Type a task
Pick Auto-safe + Tmp-copy
Run agent
Watch semantic tool/diff/validation timeline
Open proof artifacts
Build local DMG
```

That is the quickest path to a usable, impressive desktop app without compromising the CLI or Quorp’s safety model.
