## Bottom line

Build the Quorp desktop app as:

```text
Tauri 2 desktop shell
React + TypeScript + Vite UI
Rust runtime authority
Quorp crates reused directly
No WebSocket for local app control
DMG output through Tauri bundler
Signing/notarization added as a release lane, not a blocker for local testing
```

The important architectural decision is this: **do not make the desktop app a pretty wrapper around `quorp_cli::dispatch()`**. The CLI should remain a first-class product, but the desktop app should call the same Rust session/runtime crates directly through a new desktop adapter. The repo is already set up for this: Quorp has extracted `quorp_cli`, `quorp_session`, `quorp_agent_core`, `quorp_render`, `quorp_slash`, `quorp_permissions`, `quorp_sandbox`, `quorp_patch_vm`, `quorp_context`, `quorp_verify`, and related crates in the workspace. 

The “100% Rust” part should mean **100% Rust for trust, execution, sandboxing, provider calls, command policy, patching, proof receipts, context, memory, and state**. The GUI itself should be a thin webview frontend because the desired Codex-style layout, virtualized transcript, diff surfaces, settings panels, and polished Mac app flow will ship dramatically faster and better with React/TypeScript inside Tauri than with a pure Rust GUI toolkit.

## 2. Why Tauri 2 is the right call

Use **Tauri 2 + React/TypeScript**.

Tauri gives you the `.app` and `.dmg` path, native menus, macOS bundling, a Rust backend, an IPC/event bridge, security capabilities, and a smaller footprint than Electron-style desktop apps. Tauri’s DMG docs explicitly support building a macOS DMG on a Mac with `tauri build --bundles dmg`, and note that macOS GUI apps do not inherit shell `$PATH` from dotfiles, which matters a lot for Quorp because the app must run `git`, `cargo`, `node`, test commands, and language tools. ([Tauri][1])

The frontend should not talk to a local WebSocket server for normal operation. Use Tauri commands for request/response operations and Tauri events for live runtime updates. Tauri commands are designed for frontend-to-Rust calls with typed arguments/return values, while events are better for async payload streams. ([Tauri][2])

Use WebSockets only later for an optional “remote Quorp daemon” mode, multi-machine work, or browser-based replay viewer. For the local Mac app, WebSockets add avoidable complexity, port management, attack surface, and extra memory.

## 3. Existing Quorp surfaces to reuse

The desktop app can become powerful quickly because most of the hard backend pieces already exist.

**Runtime and event stream.** `quorp_agent_core` already has `RuntimeEvent`, `RuntimeEventSink`, `RuntimeEventFanout`, bounded subscriber queues, model request events, tool events, validation events, parser recovery events, policy-denied events, checkpoint events, and run-finished events. These are exactly what the desktop transcript, status bar, task view, proof view, and recovery display need. 

**Session/headless bridge.** `quorp_session` already owns the session engine, remote completion client, recording completion client, command bridge tool executor, headless recorder, usage summary, routing summary, checkpoint writing, and runtime progress sink.  

**CLI stays separate.** `quorp_cli` already has the command surface for `run`, `exec`, `doctor`, `benchmark`, `replay`, `proof`, `scan`, `permissions`, and inline CLI. The desktop app should reuse shared logic, not drive the app by spawning `quorp` except for smoke-test parity.  

**Permissions and sandboxing.** `quorp_permissions` already classifies commands into capabilities such as shell meta, compound command, dependency install, network, Docker, git remote mutation, secrets read, generated executable, MCP, and browser; it also distinguishes host vs sandbox execution and denies yolo on host.  `quorp_sandbox` already supports host, git-worktree, tmp-copy, container runtime planning, network-disabled containers, environment scrubbing, and secret env blocking. 

**Patch/diff/proof safety.** `quorp_patch_vm` already validates file hash preconditions, creates previews, requires preview IDs for high-risk patches, rejects stale hashes, and rolls back partial failed applies.  Patch receipts and rollback tokens already have domain types in `quorp_patch_model`. 

**Slash commands and rendering language.** `quorp_term` and `quorp_slash` already define the command vocabulary the desktop should expose visually: `/plan`, `/permissions`, `/status`, `/compact`, `/diff`, `/verify`, `/memory`, `/rules`, `/tasks`, `/checkpoint`, `/rollback`, `/sandbox`, `/model`, `/provider`, and more.   `quorp_render` already has deterministic shell/session rendering concepts that can inspire the desktop UI vocabulary. 

**Settings and trust model.** `quorp_config` already has user/project settings, trust merging, provider, sandbox, permission, MCP, tool policy, context, memory, rules, eval settings, and managed policy. It prevents untrusted projects from elevating permissions/network/MCP/browser/process control and downgrades unsafe full-auto host/network combinations. This should be surfaced visually in the app.

## 4. Recommended frontend technology stack

Use this exact stack for the first serious build:

```text
Vite
React
TypeScript
Zustand
TanStack Virtual
CodeMirror 6
xterm.js only for the optional terminal pane
Tauri JS API
CSS variables + plain CSS modules or lightweight scoped CSS
```

### Why this stack

**React + TypeScript** is the fastest route to a polished Codex-like desktop UI with command palette, panes, diff views, settings forms, model controls, context bars, and virtualized logs.

**Zustand** is enough state management. Keep it small. Do not add Redux. The app state is mostly append-only runtime events, current workspace metadata, UI selection state, settings, and pending approvals.

**TanStack Virtual** is critical. The transcript, diff, logs, and replay viewer must virtualize rows. Do not put the full transcript in a giant DOM.

**CodeMirror 6** is the right initial code/diff surface. It is lighter than Monaco and easier to memory-control. Monaco can be added later for a full editor mode, but the first DMG should prioritize chat, tool calls, diffs, proof, and approvals.

**xterm.js** belongs only in the Terminal pane, not in the main agent transcript. The main agent stream should be structured cards from Rust events, not a fake terminal.

**CSS variables** should mirror Quorp’s existing ANSI/render palette: cyan for agent activity, violet for system metadata, green/red for validation and diff, yellow for warnings, dim text for provenance.

## 5. Desktop architecture

The app should have four layers:

```text
React/TypeScript UI
    ↓ Tauri invoke/listen
apps/desktop/src-tauri
    ↓
crates/quorp_desktop_core + crates/quorp_desktop_api
    ↓
existing Quorp crates:
quorp_session
quorp_agent_core
quorp_permissions
quorp_sandbox
quorp_patch_vm
quorp_context
quorp_memory
quorp_rule_forge
quorp_verify
quorp_config
quorp_tools
```

### Do not wire it like this

```text
React → Tauri → spawn "quorp run ..." → parse stdout
```

That path loses structured events, approvals, checkpoint control, status, proof state, cancellation, and diff routing.

### Wire it like this

```text
React composer submit
  → invoke("start_agent_run")
    → quorp_desktop_core::SessionManager
      → quorp_session desktop wrapper
        → quorp_agent_core runtime
          → RuntimeEventSink
            → Tauri app.emit("quorp://runtime-event", event)
              → React virtualized transcript/status/tasks/diff/proof panes
```

The key is a new Rust bridge that converts runtime events into desktop events without changing the agent core.

## 9. Rust bridge: app runtime core

Add `crates/quorp_desktop_core`.

Main responsibility:

```text
Manage workspaces
Manage active runs
Bridge RuntimeEvent → DesktopEvent
Backfill ledger events
Compute diffs
Expose settings/trust/permissions
Own cancellation
Own pending approvals
```

Skeleton:

```rust
// crates/quorp_desktop_core/src/quorp_desktop_core.rs

pub mod diff_model;
pub mod event_bridge;
pub mod session_manager;
pub mod workspace_manager;

pub use session_manager::SessionManager;
pub use workspace_manager::WorkspaceManager;
```

Runtime event sink:

```rust
// crates/quorp_desktop_core/src/event_bridge.rs

use quorp_agent_core::{RuntimeEvent, RuntimeEventSink};
use quorp_desktop_api::{DesktopEvent, RunId};

pub trait DesktopEventEmitter: Send + Sync + 'static {
    fn emit_desktop_event(&self, event: DesktopEvent) -> anyhow::Result<()>;
}

pub struct DesktopRuntimeSink<E> {
    pub run_id: RunId,
    pub emitter: E,
}

impl<E> RuntimeEventSink for DesktopRuntimeSink<E>
where
    E: DesktopEventEmitter,
{
    fn emit(&self, event: RuntimeEvent) {
        let desktop_event = DesktopEvent::Runtime {
            run_id: self.run_id.clone(),
            event,
        };

        if let Err(error) = self.emitter.emit_desktop_event(desktop_event) {
            log::warn!("failed to emit desktop runtime event: {error:#}");
        }
    }
}
```

The no-`let _ =` style matters because Quorp’s own handoff rules explicitly reject ignoring fallible operations.

```rust
use quorp_desktop_core::event_bridge::DesktopEventEmitter;

#[derive(Clone)]
pub struct TauriEmitter {
    app: AppHandle,
}

impl TauriEmitter {
    pub fn new(app: AppHandle) -> Self {
        Self { app }
    }

impl DesktopEventEmitter for TauriEmitter {
    fn emit_desktop_event(&self, event: DesktopEvent) -> anyhow::Result<()> {
```

```rust
pub async fn open_workspace(
    app: AppHandle,
    state: State<'_, AppState>,
    path: std::path::PathBuf,
) -> IpcResult<WorkspaceSummary> {
    let mut workspaces = state.workspaces.write().await;
    let summary = workspaces
        .open_workspace(path)
        .await
        .map_err(to_ipc_error)?;

    let emitter = TauriEmitter::new(app);
    if let Err(error) = emitter.emit_desktop_event(
        quorp_desktop_api::DesktopEvent::WorkspaceOpened {
            workspace: summary.clone(),
        },
    ) {
        log::warn!("failed to emit workspace-opened event: {error:#}");
    }

    Ok(summary)
}

#[tauri::command]
pub async fn start_agent_run(
    app: AppHandle,
    state: State<'_, AppState>,
    request: StartRunRequest,
) -> IpcResult<StartRunResponse> {
    let emitter = TauriEmitter::new(app);
    state
        .sessions
        .start_run(request, emitter)
        .await
        .map_err(to_ipc_error)
}

#[tauri::command]
pub async fn cancel_run(state: State<'_, AppState>, run_id: RunId) -> IpcResult<()> {
    state.sessions.cancel_run(run_id).await.map_err(to_ipc_error)
}

#[tauri::command]
pub async fn get_run_status(
    state: State<'_, AppState>,
    run_id: RunId,
) -> IpcResult<DesktopStatus> {
    state
        .sessions
        .status(run_id)
        .await
        .map_err(to_ipc_error)
}

fn to_ipc_error(error: anyhow::Error) -> String {
    format!("{error:#}")
}
```

The first version of `SessionManager::start_run` should wrap the existing headless/session path in `quorp_session`, but with a desktop event sink. If the current headless function is not public enough, add a small `quorp_session::desktop` module rather than reaching into CLI internals.

```ts
// apps/desktop/src/app/eventBus.ts

import { listen } from "@tauri-apps/api/event";
import { useRunStore } from "../stores/runStore";
import { useWorkspaceStore } from "../stores/workspaceStore";

export async function installQuorpEventListener() {
  const unlisten = await listen<DesktopEvent>("quorp://event", (event) => {
    const payload = event.payload;

    switch (payload.kind) {
      case "workspaceOpened":
        useWorkspaceStore.getState().upsertWorkspace(payload.workspace);
        break;

      case "runtime":
        useRunStore.getState().appendRuntimeEvent(payload.runId, payload.event);
        break;

      case "permissionRequested":
        useRunStore.getState().setPendingPermission(payload.runId, payload);
        break;

      case "diffUpdated":
        useWorkspaceStore.getState().markDiffDirty(payload.workspaceId);
        break;

      case "statusUpdated":
        useRunStore.getState().setStatus(payload.runId, payload.status);
        break;
    }
  });

  return unlisten;
}
```

Tauri docs explicitly recommend unlistening from frontend event listeners when the component is unmounted, which matters for long-running SPAs. ([Tauri][5])

## 14. Transcript virtualization

The runtime can produce long sessions. The UI must not keep a massive DOM alive.

```tsx
// apps/desktop/src/components/AgentTranscript.tsx

import { useMemo, useRef } from "react";
import { useVirtualizer } from "@tanstack/react-virtual";
import { useRunStore } from "../stores/runStore";
import { RuntimeCard } from "./RuntimeCard";

export function AgentTranscript({ runId }: { runId: string }) {
  const parentRef = useRef<HTMLDivElement | null>(null);
  const events = useRunStore((state) => state.eventsByRun.get(runId) ?? []);

  const rows = useMemo(() => {
    return eventsToRows(events);
  }, [events]);

  const virtualizer = useVirtualizer({
    count: rows.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 72,
    overscan: 12,
  });

  return (
    <div ref={parentRef} className="agent-transcript-scroll">
      <div
        style={{
          height: virtualizer.getTotalSize(),
          position: "relative",
        }}
      >
        {virtualizer.getVirtualItems().map((item) => {
          const row = rows[item.index];

          return (
            <div
              key={item.key}
              data-index={item.index}
              ref={virtualizer.measureElement}
              style={{
                position: "absolute",
                transform: `translateY(${item.start}px)`,
                width: "100%",
              }}
            >
              <RuntimeCard row={row} />
            </div>
          );
        })}
      </div>
    </div>
  );
}
```

Events should be stored append-only, and the UI should render derived rows. For very long runs, keep a rolling hot window in memory and backfill from `run-ledger.jsonl` through `list_run_events(run_id, cursor)`.

## 15. Permission UX

When the agent tries a risky action, the app should show a blocking approval card and an inspector panel.

Use the existing classifier as the source of truth. It already distinguishes compound commands, network, dependency install, Docker, git remote mutation, generated executable, secrets read, MCP, and browser.  This matches the seriousness of Claude Code’s documented permission model and compound-command handling, but Quorp should make the decision visible and auditable. ([Claude][6])

Permission prompt design:

```text
┌─ Permission required ──────────────────────────────────────┐
│ Tool: run_command                                          │
│ Command: cargo test && curl https://example.com            │
│ Risk: compound command · network                           │
│ Surface: sandbox tmp-copy                                  │
│ Workspace: /tmp/quorp/.../workspace                        │
│                                                            │
│ [Deny] [Allow once] [Allow for session] [Allow for project] │
└────────────────────────────────────────────────────────────┘
```

Never allow “Full Permissions” to silently mean host execution. The existing config already downgrades unsafe full-auto host/network configurations, and the desktop should show that downgrade instead of hiding it.

## 16. Security posture for Tauri

Use Tauri capabilities narrowly. Tauri capabilities group permissions by windows/webviews, and Tauri’s permission model can explicitly enable or deny commands and define scopes. ([Tauri][7])

The desktop app should not enable broad frontend filesystem or shell permissions. The frontend should never run shell commands directly.

Recommended capability file:

```json
// apps/desktop/src-tauri/capabilities/main.json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "main",
  "description": "Main Quorp desktop window",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "dialog:allow-open",
    "opener:default"
  ]
}
```

Then expose Quorp-specific actions only through your own `#[tauri::command]` handlers:

```text
open_workspace
start_agent_run
cancel_run
approve_permission
deny_permission
get_workspace_diff
run_verify
export_proof_bundle
```

Do **not** expose:

```text
shell:allow-execute
fs:allow-write-text-file
fs:allow-remove
fs:allow-read-file-recursive
```

Those would bypass Quorp’s permission model.

### Fix GUI `$PATH`

Because macOS GUI apps do not inherit the shell `$PATH`, add a tiny startup repair in `path_env.rs`. Tauri explicitly calls this out for GUI apps. ([Tauri][1])

```rust
// apps/desktop/src-tauri/src/path_env.rs

pub fn repair_gui_path() {
    #[cfg(target_os = "macos")]
    {
        let current = std::env::var_os("PATH")
            .map(|value| value.to_string_lossy().to_string())
            .unwrap_or_default();

        let common = [
            "/opt/homebrew/bin",
            "/opt/homebrew/sbin",
            "/usr/local/bin",
            "/usr/bin",
            "/bin",
            "/usr/sbin",
            "/sbin",
            "$HOME/.cargo/bin",
        ];

        let mut parts = current
            .split(':')
            .filter(|part| !part.trim().is_empty())
            .map(ToOwned::to_owned)
            .collect::<Vec<_>>();

        for candidate in common {
            let expanded = candidate.replace(
                "$HOME",
                &std::env::var("HOME").unwrap_or_else(|_| String::new()),
            );
            if !parts.iter().any(|part| part == &expanded) {
                parts.push(expanded);
            }
        }

        unsafe {
            std::env::set_var("PATH", parts.join(":"));
        }
    }
}
```

Later, replace this with `fix-path-env-rs` or a stricter user-configurable path resolver.

([Tauri][9])

Tauri updater can be added once signing is in place. The updater plugin supports macOS and uses signed update artifacts; signatures verify updates and should not be disabled. ([Tauri][10])

## 22. Performance and memory rules

These should be engineering requirements from day one:

```text
No full transcript DOM.
No giant raw transcript string in React state.
No WebSocket server for local mode.
No direct frontend shell/filesystem permissions.
No broad Tauri fs/shell plugin permissions.
No loading full repo tree recursively into React at startup.
No loading entire large diff into CodeMirror at once.
No storing model raw responses in hot UI state.
No Monaco in v1 unless CodeMirror proves insufficient.
```

Targets:

```text
Cold app open: under 2 seconds on Apple Silicon dev machines
Idle memory: under ~200 MB
50k runtime events: still responsive through virtualization
Large diff: chunked/virtualized rendering
Workspace tree: lazy expansion
Run events: append-only ledger + cursor backfill
```

The backend already has bounded runtime fanout and backpressure events; the desktop should preserve that spirit rather than creating unbounded frontend queues.

## 24. The most important product idea

Make Quorp Desktop feel like **mission control for autonomous coding**, not “a chat app with a file tree.”

Codex-style layout is the baseline. Quorp can go beyond it by making these first-class visual objects:

```text
Task DAG
Runtime ledger
Permission state
Context pressure
Sandbox boundary
Patch receipt
Rollback token
Proof DAG
Memory/rule changes
Validation queue
Recovery attempts
```

That is where Quorp can feel clearly better than terminal-only agents: the user should always know what the agent is doing, why it is allowed, what it changed, what proof exists, what can be rolled back, and whether the run is drifting.
