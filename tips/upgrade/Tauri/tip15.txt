Yes ŌĆö this is a terrific fit for **Tauri 2**, but the exact repo placement and bridge design matter a lot. I would build Quorp Desktop as a **sibling app** with a **thin Tauri shell**, a **typed Rust app bridge**, and a **very lean TypeScript UI**. The CLI should remain first-class and unchanged.

I inspected current `main` via GitHub file access. I could not clone locally in this environment because GitHub DNS resolution failed, so the plan below is grounded in the files I could inspect through the repo API plus current Tauri and Apple docs.

## The core recommendation

Build this:

```text
Quorp/
  apps/
    desktop/                  # Tauri 2 desktop product
      package.json
      vite.config.ts
      src/                    # Svelte/TypeScript frontend
      src-tauri/              # Tauri Rust app shell
        Cargo.toml
        tauri.conf.json
        capabilities/
        src/
  crates/
    quorp_app/                # NEW: typed desktop/CLI app service layer
    quorp_cli/                # existing CLI stays
    quorp_session/            # existing runtime/session reuse
    quorp_agent_core/         # existing runtime events/control loop
    quorp_permissions/
    quorp_sandbox/
    quorp_render/
    quorp_term/
```

The key architectural move is: **do not make the desktop app shell out to `quorp run` as its primary runtime path**. The current repo is already split in a way that strongly supports an embedded app service: `crates/quorp/src/main.rs` is only a thin `quorp_cli::dispatch()` wrapper, and the workspace already separates CLI, session/runtime, agent core, rendering, permissions, sandboxing, verification, config, and related crates.  

Quorp already has the right raw ingredients: runtime events, bounded event fanout, checkpoints, proof receipts, command bridge execution, prompt compaction, slash commands, sandbox policies, permission classification, and settings/trust loading. The desktop app should turn those into a polished graphical ŌĆ£mission controlŌĆØ UI.

# 1. Best frontend stack

My pick:

```text
Tauri 2
Svelte 5 + TypeScript + Vite
CodeMirror 6 for rich code/diff panes
xterm.js only for an optional real terminal panel
Rust backend via Tauri commands + channels/events
No local WebSocket for the initial app
```

Tauri officially supports many frontend stacks, including Svelte, React, Vue, Solid, Angular, Preact, Yew, Leptos, and others. ([Tauri][1])

I would choose **Svelte + TypeScript** over React for this app because you want the UI to feel extremely fast and memory-conscious. React is totally viable, especially if contributor velocity is higher with it, but the default desktop UI here should be lean: minimal runtime, no heavy component framework, no Electron-style process overhead, and no giant design system.

My ranking for Quorp Desktop:

| Choice                  | Verdict                                                                     |
| ----------------------- | --------------------------------------------------------------------------- |
| **Tauri 2 + Svelte/TS** | Best balance of speed, polish, memory, and implementation velocity          |
| Tauri 2 + React/TS      | Good if the team is much faster in React                                    |
| Tauri 2 + Solid/TS      | Excellent performance, smaller ecosystem than React/Svelte                  |
| Tauri 2 + Leptos/Yew    | More Rust-pure, slower to iterate on polished IDE UX                        |
| Slint                   | Nice pure-Rust GUI, but harder to build a Codex/Cursor-like coding IDE fast |
| egui/eframe             | Great for internal tools, less ideal for a premium Mac desktop IDE          |

On WebSockets: **do not use WebSockets for the local desktop app at first**. Tauri already gives you typed command invocation from frontend to Rust plus Rust-to-frontend events/channels. TauriŌĆÖs command primitive is built for invoking Rust from the webview, and its docs specifically call out command usage and event/channel-style communication for frontend/backend interaction. ([Tauri][2])

Use WebSockets only later for one of these cases:

```text
Remote attach to a Quorp agent running on another machine
Browser-based replay viewer
Team/multiplayer session streaming
External plugin protocol
```

For local app performance and security, use:

```text
Frontend invoke() calls
        Ōåō
Tauri command bridge
        Ōåō
quorp_app session manager
        Ōåō
quorp_session / quorp_agent_core
        Ōåō
RuntimeEventFanout
        Ōåō
Tauri Channel / event stream back to frontend
```

## New crate: `crates/quorp_app`

This should be the reusable desktop-facing app layer. It should not contain Tauri-specific code. Its job is to expose stable typed APIs that both the desktop app and, later, an improved CLI can reuse.

```text

  Cargo.toml
  src/
    lib.rs
    app_state.rs
    session_manager.rs
    workspace.rs
    desktop_event.rs
    status.rs
    diff.rs
    proof.rs
    permissions.rs
    settings.rs

```

Why not put everything directly in `apps/desktop/src-tauri`? Because the repo already has serious reusable runtime crates. Keeping Tauri-specific code thin preserves QuorpŌĆÖs CLI-first identity while allowing the desktop product to mature fast.

The existing `quorp_session` crate reexports modules such as `agent_runner`, `executor`, `prompt_compaction`, `provider_config`, `run_support`, and `tui`, which is exactly the kind of functionality the desktop bridge should reuse.

## Runtime event mapping

```rust
use quorp_agent_core::{RuntimeEvent, RuntimeEventSink};

use crate::desktop_event::DesktopEvent;

pub struct DesktopRuntimeEventSink {
    run_id: String,
    tx: tokio::sync::mpsc::Sender<DesktopEvent>,
}

impl DesktopRuntimeEventSink {
    pub fn new(run_id: String, tx: tokio::sync::mpsc::Sender<DesktopEvent>) -> Self {
        Self { run_id, tx }
    }

    fn map_event(&self, event: RuntimeEvent) -> Option<DesktopEvent> {
        match event {
            RuntimeEvent::RunStarted { goal, model_id } => Some(DesktopEvent::RunStarted {
                run_id: self.run_id.clone(),
                workspace: Default::default(),
                model_id,
                objective: goal,
            }),
            RuntimeEvent::PhaseChanged { phase, detail } => Some(DesktopEvent::PhaseChanged {
                run_id: self.run_id.clone(),
                phase,
                detail,
            }),
            RuntimeEvent::ModelRequestStarted {
                step,
                request_id,
                message_count,
                prompt_token_estimate,
                completion_token_cap,
                safety_mode,
            } => Some(DesktopEvent::ModelRequestStarted {
                run_id: self.run_id.clone(),
                step,
                request_id,
                message_count,
                prompt_token_estimate,
                completion_token_cap,
                safety_mode,
            }),
            RuntimeEvent::ToolCallStarted { step, action } => Some(DesktopEvent::ToolStarted {
                run_id: self.run_id.clone(),
                step,
                action_label: action,
            }),
            RuntimeEvent::ToolCallFinished {
                step,
                action,
                status,
                action_kind,
                target_path,
                edit_summary,
            } => Some(DesktopEvent::ToolFinished {
                run_id: self.run_id.clone(),
                step,
                action_label: action,
                status,
                action_kind,
                target_path,
                edit_summary,
            }),
            RuntimeEvent::ValidationStarted { step, summary } => {
                Some(DesktopEvent::ValidationStarted {
                    run_id: self.run_id.clone(),
                    step,
                    summary,
                })
            }
            RuntimeEvent::ValidationFinished {
                step,
                summary,
                status,
            } => Some(DesktopEvent::ValidationFinished {
                run_id: self.run_id.clone(),
                step,
                summary,
                status,
            }),
            RuntimeEvent::RunFinished {
                reason,
                total_steps,
                total_billed_tokens,
                duration_ms,
            } => Some(DesktopEvent::RunFinished {
                run_id: self.run_id.clone(),
                reason: format!("{reason:?}"),
                total_steps,
                total_billed_tokens,
                duration_ms,
            }),
            RuntimeEvent::FatalError { error } => Some(DesktopEvent::Error {
                run_id: Some(self.run_id.clone()),
                title: "Fatal runtime error".to_string(),
                detail: error,
            }),
            _ => None,
        }
    }
}

impl RuntimeEventSink for DesktopRuntimeEventSink {
    fn emit(&self, event: RuntimeEvent) {
        if let Some(mapped) = self.map_event(event) {
            let _ = self.tx.try_send(mapped);
        }
    }
}
```

The current runtime already has `RuntimeEventSink`, `RuntimeEventFanout`, named subscribers, bounded queues, and subscriber backpressure reporting. That is a huge advantage for a desktop app because you can avoid log scraping entirely.

## `apps/desktop/src-tauri/src/commands.rs`

```rust
use std::path::PathBuf;

use tauri::{ipc::Channel, AppHandle, Manager, State};

use crate::state::{DesktopState, WorkspaceEntry};

#[derive(Debug, serde::Serialize)]
pub struct CommandError {
    message: String,
}

impl From<anyhow::Error> for CommandError {
    fn from(error: anyhow::Error) -> Self {
        Self {
            message: format!("{error:#}"),
        }
    }
}

#[tauri::command]
pub async fn open_workspace_dialog(
    app: AppHandle,
) -> Result<Option<PathBuf>, CommandError> {
    use tauri_plugin_dialog::DialogExt;

    let picked = app
        .dialog()
        .file()
        .set_title("Open Quorp Workspace")
        .blocking_pick_folder();

    Ok(picked.map(|p| p.to_path_buf()))
}

#[tauri::command]
pub async fn list_workspaces(
    state: State<'_, DesktopState>,
) -> Result<Vec<WorkspaceEntry>, CommandError> {
    Ok(state.workspaces.read().await.clone())
}

#[tauri::command]
pub async fn add_workspace(
    state: State<'_, DesktopState>,
    path: PathBuf,
) -> Result<WorkspaceEntry, CommandError> {
    let loaded = quorp_config::load_settings(&path)?;
    let entry = WorkspaceEntry {
        id: loaded.trust.project_id,
        label: path
            .file_name()
            .and_then(|v| v.to_str())
            .unwrap_or("workspace")
            .to_string(),
        path,
        trusted: loaded.trust.trusted,
        last_opened_unix_ms: now_unix_ms(),
    };

    state.workspaces.write().await.push(entry.clone());
    Ok(entry)
}

#[derive(Debug, serde::Deserialize)]
pub struct StartRunRequest {
    pub workspace: PathBuf,
    pub prompt: String,
    pub model_id: Option<String>,
    pub base_url_override: Option<String>,
    pub permission_mode: String,
    pub sandbox_mode: String,
    pub max_steps: Option<usize>,
    pub max_seconds: Option<u64>,
}

#[derive(Debug, serde::Serialize)]
pub struct StartRunResponse {
    pub run_id: String,
    pub result_dir: PathBuf,
}

#[tauri::command]
pub async fn start_agent_run(
    state: State<'_, DesktopState>,
    request: StartRunRequest,
    events: Channel<quorp_app::DesktopEvent>,
) -> Result<StartRunResponse, CommandError> {
    let options = quorp_app::StartSessionOptions {
        workspace: request.workspace,
        prompt: request.prompt,
        model_id: request.model_id,
        base_url_override: request.base_url_override,
        permission_mode: request.permission_mode,
        sandbox_mode: request.sandbox_mode,
        max_steps: request.max_steps.unwrap_or(80),
        max_seconds: request.max_seconds,
    };

    let session = quorp_app::start_session(options).await?;
    let run_id = session.run_id().to_string();
    let result_dir = session.result_dir().to_path_buf();

    let mut rx = session.subscribe();

    tauri::async_runtime::spawn(async move {
        while let Some(event) = rx.recv().await {
            let _ = events.send(event);
        }
    });

    state
        .sessions
        .write()
        .await
        .insert(run_id.clone(), std::sync::Arc::new(session));

    Ok(StartRunResponse { run_id, result_dir })
}

#[tauri::command]
pub async fn cancel_agent_run(
    state: State<'_, DesktopState>,
    run_id: String,
) -> Result<(), CommandError> {
    let sessions = state.sessions.read().await;
    let session = sessions
        .get(&run_id)
        .ok_or_else(|| anyhow::anyhow!("unknown run id: {run_id}"))?;
    session.cancel();
    Ok(())
}

fn now_unix_ms() -> u64 {
    use std::time::{SystemTime, UNIX_EPOCH};

    SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap_or_default()
        .as_millis()
        .try_into()
        .unwrap_or(u64::MAX)
}
```

This sample uses Tauri IPC channels for runtime event delivery. The exact API surface may need small adjustment depending on the Tauri minor version, but the design is right: one command starts the run, returns a run handle, and streams typed events to the UI.

```ts
export interface StartRunRequest {
  workspace: string;
  prompt: string;
  model_id?: string;
  base_url_override?: string;
  permission_mode: "ask" | "accept-edits" | "auto-safe" | "yolo-sandbox";
  sandbox_mode: "tmp-copy" | "host";
  max_steps?: number;
  max_seconds?: number;
}

export interface StartRunResponse {
  run_id: string;
  result_dir: string;
}

export async function startAgentRun(
  request: StartRunRequest,
  onEvent: (event: DesktopEvent) => void
): Promise<StartRunResponse> {
  const events = new Channel<DesktopEvent>();
  events.onmessage = onEvent;

  return await invoke<StartRunResponse>("start_agent_run", {
    request,
    events
  });
}

  await invoke("cancel_agent_run", { runId });
```

# 9. Permissions and sandbox UX

This has to be excellent. The desktop app should make trust state impossible to miss.

Current Quorp already has:

```text
  read-only
  ask

  find delete
  find exec
  secrets read
```

The existing permission engine classifies commands and denies yolo/full-auto behavior on the host surface.

```text

  cargo test --workspace

  RunCommand

Risk tokens:
  none

Surface:
  git-worktree sandbox

Buttons:

  Allow Once
  Allow This Session
  Allow For This Project
```

For risky commands:

```text

  cargo add anyhow && curl https://example.com/script.sh

  compound_command
  network
  dependency_install
  filesystem_write

Surface:
  sandbox

Recommendation:

```

Current config already prevents untrusted project settings from elevating permissions, network, MCP, browser, process control, or host sandbox behavior. The desktop should show that as a trust badge rather than burying it. 

## Sandbox selector

Options:

```text
tmp-copy       safest default
git-worktree   preferred when repo supports it
host           manual/trusted only
container      future/power mode
```

The sandbox crate already supports host, git-worktree, tmp-copy fallback, environment scrubbing, and container command plans with `--network=none`, `--cap-drop=ALL`, and `no-new-privileges`. 

Desktop default should be:

```text
Permission: Ask
Sandbox: tmp-copy/git-worktree
Network: off
MCP: off
Browser: off
```

Full-auto should mean:

```text
Full Auto Sandboxed
not
Full Host Permissions
```

For Apple Silicon local testing, keep:

```json
"signingIdentity": "-"
```

Tauri documents this as ad-hoc signing; it is useful for local Apple Silicon testing but does not remove macOS Privacy & Security whitelisting prompts for downloaded apps. ([Tauri][5])

That gives you the fast path:

```text
Day 1:
  tauri dev

Day 1/2:
  local ad-hoc .app and .dmg

Later:
  Developer ID signing + notarization + stapling
```

## Production signing and notarization

For a public DMG, use Developer ID signing and notarization. Apple describes notarization as a process for macOS software distributed outside the Mac App Store, and TauriŌĆÖs signing docs describe Developer ID signing, signing identity configuration, CI certificate env vars, and notarization credentials. ([Apple Developer][6])

Production pipeline:

```text
1. Build frontend
2. Compile Rust Tauri app
3. Bundle .app
4. Sign .app with Developer ID Application certificate
5. Build .dmg
6. Sign .dmg
7. Submit for notarization
8. Staple ticket
9. Publish GitHub release
10. Optional: Tauri updater manifest
```

TauriŌĆÖs updater can produce updater artifacts and signatures, but I would add updater support only after the local DMG is stable. The updater docs emphasize signed update artifacts and `createUpdaterArtifacts` configuration. ([Tauri][7])

```json

  "permissions": [

    "dialog:allow-open",
    "opener:default",
    "log:default",
```

Do **not** include broad filesystem or shell plugin permissions. The frontend should not be able to read arbitrary files or run shell commands through generic Tauri plugins. It should ask the Rust backend:

```text
frontend ŌåÆ quorp command ŌåÆ quorp_permissions ŌåÆ quorp_sandbox ŌåÆ quorp_tools/runtime
```

That preserves QuorpŌĆÖs auditability.

# 14. Important product details

## Performance rules

Use these as engineering constraints:

```text
Do not render unbounded transcript DOM.
Do not send giant logs/diffs through JSON.
Do not use generic filesystem/shell Tauri plugins.
Do not keep full file bodies in frontend stores.
Do not re-render entire mission stream on every token/event.
Do not put syntax highlighting on every collapsed card.
```

Use:

```text
Virtualized mission stream.
Paged diff hunks.
Artifact IDs for large outputs.
Small semantic events.
CSS-only status animations.
CodeMirror only for focused file/diff panes.
Rust-side event throttling for noisy command output.
```

## Memory targets

Initial targets:

```text
Cold app launch: under 150 MB RSS if possible
Normal active run: under 300 MB RSS
10,000 event replay: no crash, no UI lockup
Large diff: paged rendering, not one giant DOM tree
```

## Design language

This should look like a premium engineering instrument:

```text
Dark, high-contrast default.
Bright diff colors.
Dense but readable cards.
No toy mascot-first UI in the work surface.
Keyboard-first everything.
Status always visible.
Permission state always visible.
Sandbox state always visible.
```

# 16. The most important technical call

The desktop app should be:

```text
A GUI over QuorpŌĆÖs real typed runtime,
not a GUI over terminal logs.
```

That means:

```text
Use RuntimeEventFanout.
Use CommandBridgeRequest.
Use quorp_permissions.
Use quorp_sandbox.
Use quorp_config trust loading.
Use proof receipts.
Use artifact-backed diffs/logs.
Keep the CLI untouched.
```

Quorp already has the hard agent-runtime foundations. The DMG app should make those foundations visible, controllable, and beautiful.
