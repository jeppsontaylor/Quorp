I could not clone GitHub directly inside the container because DNS resolution failed, so I inspected the repository through the GitHub connector and current web sources. The architectural conclusion is still very clear: **build the desktop app as a Tauri 2 app under `apps/desktop`, backed by a new Rust app-service crate such as `crates/quorp_app`, and keep the existing CLI/TUI as a first-class sibling.**

The winning shape is:

```text
Quorp CLI/TUI
     ↓
quorp_cli / quorp_session / quorp_agent_core

Quorp Desktop.app / DMG
     ↓
apps/desktop  ── Tauri 2 shell + React/TypeScript UI
     ↓
crates/quorp_app  ── typed desktop/session service
     ↓
quorp_session / quorp_agent_core / quorp_config / quorp_sandbox / quorp_verify
```

Tauri is the right default here because it gives Quorp a native macOS `.app`/`.dmg` path while keeping Rust as the backend/core, using the OS WebView instead of shipping an Electron-sized runtime, and letting the frontend be any HTML/CSS/JS framework. Tauri’s own docs position it around small, fast, secure cross-platform apps with Rust logic and a frontend-agnostic UI layer, and its distribution docs explicitly support direct macOS DMG distribution. ([Tauri][1])

## 1. The key decision

Use **Tauri 2 + React + TypeScript + Vite**, not Slint or egui, for the first polished DMG app.

Quorp should remain **Rust-core-first**, but the UI should be a modern web UI because the product you described needs a Codex-style desktop layout: folder sidebar, agent transcript, diff stream, permission chips, context meters, model settings, proof panes, command palette, and virtualized long logs. Pure Rust GUI options are good, but they will slow down this particular app’s polish, layout iteration, diff rendering, and command-panel ergonomics.

Do **not** use WebSockets for the first desktop version. Use Tauri commands and Tauri events:

```text
React UI
  invoke("start_run", ...)
  listen("quorp://event", ...)
        ↓
Tauri command bridge
        ↓
Rust AppState / quorp_app::DesktopRunManager
        ↓
quorp_session::agent_runner
        ↓
quorp_agent_core runtime
```

WebSockets only become useful later if Quorp grows a separate daemon, remote web dashboard, or browser-based replay UI. For the local Mac app, WebSockets add an unnecessary local server, port management, security surface, and memory overhead.

## 2. Why this fits the current Quorp codebase

Quorp is already almost perfectly shaped for this. The repo currently describes Quorp as a high-performance TUI-first editor and agentic interface, with the CLI extracted into `quorp_cli`, the session/runtime path extracted into `quorp_session`, Patch VM-backed write receipts, bounded runtime event fanout, verification proof packets, and benchmark gates. 

The current product shape is a single Rust terminal binary with headless autonomous runs, typed tools, tmp-copy/worktree sandboxing, permission primitives, Patch VM writes, proof packets, memory/rule scaffolding, benchmark scoreboards, and proof receipts. The support doc also says release packaging/signing is not represented yet, which makes desktop packaging a clean new surface rather than a migration. 

The binary is already a thin wrapper:

```rust
fn main() {
    quorp_cli::dispatch();
}
```

That means the desktop app should **not** be wedged into `crates/quorp`. The right move is a new app shell plus a reusable service crate. 

The CLI crate already owns command dispatch and depends on `quorp_render`, `quorp_slash`, `quorp_term`, `quorp_session`, and `quorp_agent_core`.  The inline/fullscreen CLI already starts headless runs, tracks mode/permissions/sandbox, streams progress events, and renders active sessions. That is exactly the behavior the desktop app should reuse, but through typed events instead of ANSI strings. 

The runtime also already emits the events a desktop app needs: model request started/finished, tool call started/finished, validation started/finished, recovery events, parser recovery, verifier drain, policy denied, failed edits, checkpoints, and run finished. It has a bounded `RuntimeEventFanout` with named subscriptions for renderer, memory, rule forge, proof recorder, and benchmark recorder. 

The biggest gap is not “can Quorp run as a desktop app?” It can. The gap is: **Quorp needs a typed app-service layer that turns runtime/session events into durable, structured desktop state.**

## 8. Core wiring design

### Immediate v1 wiring

For fastest usable app:

```text
React Composer
  ↓ invoke("start_run")
Tauri command
  ↓
quorp_app::DesktopRunManager::start_run
  ↓
quorp_session::quorp::agent_runner::run_headless_agent_with_progress
  ↓
std::sync::mpsc::Receiver<TuiEvent>
  ↓
Tauri app.emit("quorp://event", AppEvent)
  ↓
React store updates transcript/status/diff panels
```

This reuses the existing headless progress path immediately. The CLI fullscreen shell already starts headless runs this way. 

### Correct long-term wiring

For a brilliant desktop, do not stop at progress strings. Move to structured runtime events:

```text
quorp_agent_core::RuntimeEvent
  ↓
quorp_app::events::AppEvent
  ↓
Tauri event bus
  ↓
React typed store
```

`RuntimeEvent` already has the right fidelity: tool call events, validation events, recovery events, parser recovery, verifier drain, policy denied, backpressure, checkpoints, and run finished.  The desktop needs those events in typed form, not only rendered summaries.

### Event durability rule

Every desktop event should have:

```rust
pub struct EventEnvelope<T> {
    pub seq: u64,
    pub run_id: String,
    pub session_id: String,
    pub workspace_id: String,
    pub at_unix_ms: u64,
    pub payload: T,
}
```

The backend should write important events to disk before emitting them to the UI:

```text
~/.quorp/app/events/<session_id>.jsonl
workspace/.quorp/runs/desktop/<run_id>/events.jsonl
workspace/.quorp/runs/desktop/<run_id>/proof-receipt.json
```

The UI should be able to recover after reload with:

```text
read_event_window(run_id, cursor, limit)
read_status(run_id)
read_diff(run_id)
read_proof(run_id)
```

That gives the desktop app replayability, resumability, and low memory use.

```rust
// crates/quorp_app/src/events.rs

use serde::{Deserialize, Serialize};
use std::path::PathBuf;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct AppEventEnvelope {
```

### Run manager

```rust
// crates/quorp_app/src/session.rs

use crate::events::{AppEvent, AppEventEnvelope, AppEventSink};
use anyhow::Context as _;
use quorp_agent_core::{AutonomyProfile, CompletionPolicy};
use serde::{Deserialize, Serialize};
use std::path::PathBuf;
use std::sync::{
    atomic::{AtomicU64, Ordering},
    Arc,
};
use std::time::{SystemTime, UNIX_EPOCH};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct StartRunRequest {
    pub workspace: PathBuf,
    pub objective: String,
    pub model_id: String,
    pub base_url_override: Option<String>,
    pub max_steps: usize,
    pub max_seconds: Option<u64>,
    pub permission_mode: quorp_core::PermissionMode,
    pub sandbox_mode: quorp_core::SandboxMode,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct StartRunResponse {
    pub run_id: String,
    pub result_dir: PathBuf,
}

#[derive(Debug, Clone)]
pub struct RunHandle {
    pub run_id: String,
    pub result_dir: PathBuf,
}

#[derive(Default)]
pub struct DesktopRunManager {
    seq: AtomicU64,
}

impl DesktopRunManager {
    pub fn new() -> Self {
        Self::default()
    }

    pub fn start_run(
        &self,
        req: StartRunRequest,
        sink: Arc<dyn AppEventSink>,
    ) -> anyhow::Result<StartRunResponse> {
        let workspace = std::fs::canonicalize(&req.workspace)
            .unwrap_or_else(|_| req.workspace.clone());

        let run_id = format!("desktop-{}", unix_ms());
        let result_dir = workspace
            .join(".quorp")
            .join("runs")
            .join("desktop")
            .join(&run_id);

        std::fs::create_dir_all(&result_dir)
            .with_context(|| format!("failed to create {}", result_dir.display()))?;

        let objective_file = result_dir.join("objective.md");
        std::fs::write(&objective_file, &req.objective)
            .with_context(|| format!("failed to write {}", objective_file.display()))?;

        let autonomy_profile = match (req.permission_mode, req.sandbox_mode) {
            (_, quorp_core::SandboxMode::TmpCopy) => AutonomyProfile::AutonomousSandboxed,
            (_, quorp_core::SandboxMode::Host) => AutonomyProfile::AutonomousHost,
        };

        let options = quorp_session::quorp::agent_runner::HeadlessRunOptions {
            workspace: workspace.clone(),
            objective_file,
            model_id: req.model_id.clone(),
            base_url_override: req.base_url_override.clone(),
            max_steps: req.max_steps,
            max_seconds: req.max_seconds,
            max_total_tokens: None,
            result_dir: result_dir.clone(),
            autonomy_profile,
            completion_policy: CompletionPolicy::default(),
            objective_metadata: serde_json::json!({
                "origin": "desktop",
                "permission_mode": req.permission_mode,
                "sandbox_mode": req.sandbox_mode,
            }),
            seed_context: Vec::new(),
        };

        let seq = self.seq.fetch_add(1, Ordering::Relaxed) + 1;
        sink.emit(AppEventEnvelope {
            seq,
            run_id: run_id.clone(),
            session_id: run_id.clone(),
            workspace_id: workspace.display().to_string(),
            at_unix_ms: unix_ms(),
            event: AppEvent::RunStarted {
                goal: req.objective.clone(),
                model_id: req.model_id.clone(),
                workspace: workspace.clone(),
            },
        });

        let (tx, rx) =
            std::sync::mpsc::sync_channel::<quorp_session::quorp::tui::TuiEvent>(1024);

        let run_id_for_events = run_id.clone();
        let workspace_id = workspace.display().to_string();
        let sink_for_events = Arc::clone(&sink);
        let event_seq = Arc::new(AtomicU64::new(seq));

        std::thread::Builder::new()
            .name(format!("quorp-desktop-events-{run_id}"))
            .spawn(move || {
                while let Ok(event) = rx.recv() {
                    let mapped = map_tui_event(event);
                    let seq = event_seq.fetch_add(1, Ordering::Relaxed) + 1;
                    sink_for_events.emit(AppEventEnvelope {
                        seq,
                        run_id: run_id_for_events.clone(),
                        session_id: run_id_for_events.clone(),
                        workspace_id: workspace_id.clone(),
                        at_unix_ms: unix_ms(),
                        event: mapped,
                    });
                }
            })
            .context("failed to spawn desktop event bridge")?;

        let run_id_for_worker = run_id.clone();
        let workspace_id_for_worker = workspace.display().to_string();
        let sink_for_worker = Arc::clone(&sink);
        let worker_seq = Arc::new(AtomicU64::new(0));

        std::thread::Builder::new()
            .name(format!("quorp-desktop-run-{run_id}"))
            .spawn(move || {
                let outcome =
                    quorp_session::quorp::agent_runner::run_headless_agent_with_progress(
                        options,
                        Some(tx),
                    );

                let event = match outcome {
                    Ok(outcome) => AppEvent::RunFinished {
                        reason: format!("{:?}", outcome.stop_reason),
                        total_steps: outcome.total_steps,
                        billed_tokens: outcome.total_billed_tokens,
                        duration_ms: outcome.duration_ms,
                    },
                    Err(error) => AppEvent::Error {
                        message: error.to_string(),
                    },
                };

                let seq = worker_seq.fetch_add(1, Ordering::Relaxed) + 1;
                sink_for_worker.emit(AppEventEnvelope {
                    seq,
                    run_id: run_id_for_worker.clone(),
                    session_id: run_id_for_worker,
                    workspace_id: workspace_id_for_worker,
                    at_unix_ms: unix_ms(),
                    event,
                });
            })
            .context("failed to spawn desktop agent worker")?;

        Ok(StartRunResponse { run_id, result_dir })
    }
}

fn map_tui_event(event: quorp_session::quorp::tui::TuiEvent) -> AppEvent {
    match event {
        quorp_session::quorp::tui::TuiEvent::Chat(chat) => match chat {
            quorp_session::quorp::tui::ChatUiEvent::CommandOutput(_, line) => {
                AppEvent::CommandOutput { line }
            }
            quorp_session::quorp::tui::ChatUiEvent::AssistantDelta(_, text) => {
                AppEvent::AssistantDelta { text }
            }
            quorp_session::quorp::tui::ChatUiEvent::Error(_, message) => {
                AppEvent::Error { message }
            }
            other => AppEvent::Status {
                label: "chat_event".to_string(),
                detail: format!("{other:?}"),
            },
        },
    }
}

fn unix_ms() -> u64 {
    SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .map(|duration| duration.as_millis() as u64)
        .unwrap_or(0)
}
```

This is a practical first bridge. It should then be upgraded to consume structured `RuntimeEvent` directly.

## 12. Tauri capabilities and security

Do **not** expose broad Tauri filesystem or shell permissions to the frontend.

Use narrow capabilities:


All privileged operations should be custom Rust commands:

```text
start_run
stop_run
approve_permission
reject_permission
load_workspace
read_diff
read_proof
open_result_dir
run_verify
compact_context
set_mode
```

Tauri’s permission model controls IPC access by capabilities; if there is no capability, there is no IPC access, and multiple capabilities merge permissions. Tauri also warns that remote URL access to local systems has security implications. Keep the desktop app local-only, use strict CSP, and avoid remote WebViews. ([Tauri][5])

This is especially important because Quorp is a high-agency coding agent. The frontend should never be able to run arbitrary shell commands directly. It should request actions from Rust; Rust should enforce Quorp permissions, trust state, sandbox state, Patch VM write contracts, and audit logging.

## 14. Frontend state model

Keep the store small and append-only.

```ts
// apps/desktop/src/app/store.ts

import type { AppEventEnvelope } from "./types";

export type TranscriptItem =
  | { kind: "user"; text: string; at: number }
  | { kind: "assistant"; text: string; streaming: boolean; at: number }
  | { kind: "tool"; title: string; detail: string; status: "running" | "passed" | "failed"; at: number }
  | { kind: "command_output"; text: string; at: number }
  | { kind: "validation"; summary: string; status: "running" | "passed" | "failed"; at: number }
  | { kind: "error"; message: string; at: number }
  | { kind: "receipt"; text: string; success: boolean; at: number };

export type RunState = {
  runId?: string;
  status: "idle" | "running" | "success" | "failed";
  transcript: TranscriptItem[];
  outputTail: string[];
  filesChanged: number;
  billedTokens: number;
  contextPercent?: number;
};

export function reduceRunEvent(state: RunState, envelope: AppEventEnvelope): RunState {
  const at = envelope.at_unix_ms;
  const event = envelope.event;

  switch (event.type) {
    case "run_started":
      return {
        ...state,
        runId: envelope.run_id,
        status: "running",
        transcript: [
          ...state.transcript,
          { kind: "assistant", text: `Run started with ${event.model_id}`, streaming: false, at },
        ],
      };

    case "assistant_delta":
      return {
        ...state,
        transcript: [...state.transcript, { kind: "assistant", text: event.text, streaming: true, at }],
      };

    case "command_output":
      return {
        ...state,
        outputTail: [...state.outputTail, event.line].slice(-200),
        transcript: [...state.transcript, { kind: "command_output", text: event.line, at }],
      };

    case "tool_started":
      return {
        ...state,
        transcript: [
          ...state.transcript,
          {
            kind: "tool",
            title: event.action_kind ?? "tool",
            detail: event.action,
            status: "running",
            at,
          },
        ],
      };

    case "validation_finished":
      return {
        ...state,
        transcript: [
          ...state.transcript,
          {
            kind: "validation",
            summary: event.summary,
            status: event.status === "success" ? "passed" : "failed",
            at,
          },
        ],
      };

    case "diff_updated":
      return { ...state, filesChanged: event.files_changed };

    case "context_pressure":
      return { ...state, contextPercent: event.percent ?? undefined };

    case "run_finished":
      return {
        ...state,
        status: event.reason === "Success" ? "success" : "failed",
        billedTokens: event.billed_tokens,
        transcript: [
          ...state.transcript,
          {
            kind: "receipt",
            text: `Run finished: ${event.reason} · ${event.total_steps} steps · ${event.billed_tokens} tokens`,
            success: event.reason === "Success",
            at,
          },
        ],
      };

    case "error":
      return {
        ...state,
        status: "failed",
        transcript: [...state.transcript, { kind: "error", message: event.message, at }],
      };

    default:
      return state;
  }
}
```

For memory, cap transcript DOM rendering with virtualization. Keep all events on disk. Keep only a sliding window in React state unless the user scrolls back.

## 19. Memory and performance rules

This app must stay fast over long runs.

Use these rules:

1. **No WebSocket server in v1.** Use Tauri IPC events.
2. **No broad frontend filesystem plugin.** All file access goes through Rust commands.
3. **No unbounded transcript DOM.** Use virtualization.
4. **No unbounded React event state.** Keep a ring buffer and persist JSONL on disk.
5. **No Monaco for every diff.** Lazy-load editor views.
6. **Aggregate command output.** Keep full logs on disk, show output tails in UI.
7. **Never drop critical events.** Permission requests, proof receipts, run finished, policy denied, and validation failures must be durable.
8. **Use structured events.** Avoid parsing human strings in the React app after PR1.
9. **Use CSS containment.** Transcript and diff panes should use `contain: content` where safe.
10. **Batch UI updates.** If the runtime emits many command-output lines, coalesce them to animation-frame updates.

The current runtime fanout is bounded and can emit subscriber backpressure events.  Desktop should extend that principle: if output floods, show “streaming 4,291 lines; displaying tail” while preserving full logs on disk.

## 20. Security model for desktop

## 24. What not to do

Do not:

```text
put desktop code inside crates/quorp
make the React UI shell out to `quorp`
run a local web server in production
use WebSockets for v1
expose broad fs/shell Tauri plugins
render entire transcripts as DOM forever
mount Monaco for every diff
allow project config to elevate untrusted permissions
block local testing on notarization
```

The desktop app should be a peer of the CLI, not a wrapper around the CLI executable.
