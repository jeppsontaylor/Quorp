## Bottom line

Build the desktop app as **`apps/desktop` with Tauri 2 + React + TypeScript + Vite**, while keeping Quorp’s **agent runtime, permissions, sandboxing, ledger, verification, settings, and CLI in Rust**. Do **not** make the frontend talk to a local WebSocket server for the normal desktop app. Use **Tauri commands for request/response** and **Tauri events for streaming runtime updates**. Use a WebSocket only later if Quorp grows a remote/headless daemon mode.

I could not directly clone via `git clone` in this sandbox because DNS resolution failed, so I inspected the repository through the GitHub connector and cross-checked current official Tauri/macOS docs. The repo evidence strongly supports this path: Quorp is already split into CLI/session/runtime crates, with `quorp_cli`, `quorp_session`, `quorp_agent_core`, `quorp_config`, `quorp_permissions`, `quorp_sandbox`, `quorp_render`, `quorp_slash`, and proof/ledger-related crates already available in the workspace. The current docs explicitly say release packaging/signing is not yet represented, so the desktop app should be a new first-class app target rather than a mutation of the CLI binary.   

The fastest good path is:

```text
/apps/desktop
  Tauri 2 desktop app
  React + TypeScript + Vite frontend
  macOS .app/.dmg packaging

/crates/quorp_desktop_core
  Rust desktop service layer
  session manager
  workspace manager
  runtime event projector
  permission broker
  PTY manager
  artifact/diff/proof access

/crates/quorp_desktop_model
  serde DTOs shared by Rust commands/events and TypeScript bindings

Existing crates stay reusable:
  quorp_agent_core
  quorp_session
  quorp_core
  quorp_config
  quorp_permissions
  quorp_sandbox
  quorp_patch_vm
  quorp_verify
  quorp_slash
  quorp_render
```

Tauri is the right default: official Tauri 2 docs position it as a Rust-backed, frontend-independent way to build small, fast, secure cross-platform apps using the OS web renderer, and its templates support React/TypeScript directly. Tauri’s command and event APIs are also exactly the shape Quorp needs: commands for UI actions like “start run,” “approve permission,” and “get diff,” plus events for streaming runtime updates. ([Tauri][1])

Do **not** put the desktop app under `crates/quorp`. `crates/quorp/src/main.rs` is currently a tiny CLI entrypoint that calls `quorp_cli::dispatch()`, which is exactly what we want for the terminal CLI. Keep that binary clean and stable. 

Also do **not** make `crates/quorp_cli` own desktop behavior. `quorp_cli` is command-dispatch heavy and built around clap/crossterm/headless command paths; it already dispatches `run`, `exec`, `doctor`, `session`, `replay`, `proof`, benchmark commands, and inline terminal mode. The desktop app should reuse the same underlying runtime/session crates, not route everything through CLI parsing.  

The right seam is `quorp_session` + `quorp_agent_core`. `quorp_session` already exposes `agent_runner`, `executor`, `prompt_compaction`, `provider_config`, `run_support`, and `tui` modules, while `quorp_agent_core` already has the runtime event model, run request model, completion/tool traits, checkpoint model, stop reasons, token usage, repair memory, and bounded event fanout.

## 2. Frontend technology stack

Use this stack for the first production-quality desktop app:

```text
Tauri 2
React
TypeScript
Vite
CodeMirror 6 for diff/editor surfaces
xterm.js for optional integrated terminal pane
TanStack Virtual or react-virtuoso for transcript/event virtualization
Zustand or TanStack Store for small client state
CSS modules or vanilla CSS variables
Radix UI only for menus/dialog primitives, not a huge UI kit
```

### Why React/TypeScript instead of pure Rust UI

Quorp’s **core logic should remain Rust**. The UI should be React/TypeScript because this app needs a polished, dense, fast product interface: multi-pane layout, virtualized timelines, command palette, model/settings controls, diff viewer, permission modals, proof explorer, terminal pane, and high-quality keyboard UX. Tauri lets the app logic stay in Rust while using any frontend framework that builds to HTML/CSS/JS. ([Tauri][1])

Slint or egui would be appealing for an all-Rust demo, but I would not choose them for the first polished Quorp desktop release. egui is excellent for internal tools, but Quorp needs a Codex/Cursor/Claude-Code-level product shell. Slint is cleaner than egui for native-ish UI, but React/Tauri gives faster iteration and a better ecosystem for diff panes, virtualized logs, command palettes, and terminal emulation.

### WebSockets?

No WebSocket for the normal local desktop app.

Use this:

```text
React UI
  ↓ invoke()
Tauri command IPC
  ↓
Rust desktop service
  ↓
Quorp runtime/session/core crates

Rust runtime events
  ↓ app.emit(...)
Tauri event IPC
  ↓ listen()
React event store
```

Tauri commands are designed for calling Rust functions from the frontend and returning serializable data; Tauri events are designed for backend/frontend event streams with `emit` and `listen`. For very large payloads, Tauri docs also call out that JSON can slow the app and that binary/IPC response paths are available, which matters for large diffs and logs. ([Tauri][2])

Use WebSocket only for a future mode like:

```text
Quorp desktop connected to remote quorp-agentd
Quorp desktop connected to a cloud runner
Team replay dashboard
Multi-machine benchmark monitor
```

For the local app, a localhost server adds unnecessary attack surface, port conflicts, CORS complexity, and one more thing to secure.

### Existing primitives to reuse

Create `apps/desktop/src-tauri/src/state.rs`:

```rust
use std::collections::HashMap;
use std::sync::Arc;

use tokio::sync::{mpsc, RwLock};

use quorp_desktop_core::{
    ArtifactStore, PermissionBroker, PtyManager, SessionManager, WorkspaceManager,
};
use quorp_desktop_model::DesktopEvent;

#[derive(Clone)]
pub struct AppState {
    pub workspaces: Arc<WorkspaceManager>,
    pub sessions: Arc<SessionManager>,
    pub permissions: Arc<PermissionBroker>,
    pub artifacts: Arc<ArtifactStore>,
    pub ptys: Arc<PtyManager>,
    pub event_tx: mpsc::Sender<DesktopEvent>,
    pub event_rx: Arc<RwLock<Option<mpsc::Receiver<DesktopEvent>>>>,
}

impl AppState {
    pub fn new() -> Self {
        let (event_tx, event_rx) = mpsc::channel(4096);

        Self {
            workspaces: Arc::new(WorkspaceManager::new()),
            sessions: Arc::new(SessionManager::new(event_tx.clone())),
            permissions: Arc::new(PermissionBroker::new()),
            artifacts: Arc::new(ArtifactStore::new()),
            ptys: Arc::new(PtyManager::new(event_tx.clone())),
            event_tx,
            event_rx: Arc::new(RwLock::new(Some(event_rx))),
        }
    }
}

pub fn install_event_pump(state: AppState, app: tauri::AppHandle) {
    use tauri::Emitter;

    tauri::async_runtime::spawn(async move {
        let mut rx = {
            let mut guard = state.event_rx.write().await;
            guard.take()
        };

        let Some(ref mut rx) = rx else {
            return;
        };

        let mut buffer = Vec::with_capacity(128);

        loop {
            buffer.clear();

            match rx.recv().await {
                Some(first) => buffer.push(first),
                None => break,
            }

            while buffer.len() < 128 {
                match rx.try_recv() {
                    Ok(event) => buffer.push(event),
                    Err(_) => break,
                }
            }

            // Batch events to avoid over-rendering the webview during long runs.
            if let Err(error) = app.emit("quorp://events", &buffer) {
                tracing::warn!("failed to emit desktop events: {error}");
            }
        }
    });
}
```

The batching is important. Long autonomous runs can emit many tool/model/validation events, and the React app should not re-render on every tiny event. Keep raw ledger artifacts on disk; send compact UI facts.

## 7. Runtime event projection

Add a projector that turns internal runtime events into stable UI events.

```rust
use std::sync::atomic::{AtomicU64, Ordering};

use tokio::sync::mpsc;

use quorp_agent_core::{RuntimeEvent, RuntimeEventSink};
use quorp_desktop_model::{DesktopEvent, TimelineCardDto};

pub struct DesktopRuntimeSink {
    session_id: String,
    seq: AtomicU64,
    tx: mpsc::Sender<DesktopEvent>,
}

impl DesktopRuntimeSink {
    pub fn new(session_id: String, tx: mpsc::Sender<DesktopEvent>) -> Self {
        Self {
            session_id,
            seq: AtomicU64::new(0),
            tx,
        }
    }

    fn project(&self, event: &RuntimeEvent, seq: u64) -> Vec<DesktopEvent> {
        let mut out = Vec::new();

        match event {
            RuntimeEvent::ToolCallStarted { step, action } => {
                out.push(DesktopEvent::TimelineCard {
                    session_id: self.session_id.clone(),
                    card: TimelineCardDto::ToolCall {
                        step: *step,
                        action: action.clone(),
                        status: "running".to_string(),
                        target_path: None,
                        summary: None,
                    },
                });
            }
            RuntimeEvent::ToolCallFinished {
                step,
                action,
                status,
                target_path,
                edit_summary,
                ..
            } => {
                out.push(DesktopEvent::TimelineCard {
                    session_id: self.session_id.clone(),
                    card: TimelineCardDto::ToolCall {
                        step: *step,
                        action: action.clone(),
                        status: status.clone(),
                        target_path: target_path.clone(),
                        summary: edit_summary.clone(),
                    },
                });
            }
            RuntimeEvent::ValidationStarted { step, summary } => {
                out.push(DesktopEvent::TimelineCard {
                    session_id: self.session_id.clone(),
                    card: TimelineCardDto::Validation {
                        step: *step,
                        summary: summary.clone(),
                        status: "running".to_string(),
                    },
                });
            }
            RuntimeEvent::ValidationFinished {
                step,
                summary,
                status,
            } => {
                out.push(DesktopEvent::TimelineCard {
                    session_id: self.session_id.clone(),
                    card: TimelineCardDto::Validation {
                        step: *step,
                        summary: summary.clone(),
                        status: status.clone(),
                    },
                });
            }
            RuntimeEvent::PolicyDenied {
                step,
                action,
                reason,
            } => {
                out.push(DesktopEvent::TimelineCard {
                    session_id: self.session_id.clone(),
                    card: TimelineCardDto::Recovery {
                        title: format!("Policy denied at step {step}"),
                        detail: format!("{action}: {reason}"),
                    },
                });
            }
            RuntimeEvent::RunFinished {
                reason,
                total_steps,
                total_billed_tokens,
                duration_ms,
            } => {
                out.push(DesktopEvent::TimelineCard {
                    session_id: self.session_id.clone(),
                    card: TimelineCardDto::AssistantMessage {
                        text: format!(
                            "Run finished: {reason:?}, steps={total_steps}, billed_tokens={total_billed_tokens}, duration_ms={duration_ms}"
                        ),
                        streaming: false,
                    },
                });
            }
            _ => {}
        }

        out.push(DesktopEvent::Runtime {
            session_id: self.session_id.clone(),
            seq,
            event: serde_json::to_value(event).unwrap_or(serde_json::Value::Null),
        });

        out
    }
}

impl RuntimeEventSink for DesktopRuntimeSink {
    fn emit(&self, event: RuntimeEvent) {
        let seq = self.seq.fetch_add(1, Ordering::Relaxed) + 1;
        let projected = self.project(&event, seq);

        for event in projected {
            let tx = self.tx.clone();
            tauri::async_runtime::spawn(async move {
                let _ = tx.send(event).await;
            });
        }
    }
}
```

This is intentionally a projection layer. The UI sees friendly cards, while Dev/Replay mode can still inspect raw runtime events.

## 9. Permissions and safety in the desktop app

The frontend must never get a generic “run shell command” capability.

Bad:

```text
React frontend -> shell plugin -> arbitrary command
```

Good:

```text
React frontend -> Tauri command -> Quorp PermissionBroker -> quorp_permissions -> quorp_sandbox -> typed tool executor
```

Quorp already has strong building blocks here. `quorp_config` defaults to `Ask` permission mode, `TmpCopy` sandbox mode, network/MCP/browser/process-control disabled, and managed policy that downgrades unsafe full-auto combinations such as full-auto host or full-auto with network on. It also prevents untrusted project settings from elevating permissions/network/MCP/browser/process control. 

`quorp_permissions` already classifies commands into capability tokens such as compound command, filesystem write, network, dependency install, docker, git remote mutation, find-delete, find-exec, secrets-read, generated executable, MCP, and browser. It also denies yolo/full-permission behavior on the host surface unless running inside the sandbox surface. That is exactly what the desktop permission modal should display. 

### Permission prompt UI

When a tool needs approval, show:

```text
Permission requested

Action:
  RunCommand

Command:
  cargo test --workspace

Classified capability:
  RunCommand

Risk tokens:
  none

Surface:
  Sandbox: git-worktree

Allow:
  [Deny] [Allow once] [Allow for session] [Allow for project]
```

For risky commands:

```text
Permission requested

Command:
  cargo test && curl https://example.com

Risk tokens:
  compound_command
  network

Recommended:
  Deny, or split command into validation + explicit network approval.

Allow:
  [Deny] [Allow once]
```

### Tauri capabilities

Keep `apps/desktop/src-tauri/capabilities/default.json` narrow.

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "Default Quorp desktop permissions",
  "windows": ["main"],
  "permissions": [
    "core:default"
  ]
}
```

Add dialog/opener permissions only when the app actually uses those plugins. Do not grant frontend shell execution. Tauri’s shell/sidecar capabilities can be scoped, but Quorp should keep process execution behind Rust-side Quorp permissions rather than exposing shell execution to the webview. ([Tauri][5])

## 11. Integrated terminal pane

Use `xterm.js` in the frontend and `portable-pty` in Rust.

Important separation:

```text
Agent tool commands:
  Quorp runtime -> permission engine -> sandbox/tool executor -> timeline cards

User terminal pane:
  xterm.js -> Tauri command -> Rust PTY manager -> shell
```

The terminal pane is for the human. The agent should not secretly type into the user’s terminal. Agent commands should remain typed tool calls and proofable events.

Rust PTY command sketch:

```rust
#[tauri::command]
pub async fn open_terminal(
    state: tauri::State<'_, AppState>,
    workspace_id: String,
) -> Result<String, String> {
    state
        .ptys
        .open_terminal(workspace_id)
        .await
        .map_err(|error| error.to_string())
}

#[tauri::command]
pub async fn pty_write(
    state: tauri::State<'_, AppState>,
    terminal_id: String,
    data: String,
) -> Result<(), String> {
    state
        .ptys
        .write(&terminal_id, data.as_bytes())
        .await
        .map_err(|error| error.to_string())
}

#[tauri::command]
pub async fn pty_resize(
    state: tauri::State<'_, AppState>,
    terminal_id: String,
    cols: u16,
    rows: u16,
) -> Result<(), String> {
    state
        .ptys
        .resize(&terminal_id, cols, rows)
        .await
        .map_err(|error| error.to_string())
}
```

Frontend:

```tsx
import { Terminal } from "@xterm/xterm";
import { FitAddon } from "@xterm/addon-fit";
import { invoke } from "@tauri-apps/api/core";
import { listen } from "@tauri-apps/api/event";

export function TerminalPane({ workspaceId }: { workspaceId: string }) {
  const ref = useRef<HTMLDivElement | null>(null);

  useEffect(() => {
    const terminal = new Terminal({
      convertEol: true,
      fontFamily: "var(--mono)",
      fontSize: 13,
      cursorBlink: true,
      scrollback: 10000,
    });

    const fit = new FitAddon();
    terminal.loadAddon(fit);

    let terminalId = "";

    async function boot() {
      terminal.open(ref.current!);
      fit.fit();

      terminalId = await invoke<string>("open_terminal", { workspaceId });

      terminal.onData((data) => {
        invoke("pty_write", { terminalId, data });
      });

      await listen<{ terminalId: string; data: string }>(
        "quorp://terminal-data",
        (event) => {
          if (event.payload.terminalId === terminalId) {
            terminal.write(event.payload.data);
          }
        }
      );
    }

    boot();

    return () => terminal.dispose();
  }, [workspaceId]);

  return <div ref={ref} className="terminal-pane" />;
}
```

This can power both desktop `/status` and terminal `/status`.

## 15. Performance and memory rules

This app can be very fast if the UI does not try to own the whole run.

### Rules

```text
1. React state holds visible UI state, not raw logs.
2. Rust owns artifacts, ledger, proof, raw model responses, and large diffs.
3. UI requests large data by range/chunk.
4. Timeline is virtualized.
5. Event stream is batched.
6. Diff/editor components mount only when visible.
7. Terminal component mounts only when terminal tab is open.
8. Raw provider responses never enter React state unless opened explicitly.
9. Use compact DTOs for common timeline cards.
10. Keep replay/proof artifacts on disk and show hashes/paths.
```

### Event batching target

```text
Runtime event pump:
  collect up to 128 events
  flush every recv batch or every animation-ish tick
  coalesce status snapshots
  keep only latest token/context pressure in status bar
```

### Transcript virtualization

Use virtualization for timeline cards:

```tsx
<VirtualizedTimeline
  cards={cards}
  estimateSize={(card) => estimateCardHeight(card)}
  overscan={12}
/>
```

Keep full card bodies in Rust/artifact store when huge:

```text
Timeline card:
  id
  kind
  title
  summary
  status
  artifact_ref

User expands:
  invoke("read_artifact_chunk", { artifactRef, offset, limit })
```

### Large diffs

Do not send a 10 MB diff through event IPC.

Use:

```text
DesktopEvent::DiffChanged { files: summaries }

Then frontend calls:
  get_diff_file(session_id, path)
  get_diff_hunk(session_id, path, hunk_id)
```

For very large file/diff payloads, prefer binary/chunked IPC rather than one giant JSON payload, matching Tauri’s own warning that large JSON return values can slow app calls. ([Tauri][2])

## 17. One possible “quick demo” path

For the absolute fastest usable demo, there is a pragmatic bridge:

```text
Desktop app spawns existing `quorp exec` / `quorp run`
Desktop app tails run-ledger.jsonl/events.jsonl
Desktop app renders timeline/proof/diff
```

This avoids deep runtime refactoring on day one. But it should be a **temporary alpha path**, because product-grade desktop should call the Rust runtime in-process through `quorp_session`, not parse CLI text or manage a child CLI as the main runtime.

A safe temporary command can be Rust-backend-only:

```rust
std::process::Command::new(quorp_binary)
    .arg("exec")
    .arg("--workspace")
    .arg(workspace)
    .arg("--result-dir")
    .arg(result_dir)
    .arg(task)
    .spawn()
```

Do not expose this through a frontend shell permission. The frontend says “start run”; Rust decides exactly what process/path/args are allowed.

## Final recommendation

Build Quorp Desktop as:

```text
Tauri 2
React + TypeScript + Vite
Rust desktop service crate
Typed Tauri command/event IPC
CodeMirror 6 diff/proof/editor panes
xterm.js only for the optional terminal pane
Quorp runtime in-process as soon as the session API is exposed
DMG via Tauri bundler
Ad-hoc signing for fast local testing
Developer ID signing + notarization for public release
```

The most important architectural decision is this:

```text
Do not make the desktop app a terminal wrapper.
Make it a typed UI over Quorp’s runtime ledger, events, permissions, sandbox, proofs, and patches.
```

That gives you a brilliant desktop app quickly, while preserving the terminal CLI and keeping Quorp’s high-agency safety model in Rust.
