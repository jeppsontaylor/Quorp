## Executive verdict

Build the desktop app as **`apps/desktop` using Tauri 2 + React + TypeScript + Vite**, backed by a new Rust crate called **`crates/quorp_desktop_api`** that reuses `quorp_session`, `quorp_agent_core`, `quorp_config`, `quorp_permissions`, `quorp_sandbox`, `quorp_verify`, and the existing run-ledger/proof/event machinery.

That gives Quorp a polished Mac `.app` and `.dmg` quickly, while keeping the terminal CLI intact. The current repo is already structured for this: the CLI is only a thin dispatcher, the session/runtime path has already been extracted into `quorp_session`, and the runtime already emits rich typed events and proof artifacts. The desktop app should **not** shell out to `quorp run` except as a fallback. It should call the Rust session/runtime APIs in-process.

The most important decision: **do not use WebSockets for the local desktop app.** Use Tauri commands/channels/events. WebSockets add a local server, port management, more attack surface, serialization overhead, and avoidable failure modes. Tauri’s command system is designed for calling Rust from the web frontend, and its event/channel path is the right bridge for backend-to-frontend streaming; Tauri’s docs also explicitly distinguish commands from events and note that channels are better for streaming/high-throughput data than plain events. ([Tauri][1])

The first usable Mac app can be unsigned/ad-hoc signed for local testing, with signed/notarized release builds added later. Tauri’s DMG docs show direct `tauri build --bundles dmg` support, and its macOS signing docs support both Developer ID signing/notarization and ad-hoc signing with signing identity `-`. ([Tauri][2])

## What the current Quorp repo implies

Quorp is already in the right shape for a desktop shell. The workspace has separate crates for CLI, session, runtime, config, permissions, sandbox, rendering, slash commands, verification, context, memory, and rules. The root `Cargo.toml` lists `quorp_cli`, `quorp_session`, `quorp_agent_core`, `quorp_config`, `quorp_permissions`, `quorp_sandbox`, `quorp_render`, `quorp_slash`, `quorp_term`, `quorp_verify`, and related crates as workspace members. 

The README says the current product is a Rust terminal binary with CLI extracted into `quorp_cli` and session/runtime extracted into `quorp_session`; it also says Quorp already has typed runtime actions, context hooks, Patch VM write receipts, bounded runtime event fanout, verification reports, proof packets, and benchmark gates. 

`AGENT_SUPPORT.md` is even more explicit: Quorp currently has a stream-first CLI, headless autonomous run path, typed agent tools, git-worktree/tmp-copy sandboxing, permission policy primitives, Patch VM writes, proof packets, memory/rule scaffolding, benchmark scoreboards, and proof receipts. It also says release packaging/signing is not represented yet, which makes the desktop/DMG work a clean new product surface rather than a refactor of existing release code. 

The existing binary should stay untouched. `crates/quorp/src/main.rs` simply calls `quorp_cli::dispatch()`, and `quorp_cli` itself is a small dispatch wrapper into the CLI module.  

The desktop backend should depend on `quorp_session`, because that crate already exports `agent_runner`, `executor`, `prompt_compaction`, `provider_config`, `run_support`, and the shared TUI/runtime service modules.

Tauri + React is the most pragmatic path.

```text
Store large logs/diffs in Rust-side files and fetch windows/ranges.
Use CodeMirror only for visible diff/editor panes.
```

# 6. Rust wiring design

## Existing path to reuse immediately

The CLI already has a working model:

1. Write an `objective.md`.
2. Build `HeadlessRunOptions`.
3. Spawn `run_headless_agent_with_progress`.
4. Receive `TuiEvent`.
5. Render state from events.
6. Write artifacts into the run dir.

That path is visible in `run_inline_task` inside `crates/quorp_cli/src/quorp/cli_runtime.rs`. It creates a result dir, writes `objective.md`, builds `HeadlessRunOptions`, starts `run_headless_agent_with_progress`, and receives progress events over a bounded channel. 

For **desktop v0**, use that path.

For **desktop v1**, improve it: emit raw typed `RuntimeEvent` to the desktop, not just summarized `TuiEvent`. `quorp_agent_core::RuntimeEvent` is already rich enough for a first-class desktop timeline: model request started/finished, token usage, watchdog reports, phase changes, tool call started/finished, validation, path failures, recovery, parse recovery, verifier queue/drain, policy denied, failed edit, controller reads, checkpoints, and run finished.

Then map `RuntimeEvent` into `DesktopEvent`:

```rust
use quorp_agent_core::RuntimeEvent;

pub fn map_runtime_event(
    session_id: &str,
    result_dir: std::path::PathBuf,
    event: RuntimeEvent,
) -> Option<DesktopEvent> {
    Some(match event {
        RuntimeEvent::RunStarted { goal, model_id } => DesktopEvent::RunStarted {
            session_id: session_id.to_string(),
            goal,
            model_id,
        },

        RuntimeEvent::StatusUpdate { status } => DesktopEvent::Status {
            session_id: session_id.to_string(),
            status: format!("{status:?}"),
            detail: None,
        },

        RuntimeEvent::ModelRequestStarted {
            step,
            request_id,
            message_count,
            prompt_token_estimate,
            ..
        } => DesktopEvent::ModelRequestStarted {
            session_id: session_id.to_string(),
            step,
            request_id,
            message_count,
            prompt_token_estimate,
        },

        RuntimeEvent::ModelRequestFinished {
            step,
            request_id,
            usage: Some(usage),
            watchdog,
        } => DesktopEvent::ModelRequestFinished {
            session_id: session_id.to_string(),
            step,
            request_id,
            input_tokens: usage.input_tokens,
            output_tokens: usage.output_tokens,
            total_billed_tokens: usage.total_billed_tokens,
            latency_ms: usage.latency_ms,
            near_limit: watchdog.map(|w| w.near_limit).unwrap_or(false),
        },

        RuntimeEvent::ToolCallStarted { step, action } => DesktopEvent::ToolStarted {
            session_id: session_id.to_string(),
            step,
            action,
        },

        RuntimeEvent::ToolCallFinished {
            step,
            action,
            status,
            action_kind,
            target_path,
            edit_summary,
        } => DesktopEvent::ToolFinished {
            session_id: session_id.to_string(),
            step,
            action,
            status,
            action_kind,
            target_path,
            edit_summary,
        },

        RuntimeEvent::ValidationStarted { step, summary } => DesktopEvent::ValidationStarted {
            session_id: session_id.to_string(),
            step,
            summary,
        },

        RuntimeEvent::ValidationFinished {
            step,
            summary,
            status,
        } => DesktopEvent::ValidationFinished {
            session_id: session_id.to_string(),
            step,
            summary,
            status,
        },

        RuntimeEvent::AssistantTurnSummary {
            step,
            assistant_message,
            actions,
            wrote_files,
            validation_queued,
            parse_warning_count,
        } => DesktopEvent::AssistantSummary {
            session_id: session_id.to_string(),
            step,
            text: assistant_message,
            actions,
            wrote_files,
            validation_queued,
            parse_warning_count,
        },

        RuntimeEvent::PolicyDenied {
            step,
            action,
            reason,
        } => DesktopEvent::PolicyDenied {
            session_id: session_id.to_string(),
            step,
            action,
            reason,
        },

        RuntimeEvent::CheckpointSaved { checkpoint } => DesktopEvent::CheckpointSaved {
            session_id: session_id.to_string(),
            step: checkpoint.step,
        },

        RuntimeEvent::RunFinished {
            reason,
            total_steps,
            total_billed_tokens,
            duration_ms,
        } => DesktopEvent::RunFinished {
            session_id: session_id.to_string(),
            reason: format!("{reason:?}"),
            total_steps,
            total_billed_tokens,
            duration_ms,
            result_dir,
        },

        RuntimeEvent::FatalError { error } => DesktopEvent::Error {
            session_id: Some(session_id.to_string()),
            message: error,
        },

        other => DesktopEvent::Status {
            session_id: session_id.to_string(),
            status: event_label(&other).to_string(),
            detail: Some(format!("{other:?}")),
        },
    })
}

fn event_label(event: &RuntimeEvent) -> &'static str {
    match event {
        RuntimeEvent::PhaseChanged { .. } => "phase_changed",
        RuntimeEvent::PathResolutionFailed { .. } => "path_resolution_failed",
        RuntimeEvent::RecoveryTurnQueued { .. } => "recovery_queued",
        RuntimeEvent::ParseRecoveryQueued { .. } => "parse_recovery_queued",
        RuntimeEvent::VerifierQueued { .. } => "verifier_queued",
        RuntimeEvent::VerifierDrainStarted { .. } => "verifier_drain_started",
        RuntimeEvent::VerifierDrainFinished { .. } => "verifier_drain_finished",
        RuntimeEvent::PendingValidationBlocked { .. } => "pending_validation_blocked",
        RuntimeEvent::FailedEditRecorded { .. } => "failed_edit_recorded",
        RuntimeEvent::ControllerReadInjected { .. } => "controller_read_injected",
        RuntimeEvent::SubscriberBackpressure { .. } => "subscriber_backpressure",
        _ => "runtime_event",
    }
}
```

```rust
use std::path::PathBuf;
use std::sync::{
    Arc,
    atomic::AtomicBool,
    mpsc,
};

use anyhow::Context as _;
use quorp_agent_core::{AutonomyProfile, CompletionPolicy};
use quorp_desktop_api::protocol::{DesktopEvent, SessionStartRequest, SessionStartResponse};

use crate::events::emit_event;
use crate::state::{AppState, DesktopSessionHandle};

pub fn spawn_session(
    state: AppState,
    session_id: String,
    request: SessionStartRequest,
) -> anyhow::Result<SessionStartResponse> {
    let workspace = std::fs::canonicalize(&request.workspace)
        .unwrap_or_else(|_| request.workspace.clone());

    let result_dir = quorp_session::quorp::run_support::default_run_result_dir(
        &workspace,
        "desktop",
    );
    std::fs::create_dir_all(&result_dir)?;

    let objective_file = result_dir.join("objective.md");
    std::fs::write(&objective_file, &request.prompt)?;

    let cancelled = Arc::new(AtomicBool::new(false));

    state.sessions.insert(
        session_id.clone(),
        DesktopSessionHandle {
            session_id: session_id.clone(),
            workspace: workspace.clone(),
            result_dir: result_dir.clone(),
            cancelled: cancelled.clone(),
        },
    );

    let model_id = request
        .provider
        .as_ref()
        .map(|p| p.model.clone())
        .unwrap_or_else(|| quorp_core::DEFAULT_NVIDIA_MODEL.to_string());

    let base_url_override = request.provider.as_ref().map(|p| p.base_url.clone());

    let autonomy_profile = match request.permission_mode {
        quorp_core::PermissionMode::Ask => AutonomyProfile::Interactive,
        quorp_core::PermissionMode::FullAuto => AutonomyProfile::AutonomousSandboxed,
        quorp_core::PermissionMode::FullPermissions => {
            if request.sandbox_mode == quorp_core::SandboxMode::TmpCopy {
                AutonomyProfile::AutonomousSandboxed
            } else {
                AutonomyProfile::AutonomousHost
            }
        }
    };

    let options = quorp_session::quorp::agent_runner::HeadlessRunOptions {
        workspace: workspace.clone(),
        objective_file: objective_file.clone(),
        model_id: model_id.clone(),
        base_url_override,
        max_steps: request.max_steps.unwrap_or(12),
        max_seconds: request.max_seconds,
        max_total_tokens: request.max_total_tokens,
        result_dir: result_dir.clone(),
        autonomy_profile,
        completion_policy: CompletionPolicy::default(),
        objective_metadata: serde_json::json!({
            "origin": "desktop",
            "permission_mode": request.permission_mode,
            "sandbox_mode": request.sandbox_mode,
            "run_mode": request.run_mode,
        }),
        seed_context: Vec::new(),
    };

    let app = state.app.clone();
    let event_session_id = session_id.clone();
    let event_result_dir = result_dir.clone();

    let (tx, rx) = mpsc::sync_channel::<quorp_session::quorp::tui::TuiEvent>(1024);

    std::thread::spawn(move || {
        emit_event(&app, &DesktopEvent::RunStarted {
            session_id: event_session_id.clone(),
            goal: request.prompt.clone(),
            model_id,
        });

        let app_for_events = app.clone();
        let session_id_for_events = event_session_id.clone();

        std::thread::spawn(move || {
            while let Ok(event) = rx.recv() {
                let mapped = quorp_desktop_api::event_map::map_tui_event(
                    &session_id_for_events,
                    event,
                );
                for desktop_event in mapped {
                    emit_event(&app_for_events, &desktop_event);
                }
            }
        });

        let outcome = quorp_session::quorp::agent_runner::run_headless_agent_with_progress(
            options,
            Some(tx),
        );

        match outcome {
            Ok(outcome) => emit_event(&app, &DesktopEvent::RunFinished {
                session_id: event_session_id,
                reason: format!("{:?}", outcome.stop_reason),
                total_steps: outcome.total_steps,
                total_billed_tokens: outcome.total_billed_tokens,
                duration_ms: outcome.duration_ms,
                result_dir: event_result_dir,
            }),
            Err(error) => emit_event(&app, &DesktopEvent::Error {
                session_id: Some(event_session_id),
                message: format!("{error:#}"),
            }),
        }
    });

    Ok(SessionStartResponse {
        session_id,
        result_dir,
        objective_file,
    })
}
```

This v0 uses `TuiEvent` because it is already exposed by the current CLI/session path. The stronger v1 should expose `RuntimeEvent` directly from the runtime fanout. `RuntimeEventFanout` already supports named subscribers, bounded queues, renderer/proof/benchmark subscribers, and backpressure events.

```ts
import { create } from "zustand";
import type { DesktopEvent } from "./events";

export interface WorkspaceState {
  id: string;
  path: string;
  name: string;
  trusted: boolean;
}

export interface SessionState {
  id: string;
  workspacePath: string;
  resultDir?: string;
  running: boolean;
  events: DesktopEvent[];
  totalBilledTokens: number;
  currentStep?: number;
  lastStatus?: string;
}

interface AppStore {
  workspaces: WorkspaceState[];
  activeWorkspacePath?: string;
  activeSessionId?: string;
  sessions: Record<string, SessionState>;

  runMode: "plan" | "act";
  permissionMode: "ask" | "full-auto" | "full-permissions";
  sandboxMode: "tmp-copy" | "host";

  setActiveWorkspace(path: string): void;
  setMode(mode: Partial<Pick<AppStore, "runMode" | "permissionMode" | "sandboxMode">>): void;
  createSession(id: string, workspacePath: string, resultDir?: string): void;
  appendEvent(event: DesktopEvent): void;
}

export const useAppStore = create<AppStore>((set) => ({
  workspaces: [],
  sessions: {},
  runMode: "act",
  permissionMode: "ask",
  sandboxMode: "tmp-copy",

  setActiveWorkspace: (path) => set({ activeWorkspacePath: path }),

  setMode: (mode) => set(mode),

  createSession: (id, workspacePath, resultDir) =>
    set((state) => ({
      activeSessionId: id,
      sessions: {
        ...state.sessions,
        [id]: {
          id,
          workspacePath,
          resultDir,
          running: true,
          events: [],
          totalBilledTokens: 0,
        },
      },
    })),

  appendEvent: (event) =>
    set((state) => {
      const sessionId = "session_id" in event ? event.session_id : state.activeSessionId;
      if (!sessionId) return state;

      const existing = state.sessions[sessionId] ?? {
        id: sessionId,
        workspacePath: state.activeWorkspacePath ?? "",
        running: true,
        events: [],
        totalBilledTokens: 0,
      };

      const totalBilledTokens =
        event.type === "model_request_finished"
          ? existing.totalBilledTokens + event.total_billed_tokens
          : event.type === "run_finished"
          ? event.total_billed_tokens
          : existing.totalBilledTokens;

      return {
        sessions: {
          ...state.sessions,
          [sessionId]: {
            ...existing,
            running: event.type === "run_finished" ? false : existing.running,
            events: [...existing.events, event],
            totalBilledTokens,
            currentStep: "step" in event ? event.step : existing.currentStep,
            lastStatus: event.type === "status" ? event.status : existing.lastStatus,
          },
        },
      };
    }),
}));
```

# 9. Tauri permissions and capability file

Tauri’s permission/capability model should be tight. Tauri capabilities restrict which windows/webviews can access which IPC commands and plugin permissions; windows without a matching capability have no IPC access. ([Tauri][3])

`apps/desktop/src-tauri/capabilities/default.json`:

```json
{
  "identifier": "main",
  "description": "Main Quorp desktop window permissions.",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "dialog:default",
    "opener:default",
    "process:default"
  ]
}
```

Then do **all filesystem/project mutation through Quorp’s Rust backend**, not through broad frontend filesystem permissions. The frontend should never receive arbitrary filesystem write permissions. The app should use `dialog` to select a folder, then pass the selected path to Rust, and Rust should enforce Quorp’s own trust, sandbox, permissions, and path guards.

This is important because Quorp already has a serious project-level policy engine. The desktop app should not bypass it with Tauri `fs:*` permissions.

```bash

```

Build a local DMG:

```bash
pnpm --dir apps/desktop desktop:dmg
open apps/desktop/src-tauri/target/release/bundle/dmg/
```

Tauri’s DMG docs show the `tauri build --bundles dmg` flow and describe DMG as the common macOS installer format for outside-the-App-Store distribution. ([Tauri][2])

For local unsigned/ad-hoc builds distributed through AirDrop/Slack/browser, macOS may quarantine the file. For your own testing machine:

```bash
xattr -dr com.apple.quarantine /path/to/Quorp.app
```

Do not ask early testers to do this long term. Use signed/notarized builds for real distribution.

For auto-updates, add `tauri-plugin-updater` only after the first usable desktop app is stable. Tauri’s updater plugin requires signed update artifacts and creates macOS updater artifacts such as `.app.tar.gz` and `.sig`; it also supports static GitHub release JSON. ([Tauri][5])

# 14. Security model for desktop full-auto

The desktop app must make **safe mode the default**:

```text
Default:
  permission = Ask
  sandbox = Tmp-copy
  network = Off
  MCP = Off
  browser = Off

Full Auto:
  allowed only when sandbox != Host
  network off unless explicitly approved
  show trust state
  show actual sandbox backend: git-worktree or tmp-copy
```

This aligns with existing repo behavior. `quorp_config` defaults sandbox to tmp-copy, permissions to ask, network/MCP/browser/process controls off, and managed policy requires sandbox and network-off for full-auto. 

`quorp_sandbox` already creates isolated sandboxes, prefers git worktree for git repos, falls back to tmp-copy, and has container command planning that can run with `--network=none`, `--cap-drop=ALL`, and `no-new-privileges`. 

The UI should always show:

```text
Current permission mode
Current sandbox mode
Actual sandbox backend
Trust state
Network state
MCP/browser state
Risk tokens for pending command
Changed files
Rollback/checkpoint state
```

Permission prompt card should look like:

```text
Command wants permission

Tool: run_command
Command: cargo add anyhow && curl https://example.com
CWD: /tmp/quorp/.../workspace

Detected risk:
  - Compound command
  - Dependency install
  - Network
  - Filesystem write

Execution surface:
  - Sandbox: git-worktree
  - Network: off

Choices:
  Deny
  Allow once
  Allow for session
  Add project allow rule
```

Do not expose “Full Permissions Host” as a casual pill. It should require:

```text
clean git check
trusted project
explicit typed confirmation
visible warning
audit event
```

# 18. Hard requirements for “fast and memory conscious”

Implement these from day one:

```text
Virtualize timeline.
Cap in-memory output per tool card.
Fetch full logs lazily.
Fetch diffs lazily.
Keep large artifacts on disk.
Do not store raw provider responses in React state.
Use Rust-side range APIs for logs.
Use one main webview.
Use no local web server in production.
Use no WebSockets for local desktop.
```

The existing runtime already writes model turn artifacts and raw provider responses under result directories rather than relying only on UI memory. `RecordingCompletionClient` writes per-turn artifacts under `artifacts/model_turns/request-####.json`. 

The current headless run outputs already include `request.json`, `metadata.json`, `summary.json`, `events.jsonl`, `proof-receipt.json`, and logs, so desktop should treat artifacts as durable state and the UI store as a live view, not the source of truth.

# 20. The key product feel

The desktop app should make autonomous coding feel controlled, not mysterious.

Every run should visibly answer:

```text
What is the agent trying to do?
What files is it looking at?
What commands did it run?
What changed?
Was the change validated?
What failed?
What did it recover from?
What permissions did it request?
What sandbox protected me?
Can I replay/prove/rollback this?
```

That is where Quorp can beat Codex-style desktop UX: not by having a prettier chat box, but by making the runtime’s ledger, proof, repair, sandbox, and task state visible as a coherent mission-control interface.

The existing architecture is unusually close to supporting this. The immediate job is to wrap it in a Tauri desktop product without weakening the CLI, bypassing policy, or turning the UI into a giant unbounded log buffer.
