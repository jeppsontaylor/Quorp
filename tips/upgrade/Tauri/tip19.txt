## Recommendation: build `apps/desktop` as a Tauri 2 desktop shell over Quorp’s existing Rust runtime

The right architecture is **Tauri 2 + React + TypeScript + Vite**, with the agent/runtime/security work remaining in Rust. I would not build this as a separate Electron-style app, and I would not reimplement Quorp’s runtime in the frontend. The desktop app should be a native macOS app whose webview is only the product surface; all trusted operations go through typed Rust commands.

The repo is already positioned for this: `crates/quorp/src/main.rs` is a thin binary wrapper that delegates to `quorp_cli::dispatch()`, the CLI is split into `quorp_cli`, and session/runtime support is split into `quorp_session`. That means the desktop app can reuse the same agent runner, settings, permissions, sandbox, proof, and runtime events instead of forking behavior.   

My strongest stack decision is:

```text
Quorp CLI remains:
  crates/quorp
  crates/quorp_cli

New desktop app:
  apps/desktop
    Tauri 2
    React + TypeScript + Vite
    CSS grid / virtualized timeline / native menus

New reusable Rust bridge crates:
  crates/quorp_desktop
  crates/quorp_desktop_ipc
```

Use **Tauri IPC channels**, not WebSockets, for the local desktop runtime stream. Tauri’s own docs distinguish regular events from channels and say channels are designed for ordered, high-throughput streaming patterns such as download progress, child-process output, and WebSocket messages. That maps directly to Quorp runtime events and tool-output streams. ([Tauri][1])

Tauri is a strong fit because it lets Quorp keep Rust application logic while using a modern frontend framework, and official templates support React/TypeScript-style app construction. ([Tauri][2])

## Why this fits Quorp’s current code

Quorp already has the critical backend pieces:

| Existing Quorp piece                       | Desktop reuse                                                                                          |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------ |
| `quorp_cli` command dispatch               | Keep for terminal CLI; do not entangle desktop UI with clap.                                           |
| `quorp_session`                            | Reuse session/runtime helpers from desktop bridge.                                                     |
| `run_headless_agent_with_progress` path    | Desktop starts runs through a desktop session manager instead of shelling out.                         |
| `RuntimeEvent`                             | Stream rich typed runtime events to the UI.                                                            |
| `RuntimeEventFanout`                       | Good foundation for multiple subscribers: renderer, recorder, memory/rules, desktop channel.           |
| `quorp_config`                             | Desktop settings should load/save the same `~/.quorp/settings.json` and `.quorp/settings.json`.        |
| `PermissionMode`, `SandboxMode`, `RunMode` | Directly drive UI chips and menu state.                                                                |
| `quorp_term::SlashCommand`                 | Use this as the canonical command palette model.                                                       |
| `quorp_render`                             | Reuse visual language, palette concepts, and state models; do not render ANSI in the desktop webview.  |
| `native_backend` command bridge            | Reuse typed actions and path guards; do not expose arbitrary shell execution to frontend JS.           |

The current docs explicitly say release packaging/signing is not represented yet, so the DMG app is a new product surface rather than a refactor of an existing app package.

## Frontend technology stack

Use this stack:

```text
Tauri 2
React
TypeScript
Vite
CSS variables + CSS grid
@tanstack/react-virtual
Zustand or TanStack Store
CodeMirror 6 only where code editing is needed
Optional xterm.js later, not in MVP
```

### Why React/TypeScript

React/TypeScript is the fastest route to a polished IDE-like product surface: virtualized timelines, diff panels, command palette, inspector panes, tabs, settings modals, keyboard shortcuts, and high-density stateful UI. Quorp’s runtime remains Rust. The TypeScript frontend should be treated as a thin presentation layer.

### What not to use initially

Do **not** use WebSockets for local desktop streaming. Use Tauri commands plus `tauri::ipc::Channel`.

Do **not** use a local HTTP server for the MVP. It adds unnecessary attack surface, CORS complexity, port collisions, and lifecycle problems.

Do **not** expose Tauri’s shell plugin to the frontend for agent commands. Tauri’s shell plugin can spawn child processes, but its docs also note that dangerous commands/scopes are blocked by default and must be explicitly enabled. Quorp already has a Rust-side permission/sandbox model, so privileged operations should remain inside Rust commands. ([Tauri][3])

Do **not** use Monaco as the first editor/diff dependency. It is excellent but heavy. Start with a virtualized diff viewer and lazy-load CodeMirror only for file previews or editing. That keeps memory low.

Session manager shape:

```rust
// crates/quorp_desktop/src/session_manager.rs

use std::collections::BTreeMap;
use std::sync::{Arc, Mutex};
use std::sync::atomic::{AtomicBool, Ordering};

use quorp_desktop_ipc::DesktopRunHandle;

#[derive(Debug, Clone)]
pub struct RunningDesktopSession {
    pub handle: DesktopRunHandle,
    pub cancel: Arc<AtomicBool>,
}

#[derive(Default)]
pub struct DesktopSessionManager {
    runs: Mutex<BTreeMap<String, RunningDesktopSession>>,
}

impl DesktopSessionManager {
    pub fn insert(&self, run: RunningDesktopSession) {
        self.runs
            .lock()
            .expect("desktop session manager lock")
            .insert(run.handle.run_id.clone(), run);
    }

    pub fn cancel(&self, run_id: &str) -> bool {
        let Some(run) = self
            .runs
            .lock()
            .expect("desktop session manager lock")
            .get(run_id)
            .cloned()
        else {
            return false;
        };

        run.cancel.store(true, Ordering::Relaxed);
        true
    }

    pub fn remove(&self, run_id: &str) {
        self.runs
            .lock()
            .expect("desktop session manager lock")
            .remove(run_id);
    }
}
```

```rust
// apps/desktop/src-tauri/src/commands.rs

use std::path::PathBuf;
use std::sync::atomic::AtomicBool;
use std::sync::Arc;

use quorp_core::{PermissionMode, RunMode, SandboxMode};
use quorp_desktop::session_manager::RunningDesktopSession;
use quorp_desktop_ipc::{
    DesktopBootState, DesktopEvent, DesktopRunHandle, DesktopRunRequest, WorkspaceSummary,
};
use tauri::ipc::Channel;
use tauri::State;

use crate::state::AppState;

#[tauri::command]
pub async fn boot_state() -> Result<DesktopBootState, String> {
    let user_settings_path = quorp_config::user_settings_path()
        .map_err(|error| format!("failed to resolve user settings path: {error:#}"))?;

    Ok(DesktopBootState {
        app_version: env!("CARGO_PKG_VERSION").to_string(),
        user_settings_path,
        recent_workspaces: Vec::new(),
    })
}

#[tauri::command]
pub async fn add_workspace(path: PathBuf) -> Result<WorkspaceSummary, String> {
    let root = std::fs::canonicalize(&path)
        .map_err(|error| format!("failed to canonicalize workspace {}: {error}", path.display()))?;

    let loaded = quorp_config::load_settings(&root)
        .map_err(|error| format!("failed to load settings: {error:#}"))?;

    Ok(WorkspaceSummary {
        id: loaded.trust.project_id,
        display_name: root
            .file_name()
            .and_then(|name| name.to_str())
            .unwrap_or("workspace")
            .to_string(),
        root,
        trusted: loaded.trust.trusted,
        git_branch: None,
        dirty: false,
    })
}

#[tauri::command]
pub async fn start_agent_run(
    state: State<'_, AppState>,
    req: DesktopRunRequest,
    on_event: Channel<DesktopEvent>,
) -> Result<DesktopRunHandle, String> {
    let run_id = format!(
        "desktop-{}",
        chrono::Utc::now().format("%Y%m%d-%H%M%S%.3f")
    );

    let result_dir = req
        .workspace_root
        .join(".quorp")
        .join("runs")
        .join(&run_id);

    std::fs::create_dir_all(&result_dir)
        .map_err(|error| format!("failed to create run dir {}: {error}", result_dir.display()))?;

    let handle = DesktopRunHandle {
        run_id: run_id.clone(),
        result_dir: result_dir.clone(),
        workspace_root: req.workspace_root.clone(),
    };

    let cancel = Arc::new(AtomicBool::new(false));

    state.sessions.insert(RunningDesktopSession {
        handle: handle.clone(),
        cancel: cancel.clone(),
    });

    on_event
        .send(DesktopEvent::RunRegistered {
            run_id: run_id.clone(),
            result_dir: result_dir.clone(),
        })
        .map_err(|error| format!("failed to send run registration event: {error}"))?;

    let sessions = state.sessions.clone();

    tauri::async_runtime::spawn_blocking(move || {
        let outcome = run_desktop_agent(req, handle.clone(), cancel, on_event.clone());

        if let Err(error) = &outcome {
            let _ = on_event.send(DesktopEvent::Error {
                run_id: Some(handle.run_id.clone()),
                message: format!("{error:#}"),
            });
        }

        sessions.remove(&handle.run_id);
    });

    Ok(handle)
}

#[tauri::command]
pub async fn cancel_run(state: State<'_, AppState>, run_id: String) -> Result<bool, String> {
    Ok(state.sessions.cancel(&run_id))
}

fn run_desktop_agent(
    req: DesktopRunRequest,
    handle: DesktopRunHandle,
    cancel: Arc<AtomicBool>,
    on_event: Channel<DesktopEvent>,
) -> anyhow::Result<()> {
    // This function should call the same underlying path used by
    // quorp_cli::run_inline_task / quorp run, but with a desktop event sink.
    //
    // The first implementation can reuse run_headless_agent_with_progress.
    // The better implementation is a new quorp_session API that accepts a
    // RuntimeEventSink and avoids collapsing rich RuntimeEvent values into
    // CLI-oriented summary strings.

    let _ = (req, handle, cancel, on_event);

    Ok(())
}
```

The comment in `run_desktop_agent` is important. The current CLI path converts runtime events into `TuiEvent` summaries for terminal rendering. That is useful for the terminal, but the desktop app should receive the richer `RuntimeEvent` values directly. Quorp already has serializable `RuntimeEvent` variants and a bounded fanout system, so the clean upgrade is a new session API that accepts a desktop event sink.

## Better runtime API to add for desktop

Right now `run_inline_task` constructs `HeadlessRunOptions`, creates an `mpsc::sync_channel`, spawns a worker, and renders terminal frames. 

For desktop, add a lower-level API in `quorp_session`:

```rust
// crates/quorp_session/src/quorp/desktop_run.rs

use std::path::PathBuf;
use std::sync::Arc;
use std::sync::atomic::AtomicBool;

use quorp_agent_core::{
    AgentRunOutcome, CompletionPolicy, RuntimeEventSink,
};
use quorp_core::{PermissionMode, RunMode, SandboxMode};

pub struct DesktopHeadlessRunOptions {
    pub workspace: PathBuf,
    pub prompt: String,
    pub result_dir: PathBuf,
    pub model_id: String,
    pub base_url_override: Option<String>,
    pub run_mode: RunMode,
    pub permission_mode: PermissionMode,
    pub sandbox: SandboxMode,
    pub max_steps: usize,
    pub max_seconds: Option<u64>,
    pub max_total_tokens: Option<u64>,
    pub completion_policy: CompletionPolicy,
    pub cancellation_flag: Arc<AtomicBool>,
```

Session store:

```ts
// apps/desktop/src/stores/sessionStore.ts

import { create } from "zustand";
import type { DesktopEvent, RunViewModel, TimelineItem } from "../types/quorp";

type SessionState = {
  activeRunId: string | null;
  runs: Record<string, RunViewModel>;
  applyEvent: (event: DesktopEvent) => void;
};

export const useSessionStore = create<SessionState>((set) => ({
  activeRunId: null,
  runs: {},

  applyEvent: (event) =>
    set((state) => {
      switch (event.type) {
        case "run_registered": {
          const runId = event.data.run_id;

          return {
            activeRunId: runId,
            runs: {
              ...state.runs,
              [runId]: {
                id: runId,
                resultDir: event.data.result_dir,
                status: "running",
                timeline: [],
                usage: {
                  totalBilledTokens: 0,
                },
              },
            },
          };
        }

        case "runtime": {
          const runId = event.data.run_id;
          const run = state.runs[runId];
          if (!run) return state;

          const item = runtimeEventToTimelineItem(event.data.event);

          return {
            runs: {
              ...state.runs,
              [runId]: {
                ...run,
                timeline: appendBounded(run.timeline, item, 10_000),
              },
            },
          };
        }

        case "run_finished": {
          const runId = event.data.run_id;
          const run = state.runs[runId];
          if (!run) return state;

          return {
            runs: {
              ...state.runs,
              [runId]: {
                ...run,
                status: event.data.reason === "success" ? "success" : "stopped",
                usage: {
                  totalBilledTokens: event.data.total_billed_tokens,
                },
              },
            },
          };
        }

        default:
          return state;
      }
    }),
}));

function appendBounded<T>(items: T[], item: T, max: number): T[] {
  if (items.length < max) return [...items, item];
  return [...items.slice(items.length - max + 1), item];
}

function runtimeEventToTimelineItem(event: unknown): TimelineItem {
  return {
    id: crypto.randomUUID(),
    kind: "runtime",
    timestamp: Date.now(),
    payload: event,
  };
}
```

For very long sessions, do not keep all full event payloads in React memory. Keep a visible timeline summary in memory and lazy-load raw details from `events.jsonl` or `run-ledger.jsonl` when the user opens the inspector.

```tsx
// apps/desktop/src/components/Timeline.tsx

import { useVirtualizer } from "@tanstack/react-virtual";
import { useMemo, useRef } from "react";
import { useSessionStore } from "../stores/sessionStore";
import { TimelineCard } from "./TimelineCard";

export function Timeline() {
  const parentRef = useRef<HTMLDivElement | null>(null);

  const activeRunId = useSessionStore((s) => s.activeRunId);
  const run = useSessionStore((s) =>
    s.activeRunId ? s.runs[s.activeRunId] : null,
  );

  const items = useMemo(() => run?.timeline ?? [], [run?.timeline]);

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 96,
    overscan: 12,
  });

  return (
    <div ref={parentRef} className="timeline-scroll">
      <div
        className="timeline-virtual-space"
        style={{ height: virtualizer.getTotalSize() }}
      >
        {virtualizer.getVirtualItems().map((row) => {
          const item = items[row.index];

          return (
            <div
              key={item.id}
              className="timeline-row"
              style={{
                transform: `translateY(${row.start}px)`,
              }}
            >
              <TimelineCard item={item} activeRunId={activeRunId} />
            </div>
          );
        })}
      </div>
    </div>
  );
}
```

## Permission and sandbox UI behavior

The desktop app should make trust state impossible to miss.

Use these chips:

```text
Run mode:       Plan | Act
Permissions:   Ask | Full-auto | Full permissions
Sandbox:       tmp-copy | host
Network:       off | ask | on
MCP:           off | ask | on
Browser:       off | ask | on
```

Rules:

1. **Default desktop launch:** `PermissionMode::Ask`, `SandboxMode::TmpCopy`.
2. **Full-auto:** allowed only when sandbox is `tmp-copy` unless an explicit trusted-project policy says otherwise.
3. **Full permissions / host:** show a blocking modal requiring:

   * trusted project,
   * clean git tree,
   * clear explanation of host access,
   * explicit confirmation.
4. **Untrusted project settings may not elevate privileges.** Quorp’s existing settings merge already prevents untrusted project config from escalating permission mode, network, MCP, browser, process control, or sandbox. The desktop UI should display those downgrades instead of silently applying project settings. 
5. **Frontend cannot run shell commands.** Every mutation goes through Rust and Quorp’s permission/sandbox path.

## Memory and speed rules

To keep the app fast and memory-conscious:

1. **Virtualize timeline and diff rows.**
2. **Keep raw logs on disk.**
3. **Send summarized events to React; lazy-load full artifacts.**
4. **Avoid Monaco initially.**
5. **Avoid WebSockets/local server.**
6. **Do all expensive diffing/indexing in Rust.**
7. **Debounce file tree refreshes.**
8. **Limit command output chunks in UI memory.**
9. **Use streaming channels, not polling.**
10. **Keep frontend state normalized by `run_id` and `event_id`.**

A good event retention policy:

```text
React memory:
  last 10,000 timeline summaries per run
  selected event full payload
  active command output tail

Disk:
  full events.jsonl
  full run-ledger.jsonl
  full raw command logs
  full model turn artifacts
  proof bundles
```
