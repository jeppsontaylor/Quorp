## Recommendation

Build the desktop app as a **Tauri 2 app at `/apps/desktop`**, backed by new Rust library seams, not by shelling out to the existing CLI.

The best first production path is:

```text
Tauri 2
React + TypeScript + Vite
Rust app core reused from Quorp crates
Tauri command IPC for request/response
Tauri Channel IPC for high-volume agent streaming
CodeMirror 6 for code/diff surfaces
TanStack Virtual for transcript/diff virtualization
Zustand or Jotai for small, event-driven UI state
Native Tauri menus
Tauri bundler for .app + .dmg
Signing/notarization added later
```

The Rust backend/core remains the authority. The UI should be web technology because it is the fastest path to a polished Codex-like desktop app, high-quality diff rendering, command palettes, tabs, settings panels, and fast iteration. Tauri officially supports arbitrary frontend stacks/templates, including React, Svelte, Vue, Solid, Yew, Leptos, and others; it also gives you native command IPC, streaming channels, app capabilities, native menus, and DMG output. ([Tauri][1])

The important architectural decision: **do not put the desktop app inside `crates/quorp_cli` or `crates/quorp`**. Those are terminal/CLI surfaces. Put the app in `/apps/desktop`, then add small reusable crates that expose desktop-safe core APIs.

The repo guidance also explicitly says release packaging/signing is not yet represented, which makes `/apps/desktop` the clean new surface rather than a mutation of the terminal product. # 2. Important repo-specific architecture call

The desktop app should **not** call `quorp_cli::dispatch()`.

The current CLI crate is essentially a wrapper around `crates/quorp_cli/src/quorp/cli.rs`, and that CLI file is already large and command-dispatch heavy. It parses CLI arguments, handles `run`, `exec`, benchmark commands, session launch config, sandbox flags, result directories, and then delegates into session/agent runner paths. That is good for a terminal binary, but it is not the right API for a desktop app. ([GitHub][4])

Instead, extract a desktop-safe facade:

```text
desktop UI
  ↓
Tauri commands/channels
  ↓
apps/desktop/src-tauri bridge
  ↓
crates/quorp_app_core
  ↓
quorp_session / quorp_agent_core / quorp_context / quorp_permissions / quorp_sandbox
```

The existing runtime has the right foundations: typed run requests, compaction policy, checkpoints, stop reasons, runtime event sinks, and event fanout subscribers. That makes it well-suited for a streaming desktop UI once you expose those events through a stable desktop protocol. ([GitHub][5])

# 3. Frontend stack

## Recommended stack

```text
React + TypeScript + Vite + SWC
Tauri 2
CodeMirror 6
TanStack Virtual
Zustand
CSS variables + CSS Modules
Optional Radix/Ariakit for accessible popovers/dialogs
ts-rs or specta for Rust → TypeScript type generation
```

## Why React, not pure Rust UI

For a Codex-like app with a left workspace panel, bottom composer, streaming event feed, diff hunk rendering, settings, permissions, command palette, replay viewer, context panels, and model/provider controls, React/TypeScript will get you to “usable and polished” much faster than a pure Rust GUI.

The product can still be Rust-first:

```text
Rust owns:
  agent runtime
  permissions
  sandboxing
  file writes
  command execution
  memory/rules
  verification
  workspace registry
  logs
  secrets boundary
  indexing
  diff/proof generation

TypeScript owns:
  layout
  interaction
  visual state
  virtualized rendering
  keyboard shortcuts
  settings forms
```

If “100% Rust source” becomes an absolute requirement, the plausible alternatives are:

```text
Tauri + Leptos/Yew frontend
Slint desktop app
egui/eframe internal-tool style app
SwiftUI shell + Rust core
```

But for the first excellent DMG: **Tauri + React + TypeScript**.

## WebSockets?

Do **not** use WebSockets for the local desktop app.

Use:

```text
Tauri invoke commands  → request/response operations
Tauri Channels         → streaming agent events, tool events, diffs, token chunks
Tauri events           → low-volume menu/window notifications
```

Tauri’s docs distinguish ordinary commands from channels; channels are specifically intended for streamed, ordered data and high-throughput cases, while events are simpler and not the best fit for high-volume streaming. ([Tauri][6])

Use WebSockets only later if Quorp grows a separate remote/browser UI, collaborative mode, or headless daemon. For local desktop, WebSockets add avoidable complexity and a larger attack surface.

# 6. IPC and wiring design

## Rust app state

`apps/desktop/src-tauri` should be thin.

```rust
// apps/desktop/src-tauri/src/state.rs

use std::sync::Arc;
use quorp_app_core::DesktopCore;

#[derive(Clone)]
pub struct DesktopState {
    pub core: Arc<DesktopCore>,
}
```

`DesktopCore` should live in `crates/quorp_app_core`, not in Tauri code:

```rust
// crates/quorp_app_core/src/lib.rs

pub mod app_state;
pub mod workspace_registry;
pub mod run_manager;
pub mod event_bridge;
pub mod permission_broker;
pub mod diff_service;
pub mod status_service;

pub use app_state::DesktopCore;
```

Suggested shape:

```rust
// crates/quorp_app_core/src/app_state.rs

use std::sync::Arc;

pub struct DesktopCore {
    pub workspaces: Arc<WorkspaceRegistry>,
    pub runs: Arc<RunManager>,
    pub permissions: Arc<PermissionBroker>,
    pub diffs: Arc<DiffService>,
    pub status: Arc<StatusService>,
}

impl DesktopCore {
    pub async fn new(app_data_dir: std::path::PathBuf) -> anyhow::Result<Self> {
        let workspaces = Arc::new(WorkspaceRegistry::open(app_data_dir.join("workspaces.json")).await?);
        let permissions = Arc::new(PermissionBroker::new(app_data_dir.join("permissions.json")).await?);
        let diffs = Arc::new(DiffService::new());
        let status = Arc::new(StatusService::new());
        let runs = Arc::new(RunManager::new(
            workspaces.clone(),
            permissions.clone(),
            diffs.clone(),
            status.clone(),
            app_data_dir.join("runs"),
        ).await?);

        Ok(Self {
            workspaces,
            runs,
            permissions,
            diffs,
            status,
        })
    }
}
```

## Tauri command bridge

```rust
// apps/desktop/src-tauri/src/commands.rs

use std::sync::Arc;

use quorp_desktop_protocol::{
    DesktopEvent,
    RunHandleDto,
    StartRunRequest,
    WorkspaceDto,
};
use tauri::{ipc::Channel, State};

use crate::state::DesktopState;

fn err(e: impl std::fmt::Display) -> String {
    e.to_string()
}

#[tauri::command]
pub async fn list_workspaces(
    state: State<'_, Arc<DesktopState>>,
) -> Result<Vec<WorkspaceDto>, String> {
    state.core.workspaces.list().await.map_err(err)
}

#[tauri::command]
pub async fn add_workspace(
    state: State<'_, Arc<DesktopState>>,
    path: String,
) -> Result<WorkspaceDto, String> {
    state.core.workspaces.add(path.into()).await.map_err(err)
}

#[tauri::command]
pub async fn remove_workspace(
    state: State<'_, Arc<DesktopState>>,
    workspace_id: String,
) -> Result<(), String> {
    state.core.workspaces.remove(&workspace_id).await.map_err(err)
}

#[tauri::command]
pub async fn start_agent_run(
    state: State<'_, Arc<DesktopState>>,
    req: StartRunRequest,
    on_event: Channel<DesktopEvent>,
) -> Result<RunHandleDto, String> {
    state
        .core
        .runs
        .start(req, move |event| {
            // Channel close means frontend window disappeared or run view was detached.
            // The backend run may continue or be cancelled depending on the run policy.
            let _ = on_event.send(event);
        })
        .await
        .map_err(err)
}

#[tauri::command]
pub async fn stop_run(
    state: State<'_, Arc<DesktopState>>,
    run_id: String,
) -> Result<(), String> {
    state.core.runs.stop(&run_id).await.map_err(err)
}

#[tauri::command]
pub async fn get_run_diff(
    state: State<'_, Arc<DesktopState>>,
    run_id: String,
) -> Result<quorp_desktop_protocol::DiffDto, String> {
    state.core.diffs.for_run(&run_id).await.map_err(err)
}

#[tauri::command]
pub async fn respond_permission_prompt(
    state: State<'_, Arc<DesktopState>>,
    prompt_id: String,
    approved: bool,
) -> Result<(), String> {
    state
        .core
        .permissions
        .respond(prompt_id, approved)
        .await
        .map_err(err)
}
```

Register commands:

```rust
// apps/desktop/src-tauri/src/lib.rs

mod bridge;
mod commands;
mod menus;
mod path_env;
mod state;

use std::sync::Arc;
use tauri::Manager;

use crate::state::DesktopState;

#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::Builder::default()
        .setup(|app| {
            path_env::repair_gui_path();

            let app_data_dir = app
                .path()
                .app_data_dir()
                .map_err(|e| format!("failed to resolve app data dir: {e}"))?;

            std::fs::create_dir_all(&app_data_dir)
                .map_err(|e| format!("failed to create app data dir: {e}"))?;

            let core = tauri::async_runtime::block_on(
                quorp_app_core::DesktopCore::new(app_data_dir),
            )
            .map_err(|e| format!("failed to initialize Quorp desktop core: {e}"))?;

            app.manage(Arc::new(DesktopState {
                core: Arc::new(core),
            }));

            menus::install(app)?;

            Ok(())
        })
        .invoke_handler(tauri::generate_handler![
            commands::list_workspaces,
            commands::add_workspace,
            commands::remove_workspace,
            commands::start_agent_run,
            commands::stop_run,
            commands::get_run_diff,
            commands::respond_permission_prompt,
        ])
        .run(tauri::generate_context!())
        .expect("error while running Quorp desktop");
}
```

Frontend call:

```ts
// apps/desktop/src/ipc/client.ts

import { invoke, Channel } from "@tauri-apps/api/core";
import type {
  DesktopEvent,
  RunHandleDto,
  StartRunRequest,
  WorkspaceDto,
} from "./generated";
import { useRunStore } from "../state/runStore";

export async function listWorkspaces(): Promise<WorkspaceDto[]> {
  return invoke("list_workspaces");
}

export async function addWorkspace(path: string): Promise<WorkspaceDto> {
  return invoke("add_workspace", { path });
}

export async function startAgentRun(req: StartRunRequest): Promise<RunHandleDto> {
  const onEvent = new Channel<DesktopEvent>();

  onEvent.onmessage = (event) => {
    useRunStore.getState().ingest(event);
  };

  return invoke("start_agent_run", {
    req,
    onEvent,
  });
}
```

## Event ingestion should be batched

Agent streams can produce many events. Do not set React state for every token synchronously.

```ts
// apps/desktop/src/state/runStore.ts

import { create } from "zustand";
import type { DesktopEvent } from "../ipc/generated";

const MAX_VISIBLE_EVENTS = 2_000;

type RunState = {
  events: DesktopEvent[];
  pending: DesktopEvent[];
  flushScheduled: boolean;
  ingest: (event: DesktopEvent) => void;
  flush: () => void;
};

export const useRunStore = create<RunState>((set, get) => ({
  events: [],
  pending: [],
  flushScheduled: false,

  ingest(event) {
    const state = get();
    state.pending.push(event);

    if (!state.flushScheduled) {
      set({ flushScheduled: true });
      requestAnimationFrame(() => get().flush());
    }
  },

  flush() {
    const state = get();
    const next = state.events.concat(state.pending).slice(-MAX_VISIBLE_EVENTS);

    set({
      events: next,
      pending: [],
      flushScheduled: false,
    });
  },
}));
```

The raw event log should always be persisted on disk by Rust. The frontend should only retain what it needs for current rendering.

```rust
    pub async fn start<F>(
        &self,
        req: StartRunRequest,
        event_sink: F,

        F: Fn(DesktopEvent) + Send + Sync + 'static,

            max_total_tokens: req.max_total_tokens,

        Ok(RunHandleDto {
            run_id,
            workspace_id: req.workspace_id,
            result_dir: result_dir.display().to_string(),

    }
}
```

# 8. Security and permissions model

The frontend must never get generic filesystem or shell power.

Use Tauri capabilities so the main window only has the permissions it needs. Tauri’s capability model constrains which frontend windows/webviews can access which commands and plugin permissions; permissions can allow or deny command access and scopes. ([Tauri][10])

## Desktop security rules

1. The frontend can request actions.
2. Rust validates all paths.
3. Rust enforces trust state.
4. Rust enforces sandbox.
5. Rust enforces Quorp permissions.
6. Rust executes tools.
7. Rust writes audit logs.
8. Rust sends summarized results back to UI.
9. UI never receives secrets unless explicitly needed.
10. UI cannot run arbitrary shell commands directly.

## Capability shape

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "main",
  "description": "Main Quorp desktop window",
  "windows": ["main"],
  "permissions": [
    "core:window:allow-set-title",
    "dialog:allow-open",
    "opener:default"
  ]
}
```

Add custom Quorp commands through Tauri’s command allowlist/capability system, not through broad shell/filesystem plugins.

## Permission profiles

```text
Ask
  Read allowed.
  Writes require prompt.
  Shell requires prompt.
  Network requires prompt.
  Dependency install requires prompt.

Accept Edits
  Patch VM writes allowed after diff preview.
  Shell/network still ask.

Auto Safe
  Allowed only inside tmp-copy/git-worktree sandbox.
  Safe commands allowed.
  Dangerous commands ask/block.
  Network default ask/off.

Full Permission Host
  Host workspace allowed.
  Mutating shell/write actions require clear visual state.
  Never default.
  Always auditable.

YOLO Sandbox
  Allowed only in tmp-copy/git-worktree.
  Never direct host.
  Red visible badge.
```

The desktop must preserve the existing rule that YOLO is rejected with direct host sandbox. ([GitHub][7])

## Permission prompt UI

Prompt card should include:

```text
Action: run shell command
Command: cargo test -p quorp_agent_core
Working directory: /path/to/workspace
Sandbox: git-worktree
Network: no
Writes: target/, Cargo.lock
Reason: validate patch
Risk: medium
Buttons: Allow once | Always allow in this workspace | Deny
```

No “always allow” for:

```text
curl | sh
sudo
rm -rf
chmod +x generated file then execute
network exfiltration
secret-looking env/file access
unknown MCP server
dependency install in host mode
```

## Prompt-injection protections

Workspace files can tell the agent to do unsafe things. The app should mark repo-authored config/rules as untrusted until reviewed.

In the UI:

```text
Rules:
  User rules: trusted
  Project rules: untrusted until reviewed
  Agent-created draft rules: inactive until promoted
```

## API keys and secrets

Immediate MVP:

```text
Read API keys from env/config.
Redact known secret patterns in logs.
Do not render secrets in frontend.
```

Production:

```text
macOS Keychain storage.
Settings UI only displays “configured / missing”.
Per-provider key validation.
```

## macOS GUI PATH

macOS GUI apps do not necessarily inherit the shell PATH, and Tauri’s DMG guide explicitly calls this out for apps distributed as GUI bundles. Quorp will need `git`, `cargo`, `rustc`, `node`, and other developer tools, so initialize PATH explicitly at app startup. ([Tauri][11])

Suggested implementation:

```rust
// apps/desktop/src-tauri/src/path_env.rs

pub fn repair_gui_path() {
    #[cfg(target_os = "macos")]
    {
        let current = std::env::var_os("PATH")
            .map(|s| s.to_string_lossy().to_string())
            .unwrap_or_default();

        let candidates = [
            "/opt/homebrew/bin",
            "/usr/local/bin",
            "/usr/bin",
            "/bin",
            "/usr/sbin",
            "/sbin",
            "$HOME/.cargo/bin",
        ];

        let mut parts: Vec<String> = current
            .split(':')
            .filter(|s| !s.is_empty())
            .map(ToOwned::to_owned)
            .collect();

        for candidate in candidates {
            let expanded = if let Some(home) = std::env::var_os("HOME") {
                candidate.replace("$HOME", &home.to_string_lossy())
            } else {
                candidate.to_string()
            };

            if !parts.iter().any(|p| p == &expanded) {
                parts.push(expanded);
            }
        }

        std::env::set_var("PATH", parts.join(":"));
    }
}
```

This is good enough for early testing. Later, add a `doctor` panel that shows detected paths:

```text
git: /opt/homebrew/bin/git
cargo: /Users/me/.cargo/bin/cargo
node: /opt/homebrew/bin/node
python3: /opt/homebrew/bin/python3
```

For quick testing on your Mac:


To build an unsigned local app/DMG:


```bash

```

Tauri’s macOS docs show `cargo tauri build --bundles dmg` and equivalent package-manager commands for creating a DMG bundle. ([Tauri][11])

The artifact will normally land under the Cargo target bundle directory, often one of:

```text

target/release/bundle/dmg/Quorp_0.1.0_aarch64.dmg
apps/desktop/src-tauri/target/release/bundle/dmg/...
```

The exact location depends on whether Cargo is using the workspace root target directory or the `src-tauri` target directory.

## Local unsigned behavior

For your own Mac:

```text
pnpm tauri dev       fastest iteration
pnpm desktop:dmg     local installable artifact
open Quorp.app       direct app launch
```

Unsigned apps are fine for internal testing. For sharing broadly, macOS Gatekeeper will be much happier with Developer ID signing and notarization. Apple’s Developer ID distribution path uses signing and notarization so Gatekeeper can validate software distributed outside the Mac App Store. ([Apple Developer][13])

```text

3. Bundle .app
4. Sign .app with Developer ID Application certificate
5. Create .dmg
6. Sign .dmg
7. Submit to Apple notarization
8. Staple notarization ticket

```

Tauri also provides a GitHub Action that can build native bundles across platforms and publish artifacts/releases, which can become the CI release path after local DMG testing is stable. ([GitHub][14])

# 14. Folder picker flow

Use Tauri dialog plugin only for choosing a path. Rust still validates and registers it.

```ts
// apps/desktop/src/ipc/workspaces.ts

import { open } from "@tauri-apps/plugin-dialog";
import { addWorkspace } from "./client";

export async function openAddFolderDialog() {
  const selected = await open({
    directory: true,
    multiple: false,
    title: "Add folder to Quorp",
  });

  if (typeof selected !== "string") return;

  await addWorkspace(selected);
}
```

Rust validation:

```rust
pub async fn add(&self, path: PathBuf) -> anyhow::Result<WorkspaceDto> {
    let canonical = tokio::fs::canonicalize(&path).await?;

    anyhow::ensure!(canonical.is_dir(), "workspace path is not a directory");

    let id = workspace_id_for_path(&canonical);

    let workspace = WorkspaceDto {
        id,
        path: canonical.display().to_string(),
        display_name: canonical
            .file_name()
            .and_then(|s| s.to_str())
            .unwrap_or("Workspace")
            .to_string(),
        trust: WorkspaceTrust::Untrusted,
        sandbox: SandboxMode::GitWorktree,
        index_status: IndexStatus::Pending,
        dirty_files: 0,
    };

    self.persist_add(workspace.clone()).await?;
    Ok(workspace)
}
```

# 16. Keeping the app fast and memory-conscious

Non-negotiables:

1. **Virtualize transcripts.**
2. **Virtualize diffs.**
3. **Batch token/event updates with `requestAnimationFrame`.**
4. **Persist raw events in Rust, not React memory.**
5. **Keep only current run summary + visible window in the frontend.**
6. **Use CodeMirror 6, not Monaco, for MVP.**
7. **Avoid rendering raw JSON unless expanded.**
8. **Do not keep full file contents in global state.**
9. **Stream diffs as structured hunks, not giant strings.**
10. **Use Rust for indexing, diff generation, proof parsing, and log reading.**

Memory policy:

```text
Frontend:
  active visible events: 2,000 max
  token deltas: coalesced into message blocks
  diff lines: virtualized
  file content: loaded on demand
  settings/workspace list: small persistent state

Rust:
  full event log on disk
  run ledger on disk
  receipts/proof artifacts on disk
  workspace index/cache on disk
  active run state in memory
```

```text
auto-updater
notarization

remote browser mode
```

# 20. Final product bar

A Quorp desktop app will feel better than a simple wrapped CLI when it has these visible affordances:

```text
Left panel:
  folders, sessions, tasks, trust, index state

Center:
  agent stream, tool calls, recovery, validation, bottom composer

Right:
  diff/proof/context/tasks/rules/permissions

Top status:
  model, mode, sandbox, permission, context pressure, tokens, verifier, git state

Menus:
  Add Folder, New Session, Run Full Auto, Compact, Verify, Rollback, Replay

Security:
  no direct frontend shell/filesystem power
  all writes through Rust permissions/sandbox/Patch VM
  visible host/full-auto warnings
  audit log for every mutating action

Build:
  local unsigned DMG immediately
  signed/notarized DMG later
```

The key is to treat the desktop app as **mission control for the existing Rust agent runtime**, not as a separate product and not as a shell wrapper. The app should make Quorp’s strongest internal systems visible: runtime events, checkpoints, patch receipts, verification, sandbox state, permission decisions, context pressure, and recovery behavior.
