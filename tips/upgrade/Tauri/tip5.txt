I could not clone the repo inside this environment because GitHub DNS resolution failed, so I inspected the public GitHub/raw files through the browser. The recommendation below is grounded in the current visible repo shape: Quorp is already a Rust terminal-first agent runtime with extracted `quorp_cli` and `quorp_session`, typed runtime actions, Patch VM receipts, verification proof packets, event logs, and benchmark gates. The support docs also explicitly say packaging/signing is not represented yet, so this desktop app should be added as a new first-class app surface, not squeezed into the existing CLI crates. ([GitHub][1])

# Recommendation: build `apps/desktop` with Tauri 2, React, TypeScript, and a Rust bridge crate

Use **Tauri 2 + React + TypeScript + Vite** for the DMG app. Keep Quorp’s agent core, session runtime, permissions, patching, sandboxing, verification, and proof machinery in Rust. Do **not** use WebSockets for the local app MVP. Tauri already gives you frontend-to-Rust commands and Rust-to-frontend events/channels, which are a better fit for an in-process desktop app than running a local server. Tauri supports any frontend framework while keeping backend logic in Rust, its command system supports async Rust calls from the frontend, and Rust can call the frontend through events/channels. ([Tauri][2])

The only reason to avoid React would be a hard requirement that the GUI itself be Rust-only. I would **not** make that trade for v1. Slint/egui can work, but the Codex-style product you described needs a polished multi-pane layout, virtualized transcripts, diff review, command palette, settings panes, and accessible controls. React/TypeScript gets that usable quickly while Rust remains the actual engine.

The desktop app should compete with the current Codex and Claude desktop layouts: project/workspace sidebar, parallel sessions/threads, review/diff pane, integrated controls, and agent mission control. OpenAI describes Codex app as a desktop command center with project sidebar, active thread, review pane, worktree support, automations, and Git functionality; Anthropic describes Claude Code desktop as a graphical interface for multiple sessions, sidebar management, integrated terminal/editor, visual diff review, live preview, PR monitoring, and scheduled tasks. Quorp should build directly toward that product shape. ([OpenAI Developers][3])

`apps/desktop/src-tauri/src/commands.rs`:

```rust
use std::sync::atomic::Ordering;
use std::sync::{Arc, mpsc};
use std::thread;

use tauri::{AppHandle, Emitter, State};

use quorp_desktop::{
    DesktopEvent, StartSessionRequest, StartSessionResponse,
};

use crate::app_state::{AgentSessionHandle, DesktopState};

#[tauri::command]
pub async fn start_session(
    app: AppHandle,
    state: State<'_, DesktopState>,
    req: StartSessionRequest,
) -> Result<StartSessionResponse, String> {
    let workspace = {
        let workspaces = state.workspaces.lock().map_err(lock_error)?;
        workspaces
            .get(&req.workspace_id)
            .cloned()
            .ok_or_else(|| format!("unknown workspace {}", req.workspace_id))?
    };

    let session_id = uuid::Uuid::new_v4().to_string();
    let run_dir = quorp_desktop::default_desktop_run_dir(&workspace.path, &session_id)
        .map_err(to_string_error)?;

    std::fs::create_dir_all(&run_dir).map_err(to_string_error)?;

    let prompt = req.prompt.clone().unwrap_or_default();
    let objective_file = run_dir.join("objective.md");
    std::fs::write(&objective_file, &prompt).map_err(to_string_error)?;

    let cancel = Arc::new(std::sync::atomic::AtomicBool::new(false));
    let cancel_for_worker = Arc::clone(&cancel);
    let cancel_for_forwarder = Arc::clone(&cancel);

    let (event_tx, event_rx) = mpsc::sync_channel(512);

    let session_id_for_events = session_id.clone();
    let app_for_events = app.clone();

    thread::spawn(move || {
        while let Ok(event) = event_rx.recv() {
            if cancel_for_forwarder.load(Ordering::Relaxed) {
                break;
            }

            if let Some(desktop_event) =
                quorp_desktop::map_tui_event_to_desktop_event(&session_id_for_events, event)
            {
                let _ = app_for_events.emit("quorp://event", desktop_event);
            }
        }
    });

    let options = quorp_session::quorp::agent_runner::HeadlessRunOptions {
        workspace: workspace.path.clone(),
        objective_file,
        model_id: req.model_id.unwrap_or_else(|| quorp_core::DEFAULT_NVIDIA_MODEL.to_string()),
        base_url_override: req.base_url,
        max_steps: 64,
        max_seconds: Some(3600),
        max_total_tokens: None,
        result_dir: run_dir.clone(),
        autonomy_profile: quorp_desktop::map_autonomy_profile(req.run_mode, req.sandbox_mode),
        completion_policy: quorp_agent_core::CompletionPolicy::default(),
        objective_metadata: serde_json::json!({
            "origin": "desktop",
            "workspace_id": req.workspace_id,
            "permission_mode": req.permission_mode,
            "sandbox_mode": req.sandbox_mode,
            "verify_mode": req.verify_mode,
        }),
        seed_context: Vec::new(),
    };

    let session_id_for_worker = session_id.clone();
    let app_for_worker = app.clone();

    let worker = thread::spawn(move || {
        if cancel_for_worker.load(Ordering::Relaxed) {
            anyhow::bail!("session cancelled before start");
        }

        let outcome = quorp_session::quorp::agent_runner::run_headless_agent_with_progress(
            options,
            Some(event_tx),
        )?;

        let _ = app_for_worker.emit(
            "quorp://event",
            DesktopEvent::SessionFinished {
                session_id: session_id_for_worker,
                stop_reason: format!("{:?}", outcome.stop_reason),
                billed_tokens: outcome.total_billed_tokens,
                duration_ms: outcome.duration_ms,
            },
        );

        Ok(outcome)
    });

    {
        let mut sessions = state.sessions.lock().map_err(lock_error)?;
        sessions.insert(session_id.clone(), AgentSessionHandle {
            cancel,
            worker: Some(worker),
        });
    }

    app.emit(
        "quorp://event",
        DesktopEvent::SessionStarted {
            session_id: session_id.clone(),
            workspace_id: workspace.id,
            run_dir: run_dir.clone(),
        },
    ).map_err(to_string_error)?;

    Ok(StartSessionResponse { session_id, run_dir })
}

#[tauri::command]
pub async fn cancel_session(
    state: State<'_, DesktopState>,
    session_id: String,
) -> Result<(), String> {
    let sessions = state.sessions.lock().map_err(lock_error)?;
    let handle = sessions
        .get(&session_id)
        .ok_or_else(|| format!("unknown session {session_id}"))?;

    handle.cancel();
    Ok(())
}

fn lock_error<T>(_: std::sync::PoisonError<T>) -> String {
    "desktop state lock poisoned".to_string()
}

fn to_string_error(error: impl std::fmt::Display) -> String {
    error.to_string()
}
```

This sample is intentionally close to the current CLI path, but I would add a runtime-level cancellation token soon. Ignoring events after cancellation is not enough; the agent should stop before the next model call/tool call.

```ts
  return listen<unknown>("quorp://event", (event) => {
```

```rust
use serde::{Deserialize, Serialize};
use std::path::PathBuf;

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct DiffSummary {
    pub session_id: String,
    pub files: Vec<DiffFile>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct DiffFile {
    pub path: PathBuf,
    pub status: DiffFileStatus,
    pub additions: usize,
    pub deletions: usize,
    pub hunks: Vec<DiffHunk>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "kebab-case")]
pub enum DiffFileStatus {
    Added,
    Modified,
    Deleted,
    Renamed,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct DiffHunk {
    pub old_start: usize,
    pub old_lines: usize,
    pub new_start: usize,
    pub new_lines: usize,
    pub lines: Vec<DiffLine>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct DiffLine {
    pub kind: DiffLineKind,
    pub old_line: Option<usize>,
    pub new_line: Option<usize>,
    pub text: String,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "kebab-case")]
pub enum DiffLineKind {
    Context,
    Add,
    Delete,
}
```

Implementation path:

* Initial MVP: compute diff with `git diff --no-ext-diff --unified=3`.
* Better v2: use a Rust diff parser crate and attach provenance from Patch VM receipts.
* Best v3: file-level semantic diff with AST/symbol anchors where `quorp_rust_intel` can help.

# 9. Permissions UX

Quorp already has stronger permission primitives than a typical toy app. `quorp_core` exposes `RunMode`, `PermissionMode`, and `SandboxMode`, while `quorp_permissions` has finer modes including read-only, ask, accept-edits, auto-safe, yolo-sandbox, plus command risk tokens for shell metacharacters, compound commands, filesystem writes, network, dependency installs, Docker, git remote mutation, find-delete, find-exec, secrets reads, generated executables, MCP, and browser. ([GitHub][9])

Desktop permission modes should be shown as:

```text
Read-only
  Agent can inspect files and run safe read tools only.

Ask
  Prompt before writes and commands.

Accept edits
  Allow file edits, ask before shell/network/MCP/browser.

Auto-safe
  Allow low-risk writes and safe commands, prompt for risky tokens.

Full permission in sandbox
  Allow everything only inside tmp-copy/git-worktree sandbox.

Host full permission
  Hidden behind advanced settings + typed confirmation + audit warning.
```

The composer dropdown should not simply say “Full Access.” It should say:

```text
Full permission · sandboxed
```

or:

```text
Danger: host full permission
```

For untrusted repos, default should be:

```text
Sandbox: tmp-copy
Permissions: ask
Network: off
MCP: off
Dependency install: prompt
Generated executables: prompt
Secrets reads: deny
```

Do **not** grant broad Tauri filesystem or shell plugin permissions. Use dedicated Rust commands, because those can call the Quorp permission engine before touching the host.

Permission modal layout:

```text
Permission request

Tool
  run_command

Command
  cargo test && curl https://example.com

Risk
  compound-command
  network
  shell-meta

Workspace
  ~/src/project

Surface
  tmp-copy sandbox

Buttons
  Deny
  Allow once
  Always allow this command in this session
  Always allow for this project...
```

“Always allow for project” should write a scoped Quorp config/rule file only after user approval.

# 14. Native menu implementation sample

```rust
use tauri::{
    AppHandle, Emitter, Manager,
    menu::{Menu, MenuItem, PredefinedMenuItem, Submenu},
};

pub fn install_menu(app: &AppHandle) -> tauri::Result<()> {
    let add_folder = MenuItem::with_id(
        app,
        "workspace.add-folder",
        "Add Folder…",
        true,
        Some("CmdOrCtrl+O"),
    )?;

    let new_session = MenuItem::with_id(
        app,
        "session.new",
        "New Agent Session",
        true,
        Some("CmdOrCtrl+N"),
    )?;

    let settings = MenuItem::with_id(
        app,
        "app.settings",
        "Settings…",
        true,
        Some("CmdOrCtrl+,"),
    )?;

    let command_palette = MenuItem::with_id(
        app,
        "view.command-palette",
        "Command Palette…",
        true,
        Some("CmdOrCtrl+K"),
    )?;

    let run_verify = MenuItem::with_id(
        app,
        "tools.verify",
        "Run Verify",
        true,
        Some("Shift+CmdOrCtrl+V"),
    )?;

    let file = Submenu::with_items(
        app,
        "File",
        true,
        &[&add_folder, &new_session, &PredefinedMenuItem::separator(app)?],
    )?;

    let view = Submenu::with_items(
        app,
        "View",
        true,
        &[&command_palette],
    )?;

    let tools = Submenu::with_items(
        app,
        "Tools",
        true,
        &[&run_verify],
    )?;

    let app_menu = Submenu::with_items(
        app,
        "Quorp",
        true,
        &[&settings, &PredefinedMenuItem::separator(app)?, &PredefinedMenuItem::quit(app, None)?],
    )?;

    let menu = Menu::with_items(app, &[&app_menu, &file, &view, &tools])?;
    app.set_menu(menu)?;

    let app_for_handler = app.clone();
    app.on_menu_event(move |_app, event| {
        let id = event.id().as_ref();

        let frontend_event = match id {
            "workspace.add-folder" => Some("menu:workspace-add-folder"),
            "session.new" => Some("menu:session-new"),
            "app.settings" => Some("menu:settings"),
            "view.command-palette" => Some("menu:command-palette"),
            "tools.verify" => Some("menu:run-verify"),
            _ => None,
        };

        if let Some(name) = frontend_event {
            let _ = app_for_handler.emit(name, serde_json::json!({}));
        }
    });

    Ok(())
}
```

# 17. What I would not do

I would not:

* Replace the CLI. Keep `quorp` terminal-first.
* Run the desktop app by shelling out to `quorp exec` as the main architecture.
* Give the frontend broad filesystem/shell permissions.
* Use WebSockets for local runtime streaming.
* Start with Monaco unless you need full editor features immediately.
* Build a pure Rust GUI for v1 unless the team is willing to sacrifice polish and delivery speed.
* Enable auto-update before signing.
* Enable host full-permission mode as a casual dropdown option.
* Put the app under `crates/quorp_cli` or `crates/quorp_render`.

A temporary CLI sidecar is acceptable only as a 1-day spike. The production path should call shared Rust crates directly.

# 19. The highest-leverage desktop-specific upgrades

The desktop app should not merely wrap the CLI. It should make Quorp feel obviously better:

1. **Mission-control status:** task DAG, context pressure, permission state, sandbox state, validation state always visible.
2. **Diff/proof inspector:** every edit tied to tool call, receipt, validation, and rollback option.
3. **Permission risk explainer:** show exact reason a command is risky: network, compound command, dependency install, generated executable, secrets read.
4. **Context compaction visibility:** show what will be preserved before compacting.
5. **Session replay viewer:** scrub through runtime events and proof artifacts.
6. **Rule learning pane:** show draft/shadow/active rules and evidence before promotion.
7. **Parallel agents/worktrees:** left sidebar sessions become true parallel work surfaces later.
8. **Benchmark viewer:** open a run archive and inspect cases, tokens, first-write latency, watchdog pressure, retries, and judge/proof status.
9. **Keyboard-first command palette:** every menu and slash command accessible through `Cmd+K`.
10. **No-color/high-contrast mode:** keep Quorp’s terminal accessibility discipline in the desktop app.

This is the product thesis: **Quorp desktop should be a proof-backed autonomous coding mission-control app, not just a prettier chat window.**
