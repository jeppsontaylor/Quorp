## Bottom line

Build the Quorp desktop app as **`/apps/desktop` using Tauri 2 + React + TypeScript + Vite**, with the **agent/runtime/session/backend remaining Rust-native and in-process**. Do **not** make the desktop app a wrapper that shells out to `quorp run` except as an emergency fallback. Quorp already has the right Rust foundations: a thin binary entrypoint, extracted CLI/session crates, typed agent runtime requests, runtime event fanout, sandbox/permission modes, Patch VM receipts, proof artifacts, and headless run artifacts. The desktop app should become a polished GUI shell over those same core services, not a second product. ([GitHub][1])

One caveat: I could not run a local clone/build in this environment because GitHub DNS resolution failed. The plan below is based on direct inspection of the public repo files through GitHub plus current official Tauri and Apple documentation.

# 1. Product and stack decision

Use:

```text
Tauri 2
React + TypeScript
Vite
Rust backend using quorp_session + quorp_agent_core + existing crates
Tauri command IPC for frontend → Rust calls
Tauri channels / batched events for Rust → frontend streaming
No local WebSocket server for the normal desktop path
```

Tauri is the right desktop shell because it lets Quorp keep Rust as the application core while using a high-quality web UI inside a native macOS bundle. Tauri officially supports using essentially any frontend framework, with application logic in Rust, and can build for macOS, Windows, Linux, iOS, and Android from one codebase. ([Tauri][2])

React/TypeScript is the right first frontend because this product needs a dense, polished, fast, Codex-like interface: virtualized event streams, diff views, command palette, settings panes, multi-column layout, accessibility, keyboard shortcuts, and fast iteration. A pure-Rust UI with Slint, egui, Leptos, or Yew is possible, and Tauri templates include Rust-web options like Yew/Leptos/Sycamore, but React + TypeScript is the fastest route to a brilliant first DMG. ([Tauri][3])

**Do not use WebSockets for the local desktop app.** Tauri already provides command IPC for calling Rust from the frontend, and Rust can call the frontend using events or channels. Commands are ideal for `start_run`, `cancel_run`, `open_workspace`, `approve_permission`, and `get_status`; channels or batched events are better for streaming agent events. Tauri’s own docs note that events are not meant for high-throughput streaming and point to channels for streaming data, so Quorp should batch or channel runtime events rather than fire one JSON event per token/tool line. ([Tauri][4])

```text
/crates/quorp_desktop_bridge

```

```toml

[workspace]

]
```

Why exclude at first? It lets the desktop app iterate quickly without making every `cargo check --workspace` pull in Tauri immediately. Once the app is stable, move `apps/desktop/src-tauri` into `workspace.members` and add a dedicated desktop CI job.

# 3. How the desktop app should wire into Quorp

Quorp already exposes most of the machinery a desktop app needs.

Existing surfaces that matter:

```text
crates/quorp_agent_core/src/runtime.rs
crates/quorp_agent_core/src/runtime/*.rs
crates/quorp_session/src/quorp/agent_runner.rs
crates/quorp_session/src/quorp/executor.rs
crates/quorp_session/src/quorp/prompt_compaction.rs
crates/quorp_permissions
crates/quorp_sandbox
crates/quorp_patch_vm
crates/quorp_verify
crates/quorp_memory
crates/quorp_rule_forge
crates/quorp_context
```

The agent runtime already has `AgentRunRequest` fields for session ID, goal, model, base URL override, max iterations, max total tokens, max seconds, autonomy profile, project root, cwd, rollback behavior, metadata, and cancellation. That maps directly to a desktop run form. ([GitHub][6])

The runtime already emits rich events: status updates, phase changes, model request start/finish, tool call start/finish, validation start/finish, path resolution failure, recovery turns, parser recovery, verifier queue/drain, and backpressure-related events. That is exactly what the desktop stream should render as cards. ([GitHub][6])

`RuntimeEventFanout` already supports named bounded subscribers. The desktop backend should subscribe to runtime events with a named subscriber like `"desktop-renderer"` and stream batches to the frontend. ([GitHub][6])

`crates/quorp_session/src/quorp/agent_runner.rs` already has a `CommandBridgeToolExecutor` pattern that relays typed tool execution requests through a command bridge. That is the pattern to reuse for desktop permission prompts, tool status, and approval UX. ([GitHub][7])

## Proposed Rust service layer

Add this facade:

```text
crates/quorp_session/src/desktop.rs
```

or, if you want to keep `quorp_session` cleaner:

```text
crates/quorp_desktop_bridge/src/quorp_desktop_bridge.rs
```

Preferred first implementation:

```rust

    ReadOnly,
```

```rust
pub async fn start_run(
    app: AppHandle,
    state: &AppState,
    request: DesktopStartRunRequest,
) -> anyhow::Result<DesktopRunStarted> {
    validate_workspace_and_policy(&request)?;

    let run_id = Uuid::new_v4().to_string();
    let session_id = Uuid::new_v4().to_string();
    let result_dir = app_data_run_dir(&app, &request.workspace_root, &run_id)?;

    std::fs::create_dir_all(&result_dir)
        .with_context(|| format!("failed to create run dir {}", result_dir.display()))?;

    let cancel = Arc::new(AtomicBool::new(false));

    let app_for_task = app.clone();
    let request_for_task = request.clone();
    let run_id_for_task = run_id.clone();
    let session_id_for_task = session_id.clone();
    let result_dir_for_task = result_dir.clone();
    let cancel_for_task = cancel.clone();

    let join = tokio::spawn(async move {
        run_quorp_agent_in_process(
            app_for_task,
            run_id_for_task,
            session_id_for_task,
            result_dir_for_task,
            request_for_task,
            cancel_for_task,
        )
        .await
    });

    let handle = RunHandle {
        run_id: run_id.clone(),
        session_id: session_id.clone(),
        result_dir: result_dir.clone(),
        cancel,
        join,
    };

    state.runs.insert(run_id.clone(), handle);

    Ok(DesktopRunStarted {
        run_id,
        session_id,
        result_dir,
    })
}

async fn run_quorp_agent_in_process(
    app: AppHandle,
    run_id: String,
    session_id: String,
    result_dir: PathBuf,
    request: DesktopStartRunRequest,
    cancel: Arc<AtomicBool>,
) -> anyhow::Result<()> {
    // This is intentionally a shape, not a blind exact call.
    // The implementation should call the existing quorp_session headless runner
    // or a new quorp_session::desktop facade that constructs AgentRunRequest.

    let (event_tx, mut event_rx) = mpsc::channel::<serde_json::Value>(4096);

    let emit_task = {
        let app = app.clone();
        let run_id = run_id.clone();

        tokio::spawn(async move {
            let mut seq: u64 = 0;
            let mut batch = Vec::with_capacity(64);
            let mut tick = tokio::time::interval(std::time::Duration::from_millis(33));

            loop {
                tokio::select! {
                    maybe_event = event_rx.recv() => {
                        match maybe_event {
                            Some(event) => {
                                batch.push(event);

                                if batch.len() >= 64 {
                                    let seq_start = seq;
                                    seq += batch.len() as u64;
                                    let payload = DesktopEventBatch {
                                        run_id: run_id.clone(),
                                        seq_start,
                                        seq_end: seq,
                                        events: std::mem::take(&mut batch),
                                    };
                                    let _ = app.emit("runtime:event-batch", payload);
                                }
                            }
                            None => break,
                        }
                    }
                    _ = tick.tick() => {
                        if !batch.is_empty() {
                            let seq_start = seq;
                            seq += batch.len() as u64;
                            let payload = DesktopEventBatch {
                                run_id: run_id.clone(),
                                seq_start,
                                seq_end: seq,
                                events: std::mem::take(&mut batch),
                            };
                            let _ = app.emit("runtime:event-batch", payload);
                        }
                    }
                }
            }
        })
    };

    // Target implementation:
    //
    // 1. Build quorp_agent_core::AgentRunRequest.
    // 2. Create RuntimeEventFanout.
    // 3. Subscribe a named desktop subscriber.
    // 4. Forward runtime events into event_tx.
    // 5. Run quorp_session headless/runtime path.
    // 6. Write the same artifacts CLI writes.
    //
    // Do not shell out to the CLI unless a temporary dev-only fallback is enabled.

    let _ = event_tx
        .send(serde_json::json!({
            "kind": "run-started",
            "run_id": run_id,
            "session_id": session_id,
            "result_dir": result_dir,
            "workspace_root": request.workspace_root,
            "prompt": request.prompt,
        }))
        .await;

    // TODO: call quorp_session::desktop::run_agent(...)
    // quorp_session already contains the headless agent runner and bridge patterns.

    drop(event_tx);
    emit_task.await??;

    Ok(())
}

fn validate_workspace_and_policy(request: &DesktopStartRunRequest) -> anyhow::Result<()> {
    if !request.workspace_root.exists() {
        anyhow::bail!(
            "workspace does not exist: {}",
            request.workspace_root.display()
        );
    }

    if request.permission_mode == "direct-host-danger" {
        // Require explicit trust record in real implementation.
        anyhow::bail!("direct host writes require explicit workspace trust");
    }

    Ok(())
}

fn app_data_run_dir(
    app: &AppHandle,
    workspace_root: &std::path::Path,
    run_id: &str,
) -> anyhow::Result<PathBuf> {
    let base = app
        .path()
        .app_data_dir()
        .context("failed to resolve app data dir")?;

    let workspace_hash = quorp_workspace_hash(workspace_root);
    Ok(base.join("sessions").join(workspace_hash).join("runs").join(run_id))
}

fn quorp_workspace_hash(path: &std::path::Path) -> String {
    use std::hash::{Hash, Hasher};

    let mut hasher = std::collections::hash_map::DefaultHasher::new();
    path.display().to_string().hash(&mut hasher);
    format!("{:016x}", hasher.finish())
}
```

In the real implementation, replace the placeholder with a direct call into a small `quorp_session::desktop` facade that constructs `AgentRunRequest` and uses `RuntimeEventFanout::subscribe_named_with_capacity`. The runtime already has a bounded fanout design, so the desktop app should preserve that backpressure model instead of inventing a new streaming layer. ([GitHub][6])

# 11. Frontend state and memory policy

The frontend must not store giant raw logs forever.

Use a normalized store:

```ts
interface RuntimeState {
  activeRunId: string | null;

  runOrder: string[];

  runsById: Record<string, {
    runId: string;
    status: "idle" | "running" | "paused" | "failed" | "succeeded";
    eventIds: string[];
    latestSeq: number;
    tokenPressure?: number;
    contextPressure?: number;
    activeToolCallIds: string[];
    diffIds: string[];
  }>;

  eventsById: Record<string, RuntimeEventViewModel>;

  appendBatch: (batch: DesktopEventBatch) => void;
  compactUiEvents: (runId: string) => void;
}
```

Memory rules:

```text
Keep raw events in Rust files and receipts.
Keep only view models in the renderer.
Virtualize the stream.
Batch event updates at 30–60Hz.
Collapse old tool output by default.
Fetch large logs/diffs lazily by range.
Keep only the current session hot.
Keep previous sessions as summaries until opened.
```

A 10,000-event run should not make the renderer sluggish. The UI should render the visible 40–100 cards, not the full DOM.

```text

3. Bundle .app
```

# 16. Security model for the desktop app

The desktop app must not become a permission bypass.

Default modes:

```text
Read Only          safe inspection only
Ask                ask before edits / risky commands
Accept Edits       allow Patch VM edits, ask for risky shell/network
Auto Safe          allow safe commands and sandboxed edits
YOLO Sandbox       full-auto only inside tmp-copy or git-worktree
Direct Host        danger-zone only, explicit trust, never default
```

Desktop-specific rules:

```text
No direct JS filesystem access to project files.
No direct JS shell access.
No frontend-provided command string runs without Rust policy.
No untrusted repo config can change global permissions.
No Direct Host mode unless the workspace is explicitly trusted.
YOLO must mean sandboxed YOLO, not host YOLO.
Every mutating action gets an audit event.
Every patch links to a receipt/proof artifact.
Every permission decision is persisted.
```

Use Tauri capabilities as the first boundary, then Quorp permissions/sandboxing as the second boundary. Tauri IPC is message passing, and the core process can reject malicious requests rather than exposing direct function access. ([Tauri][14])

The app should show a persistent trust pill:

```text
Trusted · Git Worktree · Auto Safe
Untrusted · Tmp Copy · Ask
Danger: Direct Host Writes
```

Clicking it opens `/permissions`.

# 22. The most important engineering choices

## 1. In-process Rust, not CLI subprocess

Use:

```text
Desktop UI → Tauri commands → Rust AgentService → quorp_session/quorp_agent_core
```

Avoid:

```text
Desktop UI → spawn `quorp run` → parse terminal output
```

The CLI is still important, but the desktop app should share the runtime, not scrape it.

## 2. Channels/batches, not per-token events

Tauri events are okay for lifecycle events, but the docs explicitly say events are not designed for high-throughput streaming. Use channels or batching for runtime streams. ([Tauri][17])

## 3. Virtualize everything

Virtualize:

```text
agent event stream
diff hunks
logs
transcripts
benchmark replay
file tree for large repos
```

## 4. Rust owns durable truth

The frontend owns:

```text
visible layout
selection state
expanded/collapsed cards
local command palette state
```

Rust owns:

```text
workspace trust
run state
event logs
proof receipts
permissions
settings
secrets
sandbox policy
artifacts
```

## 5. Desktop should expose Quorp’s strengths

Quorp’s differentiators are not just “chat with code.” They are:

```text
Patch VM receipts
proof packets
runtime recovery
event fanout
sandbox modes
benchmark artifacts
context/rule/memory machinery
```

The desktop app should make those visible.

# 24. My strongest recommendation

Build the first desktop version as:

```text
/apps/desktop
  Tauri 2
  React + TypeScript + Vite
  In-process Rust agent service
  Batched/channel runtime events
  Virtualized stream and diff UI
  Strict Tauri capabilities
  Ad-hoc DMG for local testing
  Signed/notarized DMG after UX/runtime wiring is proven
```

This gives you a usable Mac app quickly without weakening Quorp’s Rust-native architecture. It also sets up the product to become more than “CLI in a window”: a real agent mission-control desktop app with visible tasks, diffs, proof, permissions, context pressure, recovery, replay, and benchmark artifacts.
