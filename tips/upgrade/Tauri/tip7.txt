## 1. Core recommendation

Build the desktop app as:

```text
Tauri 2 + React + TypeScript + Vite
Rust runtime host using existing Quorp crates
Tauri IPC commands for request/response
Tauri Channel streams for live agent events
No local WebSocket server for the first app
```

The repo should gain a new product app at:

```text
apps/desktop
```

and two small Rust crates:

```text
crates/quorp_app_protocol
crates/quorp_app_host
```

Keep the CLI exactly where it belongs today: `crates/quorp`, `crates/quorp_cli`, and `crates/quorp_session`. The desktop app should not replace the CLI. It should become another client of the same runtime/session/ledger/sandbox/permission system.

This is the fastest serious path to a polished `.app` and `.dmg` while keeping the important parts Rust-native. Tauri’s model is a Rust backend plus a webview frontend, with IPC between the web UI and Rust, and it is specifically designed for small native desktop bundles using the OS webview rather than shipping Chromium. ([Tauri][1])

The app should **not** spawn the `quorp` CLI as its primary runtime. That is acceptable only as a temporary smoke-test adapter. The durable architecture should be:

```text
React/TypeScript UI
  ↓ Tauri commands + streaming Channels
quorp_desktop Tauri shell
  ↓
quorp_app_host
  ↓
quorp_session / quorp_agent_core / quorp_provider
  ↓
quorp_permissions / quorp_sandbox / quorp_patch_vm / quorp_verify / ledger
```

That gives Quorp one runtime and two surfaces: terminal CLI and desktop mission control.

## 2. What the repo already has that the desktop app should reuse

Quorp is already positioned well for this. The repo is not just a terminal app; it already has the right runtime pieces to become a desktop app.

The current project is described as a Rust workspace centered on a terminal binary, with `quorp_cli`, `quorp_session`, smart tooling, typed runtime actions, context expansion, Patch VM write receipts, bounded event fanout, proof packets, benchmark gates, and session artifacts. ([GitHub][2])

The workspace already contains the core crates a desktop app should depend on, including `quorp_agent_core`, `quorp_agent_protocol`, `quorp_cli`, `quorp_session`, `quorp_config`, `quorp_context`, `quorp_memory`, `quorp_permissions`, `quorp_patch_vm`, `quorp_provider`, `quorp_render`, `quorp_sandbox`, `quorp_slash`, `quorp_tools`, and `quorp_verify`. ([GitHub][3])

The CLI already exposes important product surfaces: `run`, `exec`, `doctor`, `replay`, `proof`, `diagnostics`, `agent`, `benchmark`, and render demos. It also already enforces an important safety boundary: `--yolo` maps to sandboxed autonomous execution and rejects host sandboxing. ([GitHub][4])

The runtime already tracks state the desktop app can surface beautifully: goal, mode, autonomy profile, policy, acceptance criteria, working set, workspace root, last tool summary, failing verifier, safe checkpoint, parse errors, failed tools, stall counts, mutating changes, verified-green state, validation queue, total billed tokens, and repair recovery. ([GitHub][5])

The renderer already has a useful TUI mental model: task rows, command cards, pass/fail/warn states, headers, footers, no-color support, and a status footer like `qwen3-coder@nvidia · yolo sandbox · ctx 12.4k/64k`. ([GitHub][6])

So the desktop app should not invent a parallel system. It should turn these existing runtime facts into a high-density graphical mission-control UI.

### Why `apps/desktop`, not `crates/quorp_desktop`?

Because this is a product app, not just a Rust library. It has a frontend package, icons, entitlements, Tauri config, DMG assets, updater config, macOS-specific bundle settings, and test fixtures. Keeping it under `apps/desktop` avoids polluting the crate layer with product packaging.

The Rust portion of the Tauri app still lives in `apps/desktop/src-tauri`, but the reusable logic goes into `crates/quorp_app_host`.

### 4.2 IPC: Tauri commands plus Channels, not WebSockets

Use Tauri commands for normal calls:

```text
list_workspaces
trust_workspace
open_workspace
start_agent_turn
stop_agent_turn
resume_session
get_session_status
get_diff
get_proof
save_settings
```

Use Tauri `Channel<T>` for streaming long-running agent events. Tauri documents commands as the normal way to call Rust from the web frontend, including async commands, and Channels as the better fit for ordered streaming data from Rust to the frontend. ([Tauri][7])

Avoid WebSockets for the local desktop MVP. A local WebSocket server adds another port, another security boundary, extra lifecycle management, and more failure modes. Tauri Channels are enough for local streaming.

A future `quorp app-server` can expose a JSON-RPC or JSONL protocol for remote clients, VS Code, browser, or multi-process mode. That would follow the same broad pattern Codex describes: one core agent loop/runtime exposed to multiple clients through a long-lived app server, with client requests producing many UI-ready events. ([OpenAI][8])

### 4.3 Core command flow

```text
User enters prompt
  ↓
React calls startAgentTurn(request, channel)
  ↓
Tauri command validates workspace trust + permission profile
  ↓
quorp_app_host creates or resumes session
  ↓
runtime starts agent turn in async task
  ↓
runtime/tool/ledger events are mapped to UiEvent
  ↓
UiEvent is sent through Tauri Channel
  ↓
React timeline appends virtualized cards
  ↓
diff/proof/status inspectors update
```

## 6. Frontend technology stack

### Recommended stack

```text
React 19
TypeScript
Vite
Tauri 2
Zustand
TanStack Virtual
CodeMirror 6
xterm.js, lazy-loaded
CSS variables + plain CSS modules
Playwright for UI tests
Vitest for TS unit tests
```

### Why this stack

React + TypeScript is the fastest path to a polished, high-density, Codex-like desktop interface. The runtime stays Rust. The UI should not be pure Rust if the goal is a beautiful, quickly shippable desktop IDE with rich layout, virtualized timelines, menus, popovers, diff views, settings panels, keyboard palette, and future cross-platform support.

Use CodeMirror 6 for prompt editing, small file previews, search boxes, and diff text because it is modular and lower overhead than embedding a full VS Code-like editor everywhere. CodeMirror’s core packages are split into state and view modules, which fits this app’s need for lightweight editor surfaces. ([CodeMirror][12])

Use Monaco only as an optional lazy-loaded “open full editor” feature later. Monaco is powerful and VS Code-derived, but it is heavier than Quorp needs for the first desktop release. ([Microsoft GitHub][13])

Use xterm.js only for the terminal drawer or raw command streams, and lazy-load it. xterm.js is a full terminal component used by major browser-based developer tools, but always mounting it for every command card would waste memory. ([GitHub][14])

### What not to use initially

Avoid:

```text
Electron
A local web server for the local app
WebSockets for local app IPC
Redux
Heavy CSS-in-JS runtime systems
Always-mounted Monaco
Always-mounted xterm instances
```

### Frontend state model

Use stores like this:

```ts
// src/state/sessionStore.ts
type SessionStore = {
  activeSessionId: string | null;
  sessions: Record<string, SessionSummary>;
  activeRun: RunState | null;
  startRun: (request: StartTurnRequest) => Promise<void>;
  stopRun: (sessionId: string) => Promise<void>;
};

// src/state/timelineStore.ts
type TimelineStore = {
  bySession: Record<string, TimelineEvent[]>;
  appendEvent: (sessionId: string, event: UiEvent) => void;
  clearSession: (sessionId: string) => void;
};

// src/state/permissionStore.ts
type PermissionStore = {
  profile: PermissionProfile;
  sandbox: SandboxMode;
  pendingRequests: PermissionRequest[];
  approve: (id: string) => Promise<void>;
  deny: (id: string) => Promise<void>;
};
```

For very long sessions, keep full raw logs in Rust/ledger storage and only keep the visible timeline projection in the frontend.

## 9. `quorp_app_host` design

This crate should become the desktop-safe adapter around existing Quorp runtime pieces.

```rust
// crates/quorp_app_host/src/lib.rs

mod cancellation;
mod error;
mod event_mapper;
mod host;
mod ledger_adapter;
mod permission_manager;
mod runtime_adapter;
mod session_manager;
mod settings_store;
mod workspace_manager;

pub use error::{DesktopHostError, DesktopHostResult};
pub use host::DesktopHost;
```

```rust
// crates/quorp_app_host/src/host.rs

use std::collections::HashMap;
use std::sync::Arc;

use tokio::sync::RwLock;

use quorp_app_protocol::{
    DiffEvent, SessionId, StartTurnRequest, StartTurnResponse, UiEvent, WorkspaceId,
    WorkspaceSummary,
};

use crate::error::{DesktopHostError, DesktopHostResult};
use crate::permission_manager::PermissionManager;
use crate::session_manager::SessionManager;
use crate::workspace_manager::WorkspaceManager;

pub struct DesktopHost {
    workspaces: WorkspaceManager,
    sessions: SessionManager,
    permissions: PermissionManager,
    running: RwLock<HashMap<SessionId, RunningTurn>>,
}

struct RunningTurn {
    cancel: tokio::sync::watch::Sender<bool>,
}

impl DesktopHost {
    pub fn new() -> Self {
        Self {
            workspaces: WorkspaceManager::new(),
            sessions: SessionManager::new(),
            permissions: PermissionManager::new(),
            running: RwLock::new(HashMap::new()),
        }
    }

    pub async fn list_workspaces(&self) -> DesktopHostResult<Vec<WorkspaceSummary>> {
        self.workspaces.list().await
    }

    pub async fn trust_workspace(&self, path: String) -> DesktopHostResult<WorkspaceSummary> {
        self.workspaces.trust(path).await
    }

    pub async fn open_workspace(
        &self,
        workspace_id: WorkspaceId,
    ) -> DesktopHostResult<WorkspaceSummary> {
        self.workspaces.open(workspace_id).await
    }

    pub async fn start_agent_turn<F>(
        &self,
        request: StartTurnRequest,
        emit: F,
    ) -> DesktopHostResult<StartTurnResponse>
    where
        F: Fn(UiEvent) + Send + Sync + 'static,
    {
        let workspace = self
            .workspaces
            .get_trusted(&request.workspace_id)
            .await?;

        self.permissions
            .validate_start_request(&workspace, &request)
            .await?;

        let session = self
            .sessions
            .create_or_resume(&workspace, request.session_id.clone())
            .await?;

        let response = self
            .sessions
            .start_runtime_turn(session, request, Arc::new(emit))
            .await?;

        Ok(response)
    }

    pub async fn stop_agent_turn(&self, session_id: SessionId) -> DesktopHostResult<()> {
        self.sessions.stop(session_id).await
    }

    pub async fn get_diff(
        &self,
        session_id: SessionId,
        path: String,
    ) -> DesktopHostResult<DiffEvent> {
        self.sessions.diff(session_id, path).await
    }
}
```

### Important implementation rule

`quorp_app_host` should depend on runtime/session crates. It should **not** depend on `quorp_cli`. The CLI and desktop should be siblings, not parent/child.

Bad:

```text
desktop → quorp_cli → runtime
```

Good:

```text
quorp_cli       → quorp_session / quorp_agent_core
quorp_app_host  → quorp_session / quorp_agent_core
```

## 15. Permission model for desktop

Codex’s desktop docs emphasize sandbox boundaries, approvals when crossing those boundaries, and reducing approval fatigue through controlled sandbox behavior. ([OpenAI Developers][16]) Claude Code similarly exposes permission management through slash commands and fine-grained allow/ask/deny rules. ([Claude][17])

Quorp should use these desktop permission profiles:

### Read-only

```text
Can:
  read trusted workspace files
  inspect git state
  search
  plan
  run non-mutating diagnostics if explicitly safe

Cannot:
  write files
  run shell commands that mutate
  install dependencies
  access network
  use MCP/browser
```

### Ask before writes

```text
Can:
  read
  search
  propose edits
  ask before patch/write
  ask before shell command

Default for newly trusted workspace.
```

### Auto-safe sandbox

```text
Can:
  write inside sandbox/worktree
  run safe local commands inside sandbox
  run tests inside sandbox
  create checkpoints
  produce diffs/proofs

Cannot without approval:
  host writes
  network
  dependency installs
  MCP
  browser
  secrets/env access
  destructive commands
```

### Full-auto sandbox

```text
Can:
  autonomous loop
  sandbox/worktree writes
  local validation
  rollback/checkpoint

Cannot:
  host full-auto
  unrestricted network
  unrestricted dependency installs
  unrestricted MCP/browser
```

This should map to the existing CLI behavior where full-auto `--yolo` is sandboxed and host-yolo is rejected. ([GitHub][4])

### Do not ship host full-permission as a normal default

A “Host full permission” option can exist behind an advanced setting, but not in the normal permission picker for the initial desktop app. Full-auto host execution against arbitrary repositories is the wrong default.

## 16. Workspace trust flow

When the user clicks **Add Folder…**:

```text
1. Native folder picker opens.
2. Backend canonicalizes path.
3. Backend detects git root.
4. Backend scans for:
   - AGENTS.md
   - .rules
   - .codex
   - package scripts
   - shell scripts
   - config that could influence agent behavior
5. UI shows trust prompt:
   - path
   - git remote
   - branch
   - detected repo-authored instructions
   - initial permission profile
6. Until trusted:
   - no shell
   - no writes
   - no repo-authored config execution
7. On trust:
   - persist workspace trust record
   - enable selected permission profile
```

Codex distinguishes user-level config from project-level config and only applies project config for trusted projects; Quorp should adopt the same security principle. ([OpenAI Developers][18])

Persist trust records in:

```text
~/Library/Application Support/Quorp/workspaces.json
```

Example:

```json

      "id": "ws_01j...",
      "path": "/Users/taylor/dev/Quorp",
      "trusted": true,
      "trustedAt": "2026-04-29T18:22:00Z",
      "gitRemote": "git@github.com:jeppsontaylor/Quorp.git",
      "defaultPermissionProfile": "autoSafeSandbox",
      "defaultSandboxMode": "gitWorktree"

}
```

## 17. Tauri security configuration

Tauri capabilities define what the frontend is allowed to access from the Rust core and plugin system. Capability files live under `src-tauri/capabilities`, and permissions/scopes should be explicit. ([Tauri][19])

Start with a narrow capability file:

```json
// apps/desktop/src-tauri/capabilities/desktop.json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "desktop",
  "description": "Main Quorp desktop window permissions",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "dialog:default",
    "opener:default"
  ]
}
```

Do **not** grant broad filesystem or shell plugin permissions to the frontend.

Bad for MVP:

```json
"fs:default",
"shell:allow-execute"
```

Instead, expose Quorp-specific Rust commands:

```text
trust_workspace
open_workspace
start_agent_turn
approve_permission_request
deny_permission_request
get_diff
get_proof
```

Those commands enforce Quorp’s own permission model.

### 19.1 Local development, no signing delay

### 19.2 Local `.app` and `.dmg`, unsigned

```bash

pnpm tauri bundle --bundles app,dmg --no-sign
```

Tauri documents `tauri bundle --bundles app,dmg`, `tauri build --no-bundle`, and a `--no-sign` flag for skipping signing. ([Tauri][20])

This may require macOS Privacy & Security approval when opening the app. That is fine for rapid testing.

### 19.3 Local ad-hoc signed build

For Apple Silicon local testing, use ad-hoc signing:

```bash
cd apps/desktop
APPLE_SIGNING_IDENTITY="-" pnpm tauri build --no-bundle
APPLE_SIGNING_IDENTITY="-" pnpm tauri bundle --bundles app,dmg
```

Tauri documents using `"-"` as the signing identity for ad-hoc signing, which is useful for local Apple Silicon builds but does not replace Developer ID signing and notarization for distribution. ([Tauri][21])

### 19.4 Proper signed and notarized release

Prerequisites:

```bash
security find-identity -v -p codesigning
```

Then either use App Store Connect API credentials:

```bash
export APPLE_SIGNING_IDENTITY="Developer ID Application: Your Name (TEAMID)"
export APPLE_API_KEY_PATH="/path/to/AuthKey_XXXXXX.p8"
export APPLE_API_KEY="XXXXXX"
export APPLE_API_ISSUER="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

cd apps/desktop
pnpm build
pnpm tauri build --no-bundle
pnpm tauri bundle --bundles app,dmg
```

or Apple ID credentials:

```bash
export APPLE_SIGNING_IDENTITY="Developer ID Application: Your Name (TEAMID)"
export APPLE_ID="you@example.com"
export APPLE_PASSWORD="app-specific-password"
export APPLE_TEAM_ID="TEAMID"

cd apps/desktop
pnpm build
pnpm tauri build --no-bundle
pnpm tauri bundle --bundles app,dmg
```

Tauri documents the relevant signing and notarization environment variables, including signing identity, Apple ID credentials, and App Store Connect API credentials. ([Tauri][21])

For public distribution outside the Mac App Store, Apple notarization is the right release path; Tauri’s macOS docs also call out signing/notarization for DMG distribution. ([Tauri][20])

```tsx
export function App() {

    />
  );
}
```


```text

- Auto-updater
```

## 26. Memory and performance rules

The app should feel extremely fast. Concrete constraints:

```text
Startup:
  cold launch under 1.5s on modern Mac
  no runtime session boot until workspace opened

Frontend:
  virtualized timeline
  no always-mounted Monaco
  no always-mounted xterm
  no giant JSON blobs in React state
  large artifacts fetched lazily by artifact id
  command output collapsed by default after N lines
  diff hunks virtualized for large files

Rust:
  session source of truth is ledger/storage, not frontend
  event projection is bounded
  command output streamed and chunked
  cancellation tokens for active runs
  no unbounded broadcast buffers
  no raw artifact duplication across stores
```

Timeline event payloads should carry small summaries plus artifact references:

```ts
type CommandEvent = {
  commandId: string;
  commandPreview: string;
  cwd: string;
  sandbox: string;
  status: "started" | "running" | "passed" | "failed";
  exitCode?: number;
  durationMs?: number;
  stdoutPreview?: string;
  stderrPreview?: string;
  stdoutArtifactId?: string;
  stderrArtifactId?: string;
};
```

Fetch full stdout/stderr only when expanded.

## 28. Important product details borrowed from Codex-like desktop UX

The design should copy the good ideas, not the implementation:

```text
- folder/project picker
- left project sidebar
- central agent timeline
- bottom composer
- permission/sandbox selector
- settings inside the app
- worktree-aware sessions
- terminal/actions surface
- artifact/sidebar surfaces
```

Codex’s app docs describe choosing a project folder, running local tasks from a desktop app, multitasking across projects, worktree support, terminal/actions, artifacts, and sidebar workflows. ([OpenAI Developers][22])

Quorp can differentiate by making the following more explicit than Codex/Claude-style tools:

```text
- visible proof DAG
- visible Patch VM receipts
- visible context pressure
- visible context compaction history
- visible task DAG
- visible sandbox root
- visible permission policy
- visible replay path
- visible benchmark/replay viewer
```

This leverages what Quorp already has rather than chasing generic chat UX.

## 29. The biggest architectural choice: in-process host now, app-server later

### Phase 1

Use in-process Tauri host:

```text
Tauri process contains quorp_app_host and runtime adapter.
```

Pros:

```text
fast
simple
no port
no subprocess lifecycle
easy DMG
easy local testing
```

### Phase 2

Add optional app server:

```text
quorp app-server --stdio
quorp app-server --socket <path>
```

Then:

```text
desktop → app-server
tui     → app-server
cli     → app-server optionally
```

This becomes useful for:

```text
multiple windows
remote workspaces
crash isolation
agent process restart
plugin/client ecosystem
VS Code extension
web dashboard
```

Codex has moved toward a shared App Server pattern where all clients consume one evented agent runtime; Quorp should eventually get the same benefit, but the first DMG does not need that complexity. ([OpenAI][8])

## 31. The exact “best” first implementation target

The first desktop release should be named:

```text
Quorp Desktop Alpha
```

It should include:

```text
- Add Folder
- Trust Workspace
- Left workspace/thread sidebar
- Bottom chat composer
- Permission picker
- Model/provider picker
- Runtime event timeline
- Bright diff cards
- Command cards
- Validation cards
- Stop button
- Status inspector
- Context pressure display
- Local unsigned/ad-hoc DMG
```

It should deliberately exclude:

```text
- host full-auto default
- broad frontend filesystem permissions
- frontend shell execution plugin
- unrestricted network/MCP/browser access
- heavy always-on editor/terminal components
```

That gives you a fast, memory-conscious, safe, buildable desktop app without delaying testing on your Mac.
