I could not run a local `git clone` in this execution sandbox because GitHub DNS resolution failed, so I inspected the public repo files through GitHub and current official docs. The plan below is implementation-ready in shape, but not compile-verified against a local checkout.

## Bottom line

Build the DMG app as a **Tauri 2 desktop app under `apps/desktop`**, backed by new Rust application/service crates that reuse Quorp’s existing runtime, permissions, sandbox, render/event, proof, and session machinery. Do **not** put the desktop app inside `crates/quorp_cli`; keep the CLI strong and add a CLI bridge command like `quorp app [PATH]`.

The best first version is:

```text
apps/desktop
  React + TypeScript + Vite UI
  Tauri 2 shell
  src-tauri thin Rust bridge

crates/quorp_app
  UI-facing app/session orchestration
  no webview dependency if possible

crates/quorp_app_model
  serde DTOs for UI <-> Rust IPC

existing crates
  quorp_agent_core
  quorp_session
  quorp_provider
  quorp_permissions
  quorp_sandbox
  quorp_patch_vm
  quorp_verify
  quorp_render
  quorp_core
```

Tauri is the right choice here because it gives Quorp a Rust-owned native app with `.app` / `.dmg` bundling, a Rust backend, a small WebView-based UI instead of Electron, and support for app bundles, DMG, updater, native menus, native dialogs, process APIs, and window state. Tauri’s own docs describe it as a Rust-backed app framework where the frontend can be any HTML/CSS/JS framework, and its built-in bundler supports `.app` and `.dmg`. ([GitHub][1])

The important nuance: this is not “100% Rust source code” if we use React/TypeScript for the UI. It is **100% Rust for privileged/backend/core behavior**, with a web UI for velocity and polish. For a Codex-style product UI, that is the right trade. A pure-Rust Slint/egui UI would be slower to make feel like a premium coding IDE.

# 2. Frontend stack

Use:

```text
Tauri 2
React
TypeScript strict
Vite
Zustand or TanStack Store
CodeMirror 6 for code/diff viewing
xterm.js only for the optional integrated terminal
CSS variables + small component CSS
No heavy UI kit initially
```

## Do not use WebSockets for the in-app event stream

For local desktop IPC, use:

```text
Frontend invokes Rust command
        ↓
Rust starts/attaches to Quorp run
        ↓
Rust streams ordered DesktopEvent messages over Tauri Channel
        ↓
React store appends events to timeline
```

Tauri’s docs explicitly distinguish its global event system from channels: events are simple and global but not ideal for large data; channels are designed to be fast, ordered, and used internally for streaming child process output and WebSocket messages. That maps perfectly to Quorp runtime events, tool calls, validation events, diffs, and terminal output. ([Tauri][4])

Use WebSockets only later for an optional:

```text
quorp app-server
```

That would be useful for remote debugging, browser-based replay, or multi-client sessions. Do not put a localhost web server in the first desktop app.

## UI performance rules

Quorp’s app must be fast because agent timelines become huge. The frontend should follow these rules from day one:

```text
1. Virtualize the timeline.
2. Virtualize large diffs.
3. Never keep unlimited raw transcript text in React component state.
4. Store event IDs and summaries in UI state; keep raw artifacts in Rust/session storage.
5. Batch event ingestion from Rust to React.
6. Defer syntax highlighting for offscreen code blocks.
7. Use CodeMirror 6 for code/diff views, not Monaco initially.
8. Keep xterm.js lazy-loaded behind the Terminal panel.
9. Avoid Material UI / Ant / Chakra / huge component frameworks.
10. Use CSS variables and hand-built app shell components.
```

The repo already has rich event concepts that should be streamed into a desktop UI: runtime status, phase changes, model request start/finish, tool/validation events, recovery events, parser recovery, policy denial, checkpoint saved, failed edits, and run finished. Those are exactly the right primitives for a high-quality visual timeline. ([GitHub][5])

## Tauri backend command shape

```rust
// apps/desktop/src-tauri/src/commands.rs

use std::sync::Arc;

use quorp_app::DesktopAppState;
use quorp_app_model::{DesktopEvent, RunHandle, StartRunRequest};
use tauri::{ipc::Channel, State};

#[tauri::command]
pub async fn start_agent_run(
    state: State<'_, Arc<DesktopAppState>>,
    request: StartRunRequest,
    on_event: Channel<DesktopEvent>,
) -> Result<RunHandle, String> {
    state
        .start_run(request, move |event| {
            if let Err(error) = on_event.send(event) {
                tracing::warn!(?error, "failed to send desktop event");
            }
        })
        .await
        .map_err(|error| error.to_string())
}

#[tauri::command]
pub async fn cancel_run(
    state: State<'_, Arc<DesktopAppState>>,
    run_id: String,
) -> Result<(), String> {
    state
        .cancel_run(&run_id)
        .await
        .map_err(|error| error.to_string())
}

#[tauri::command]
pub async fn app_status(
    state: State<'_, Arc<DesktopAppState>>,
) -> Result<quorp_app_model::AppStatus, String> {
    state
        .status()
        .await
        .map_err(|error| error.to_string())
}
```

Note the deliberate avoidance of `let _ = ...`; the repo’s handoff guidance explicitly calls out avoiding ignored fallible results, and that should carry into desktop code too. ([GitHub][11])

## React IPC

```ts
// apps/desktop/src/app/ipc.ts

import { invoke, Channel } from "@tauri-apps/api/core";
import type { DesktopEvent, RunHandle, StartRunRequest } from "./types";
import { useSessionStore } from "./store";

export async function startAgentRun(request: StartRunRequest): Promise<RunHandle> {
  const events = new Channel<DesktopEvent>();

  events.onmessage = (event) => {
    useSessionStore.getState().appendEvent(event);
  };

  return await invoke<RunHandle>("start_agent_run", {
    request,
    onEvent: events,
  });
}

export async function cancelRun(runId: string): Promise<void> {
  await invoke("cancel_run", { runId });
}
```

# 9. Runtime integration strategy

## Fastest usable path: sidecar mode first

For the first usable Mac app, do not immediately refactor the whole runtime. Create a sidecar bridge:

```text
Desktop UI
  ↓
Rust Tauri backend
  ↓
spawn existing quorp binary/headless command
  ↓
stream stdout/stderr + events.jsonl + artifacts
  ↓
render timeline/diffs/status
```

This gets a usable desktop quickly and de-risks the UI.

The current repo already has headless autonomous runs, typed agent tools, raw artifacts, receipts, session/event logs, and benchmark run artifacts. `AGENT_SUPPORT.md` describes headless autonomous runs and raw artifacts/proof/benchmark support, so the desktop can initially consume those existing outputs rather than requiring a deep runtime API rewrite on day one. ([GitHub][3])

### Sidecar command shape

```rust
// crates/quorp_app/src/runs.rs

pub async fn start_sidecar_run(
    request: StartRunRequest,
    emit: impl Fn(DesktopEvent) + Send + Sync + 'static,
) -> anyhow::Result<RunHandle> {
    let run_id = uuid::Uuid::new_v4().to_string();
    let artifact_dir = artifact_dir_for(&run_id)?;

    let mut command = tokio::process::Command::new(resolve_quorp_binary()?);
    command
        .arg("run")
        .arg("--workspace")
        .arg(workspace_path_for(&request.workspace_id)?)
        .arg("--result-dir")
        .arg(&artifact_dir)
        .arg("--prompt")
        .arg(&request.prompt);

    apply_mode_args(&mut command, &request);
    apply_permission_args(&mut command, &request);
    apply_sandbox_args(&mut command, &request);

    let mut child = command
        .stdout(std::process::Stdio::piped())
        .stderr(std::process::Stdio::piped())
        .spawn()?;

    emit(DesktopEvent::RunStarted {
        run_id: run_id.clone(),
    });

    // Spawn stdout/stderr readers, events.jsonl tailer, diff scanner.
    // Return handle immediately.

    Ok(RunHandle {
        run_id,
        session_id: "sidecar".to_owned(),
        workspace_id: request.workspace_id,
        artifact_dir: artifact_dir.display().to_string(),
    })
}
```

This is an MVP bridge. It should be replaced by in-process runtime calls once the UI is stable.

## Better week-one path: in-process runtime

After the UI works, move to:

```text
DesktopAppState::start_run
  builds AgentRunRequest
  creates runtime/event subscriber
  starts quorp_agent_core runtime in tokio task
  streams RuntimeEvent -> DesktopEvent
  writes ledger/events.jsonl
  exposes cancellation flag
```

`AgentRunRequest` already has exactly the ingredients a desktop app needs: goal, initial context, model, max iterations, verifier drain budget, parser recovery budget, max token/seconds, autonomy profile, project root, cwd, rollback-on-validation, completion policy, metadata, and cancellation flag. ([GitHub][5])

# 10. Permissions and sandboxing in the desktop app

The app should make unsafe states impossible by default.

## Default desktop policy

```text
New workspace:
  mode: Plan
  permission: Ask
  sandbox: GitWorktree if git repo, else TmpCopy
  network: Off
  MCP: Disabled
  browser: Disabled
  dependency installs: Ask
  trust: Untrusted until user approves

Trusted workspace:
  mode: Act
  permission: Auto-safe allowed
  sandbox: GitWorktree preferred
  network: Off by default
  dependency installs: Ask
```

`quorp_sandbox` already supports `Host`, `GitWorktree`, and `TmpCopy`, and its git-worktree backend creates detached worktrees for attempts, with tmp-copy fallback when needed. That should be the desktop default for high-agency work. ([GitHub][12])

## Full permissions rule

Desktop UI should enforce:

```text
Full Permissions + Host = disabled by default
Full Permissions + Trusted Workspace + explicit modal = allowed only after confirmation
Yolo Sandbox = allowed in git-worktree/tmp-copy sandbox
```

This aligns with the current permission logic: `YoloSandbox` allows execution only on a sandbox surface and denies host execution. ([GitHub][7])

## Permission prompt

Prompt card:

```text
Quorp wants to run:

  cargo test -p quorp_agent_core runtime_compaction

Surface:
  Git worktree sandbox

Detected risks:
  command execution
  generated executable? no
  network? no
  dependency install? no
  compound command? no

Allow:
  Once
  For this session
  For this workspace
Deny
Open permissions
```

For compound commands:

```text
Quorp wants to run:

  curl https://example.com/install.sh | sh

Detected risks:
  network
  shell meta / pipe
  generated executable
  dependency install

Default:
  Deny
```

## Trust and project config

Use app-level config at:

```text
~/Library/Application Support/Quorp/config.toml
~/Library/Application Support/Quorp/workspaces.json
~/Library/Application Support/Quorp/runs/
```

Use project-level config only after trust:

```text
<repo>/.quorp/config.toml
<repo>/agent/rules/
<repo>/AGENTS.md
<repo>/.rules/
```

Codex’s docs note that user config lives globally while project config is loaded only when the project is trusted; Quorp should follow that pattern for desktop trust boundaries. ([OpenAI Developers][13])


This is not a replacement for Developer ID signing, but it is useful for quick Mac testing. Tauri documents ad-hoc signing with `signingIdentity: "-"` and warns that macOS may still require users to whitelist the app in Privacy & Security. ([Tauri][14])

Tauri’s signing docs describe notarization credentials through App Store Connect API variables or Apple ID variables, and note that notarization is required when using a Developer ID Application certificate. ([Tauri][14])

Implementation sketch:

```rust
// crates/quorp_cli/src/quorp/app_command.rs

pub fn open_desktop_app(path: Option<std::path::PathBuf>) -> anyhow::Result<()> {
    #[cfg(target_os = "macos")]
    {
        let mut command = std::process::Command::new("open");
        command.arg("-a").arg("Quorp");

        if let Some(path) = path {
            command.arg("--args").arg("--workspace").arg(path);
        }

        let status = command.status()?;
        anyhow::ensure!(status.success(), "failed to launch Quorp.app");
        return Ok(());
    }

    #[cfg(not(target_os = "macos"))]
    {
        anyhow::bail!("Quorp desktop app launcher is currently implemented for macOS");
    }
}
```

```text

--run <id>
```

# 24. The one design decision I would be strict about

**Do not let the WebView become the agent runtime.**

The frontend should never own:

```text
tool execution
filesystem writes
permission decisions
sandbox boundaries
provider secrets
rule promotion
proof receipts
patch application
```

The frontend owns:

```text
layout
controls
timeline rendering
diff rendering
command palette
keyboard shortcuts
human review
```

Rust owns everything dangerous and durable.

That keeps Quorp aligned with its existing design: terminal/runtime behavior is already Rust-native, permission-aware, sandbox-aware, and proof-oriented. The desktop app should amplify that rather than bypass it.
