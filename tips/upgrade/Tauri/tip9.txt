## Recommendation

Build the desktop app as a **new Tauri 2 app at `apps/desktop`**, backed by the existing Rust session/runtime crates. Keep the current `quorp` terminal CLI exactly where it is.

The important architectural decision is this:

```text
Do not wrap the CLI as the desktop app.
Do not fork the runtime.
Do not rewrite Quorp in React.

Build a native desktop shell that links directly to Quorp’s Rust crates,
streams runtime events into a React/TypeScript UI, and preserves the CLI as a first-class sibling.
```

Quorp is already shaped well for this. The repo describes itself as a “high-performance, TUI-first text editor and agentic interface,” with native Rust foundations, CLI/runtime split into reusable crates, Patch VM write receipts, bounded runtime event fanout, verification reports, and benchmark gates. ([GitHub][1]) The current `crates/quorp/src/main.rs` is only:

```rust
fn main() {
    quorp_cli::dispatch();
}
```

That means the desktop app should not live inside `crates/quorp`; that crate should remain the CLI binary. The right place is a new app package.

# 2. Keep the CLI and desktop as peers

The repo already has a real terminal product. `quorp_term` exposes slash commands for `/plan`, `/act`, `/auto`, `/full-auto`, `/full-permissions`, `/permissions`, `/sandbox`, `/memory`, `/rules`, `/status`, `/diff`, `/verify`, `/compact`, `/tasks`, `/checkpoint`, `/rollback`, `/theme`, and more. ([GitHub][3])

Do not lose that.

The product should have two first-class entrypoints:

```text
Terminal:
  quorp
  quorp run
  quorp exec
  quorp doctor
  quorp benchmark
  quorp replay
  quorp proof

Desktop:
  Quorp.app
  Quorp.dmg
```

The desktop should eventually include an **“Install CLI…”** menu item that installs or symlinks the `quorp` binary into a selected path such as `/usr/local/bin`, `~/.local/bin`, or `~/bin`, but the first MVP can simply keep CLI installation manual.

# 3. Best frontend stack

Use:

```text
Tauri 2
React + TypeScript
Vite
CSS variables + focused component primitives
TanStack Virtual for long event feeds
CodeMirror 6 for code/diff views
xterm.js only when a true terminal pane is needed
Zustand or TanStack Store for lightweight client state
```

I would **not** use Electron. I would also avoid a huge SPA framework, a heavy design system, or Monaco as the default editor. Monaco is excellent but heavier than needed for an event-streaming agent console. Use CodeMirror 6 first for diffs, read-only code, and lightweight editors.

I would also **not use WebSockets for local runtime streaming**. For the desktop app, use **Tauri commands and channels**. Tauri’s docs describe commands as the bridge for type-safe Rust calls from the frontend, including async commands, and describe channels as optimized for fast, ordered streaming from Rust to the frontend. Tauri events are useful, but the docs explicitly distinguish events from lower-latency/high-throughput channel use. ([Tauri][4])

Use WebSockets only later if you add:

```text
quorp daemon --listen 127.0.0.1:...
remote browser UI
team-shared run monitor
headless server mode
```

For the desktop app, the ideal IPC is:

```text
React UI
  invoke("start_agent_run", { request, eventsChannel })
       ↓
Tauri command
       ↓
Rust desktop session host
       ↓
quorp_session / quorp_agent_core runtime
       ↓
RuntimeEventSink
       ↓
Tauri Channel<DesktopEvent>
       ↓
React event reducer + virtualized feed
```

## Center feed

The feed should be append-only, virtualized, and typed.

Card types:

```ts
type FeedCard =
  | { kind: "assistant_delta"; text: string }
  | { kind: "tool_call"; tool: string; argsPreview: string; state: "running" | "done" | "failed" }
  | { kind: "shell"; command: string; cwd: string; exitCode?: number; outputId: string }
  | { kind: "diff"; file: string; additions: number; deletions: number; diffId: string }
  | { kind: "validation"; command: string; status: "passed" | "failed" | "running"; reportId: string }
  | { kind: "permission_prompt"; promptId: string; risk: "low" | "medium" | "high" }
  | { kind: "checkpoint"; checkpointId: string; summary: string }
  | { kind: "rollback"; checkpointId: string; reason: string }
  | { kind: "proof"; receiptId: string; status: "ready" | "incomplete" };
```

Quorp’s existing runtime already has `RuntimeEvent`, `AgentTaskStateSnapshot`, `AgentCheckpoint`, and `RuntimeEventFanout`, which are exactly the backend primitives the desktop feed should consume. ([GitHub][5])

## Composer

The composer should be the command center.

```text
┌──────────────────────────────────────────────────────────┐
│ Ask Quorp to change, inspect, test, explain, or continue │
│                                                          │
│ /plan refactor provider routing and add tests            │
└──────────────────────────────────────────────────────────┘

[Plan] [Act] [Full Auto]     Permission: Ask ▾   Sandbox: Git worktree ▾
Model: GPT-5.5 Thinking ▾    Context: 18.2k / 128k  ▾   Attach ▾   Send
```

Controls:

| Control      | Behavior                                                                      |
| ------------ | ----------------------------------------------------------------------------- |
| `Plan`       | Read-only planning mode.                                                      |
| `Act`        | Can edit with approvals.                                                      |
| `Full Auto`  | Autonomous, but only inside configured sandbox and permission policy.         |
| `Permission` | Read-only, Ask, Accept edits, Auto-safe, Full-permission/yolo-sandbox.        |
| `Sandbox`    | Host, tmp-copy, git-worktree. Host should never be the default for full-auto. |
| `Model`      | Provider/model/router profile.                                                |
| `Context`    | Shows pressure, opens context inspector.                                      |
| `Attach`     | Add files, selected folders, screenshots later, logs, benchmark artifacts.    |
| `Send`       | Starts or continues a session.                                                |

## Diff design

Diff cards should use high-contrast professional colors:

```text
src/foo.rs
────────────────────────────────────────────────────────────
- old line with bright red background
+ new line with bright green background
```

Requirements:

```text
sticky file header
unified and side-by-side toggle
word-level highlights
copy patch
open file
revert file
apply selected hunk later
show Patch VM receipt
show hash precondition/stale hash state
```

The desktop should not just render a string diff. It should store diff metadata in Rust and lazy-load full diff details by `diffId`.

## Desktop state

```rust
// apps/desktop/src-tauri/src/desktop_state.rs

use std::{
    collections::HashMap,
    path::PathBuf,
    sync::{Arc, Mutex},
};

use uuid::Uuid;

#[derive(Default)]
pub struct DesktopState {
    inner: Arc<Mutex<DesktopStateInner>>,
}

#[derive(Default)]
pub struct DesktopStateInner {
    pub workspaces: HashMap<WorkspaceId, WorkspaceRecord>,
    pub sessions: HashMap<SessionId, SessionHandle>,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct WorkspaceId(pub Uuid);

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct SessionId(pub Uuid);

#[derive(Debug, Clone)]
pub struct WorkspaceRecord {
    pub id: WorkspaceId,
    pub path: PathBuf,
    pub trusted: bool,
}

pub struct SessionHandle {
    pub id: SessionId,
    pub workspace_id: WorkspaceId,
    pub cancel: tokio::sync::watch::Sender<bool>,
}
```

Use a `Mutex<HashMap<...>>` first. Move to `DashMap` only when needed. Keep the first PR boring and reliable.

# 7. IPC DTOs

Create a narrow API between React and Rust.

```rust
// crates/quorp_desktop_api/src/lib.rs

use serde::{Deserialize, Serialize};
use std::path::PathBuf;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DesktopRunRequest {
    pub workspace_path: PathBuf,
    pub objective: String,
    pub provider: Option<String>,
    pub model: Option<String>,
    pub mode: DesktopRunMode,
    pub permission_mode: DesktopPermissionMode,
    pub sandbox_mode: DesktopSandboxMode,
    pub max_steps: Option<u32>,
    pub max_seconds: Option<u64>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum DesktopRunMode {
    Plan,
    Act,
    FullAuto,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum DesktopPermissionMode {
    ReadOnly,
    Ask,
    AcceptEdits,
    AutoSafe,
    YoloSandbox,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum DesktopSandboxMode {
    Host,
    TmpCopy,
    GitWorktree,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum DesktopEvent {
    RunStarted {
        session_id: String,
    },
    AssistantDelta {
        session_id: String,
        text: String,
    },
    ToolCallStarted {
        session_id: String,
        call_id: String,
        tool: String,
        preview: String,
    },
    ToolCallFinished {
        session_id: String,
        call_id: String,
        ok: bool,
        summary: String,
    },
    ShellOutput {
        session_id: String,
        command_id: String,
        chunk: String,
    },
    DiffReady {
        session_id: String,
        diff_id: String,
        file: String,
        additions: u32,
        deletions: u32,
    },
    Validation {
        session_id: String,
        command: String,
        status: ValidationStatus,
    },
    PermissionPrompt {
        session_id: String,
        prompt_id: String,
        title: String,
        body: String,
        risk: RiskLevel,
    },
    ContextPressure {
        session_id: String,
        used_tokens: u32,
        max_tokens: u32,
        pressure: f32,
    },
    Checkpoint {
        session_id: String,
        checkpoint_id: String,
        summary: String,
    },
    ProofReceipt {
        session_id: String,
        receipt_id: String,
    },
    RunFinished {
        session_id: String,
        success: bool,
        summary: String,
    },
    Error {
        session_id: Option<String>,
        message: String,
    },
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum ValidationStatus {
    Running,
    Passed,
    Failed,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum RiskLevel {
    Low,
    Medium,
    High,
}
```

```rust
// apps/desktop/src-tauri/src/event_bridge.rs

use quorp_agent_core::runtime::{RuntimeEvent, RuntimeEventSink};
use quorp_desktop_api::DesktopEvent;
use tauri::ipc::Channel;

pub struct DesktopEventSink {
    session_id: String,
    channel: Channel<DesktopEvent>,
}

impl DesktopEventSink {
    pub fn new(session_id: String, channel: Channel<DesktopEvent>) -> Self {
        Self { session_id, channel }
    }

    fn send(&self, event: DesktopEvent) {
        let _ = self.channel.send(event);
    }
}

impl RuntimeEventSink for DesktopEventSink {
    fn on_event(&self, event: RuntimeEvent) {
        match event {
            RuntimeEvent::AssistantDelta { text, .. } => {
                self.send(DesktopEvent::AssistantDelta {
                    session_id: self.session_id.clone(),
                    text,
                });
            }

            RuntimeEvent::CheckpointCreated { checkpoint, .. } => {
                self.send(DesktopEvent::Checkpoint {
                    session_id: self.session_id.clone(),
                    checkpoint_id: checkpoint.id.to_string(),
                    summary: checkpoint.summary,
                });
            }

            other => {
                // Start conservative. Add typed mappings as needed.
                self.send(DesktopEvent::ToolCallFinished {
                    session_id: self.session_id.clone(),
                    call_id: "runtime".into(),
                    ok: true,
                    summary: format!("{other:?}"),
                });
            }
        }
    }
}
```

The exact enum fields may differ, but the target design is correct: **one runtime event sink translates Rust runtime events into normalized desktop events**.

## Tauri command

```rust
// apps/desktop/src-tauri/src/commands.rs

use tauri::{ipc::Channel, State};
use uuid::Uuid;

use quorp_desktop_api::{DesktopEvent, DesktopRunRequest};

use crate::{
    desktop_state::{DesktopState, SessionId},
    event_bridge::DesktopEventSink,
    session_host,
};

#[tauri::command]
pub async fn start_agent_run(
    state: State<'_, DesktopState>,
    request: DesktopRunRequest,
    events: Channel<DesktopEvent>,
) -> Result<String, String> {
    let session_id = SessionId(Uuid::new_v4());
    let session_id_string = session_id.0.to_string();

    events
        .send(DesktopEvent::RunStarted {
            session_id: session_id_string.clone(),
        })
        .map_err(|err| err.to_string())?;

    let sink = DesktopEventSink::new(session_id_string.clone(), events);

    let (cancel_tx, cancel_rx) = tokio::sync::watch::channel(false);

    state.insert_session(session_id, request.workspace_path.clone(), cancel_tx)?;

    tauri::async_runtime::spawn(async move {
        let result = session_host::run_desktop_session(request, sink, cancel_rx).await;

        // In production, send final event through sink/channel.
        if let Err(err) = result {
            tracing::error!(?err, "desktop agent run failed");
        }
    });

    Ok(session_id_string)
}
```

# 11. Virtualized mission feed

This is critical for memory use. Do **not** render thousands of events normally.

```tsx
// apps/desktop/src/components/MissionFeed.tsx

import { useRef } from "react";
import { useVirtualizer } from "@tanstack/react-virtual";
import { useSessionStore } from "../state/sessionStore";
import { FeedCard } from "./FeedCard";

export function MissionFeed({ sessionId }: { sessionId?: string | null }) {
  const parentRef = useRef<HTMLDivElement | null>(null);

  const cards = useSessionStore((s) =>
    sessionId ? s.sessions[sessionId]?.cards ?? [] : [],
  );

  const virtualizer = useVirtualizer({
    count: cards.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 88,
    overscan: 12,
  });

  return (
    <div ref={parentRef} className="mission-feed">
      <div
        style={{
          height: `${virtualizer.getTotalSize()}px`,
          position: "relative",
        }}
      >
        {virtualizer.getVirtualItems().map((item) => {
          const card = cards[item.index];
          return (
            <div
              key={card.id}
              data-index={item.index}
              ref={virtualizer.measureElement}
              className="mission-feed-row"
              style={{
                transform: `translateY(${item.start}px)`,
              }}
            >
              <FeedCard card={card} />
            </div>
          );
        })}
      </div>
    </div>
  );
}
```

The desktop should keep only compact event/card state in React. Full shell output, full transcript JSON, raw diffs, and proof bundles should stay on the Rust side and be fetched lazily by ID.

# 12. State reducer

```ts
// apps/desktop/src/state/sessionStore.ts

import { create } from "zustand";
import type { DesktopEvent } from "../ipc/events";

type FeedCard = {
  id: string;
  kind: string;
  sessionId: string;
  title?: string;
  text?: string;
  meta?: Record<string, unknown>;
};

type SessionState = {
  id: string;
  cards: FeedCard[];
  status: {
    running: boolean;
    permissionMode: string;
    sandboxMode: string;
    contextUsed?: number;
    contextMax?: number;
    costUsd?: number;
    provider?: string;
    model?: string;
  };
};

type Store = {
  activeSessionId?: string;
  sessions: Record<string, SessionState>;
  ingestEvent: (event: DesktopEvent) => void;
};

export const useSessionStore = create<Store>((set) => ({
  sessions: {},

  ingestEvent: (event) =>
    set((state) => {
      const sessionId =
        "session_id" in event && event.session_id
          ? event.session_id
          : state.activeSessionId;

      if (!sessionId) return state;

      const session =
        state.sessions[sessionId] ??
        ({
          id: sessionId,
          cards: [],
          status: {
            running: false,
            permissionMode: "ask",
            sandboxMode: "tmp_copy",
          },
        } satisfies SessionState);

      const next = reduceEvent(session, event);

      return {
        activeSessionId: state.activeSessionId ?? sessionId,
        sessions: {
          ...state.sessions,
          [sessionId]: next,
        },
      };
    }),
}));

function reduceEvent(session: SessionState, event: DesktopEvent): SessionState {
  switch (event.type) {
    case "run_started":
      return {
        ...session,
        status: { ...session.status, running: true },
        cards: [
          ...session.cards,
          {
            id: crypto.randomUUID(),
            kind: "run_started",
            sessionId: event.session_id,
            title: "Run started",
          },
        ],
      };

    case "assistant_delta":
      return appendCard(session, {
        kind: "assistant_delta",
        text: event.text,
      });

    case "diff_ready":
      return appendCard(session, {
        kind: "diff",
        title: event.file,
        meta: {
          diffId: event.diff_id,
          additions: event.additions,
          deletions: event.deletions,
        },
      });

    case "context_pressure":
      return {
        ...session,
        status: {
          ...session.status,
          contextUsed: event.used_tokens,
          contextMax: event.max_tokens,
        },
      };

    case "run_finished":
      return {
        ...appendCard(session, {
          kind: "run_finished",
          title: event.success ? "Run finished" : "Run failed",
          text: event.summary,
        }),
        status: { ...session.status, running: false },
      };

    default:
      return appendCard(session, {
        kind: event.type,
        text: JSON.stringify event),
      });
  }
}

function appendCard(
  session: SessionState,
  partial: Omit<FeedCard, "id" | "sessionId">,
): SessionState {
  return {
    ...session,
    cards: [
      ...session.cards,
      {
        id: crypto.randomUUID(),
        sessionId: session.id,
        ...partial,
      },
    ],
  };
}
```

There is a small typo in the default branch above; it should be:

```ts
text: JSON.stringify(event)
```

I left that visible because this is the kind of reducer that should get a simple test immediately.

# 13. Performance and memory rules

The desktop app should feel instant even during long autonomous runs.

## Hard rules

```text
1. React never owns raw transcript.json.
2. React never owns full events.jsonl.
3. React never owns giant shell output buffers.
4. React never owns all diffs eagerly.
5. Rust owns durable artifacts.
6. UI owns visible summaries and fetches details by ID.
7. Event streaming is batched.
8. The feed is virtualized.
9. Diffs/code editors are lazy-mounted.
10. Shell output cards are capped and expandable.
```

## Event batching

On the Rust side, batch high-frequency token/output chunks into 16–33ms windows.

```rust
pub struct EventBatcher {
    pending: Vec<DesktopEvent>,
    last_flush: std::time::Instant,
}

impl EventBatcher {
    pub fn push(&mut self, event: DesktopEvent) -> Option<Vec<DesktopEvent>> {
        self.pending.push(event);

        if self.pending.len() >= 64 || self.last_flush.elapsed().as_millis() >= 33 {
            self.last_flush = std::time::Instant::now();
            Some(std::mem::take(&mut self.pending))
        } else {
            None
        }
    }
}
```

Then send either single events or a `DesktopEvent::Batch { events }`.

## Lazy payload model

```text
DesktopEvent::DiffReady { diff_id, file, additions, deletions }
  UI card appears immediately
  User opens card
  UI calls read_diff(diff_id)
  Rust returns diff text or structured hunks

DesktopEvent::ShellOutputReady { output_id, command, preview }
  UI card appears immediately
  User expands output
  UI calls read_shell_output(output_id, offset, limit)
```

This is how you get a beautiful UI without turning the WebView into a memory sink.

# 14. Security model for desktop

The desktop app must be safer than the CLI by default, not more dangerous.

Quorp already has permission and sandbox concepts, including read-only, ask, accept-edits, auto-safe, yolo-sandbox, and sandbox modes such as host/tmp-copy/git-worktree semantics in the runtime documentation. ([GitHub][9])

Desktop defaults should be:

```text
New folder:
  Trust: Untrusted
  Permission: Ask
  Sandbox: Tmp Copy
  Network: Off unless requested
  Dependency installs: Ask
  MCP: Off until configured
  Browser: Ask
  Generated executables: Ask / sandbox only

Full Auto:
  Allowed only with Tmp Copy or Git Worktree sandbox
  Never host by default
```

Permission prompt design:

```text
Quorp wants to run:

cargo test -p quorp_agent_core

cwd:
~/Code/Quorp

Sandbox:
git-worktree /var/folders/.../quorp-run-abc123

Risk:
Medium — command executes project tests and build scripts.

[Allow once] [Allow for this session] [Deny] [Inspect]
```

Never expose a broad frontend shell API. The frontend should not be able to run arbitrary commands. It should only call typed Rust commands, and the Rust side should enforce Quorp’s permission model.

# 21. Settings model

Desktop settings should be project-aware.

```text
Global settings:
  provider configs
  model defaults
  UI theme
  telemetry preference
  update channel
  CLI install path

Workspace settings:
  trusted/untrusted
  default sandbox mode
  default permission mode
  allowed commands
  network policy
  dependency install policy
  MCP policy
  browser policy
  context include/exclude patterns

Session settings:
  current objective
  model override
  max steps
  max seconds
  max tokens
  autonomy profile
```

Storage:

```text
~/Library/Application Support/Quorp/settings.json
~/Library/Application Support/Quorp/workspaces.json
~/Library/Application Support/Quorp/sessions/<session-id>/
  events.jsonl
  transcript.json
  proof-receipt.json
  diffs/
  shell-output/
```

For API keys, do not store plaintext JSON long-term. Use macOS Keychain or a secure Tauri-compatible secret storage path. For the MVP, reading existing environment variables is acceptable, but the desktop PATH/env issue means you will want UI-managed provider settings quickly.

# 23. What makes this better than a basic Tauri app

The breakthrough is not “Quorp has a GUI.”

The breakthrough is:

```text
A desktop mission-control surface over a ledgered, permissioned, replayable,
Patch-VM-backed coding agent runtime.
```

That means the desktop app should make these things visible:

```text
What is the agent trying to do?
What files matter?
What changed?
What validation has passed?
What validation failed?
What context is loaded?
What context is stale?
What permission boundary is active?
What sandbox is active?
What checkpoint can I roll back to?
What proof receipt exists?
What did the agent learn from the mistake?
```

The terminal CLI can remain power-user fast. The desktop app should become the high-resolution control room.

# 24. Final build recommendation

Build this:

```text
/apps/desktop
  Tauri 2 desktop app
  React + TypeScript + Vite frontend
  Tauri channels for runtime event streaming
  Tauri commands for typed actions
  Rust runtime linked directly through quorp_session/quorp_agent_core
  DMG target through Tauri bundler
```

Do **not** start with:

```text
Electron
Swift rewrite
CLI subprocess wrapper as the main architecture
WebSocket local server
Monaco-heavy IDE clone
unrestricted Tauri shell plugin
host full-auto by default
```

The first usable milestone should be:

```text
Open Quorp.app
Add Folder → select repo
Type task in bottom composer
Choose Plan / Act / Full Auto
Choose Ask / Auto Safe / Yolo Sandbox
Watch runtime events, tool calls, diffs, validation, and proof cards stream above
Build a local unsigned/ad-hoc DMG
```

That is achievable quickly because Quorp already has the hard Rust-side primitives: reusable session crates, typed runtime actions, event fanout, permission/sandbox concepts, Patch VM receipts, and verification/proof infrastructure. The desktop app should expose those strengths rather than reinvent them.
