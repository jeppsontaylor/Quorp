I couldn’t clone the repo locally because this environment cannot resolve `github.com`, but I reviewed the public repo, raw files, docs, handoff notes, and representative core crates through GitHub/web sources. The strongest takeaway: **Quorp already has unusually good bones for a Rust-native “proof-carrying coding agent,” but it needs a major jump in orchestration, durable memory, real retrieval, verification depth, permissions, sandboxing, and TUI polish to become truly world-class.**

## Executive verdict

Quorp’s best existing advantage is not “chat in a terminal.” It is the emerging combination of **Rust-native runtime + typed tools + Patch VM receipts + proof packets + verification + memory/rule scaffolding + event logs**. That is a better foundation than trying to clone Claude Code, Codex CLI, OpenCode, Goose, or Crush. The winning product direction should be:

> **A Rust-native, proof-carrying coding agent that can run for hours, preserve state, explain every side effect, verify every patch, learn from failures, and resume/replay any run deterministically.**

The current repo is already pointed this way: the README describes Quorp as a “TUI-first text editor and agentic interface,” in transition from a Zed-derived backend toward a standalone terminal-first agent runtime; the handoff says the tree now includes CLI/session crates, typed runtime actions, context hooks, Patch VM receipts, verification reports/proof packets, and benchmark gates. The agent support docs also describe stream-first CLI, headless autonomous runs, typed tools, git-worktree/tmp-copy sandboxing, permission primitives, Patch VM writes, proof packets, memory/rule scaffolding, and benchmark scoreboards. ([GitHub][1])

The biggest issue: several raw source files appear compressed into extremely long physical lines, including `Cargo.toml`, CLI, repo graph, Rust intelligence, MCP, and LSP files. Even if this compiles, it is a serious maintainability and diff-quality problem for a tool that wants “incredible code layout.” A world-class Rust agent should never optimize a LOC cap by minifying Rust; it should split modules semantically and keep `rustfmt`-normal, readable source. ([GitHub][2])

## What Quorp already has that is genuinely strong

Quorp has a much stronger architectural direction than a normal prototype. The handoff describes a runtime with `ExpandContext`, `RecallMemory`, and `ProposeRule`; bounded runtime event fanout/backpressure; Patch VM receipts for writes; `RunValidation` mapped to `quorp_verify`; and regression coverage. The same handoff identifies remaining work as durable memory/rule subscriber workers, deeper verify executors, broader replay/proof CLI, and benchmark/runtime polish. That is exactly the right problem list. ([GitHub][3])

The crate split is also promising: `quorp_agent_core`, `quorp_context`, `quorp_memory`, `quorp_rule_forge`, `quorp_patch_vm`, `quorp_verify`, `quorp_repo_graph`, `quorp_repo_scan`, `quorp_rust_intel`, `quorp_storage`, `quorp_permissions`, `quorp_plan_mode`, `quorp_slash`, `quorp_render`, `quorp_provider`, `quorp_sandbox`, `quorp_tools`, and `quorp_mcp` are all present in the workspace. That means Quorp is not merely a wrapper around an LLM; it is already becoming an agent operating system. ([GitHub][3])

The current agent runtime already models long-run realities: it has max iterations, verifier drain budget, parser recovery budget, max total tokens, max seconds, autonomy profiles, rollback-on-validation-failure, cancellation, checkpoint events, first-token timeout, stream-idle timeout, model-request timeout, stalled stop reasons, and a detailed repair/scorecard structure. That is a serious advantage. ([GitHub][4])

The provider default is also aligned with your goal: Quorp’s core defaults point at NVIDIA/OpenAI-compatible Qwen3-Coder-480B-A35B-Instruct, and the provider crate uses an OpenAI-compatible chat-completions path. ([GitHub][5])

## Highest-priority fixes before feature expansion

### 1. Fix repo hygiene and source layout immediately

The raw repo currently looks like it violates the spirit of its own “incredible code layout” goal. The `.rules` file emphasizes correctness, clarity, no panic/unwrap/indexing, no fallible `let _ =`, no `mod.rs`, and disciplined rule hygiene, but much of the raw source appears packed into huge lines. This will hurt code review, agent editing, blame, merge conflict resolution, and context chunking. ([GitHub][6])

I would change the codebase rule from “2,000 LOC cap” to a more precise standard:

```text
Rust source policy:
- rustfmt-normal source only; no minified physical lines.
- Max 600–900 meaningful lines per file unless explicitly waived.
- One domain concept per module.
- Split by behavior, not by artificial LOC quotas.
- CI rejects files with average line length > 140 or max line length > 300 except generated fixtures.
- Generated or compressed artifacts must live under generated-zones.toml and be ignored by the agent unless explicitly requested.
```

This matters more than it sounds. Coding agents are extremely sensitive to source shape. Clean module boundaries and normal formatting make semantic retrieval, patch generation, and review dramatically better.

### 2. Resolve the license inconsistency

The README says GPL-3.0-or-later, package manifests show GPL-3.0-or-later, but the GitHub repo metadata displays MIT. That should be fixed before building community momentum, shipping binaries, or integrating with commercial users. ([GitHub][1])

### 3. Turn the event system into a durable replay kernel

The runtime already has named bounded fanout subscribers for memory, rule forge, renderer, proof recorder, and benchmarks. Bounded queues are fine for UI rendering, but **not** for proof, memory, or rule learning. Those subscribers must not lose critical events under pressure. ([GitHub][4])

Make the event log the center of the architecture:

```text
quorp run kernel
  -> append-only event log, SQLite/WAL + JSONL mirror
  -> deterministic event ids
  -> run/session/task/attempt ids
  -> every tool call has request, decision, result, duration, digest
  -> every edit has Patch VM receipt
  -> every validation has proof packet
  -> every model call has provider/model/profile/token/budget metadata
  -> every learned memory/rule has evidence hashes
```

Then every surface—TUI, headless CLI, benchmark runner, memory writer, proof recorder, rule forge—subscribes from the same durable log. UI subscribers can drop frames. Proof and memory subscribers should checkpoint offsets and replay missed events.

This unlocks the killer feature: **`quorp replay <run>`** that can reconstruct the entire agent trajectory, show why it acted, verify what it touched, and resume from a checkpoint.

## Model strategy for Qwen3-Coder and beyond

Qwen3-Coder-480B-A35B-Instruct is a very reasonable default. Qwen describes it as a 480B-parameter MoE with 35B active parameters, 256K native context, up to 1M with extrapolation, and strong coding/agentic performance; Qwen also says it trained with execution-driven Code RL and long-horizon Agent RL involving planning, tool use, feedback, and decisions. ([Qwen][13])

The Hugging Face model card adds an important implementation detail: Qwen3-Coder supports only non-thinking mode and does not emit `<think>` blocks. That means Quorp should not build behavior around hidden/visible chain-of-thought parsing for Qwen. It should instead use explicit structured plans, tool calls, state packets, verification feedback, and reflection summaries. ([Hugging Face][14])

NVIDIA’s NIM docs confirm Qwen3-Coder’s use for code generation, agentic coding, repository understanding, tool integration, code review, docs, and function calling/tool choice. So the provider profile should be Qwen-specific: strict JSON/tool-call repair, no `<think>` assumptions, long-context budget presets, and a tool schema tuned to Qwen’s function-call format. ([NVIDIA Docs][15])

Also prepare for model portfolios. Qwen3-Coder-Next’s technical report describes an 80B total / 3B active coding-agent model trained with executable environments and environment feedback for multi-step code editing, tool usage, and fault recovery. That suggests Quorp should support “small fast local scout + huge remote implementer + strong reviewer” workflows instead of one-model-only thinking. ([arXiv][16])

## Context management: this is where Quorp can leap ahead

The current `quorp_context` already describes token-budgeted `ContextPack`s, multi-source retrieval, knapsack selection, progressive disclosure via handles, memory recall, workspace contracts, anchors/file excerpts, lexical search, budgeted items, and handles. That is the right API shape. But today it still looks like deterministic/mock and lexical-heavy scaffolding rather than a fully mature context compiler. ([GitHub][17])

Build **Context Compiler v2** as a real subsystem:

```text
Inputs:
- user request
- current task graph node
- repo graph
- LSP symbols
- AST/tree-sitter chunks
- lexical/BM25 hits
- vector hits
- recent files
- changed files
- failing diagnostics
- failing tests
- owner/test/proof contracts
- memory/rules
- prior attempts and negative memories

Outputs:
- state packet
- minimal plan context
- ranked code excerpts
- handles for overflow content
- missing-context questions for the agent
- “do not load yet” deferred context
- citations/evidence ids for every included fact
```

The key design: **never stuff the transcript. Compile context fresh each turn.** Keep the conversation short; keep the state rich.

Use these tiers:

```text
Tier 0: Run state packet
- objective
- current plan
- changed files
- last verifier result
- known constraints
- active permissions
- remaining budget

Tier 1: Hot code
- exact functions/classes/modules being edited
- failing tests and diagnostics
- direct callers/callees
- user-mentioned files

Tier 2: Repo intelligence
- owner map
- test map
- dependency graph
- generated zones
- unsafe ledger
- proof lanes

Tier 3: Long-term memory/rules
- only scoped, evidence-backed, high-confidence memories
- negative memories relevant to current failure signatures

Tier 4: Handles
- huge files, logs, docs, full test outputs, prior trajectories
- load on demand by handle id
```

Also replace the rough token heuristic with model-aware token estimation. The current context code uses a simple bytes/token heuristic. That is okay for tests, but not for 256K/1M-token production runs. ([GitHub][17])

## Memory that learns from mistakes without becoming slow or noisy

Quorp already has the right conceptual memory tiers: Working, Episodic, Semantic, Procedural, Negative, and Rule. It also has SQLite durability, WAL, FTS5 fallback behavior, failed-attempt recording, and retry-decision logic that can block repeated failed fixes. ([GitHub][18])

Turn that into a proper **Mistake Memory + Rule Forge** pipeline:

```text
1. Observe failure
   - compiler/test/security/verifier error
   - patch receipt
   - attempted fix summary
   - affected files/symbols
   - command output digest

2. Fingerprint failure
   - normalized diagnostic
   - stack/test name
   - symbol/path scope
   - patch-intent hash
   - model/tool profile

3. Decide retry policy
   - same failure + same attempted fix => block/replan
   - same failure + new evidence => allow
   - new failure => continue

4. Reflect
   - produce short procedural lesson
   - never store raw giant logs
   - attach evidence hashes and expiry

5. Promote
   Candidate -> Draft -> Shadow -> Verified -> Active

6. Persist
   - personal memory in local DB
   - project rules as lightweight git-checkable TOML/Markdown
```

Reflexion supports this direction: it improves agents without weight updates by having them reflect on feedback and store reflective text in episodic memory. Self-Refine similarly shows test-time feedback/refinement loops can improve outputs without additional training. MemGPT supports the broader memory model: tiered memory and virtual context management rather than trying to place everything in the prompt. ([arXiv][19])

The important guardrail: **do not let memory become vibes.** Every memory should be scoped, evidence-backed, ranked, and expirable.

Suggested git-checkable rule object:

```toml
# .quorp/rules/borrow-move-before-async.toml
id = "rule_2026_04_28_borrow_move_before_async"
status = "shadow" # candidate | draft | shadow | verified | active
scope = { language = "rust", crates = ["quorp_agent_core"] }
trigger = { diagnostic_codes = ["E0382"], path_globs = ["crates/**/*.rs"] }
summary = "When moving a value into an async task, clone or Arc-wrap it before the move if later code still needs it."
avoid = [
  "Do not blindly clone large structures in hot paths.",
  "Do not silence the error by widening lifetimes."
]
preferred_fixes = [
  "Use Arc<T> when shared ownership is intentional.",
  "Narrow the moved value scope when possible.",
  "Clone small config values before spawn."
]
evidence = [
  { run_id = "...", event_id = "...", patch_hash = "...", verifier = "cargo check" }
]
confidence = 0.72
last_seen = "2026-04-28"
expires_after_days = 180
```

Rule injection should be lazy: include a rule only when its trigger matches the current task context. Claude Code’s skills docs make the same product point: long procedural content should cost almost nothing until needed. ([Claude API Docs][20])

## Rule Forge: make it rigorous

The current `quorp_rule_forge` already clusters failure signals into draft rules and has a Candidate/Draft/Verified/Active lifecycle, but the comment indicates the shadow harness is not fully landed. ([GitHub][21])

I would make Rule Forge one of Quorp’s flagship features:

```text
Rule Forge stages:
1. Candidate
   - generated from repeated failure cluster
   - not shown to model by default

2. Draft
   - human-readable rule created
   - evidence attached
   - scope inferred

3. Shadow
   - rule is evaluated silently
   - Quorp records: would this rule have helped?
   - no model behavior change yet

4. Verified
   - passed N shadow opportunities
   - no known harmful false positives

5. Active
   - injected only when scoped trigger fires
   - visible in /rules and final proof receipt
```

Quorp’s `.rules` file already says rules should be high-signal, non-obvious, repeated, specific, and not drive-by additions. Keep that philosophy. ([GitHub][6])

## Verification: go far beyond “tests passed”

The current `quorp_verify` has a staged verification ladder and content-addressed result cache, but the source comment says the API is still minimal and deeper cargo/rustfmt/miri executors are future work. ([GitHub][22])

World-class Quorp needs a **validation planner**, not just a shell command runner:

```text
L0 — cheap static checks
- format check
- parse check
- changed-file syntax
- generated-zone violations
- secret scan
- permission diff

L1 — targeted checks
- changed package cargo check / npm typecheck / pytest selected
- tests mapped from test-map
- LSP diagnostics for touched files

L2 — affected graph checks
- reverse dependency tests
- caller/callee packages
- owner-required lanes

L3 — full project checks
- full test suite
- full clippy/lint/typecheck
- doc tests

L4 — deep / optional
- miri, sanitizers, mutation tests
- security scanners
- benchmark gates
- containerized reproducibility
```

The research strongly supports this. SWE-agent showed that agent-computer interface design materially improves an agent’s ability to navigate, edit, and test code. UTBoost found that SWE-bench tests can be insufficient, allowing patches to pass without fixing the issue, and identified hundreds of erroneous patches labeled as passing. METR found roughly half of test-passing SWE-bench Verified PRs from mid-2024 to mid/late-2025 agents would not be merged by maintainers. ([arXiv][23])

So Quorp should not say “success” merely because one command passed. It should say:

```text
Success level:
- Verified targeted fix
- Full suite clean
- Maintainer-quality review clean
- Residual risks: ...
- Unverified assumptions: ...
```

## Agent loop: simple core, powerful tools

Do not overcomplicate the core loop. The mini-SWE-agent project is a useful warning: a very small agent loop can perform surprisingly well when the environment and tools are good. Its README emphasizes radical simplicity and strong SWE-bench Verified performance. ([GitHub][25])

Make the tools excellent:

## Permissions and security

Quorp already has permission primitives such as read-only, ask, accept-edits, auto-safe, and yolo-sandbox; the docs say `--yolo` is rejected with direct host mode. That is a good start. ([GitHub][26])

But to beat OpenCode/Codex/Claude, make permissions granular and auditable:

```toml
[permissions]
default = "ask"

[permissions.read]
"*" = "allow"
".env*" = "deny"
"**/secrets/**" = "deny"

[permissions.edit]
"src/**" = "ask"
"tests/**" = "ask"
"Cargo.toml" = "ask"
".github/workflows/**" = "deny"

[permissions.shell]
"git status*" = "allow"
"git diff*" = "allow"
"cargo check*" = "allow"
"cargo test*" = "ask"
"curl *" = "deny"
"rm -rf *" = "deny"

[permissions.network]
default = "deny"
"crates.io" = "ask"
"github.com" = "ask"

[permissions.mcp]
default = "ask"
trusted_servers = ["filesystem-local"]
```

Sandboxing should be real OS/process enforcement, not only policy checks. OpenAI’s Codex docs correctly separate sandbox boundaries from approval policy, and state that spawned commands inherit sandbox limits. That is the bar. ([OpenAI Developers][27])

Recommended sandbox model:

```text
macOS:
- seatbelt sandbox profile if feasible
- fallback: Docker/Colima/Podman
- no host network by default
- workspace-only read/write

Linux:
- bubblewrap/firejail/user namespaces where available
- Docker/Podman fallback
- seccomp, rlimits, read-only mounts

All platforms:
- git worktree or tmp-copy isolation
- no secrets mounted by default
- explicit network allowlist
- symlink-safe canonicalization
- command allowlist/denylist
- environment variable redaction
- proof receipt records effective sandbox
```

MCP needs special caution. Claude’s MCP docs warn that third-party MCP servers may expose prompt-injection risk, especially if they fetch untrusted content. Quorp should treat MCP servers as untrusted plugins unless explicitly managed. ([Claude API Docs][28])

```text
Feature Matrix Runner
- run checks across feature combinations
- cache successful lanes

- cargo semver-checks integration
- detect breaking API changes
```

Make these tools available to the model as structured diagnostics, not as giant logs.

Architecture:

```text
quorp daemon / app-server

  -> IDE/ACP client
```

## The top 10 changes I would make first

1. **Reformat and split the codebase** so it is genuinely readable, rustfmt-normal, and agent-editable.
2. **Make the event log durable and replayable**; proof/memory/rule subscribers must never silently drop important events.
3. **Upgrade Context Compiler into the core intelligence layer** with hybrid retrieval, LSP/AST/repo graph, test/owner/proof maps, and handles.
4. **Build Mistake Memory as evidence-backed negative/procedural memory**, not vague chat summaries.
5. **Finish Rule Forge shadow validation** and store lightweight project rules under `.quorp/rules/*.toml`.
6. **Replace shell-only validation with a verification planner** that chooses targeted/full/deep checks and emits proof packets.
7. **Implement a granular permission DSL** for tools, paths, commands, network, env, MCP, and external dirs.
8. **Harden sandboxing** with worktree/tmp-copy/container modes, network deny by default, symlink-safe paths, and secret isolation.
9. **Make the TUI feel like Claude/Codex/OpenCode but with proof panels**: plan/build toggle, diff review, task graph, context meter, memory/rule viewer, replay.
10. **Use Qwen3-Coder as a model profile, not just a model string**: non-thinking mode, Qwen tool protocol, long-context budgets, structured-turn repair, and strong verifier feedback loops.

## Final product north star

The product should feel magical, but its internals should be boringly reliable:

```text
Fast Rust core.
Typed tools.
No mystery writes.
No lost state.
No unverified success.
No repeated failed fixes.
No giant context soup.
No unsafe host execution by default.
Every run replayable.
Every edit receipted.
Every learned rule evidence-backed.
```

That is the path where Quorp can become more than “another vibe coding CLI.” It can become the **trustworthy, Rust-native, proof-carrying coding agent for serious long-horizon software engineering**.
