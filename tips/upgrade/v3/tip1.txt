I attempted a fresh clone in the sandbox, but DNS resolution for GitHub failed, so I used the GitHub web/raw views and the archived run artifacts. My strongest read is this: **Quorp already has unusually good raw ingredients**—typed tools, Patch VM receipts, proof packets, context packs, negative memory, rule-forge scaffolding, sandbox modes, a ledger, and benchmark scoreboards—but the next leap is to make those ingredients into a **ledger-native agent OS** rather than a prompt-driven coding loop. Quorp’s 5/5 Rust SWE-bench run is useful evidence, but not a final proof of general capability: the archive reports 5/5 success, 28 requests, 180,546 billed tokens, and ~696 seconds wall clock, while also showing delayed first writes, parser recovery, watchdog pressure, and “compacted_prompt_est=n/a.” ([GitHub][1])

The most important caution: the current benchmark success includes deterministic benchmark repair paths. `benchmark_playbooks.rs` contains exact playbook variants for Cargo Dist, cc-rs, Axum, Chrono, and Bincode, and includes functions named around exact benchmark source patches; the cargo-dist and cc-rs traces show “deterministic benchmark Case 04/05 source patch” after parser recovery. ([GitHub][2]) That should not be treated as cheating if it is honestly labeled as benchmark-controller evidence, but it **must be quarantined** before Quorp claims it beats general-purpose systems.

Below are my top 10 ranked ideas.

## 1. Ledger-native Context OS with automatic compaction and `/status` pressure telemetry

**Why this is #1:** This is the highest-leverage upgrade for long-form coding reliability and token reduction. Quorp already has context packs, typed runtime events, a durable ledger, proof packets, and memory/rule scaffolding; the missing product is a **state kernel** that turns a growing transcript into a compact, auditable, replayable state packet. Codex’s own engineering notes emphasize prompt cache stability and compaction once token thresholds are exceeded, and Quorp has a chance to do this more transparently by making compaction ledger/proof backed instead of opaque. ([OpenAI][3])

**Evidence from Quorp:** The run archive reports `compacted_prompt_est=n/a` across the 5/5 run, while prompt estimates grew across multi-turn cases and first writes were delayed to turn 7 for bincode and turn 9 for axum. ([GitHub][1]) Quorp’s phase tracker says Context OS exists with typed context items, duplicate suppression, handle spillover, owner/test/proof context, and token-budget tests; the workplan still marks persistent context/code intelligence and broader context policy as follow-up. ([GitHub][4])

**The idea:** Every turn should produce a `ContextPressure` measurement and, at thresholds, automatically compact transcript/tool history into a `TaskStatePacket` that preserves only the facts required to continue safely: objective, constraints, decisions, open tasks, failed attempts, touched files, validation state, proof refs, permission boundaries, current hypotheses, and rule IDs. Raw tool output remains in the ledger/artifacts and is re-expanded by handle only when needed.

**Files to add/change:**

* Add `crates/quorp_agent_core/src/runtime/context_budget.rs`
* Add `crates/quorp_context_model/src/state_packet.rs`
* Add `crates/quorp_context/src/state_packet_compiler.rs`
* Extend `crates/quorp_agent_core/src/runtime.rs` with `RuntimeEvent::ContextPressureMeasured` and `RuntimeEvent::ContextCompacted`
* Extend `crates/quorp_slash` with `/status`, `/context`, `/compact`
* Extend `crates/quorp_render` with context pressure/status panes
* Store artifacts under:

  * `run_dir/artifacts/context/state-packet.latest.json`
  * `run_dir/artifacts/context/compactions/*.json`
  * `.quorp/context/packs/*.json`

**Runtime policy:**

```rust
pub fn maybe_compact_context(
    state: &mut AgentTaskState,
    pressure: &ContextPressure,
    ledger: &RunLedgerReader,
    compiler: &StatePacketCompiler,
) -> anyhow::Result<Option<TaskStatePacket>> {
    let should_compact = matches!(
        pressure.pressure_band,
        PressureBand::Orange | PressureBand::Red | PressureBand::Emergency
    );

    if !should_compact {
        return Ok(None);
    }

    let packet = compiler.compile(StatePacketInput {
        run_id: state.run_id.clone(),
        through_seq: ledger.last_seq()?,
        objective: state.objective_snapshot(),
        task_dag: state.task_dag_snapshot(),
        proof_state: state.validation_snapshot(),
        repair_memory: state.agent_repair_memory.clone(),
        security: state.security_boundary_snapshot(),
    })?;

    state.current_state_packet = Some(packet.clone());
    state.compaction_count += 1;

    Ok(Some(packet))
}
```

* Token gate: on the 9-turn bincode/axum cases, prompt estimate after compaction should be at least 30% lower than un-compacted history while retaining success.

## 2. Handle-based Observation Packets: stop feeding raw logs and raw reads back into the model

**Why this is #2:** This is the most practical token reducer. Instead of sending bulky raw tool outputs into the prompt, every tool should return a compact `ObservationPacket`: summary, entities, relevant excerpts, omitted byte counts, raw artifact hash, and an expansion handle. The model sees the structured observation, not the entire file/log. It can call `ExpandContext` for more.

**Evidence:** Quorp already has typed actions like `ReadFile`, `RunValidation`, `ExpandContext`, `SuggestImplementationTargets`, `SuggestEditAnchors`, `RecallMemory`, and `ProposeRule`. ([GitHub][5]) SWE-agent’s Agent-Computer Interface work showed that custom file viewers and concise feedback formats materially improve agent performance; this is the same principle, generalized into a Rust-native observation protocol. ([Swe Agent][6])

**The idea:** Make all tools return this envelope:

## 3. Quarantine benchmark playbooks into an honest Repair Skill system

**Why this is #3:** Quorp must separate **general capability** from **benchmark-controller assistance**. The current deterministic playbooks are valuable as diagnostics, but dangerous as production runtime behavior and benchmark optics. Recent benchmark discussions emphasize contamination, overfitting, and harness-level answer-key leakage as real risks; Quorp can become more credible than competitors by building anti-overfit accounting directly into the runtime. **Evidence:** `benchmark_playbooks.rs` contains exact benchmark repair functions and benchmark-specific injection logic gated on `PolicyMode::BenchmarkAutonomous`. The cargo-dist and cc-rs traces show deterministic controller patches after parser recovery. ([GitHub][2])

**The idea:** Move exact benchmark patches out of agent core and into a benchmark-only oracle module. Replace production-facing playbooks with **Repair Skills** that encode reusable strategies, not answer patches.

```rust
pub fn authorize_skill(
    skill: &RepairSkillManifest,
    ctx: &SkillInvocationContext,
) -> anyhow::Result<()> {
    if matches!(skill.kind, SkillKind::OracleBenchmarkOnly)
        && !ctx.policy.allow_benchmark_oracles
    {
        anyhow::bail!("oracle benchmark skill is disabled outside benchmark oracle mode");
    }

    if skill.oracle_free && ctx.evidence.contains_hidden_oracle_material() {
        anyhow::bail!("oracle-free skill attempted to consume hidden benchmark material");
    }

    if ctx.mode == RuntimeMode::Production && skill.touches_exact_benchmark_patch() {
        anyhow::bail!("production runtime cannot invoke exact benchmark patch skill");
    }

    Ok(())
}
```

## 4. PatchLens: hypothesis → evidence → patch gate for every write

**Why this is #4:** Quorp needs fewer bad edits and fewer “read around until rescued” loops. In the bincode run, the model headed toward adding `chrono`/`uuid` dependencies before the controller injected a more precise source repair; in harder tasks this kind of target drift will waste tokens or corrupt the repo. PatchLens would require every mutating action to attach to a live hypothesis grounded in validation evidence.

**Evidence:** The run summary shows bincode first wrote only at turn 7 and axum at turn 9, while the detailed logs show controller involvement and repair scaffolding. ([GitHub][8]) Quorp already has proof packets, validation plans, `SuggestImplementationTargets`, `SuggestEditAnchors`, Patch VM preconditions, and negative memory; PatchLens connects them into a write contract. ([GitHub][9])

**The idea:** Before a write can execute, the runtime checks that the action is tied to a current `PatchHypothesis`: what failed, why this target file, what evidence supports it, what exact check should improve, and what rollback point exists.

**Files to add/change:**

* Add `crates/quorp_plan_mode/src/patch_lens.rs`
* Add `crates/quorp_agent_core/src/runtime/write_intent.rs`
* Extend `AgentAction::PreviewEdit`, `ReplaceRange`, `ModifyToml`, `ApplyPatch`, `WriteFile` with optional `intent_id`
* Extend Patch VM receipts with `intent_id`, `hypothesis_id`, `evidence_refs`
* Add `/diff --why`, `/proof --intent`, `/tasks --hypotheses`

**Core data structure:**

```rust
    pub failure_kind: FailureKind,
    pub diagnostic_summary: String,

    pub target_files: Vec<TargetFile>,

    pub status: HypothesisStatus,

    pub required_validation: ValidationPlan,

#[derive(Debug, Clone, Serialize, Deserialize)]

    Warn { reason: String },
```

**Hard gate:**

```rust
pub fn authorize_write_action(
    action: &AgentAction,
    state: &AgentTaskState,
    hypotheses: &HypothesisStore,
) -> WriteAuthorization {
    let Some(intent_id) = action.intent_id() else {
        return WriteAuthorization::Deny {
            reason: "write action lacks intent_id".into(),
            suggested_next_action: Some(AgentAction::SuggestImplementationTargets {
                command: state.last_failing_command(),
                output: state.last_failing_output_excerpt(),
                failing_path: state.last_failing_path(),
                failing_line: state.last_failing_line(),
            }),
        };
    };

    let Some(h) = hypotheses.get(intent_id) else {
        return WriteAuthorization::Deny {
            reason: format!("unknown write intent {intent_id}"),
            suggested_next_action: None,
        };
    };

    if !h.target_files.iter().any(|t| t.path == action.target_path()) {
        return WriteAuthorization::Deny {
            reason: format!(
                "write target `{}` is not justified by hypothesis `{}`",
                action.target_path().unwrap_or("<none>"),
                h.id
            ),
            suggested_next_action: Some(AgentAction::SuggestEditAnchors {
                path: h.target_files[0].path.clone(),
                range: None,
                search_hint: Some(h.diagnostic_summary.clone()),
            }),
        };
    }

    WriteAuthorization::Allow { hypothesis_id: h.id.clone() }
}
```

**Acceptance tests:**

* A Rust assertion/test failure cannot trigger a `Cargo.toml` dependency edit unless the diagnostic references missing crate/import resolution.
* Broad `WriteFile` is denied when a hash-preconditioned `ReplaceRange` can satisfy the hypothesis.
* If validation fails after a patch, PatchLens records the failed hypothesis and negative memory blocks identical retries without new evidence.
* `/diff --why` shows the validation packet, target ranking, expected effect, and proof result.

```rust
    UserApproved,
    RepoReviewed { commit: String },
    RepoUntrusted,
    GeneratedCandidate,

impl RuleTrust {
    pub fn may_affect_permissions(self) -> bool {

                | RuleTrust::GeneratedCandidate

        matches!(self, RuleTrust::ManagedPolicy | RuleTrust::UserApproved)
    }
}
```

## 6. Proof-backed Task DAG and disposable worktree subagents

**Why this is #6:** This is the upgrade that makes Quorp feel like a serious autonomous engineering runtime instead of a single chat transcript. Codex App emphasizes parallel threads and built-in worktrees, and Claude Code has built-in and custom subagents. ([OpenAI Developers][11]) Quorp can beat both by making subagent returns **proof-backed, patch-bundled, ledgered, and replayable**.

**Evidence:** Quorp’s workplan explicitly marks “Ledgered task DAG and subagents” as open follow-up, while the phase tracker says the full-auto SWE loop has Understand → Plan → Inspect → Patch → Verify → Review → Learn stages, budgets, stall detection, and optional parallel worktree attempts. ([GitHub][12])

**The idea:** Every autonomous task becomes a DAG. Nodes can be exploration, hypothesis, patch, verification, security review, or benchmark replay. Subagents run in disposable worktrees with scoped permissions and return patch/proof bundles. The parent cannot merge a subagent result unless its proof DAG validates.

## 8. Taint-aware permissions, sandbox leases, and permission UX that can beat Claude/Codex

**Why this is #8:** Full-auto against untrusted repos is only acceptable if the permission model is stronger than “ask less often.” Codex emphasizes sandboxing spawned commands, approvals, and network boundaries; Claude Code exposes fine-grained allow/ask/deny rules, compound command policy, hooks, permission modes, and managed settings. ([OpenAI Developers][14]) Quorp should combine both ideas with ledgered leases, taint tracking, and proof receipts.

**Evidence:** Quorp already has sandbox modes, permission primitives, typed action classification, and rejects `--yolo --sandbox host`, but its workplan still calls richer trust/settings/compound-command policy future work. ([GitHub][9])

**The idea:** Treat every permission as a scoped, expiring, ledgered lease. Every repo-originated instruction/log/tool result gets a taint label. Tainted content can suggest work but cannot grant capabilities, alter sandbox policy, or suppress verification.

```rust
    pub mcp: McpPolicy,
    pub expires_at_ms: u64,
    pub max_invocations: u32,

}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CommandRisk {
    pub compound: bool,
    pub destructive: bool,
    pub writes_outside_workspace: bool,
    pub network_access: bool,
    pub dependency_install: bool,
    pub generated_executable: bool,

```

**Acceptance tests:**

* `cargo test && curl ...` is parsed as compound and networked, not treated as a safe cargo test.
* `build.rs`, generated binaries, and downloaded scripts require a higher-risk lease before execution.
* Repo `.quorp/config`, `agent/rules`, or prompt-injected logs cannot grant network/MCP/browser permissions.
* Secrets paths like `.env`, credentials files, and configured deny globs cannot be read even in full-auto unless explicitly leased by the user/managed policy.
* `/permissions` shows exact active leases, denied actions, risk reasons, and sandbox backend.

## 10. Anti-overfit evaluation program: mutated prompts, replay lanes, Rust suites, and oracle labels

**Why this is #10:** Quorp should not optimize itself into a Rust top-5 specialist. The next public credibility jump is to prove that improvements survive broader, more realistic, and contamination-resistant evals. SWE-bench Verified is useful but limited; newer work argues formal GitHub issue prompts can overestimate real chat-agent capability, and Terminal-Bench-style tasks test end-to-end terminal competence. ([SWE-bench][13])

**Evidence:** Quorp has benchmark scoreboards and regression gates, but the workplan still marks broader replayable eval and release lanes open. ([GitHub][9]) The current 5/5 should be split into normal/model vs controller/oracle categories because deterministic controller patches materially affected at least two cases. ([GitHub][15])

**Eval lanes to add:**

1. **No-oracle Rust SWE top-5 replay**
   Disable `benchmark-oracle-playbooks`; record true model/controller capability.

2. **Mutated user-prompt lane**
   Convert formal benchmark descriptions into terse real-user requests: “tests fail around fallback merging; figure it out.” This directly targets benchmark over-specification risk.

3. **Rust crate maintenance suite**
   Add tasks from real Rust crates: feature additions, migration, compile-fix, Clippy/fmt/test, generated code zones.

4. **Terminal-Bench-style lane**
   Sandboxed tasks where the agent must use shell/tools effectively, not just patch one file.

5. **Context compaction lane**
   Artificially constrain context and require successful resume after compaction.

6. **Rule-learning lane**
   Replay known failures with rules off, shadow, and active; score helped/hurt.

7. **Security lane**
   Malicious repo instructions, poisoned `AGENTS.md`, fake tool logs, secret files, dependency-install traps, MCP escalation attempts.

8. **Replay determinism lane**
   Given `run-ledger.jsonl`, reconstruct state, proof, task DAG, memory effects, and UI timeline.

# The final product bet

The best version of Quorp is not “Claude Code but Rust” or “Codex CLI but terminal-first.” The best version is:

**A ledger-native coding agent OS where every thought-relevant state transition, tool observation, write, proof, permission, memory update, rule promotion, context compaction, and subagent return is replayable, inspectable, compactable, and testable.**

That gives Quorp unique advantages:

1. **Lower tokens without lower capability** through state packets and observation handles.
2. **Better long-horizon reliability** through task DAGs, PatchLens, and proof-backed subagents.
3. **Safer full-auto** through sandbox leases and taint-aware permissions.
4. **Real-time learning** through durable, scoped, shadow-evaluated rules.
5. **A terminal UX competitors do not have**: mission-control UI, proof debugger, replay timeline, context pressure graphics, and permission/trust visibility.
6. **Credible benchmarks** because Quorp labels oracle assistance instead of hiding it.

If I had to start with only three PRs, I would do them in this order:

1. **ContextPressure + StatePacket + `/status` + auto-compaction hooks.**
2. **Oracle quarantine + scoreboard labels for model/skill/oracle success.**
3. **ObservationPacket handles for read/validation outputs.**

Those three together attack Quorp’s biggest current risks: token growth, benchmark-specific overclaiming, and bulky tool feedback.
