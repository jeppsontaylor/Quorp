Here are my **8 best ideas**, ranked by expected impact on coding capability, token efficiency, memory/recovery quality, and terminal product feel. I’m treating Quorp’s 5/5 Rust SWE-bench run as useful evidence, not proof that the system is done: the run succeeded, but the archive shows high token use, late writes in harder cases, controller repair, benchmark-state packets, and at least one large full-file rewrite. The batch report confirms **5/5 success, 28 total requests, 180,546 billed tokens, and about 696 seconds wall clock**. ([GitHub][1])

The big competitive bar is clear: Codex is moving toward multi-agent command-center workflows with isolated worktrees, verifiable logs, and task delegation; Claude Code has strong slash commands, memory files, hooks, permissions, subagents, and compaction; and recent Claude Code architecture analysis argues that most agent advantage comes from the systems around the model loop: permissions, compaction, extensibility, subagents, and append-oriented session storage. ([OpenAI][2])

## 1. Ledger-Native Context OS: turn Quorp into the best long-session coding agent

**Why this is #1:** Quorp already has the raw ingredients: runtime token telemetry, prompt compaction policy variants, task-state snapshots, context index provenance, checkpoints, and a hash-chain ledger. But the current design still looks more like “transcript plus repair state” than a true **context operating system**. The 5/5 run shows prompt growth and expensive late-turn behavior: case 08 used **9 requests / 62,702 tokens** and did not write until step 9; case 06 used **9 requests / 52,083 tokens** with repair submode and controller assistance. ([GitHub][3])

**Existing evidence:** `runtime.rs` already tracks `TokenUsage`, `ModelRequestWatchdogReport`, `prompt_token_estimate`, `PromptCompactionPolicy`, checkpoints, validation queues, failed edit records, and `AgentRepairMemory`.  The context index already stores lexical chunks, symbols, tests, imports, proof history, and context-pack provenance in `.quorp/index/index.sqlite`.  The ledger already supports hash-chained events, snapshots, subscriber cursors, and tamper tests. 

**Build this:** a `MissionStatePacket` compiler that replaces stale transcript bulk with a compact, replayable, provenance-backed state packet. It should preserve objective, constraints, decisions, task DAG, open files, failed attempts, verifier state, security boundary, diff summary, and “do not repeat” memories.

## 2. Replace benchmark playbooks with a general Failure-to-Action Compiler

**Why this is #2:** The 5/5 run succeeded partly because Quorp has strong repair scaffolding. That is valuable, but some of it is explicitly benchmark-specific. `benchmark_playbooks.rs` is labeled “Benchmark-specific deterministic repair playbooks,” hardcodes the five Rust SWE-bench cases, and contains exact source transforms for cargo-dist, cc-rs, axum, chrono, and bincode.  That must be quarantined if Quorp wants credibility beyond the top-5 cohort.

**Existing evidence:** The runtime turn path invokes benchmark repair injection and required read injection in `BenchmarkAutonomous` paths. ([GitHub][4]) Case 06 shows controller-injected reads and an assistant message still focused on Cargo.toml while the actual patch modified `de_owned.rs`, a strong sign that the controller saved the run from drift. ([GitHub][5])

**Build this:** a general `FailureToActionCompiler` that converts proof packets and failed tool events into constrained next actions:

```text
failure evidence → failure class → owner candidates → required read lease
→ allowed patch shapes → verifier plan → no-repeat constraints
```

This should generalize the *behavior* that worked in the benchmark—target leasing, minimal evidence packets, repair requirements—without hardcoded patch content.

## 3. Patch VM 2.0: semantic edit contracts, diff budgets, and no broad source rewrites

**Why this is #3:** Quorp’s Patch VM is already a strength: hash preconditions, preview IDs, touched paths, rollback tokens, high-risk preview requirements, and full-file rewrite policy exist.  But case 08 still did `write_file axum/src/routing/mod.rs` with “write 705 lines” for a small net change.  That is dangerous in real repos: it risks formatting churn, lost concurrent edits, hidden regressions, and token-heavy output.

**Build this:** make every write pass through a **semantic delta budget**. Full-file writes to source files should be refused unless the file is tiny, generated, or explicitly approved. Normal source edits should use `ReplaceRange`, `ReplaceBlock`, `ModifyToml`, or AST edit previews.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SemanticDeltaBudget {
    pub max_changed_lines_without_preview: usize,
    pub max_changed_ratio_without_preview: f32,
    pub max_source_full_rewrite_lines: usize,
    pub require_ast_or_range_for_source: bool,
    pub generated_file_globs: Vec<String>,
}

pub fn classify_write_intent(
    path: &Path,
    old_text: &str,
    new_text: &str,
    policy: &SemanticDeltaBudget,
) -> Result<WriteIntent, PatchPolicyError> {
    let delta = line_diff_stats(old_text, new_text);
    let is_source = looks_like_source_file(path);
    let is_generated = matches_generated_glob(path, &policy.generated_file_globs);

    if is_source && !is_generated && delta.new_total_lines > policy.max_source_full_rewrite_lines {
        return Err(PatchPolicyError::NeedsStructuredEdit {
            path: path.to_path_buf(),
            reason: format!(
                "source full-file rewrite has {} lines; use ReplaceRange/AST edit",
                delta.new_total_lines
            ),
        });
    }

    if delta.changed_ratio > policy.max_changed_ratio_without_preview {
        return Ok(WriteIntent::HighRiskPreviewRequired { delta });
    }

    Ok(WriteIntent::LowRisk { delta })
}
```

## 4. Durable, git-trackable rule learning with shadow evaluation and trust separation

**Why this is #4:** Quorp has the beginnings of something excellent: `RuleForge` clusters failures, emits candidate rules, and promotes via shadow results. But the current forge is in-memory with lifecycle mechanics; the comment explicitly says the shadow-validation harness still needs wiring.  Memory has SQLite persistence, FTS, negative signatures, failed-attempt tracking, and retry blocking.  The missing breakthrough is making rules **tiny, reviewable, scoped, git-trackable, and safe from repo-authored escalation**.

```text
agent/rules/                 # repo-visible, untrusted/project-authored candidates
```

Project-authored rules can influence coding style and workflows, but cannot grant permissions, disable sandboxing, allow network, or suppress security checks.

```rust
pub enum RuleTrustDecision {
    LoadAsActive,
    LoadAsShadow,
    LoadAsUntrustedContextOnly,
    Reject { reason: String },
}

pub fn evaluate_rule_trust(rule: &RuleFile, source: RuleSource) -> RuleTrustDecision {
    if rule.effect.requests_permission_change() {
        return RuleTrustDecision::Reject {
            reason: "rules may not grant permissions or weaken sandbox policy".into(),
        };
    }

    match source {
        RuleSource::Enterprise | RuleSource::UserTrusted => RuleTrustDecision::LoadAsActive,
        RuleSource::RepoCommitted => {
            if rule.trust.repo_authored && rule.state == RuleState::Active {
                RuleTrustDecision::LoadAsShadow
            } else {
                RuleTrustDecision::LoadAsUntrustedContextOnly
            }
        }
        RuleSource::RuntimeGenerated => RuleTrustDecision::LoadAsShadow,
    }
}
```

## 5. Proof-backed Task DAG and subagent worktrees

**Why this is #5:** Claude Code has `/agents` and subagent-related hooks; Codex’s app is explicitly moving toward multiple agents in isolated worktrees. ([Claude API Docs][6]) Quorp should not merely imitate that. It should make subagents **proof-returning, ledger-native, and context-cheap**.

**Build this:** every long task becomes a DAG of `TaskNode`s. Subagents run in isolated worktrees/sandboxes and return a compact, validated `SubagentReturn`: patch receipt, proof packet hash, touched files, failures learned, and open questions. The parent agent never ingests the child transcript unless requested; it consumes the proof-backed summary.

```rust
    pub class: FailureClass,
    pub primary_diagnostics: Vec<DiagnosticSlice>,
    pub likely_owner_files: Vec<RankedOwner>,
    pub minimal_required_reads: Vec<ReadLease>,
    pub exact_rerun: Vec<ValidationPlan>,

    pub forbidden_actions: Vec<ActionFingerprint>,

pub fn compile_failure_atlas(
    packet: &ProofPacket,
    index: &IndexReader,
    memory: &Memory,
    budget: TokenBudget,
) -> anyhow::Result<FailureAtlas> {
    let diagnostics = extract_primary_diagnostics(packet)?;
    let owners = rank_owner_files(&diagnostics, index)?;
    let previous = memory.failed_attempts_for_signature(&failure_signature(packet)?)?;

        class: classify_failure(packet),

        minimal_required_reads: choose_read_leases(&owners, budget),
        exact_rerun: verifier_rerun_plan(packet),
        previous_failed_attempts: previous.into_iter().map(Into::into).collect(),
        forbidden_actions: repeated_action_fingerprints(packet, memory)?,
        token_cost_estimate: 0,
    })
}
```

## 8. Capability-token permissions with project-specific safe workflow learning

**Why this is #8:** Quorp’s permission engine is already unusually good: it classifies read/write/run/network/MCP/browser capabilities; detects shell meta, compound commands, dependency installs, Docker, git remote mutation, `find -delete`, secrets reads, generated executables; and `YoloSandbox` denies host execution.  But the next leap is to make permissions **auditable, lease-based, and learned per project** without letting untrusted repos escalate privileges.

Claude Code supports hierarchical settings, permission allow/ask/deny rules, enterprise precedence, hooks, and memory files. ([Claude API Docs][8]) Codex CLI’s Full Auto mode is documented as autonomous inside a sandboxed, network-disabled environment scoped to the current directory. ([OpenAI Help Center][9]) Quorp should go further: every risky capability gets a short-lived lease with proof and expiry.

**Build this:**

```text
CapabilityToken + ExecutionSurface + TrustSource + Lease
```

Examples:

```text
cargo test                     → allow in auto-safe
cargo add anyhow               → dependency_install lease required
curl https://...               → network lease required
python /tmp/generated.py       → generated_executable lease + sandbox required
git push                       → deny unless explicit host/manual approval
MCP github.create_pr           → MCP lease scoped to tool + repo + duration
```

```rust
pub struct CapabilityLease {
    pub id: String,
    pub token: CapabilityToken,
    pub surface: ExecutionSurface,
    pub command_pattern: Option<String>,
    pub path_scope: Vec<String>,
    pub granted_by: TrustSource,
    pub expires_at_ms: u128,
    pub max_uses: u32,
    pub used: u32,

pub fn decide_with_lease(
    permissions: &Permissions,
    action: &Action,
    surface: ExecutionSurface,
    leases: &[CapabilityLease],
    now_ms: u128,

    let base = permissions.check_on_surface(action, surface);
    if base == Decision::Allow {
        return Decision::Allow;
    }

    if action.tokens.iter().all(|token| {
        leases.iter().any(|lease| {
            lease.token == *token
                && lease.surface == surface
                && lease.expires_at_ms > now_ms
                && lease.used < lease.max_uses
        })
    }) {
        return Decision::Allow;
    }

    base
}
```
