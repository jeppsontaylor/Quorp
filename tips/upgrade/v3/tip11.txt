I could not use a local `git clone` in this container because DNS resolution for GitHub failed, so I inspected Quorp through the public GitHub/raw source and benchmark artifacts. The ranking below is my best final set of ideas, optimized for coding capability, lower token burn, long-session memory, stronger tooling, and a terminal UX that can plausibly feel beyond Claude Code CLI and Codex CLI.

The important pattern from the evidence: Quorp already has strong primitives—typed runtime tools, sandbox modes, Patch VM receipts, context/memory/rule scaffolding, proof packets, and benchmark scoreboards—but the 5/5 Rust SWE-bench run also shows token pressure, late writes, benchmark-specific repair pressure, parser-recovery cost, and a need to turn “support crates” into first-class runtime policy. Quorp’s archive reports 5 cases run, 5 successes, 0 failures, 28 requests, 180,546 billed tokens, and ~696 seconds wall clock; case 08 alone took 9 requests and 62,702 tokens with first write at step 9, while case 06 also took 9 requests and first write at step 7. ([GitHub][1])

## Ranked top 10 ideas

### 1. Ledger-native Mission State + automatic Context OS compaction

**Why this is #1:** This is the highest-leverage upgrade for long-form coding reliability and token reduction. Quorp currently sends runtime state back into the model every turn, and the benchmark reports show prompts growing across turns: case 08 grew from 2,554 prompt-estimated tokens to 7,747, and case 06 grew from 2,127 to 8,237. ([GitHub][2])

Quorp already has `PromptCompactionPolicy` variants and runtime state snapshots in `crates/quorp_agent_core/src/runtime.rs`, plus a persistent context index under `.quorp/index/index.sqlite` with lexical chunks, symbols, imports, tests, proof history, and context-pack provenance in `crates/quorp_context/src/index.rs`. ([GitHub][3]) The missing breakthrough is making compaction **semantic, visible, ledger-replayable, and automatic**, not just “last N messages plus a state packet.”

**What to build:** Add a `MissionStateFrame` that is regenerated from the ledger and runtime facts. It should preserve the objective, hard constraints, open tasks, decisions, failed attempts, watched files, validation state, proof refs, security boundary, current context pressure, and provenance handles. The prompt should carry this small frame plus only the relevant context pack, not the entire transcript.

**Files to add/change:**

* Add `crates/quorp_context_model/src/mission_state.rs`
* Add `crates/quorp_agent_core/src/runtime/context_budget.rs`
* Add `crates/quorp_agent_core/src/runtime/compaction.rs`
* Change `crates/quorp_agent_core/src/runtime/recovery.rs` to replace `transcript.clone() + runtime_summary()` with `MissionStateFrame::for_prompt(...)` once pressure crosses a soft threshold.
* Change `crates/quorp_agent_core/src/runtime/state_record.rs` so `runtime_summary()` becomes a human/debug renderer, not the primary prompt-state transport.
* Add UI plumbing in `crates/quorp_render/src/status_footer.rs`, `crates/quorp_term`, and `crates/quorp_slash` for `/status`, `/context`, and `/compact`.

### 2. Replace broad writes with a Patch-Intent VM and “write amplifier” rejection

**Why this is #2:** The best coding agents are not just agents that edit; they edit minimally, auditably, and reversibly. Quorp already has Patch VM, `ReplaceRange`, `ModifyToml`, previews, receipts, rollback tokens, and large full-file rewrite rejection in the roadmap/tracker, but the case 08 event log still shows `write_file axum/src/routing/mod.rs` with `edit_summary: "write 705 lines"` even though the final diff was 8 added / 3 removed. ([GitHub][4])

**What to build:** Treat full-file `WriteFile` as a last-resort artifact path, not a normal source-edit path. Add a `PatchIntent` layer that turns edits into preconditioned, minimal, auditable operations. If the model proposes a huge write but the diff is small, reject it before apply and ask for `ReplaceRange`, `ApplyPreview`, `ModifyToml`, or `AstTransform`.

**Files to add/change:**

* Add `crates/quorp_patch_model/src/intent.rs`
* Add `crates/quorp_patch_vm/src/intent_apply.rs`
* Change `crates/quorp_agent_core/src/agent_protocol.rs` to deprecate source `WriteFile` in autonomous mode.
* Change native action dispatch in `crates/quorp_tools` and runtime turn handling in `crates/quorp_agent_core/src/runtime/turn.rs`.
* Extend `script/quorp-write-path-audit` and `agent/write-path-allowlist.toml`.
* Add benchmark metric `write_amplification_ratio = written_lines / diff_changed_lines`.

```rust
pub fn reject_amplified_source_write(
    path: &str,
    attempted_content: &str,
    before: &str,
    policy: &PatchPolicy,
) -> Option<WriteAmplification> {
    if policy.allow_full_file_rewrite(path) {
        return None;
    }

    let attempted_written_lines = attempted_content.lines().count();
    let resulting_diff_lines = diff_changed_line_count(before, attempted_content);

    if attempted_written_lines >= policy.full_file_line_threshold
        && resulting_diff_lines <= policy.small_diff_line_threshold
    {
        return Some(WriteAmplification {
            path: path.to_string(),
            attempted_written_lines,
            resulting_diff_lines,
            ratio: attempted_written_lines as f32 / resulting_diff_lines.max(1) as f32,
        });
    }

    None
}
```

**Acceptance gates:**

* Case 08 must no longer allow a 705-line source `WriteFile`; it must use `ReplaceRange`, `ApplyPreview`, or `PatchIntent`.
* `cargo test -p quorp_patch_vm --lib` must include stale hash, protected path, full-file rewrite, and rollback-token tests.
* `script/quorp-write-path-audit` must fail any new production workspace mutation outside Patch VM unless allowlisted.
* Benchmark reports must include `write_amplification_ratio`, and release gates should fail on broad writes unless explicitly justified.

### 3. Ledgered task DAG + proof-backed subagents + worktree patch tournament

**Why this is #3:** Claude Code and Codex now expose subagents or separate review/parallel work surfaces; Claude Code’s docs describe subagents with restricted tools and model selection, while Codex CLI docs list subagents as part of its CLI workflows. ([Claude][5]) Quorp has a `SweController` stage machine and roadmap language for task DAGs, but the next jump is to make tasks ledger-native and proof-backed. ([GitHub][4])

**What to build:** A task DAG where every subtask has objective, assigned files, capability set, context budget, sandbox/worktree policy, proof requirement, and file leases. Subagents return artifacts, Patch VM receipts, proof receipts, confidence, and next actions—not raw hidden context.

The novel differentiator is a **patch tournament**: spawn two or three isolated implementers for hard tasks, then a verifier/reviewer subagent ranks candidates by proof success, patch minimality, security policy, and diff scope.

**Files to add/change:**

* Add `crates/quorp_agent_core/src/task_dag.rs`

* Add `crates/quorp_cli/src/quorp/agents.rs`
* Add `/agents` and `/tasks` support in `crates/quorp_slash` and TUI panes in `crates/quorp_render`.

### 4. Git-trackable Rule Cards with shadow evaluation and negative memory

**Why this is #4:** Claude Code has `CLAUDE.md`, `.claude/rules/`, auto memory, and `/memory`, but Claude docs explicitly frame memory as context rather than enforced configuration. ### 5. Action Value Controller: no-progress detector that prices reads, searches, validation, and writes

**Why this is #5:** Case 08 shows the exact failure mode: list directory, read implementation, read tests, four searches, a failed fast-loop command, two more reads, then finally a write on request 9. The case still succeeded, but the behavior burned tokens and time. ([GitHub][8]) Case 06 similarly succeeded only after 8 reads and first write at step 7. ([GitHub][9])

**What to build:** A runtime-side action value model, not an LLM classifier. It should score each proposed action against current evidence, known owner files, prior reads, failed attempts, validation state, and “write readiness.” Low-value repeated reads/searches are not forbidden forever, but they require a reason or new target. Validation before a patch should be blocked when the failure is already known and no edit has happened.

**Files to add/change:**

* Add `crates/quorp_agent_core/src/runtime/action_value.rs`
* Change `crates/quorp_agent_core/src/runtime/turn.rs` before dispatch.
* Add runtime events:

  * `agent.low_value_action_blocked`
  * `agent.write_readiness_updated`
  * `agent.no_progress_escalated`
* Add report fields in `crates/quorp_benchmark`:

  * `low_value_actions_blocked`
  * `tokens_before_first_write`
  * `first_write_readiness_score`
  * `validation_before_patch_count`

**Hard core snippet:**

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct ActionValueDecision {
    pub action_allowed: bool,
    pub score: f32,
    pub reason: String,
    pub recovery_hint: Option<String>,
}

pub fn evaluate_action_value(
    action: &AgentAction,
    state: &AgentTaskState,
    ledger: &ActionHistory,
) -> ActionValueDecision {
    let signature = ActionSignature::from(action);

    if ledger.same_successful_observation_count(&signature) >= 2 {
        return ActionValueDecision {
            action_allowed: false,
            score: 0.05,
            reason: "same observation already captured twice".to_string(),
            recovery_hint: Some(
                "Use existing evidence, read a new targeted range, or emit a patch intent."
                    .to_string(),
            ),
        };
    }

    if action.is_validation_like()
        && state.has_known_failure()
        && !state.has_new_patch_since_last_validation()
    {
        return ActionValueDecision {
            action_allowed: false,
            score: 0.10,
            reason: "validation before any new patch would repeat known failure".to_string(),
            recovery_hint: Some("Patch the leased implementation target first.".to_string()),
        };
    }

    ActionValueDecision {
        action_allowed: true,
        score: 0.70,
        reason: "action has plausible new information value".to_string(),
        recovery_hint: None,
    }
}
```

**Acceptance gates:**

* Synthetic repeated-read tasks must block the third identical read unless new evidence changed.
* Case 08 should hit first write earlier or require an explicit model justification for continued inspection.
* No-progress events must be visible in `/status`.
* Regression gate should track `tokens_before_first_write` and `read_count_before_first_write`.

### 9. Eval mesh + anti-overfit quarantine for benchmark playbooks

### 10. Provider-native structured action ABI + parser recovery eliminator

**Why this is #10:** The 5/5 run still needed parser recovery in cases 07, 09, and 10. ([GitHub][15]) Quorp already supports `strict_json_v1` and native tool-call flags, but the next improvement is to make structured action reliability testable across providers rather than handled mostly by prompt repair.

SWE-bench Verified notes that mini-SWE-agent v2 moved to tool calling while earlier versions parsed actions from output strings, which is a useful external signal: action interface reliability matters enough that benchmark configurations track it. ([SWE-bench][13])

**What to build:** A provider-independent Action ABI with generated JSON Schema, provider-native tool definitions when available, grammar/constrained decoding when supported, and a parser fuzz/replay suite.

**Files to add/change:**

* Add `crates/quorp_agent_protocol_schema`

* Change `crates/quorp_provider`
* Add `tests/action_contract_fuzz.rs`
* Add benchmark report fields:

  * `native_tool_success_rate`
  * `parser_recovery_tokens`
  * `invalid_turn_class`
  * `schema_retry_count`

pub trait ActionAbi {

* Runtime should emit `AbiParseReport` into the ledger and benchmark reports.

## The most important implementation sequence

I would not implement these in arbitrary order. The highest-return sequence is:

1. **Patch-Intent VM + write amplification gate** — because it reduces catastrophic edit risk immediately.
2. **Mission State + context pressure + `/status`** — because it reduces token bloat and enables long-session autonomy.
3. **Action Value Controller** — because it attacks the observed late-write/read-loop behavior.
4. **Rule Cards + negative memory injection** — because it converts mistakes into scoped, reviewable learning.
5. **TUI Mission Control** — because it makes the above systems visible and controllable.
6. **Task DAG/subagents** — because parallelism before ledger/context/proof stability can amplify confusion.
7. **Eval mesh and playbook quarantine** — because Quorp’s credibility depends on proving generality beyond the 5/5 cohort.

## My single strongest “moonshot” recommendation

Build Quorp around **ledger-native agent cognition**: every thought-worthy system fact becomes an event, every event can be replayed, every replay can regenerate mission state, every mission state can compact context, every compacted context has provenance, every write has a Patch VM receipt, every validation has a proof packet, every mistake can propose a scoped rule, and every rule can be shadow-evaluated against old ledgers before affecting live work.

That combination—**Context OS + Patch VM + Proof DAG + Rule Cards + Mission Control Replay**—is the unique path where Quorp can be more than another Claude/Codex-style terminal chat loop. It can become a replayable, inspectable, self-improving autonomous engineering runtime.
