One caveat up front: I could not complete a literal local `git clone` in the container because DNS resolution failed, so I inspected the repo through GitHub/raw file access plus current web sources. The code and log findings below are from the repository artifacts and current public docs.

# Final ranked ideas

## 1. Build a **Ledger-Native Context OS**: lossless compaction, visible context pressure, and replayable state capsules

This is the highest-leverage upgrade. Quorp already has the ingredients: `RuntimeEvent`, bounded event fanout, `AgentTaskStateSnapshot`, checkpoint events, parse recovery events, validation events, token accounting, and context-pack provenance. But the current context story is still split between prompt compaction policy, context packs, memory recall, and verbose repair messages. The upgrade is to make the **ledger the source of truth**, and make every model request a compact projection of that ledger.

The 5/5 run succeeded, but it also showed exactly why this matters: total billed tokens were 180,546 over only 28 requests, with 52,083 tokens in bincode and 62,702 in axum; first writes happened as late as step 7 for bincode and step 9 for axum; three cases had parser recovery incidents; and the batch had two watchdog-near-limit cases. Those are not “model got smarter” problems. They are context-operating-system problems.  

**What to add**

Create:

```text

  src/budget.rs
  src/state_capsule.rs
  src/compactor.rs
  src/retention.rs
  src/status.rs

crates/quorp_context_model/src/session_context.rs
crates/quorp_agent_core/src/runtime/context_os.rs
crates/quorp_cli/src/quorp/status.rs
```

```text

.quorp/sessions/<session_id>/compaction-receipts.jsonl
```

**Core design**

Every turn should compute a `ContextPressureSnapshot` before calling the model. When pressure crosses thresholds, Quorp should compact automatically, but only by replacing lossy transcript history with a **state capsule that preserves invariants**:

## 2. Replace verbose repair prose with **typed controller packets and action contracts**

The current runtime is very strong, but too much high-value control is encoded as natural-language prompt packets. `runtime/turn.rs` already injects controller reads, benchmark repair reads, parse recovery messages, validation queues, and write pressure; `state_messages.rs` contains detailed benchmark repair prompts and next-step contracts. That worked on the top-5 run, but it costs tokens and invites parser failures.  

The upgrade: move repair instructions out of prose and into a compact, typed `ControllerPacket` that the runtime and tools enforce.

**Files to change**

```text
crates/quorp_agent_core/src/runtime/controller_packet.rs       # new
crates/quorp_agent_core/src/runtime/state_messages.rs          # shrink

crates/quorp_render/src/controller_panel.rs                    # display packet status
```

**Hard part**

```rust
#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq, Eq)]
pub enum RuleTrust {
    UserTrusted,        // ~/.quorp/rules or explicitly approved
    ProjectReviewed,    // committed but approved in this workspace
    ProjectUntrusted,   // visible, not active
    GeneratedCandidate, // agent-authored, shadow only
}

pub fn effective_rule_state(rule: &RuleFile, trust: RuleTrust) -> EffectiveRuleState {
    match (rule.state, trust) {
        (RuleState::Active, RuleTrust::UserTrusted | RuleTrust::ProjectReviewed) => {
            EffectiveRuleState::Active
        }
        (RuleState::Active, RuleTrust::ProjectUntrusted) => {
            EffectiveRuleState::ShadowOnly
        }
        (RuleState::Candidate | RuleState::Shadow, _) => EffectiveRuleState::ShadowOnly,
        (RuleState::Rejected | RuleState::Deprecated | RuleState::Retired, _) => {
            EffectiveRuleState::Ignored
        }
        _ => EffectiveRuleState::ShadowOnly,
    }
}
```

## 5. Create a **Task DAG with proof-backed subagents and worktree returns**

Quorp’s runtime tracks mode, working set, validation queues, failed edits, repair state, and checkpoints, but it does not yet expose a durable task graph as the core planning object. `AgentTaskStateSnapshot` is close; turn it into a first-class DAG.

## 6. Upgrade Patch VM into **Edit Intent IR** with AST anchors and semantic postconditions

The Patch VM is already a major strength: it validates hash preconditions, previews file changes, applies with rollback tokens, classifies high-risk patches, and blocks large full-file rewrites unless explicitly allowed.

## 8. Build **TUI Mission Control**: context pressure, task DAG, proof, rules, sandbox, and replay debugger

Quorp has a promising renderer: pure rendering primitives, custom ANSI over crossterm, color capability detection, no-color support, session frames, task rows, command cards, shimmer, status footer, permission modal, and deterministic render tests. ## 9. Add **adaptive model routing and tool-first turns** to reduce billed tokens without losing capability

The top-5 run used 28 total requests and 180,546 billed tokens. That is not catastrophic, but it is expensive for five small cases. The highest-value token reduction after compaction is model routing: do not call the strongest model for deterministic inspection, validation interpretation, status summarization, or already-constrained patch applications.

**Files to add/change**

```text
crates/quorp_provider/src/router.rs
crates/quorp_agent_core/src/runtime/model_router.rs
crates/quorp_agent_core/src/runtime/tool_first.rs
crates/quorp_context_os/src/budget.rs
crates/quorp_cli/src/quorp/model.rs
```

**Core design**

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ModelRouteDecision {
    pub route: ModelRoute,
    pub reason: RouteReason,
    pub max_prompt_tokens: u32,
    pub max_output_tokens: u32,
    pub cache_key: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum ModelRoute {
    DeterministicToolOnly,
    SmallPlanner,
    FastRepair,
    StrongPatchSynthesis,
    StrongFinalReview,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum RouteReason {
    NoNewInformationNeeded,
    ValidationFailureClassifiable,
    ControllerPacketRequiresSpecificTool,
    PatchRequiresSynthesis,
    FinalProofSummary,
}
```

**Tool-first turn examples**

No model call needed when:

```text
- validation queue is non-empty and last write succeeded
- context index is stale and policy says rebuild before retrieval
- controller packet requires a specific read before write
- permission receipt requires user prompt
- patch preview has not been applied yet
```

This is especially relevant because mini-SWE-agent demonstrates that simple, transparent action loops remain very competitive; Quorp should reserve richer model/tool scaffolding for when it actually buys reliability. ([GitHub][2])

## 10. Build an **Evaluation Lab** that measures generality, compaction, security, and rule learning

The 5/5 run is valuable evidence, but it is too small and too close to benchmark-specific repair scaffolding. SWE-bench Verified is a human-validated 500-instance subset intended for reliable evaluation, while Terminal-Bench 2.0 tests terminal agents across 89 software engineering, ML, security, data-science, and system tasks. Quorp needs both kinds of evaluation, plus its own replay-based regression gates. ([SWE-bench][6])

**Anti-overfitting gates**

```text
1. Production crates cannot contain known benchmark case IDs or exact oracle patch strings.
2. Benchmark fixture playbooks compile only under benchmark-fixture-oracles.
3. Every benchmark run must produce replayable events, proof receipts, and context-pressure traces.
4. Release gates include one fixed suite, one randomly sampled suite, and one newly generated local suite.
5. Scoreboards show token/cost/recovery metrics, not just pass/fail.
```

OpenHands already positions itself across CLI, SDK, local GUI, cloud, integrations, and enterprise permissions; Quorp’s open-source advantage should be that every run is replayable, every decision is auditable, and every claimed improvement has a regression gate. ([GitHub][7])

# My top three PRs to start immediately

**PR 1: Context pressure + `/status` visibility.** Add `ContextPressureSnapshot`, persist `context-pressure.jsonl`, show context pressure in `/status` and the footer. This is the smallest PR with the biggest UX and debugging win.

**PR 2: Quarantine benchmark playbooks.** Move exact benchmark repairs out of production core behind a fixture-only feature and add the source-level overfit gate. This makes Quorp’s claims much more credible.

**PR 3: ControllerPacket v1.** Replace the largest repair prompt paths with typed packets for “required read,” “required write,” and “required validation.” This should directly reduce tokens, delayed writes, parser recovery, and no-progress loops.

The moonshot version is: Quorp becomes the first terminal coding agent where autonomous cognition is not hidden in a chat transcript. It becomes a replayable, proof-backed, context-aware operating system for repository work.
