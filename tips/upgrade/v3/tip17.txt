One caveat: my execution environment could not complete a local `git clone` because DNS resolution to GitHub failed, so I audited Quorp through GitHub/raw file access and the run archive rather than by running `cargo test` locally. The recommendations below are based on the repo files, benchmark artifacts, and current public sources I could inspect, not on a local build.

## The highest-signal facts behind this ranking

Quorp already has unusually strong bones: a typed agent runtime, sandbox/permission primitives, Patch VM write receipts, proof packets, memory/rule scaffolding, benchmark scoring gates, headless run/exec modes, and a terminal-first product direction. The roadmap and phase tracker also show durable ledger/proof work is in place, while major gaps remain around write-path enforcement, managed trust, durable rule policy, task DAG/subagents, TUI mission control, and broader replayable evals. ([GitHub][1])

The 5/5 Rust SWE-bench run is valuable evidence, but it also exposes the next bottlenecks: 28 requests, 180,546 billed tokens, about 696 seconds wall clock, late first writes in cases 06 and 08, parser recovery in three cases, and benchmark repair/patch-packet behavior in multiple cases. Case 06 took 9 requests and first wrote at step 7; case 08 took 9 requests and first wrote at step 9; both hit watchdog pressure. ([GitHub][2])

The strongest opportunity is not “add more agent prompts.” It is to turn Quorp’s existing ledger, context index, memory, Patch VM, verifier, and terminal renderer into a **proof-carrying Context OS** where the model sees less text but has more reliable state, better tools, safer autonomy, and a stronger terminal control surface. SWE-agent’s ACI work shows interface design materially affects coding-agent performance, while Codex and Claude Code have moved toward sandboxing, skills, hooks, scoped memory, subagents, and progressive context disclosure. ([arXiv][3])

# 1. Ledger-native Context OS with lossless compaction and visible context pressure

**Why this is my #1 idea:** Quorp’s biggest route to better coding and lower tokens is to stop treating the prompt as the session state. The session state should live in the ledger, context index, memory store, task DAG, proof DAG, and Patch VM receipts. The prompt should be a compact, auditable projection of that state. Quorp already has a hash-chain run ledger, persistent context index under `.quorp/index/index.sqlite`, durable memory, and runtime events; the missing product is a first-class state packet and context budget supervisor that decides what the model actually sees. ([GitHub][4])

**What it unlocks beyond Claude CLI / Codex CLI:** Claude has project memory and compaction behavior, Codex uses progressive disclosure for skills, and the 2026 Claude Code architecture analysis specifically calls out compaction and append-oriented storage as core system design pieces. Quorp can go further by making compaction **ledger-verifiable, replayable, typed, diffable, and shown live in the TUI** instead of being an opaque chat summarization step. ([Claude API Docs][5])

pub enum ContextPressure {
    Green,
    Yellow,
    Red,
    Critical,
}

impl PromptBudgetState {
    pub fn pressure_for(
        estimated_prompt_tokens: usize,
        target_prompt_tokens: usize,
        hard_limit: usize,
    ) -> ContextPressure {
        let ratio = estimated_prompt_tokens as f64 / target_prompt_tokens.max(1) as f64;
        if estimated_prompt_tokens >= hard_limit {
            ContextPressure::Critical
        } else if ratio >= 0.92 {
            ContextPressure::Red
        } else if ratio >= 0.72 {
            ContextPressure::Yellow
        } else {
            ContextPressure::Green
        }
    }
}

// crates/quorp_agent_core/src/runtime/context_budget.rs

pub struct ContextBudgetController {
    pub soft_target_tokens: usize,
    pub hard_limit_tokens: usize,
    pub recent_ledger_tail_events: usize,
    pub max_failed_attempts: usize,
}

impl ContextBudgetController {
    pub fn prepare_prompt_inputs(
        &self,
        run_id: &str,
        ledger: &RunLedgerReader,
        context: &ContextCompiler,
        memory: &Memory,
        previous_packet: Option<RunStatePacketV1>,
    ) -> anyhow::Result<PromptInputs> {
        let tail = ledger.read_tail(run_id, self.recent_ledger_tail_events)?;
        let estimate = estimate_prompt_tokens(&tail, previous_packet.as_ref());

        let pressure = PromptBudgetState::pressure_for(
            estimate,
            self.soft_target_tokens,
            self.hard_limit_tokens,
        );

        if matches!(pressure, ContextPressure::Red | ContextPressure::Critical) {
            let packet = RunStatePacketV1::compact_from_sources(
                run_id,
                ledger,
                context,
                memory,
                previous_packet,
                CompactionPolicy {
                    preserve_open_tasks: true,
                    preserve_failed_attempts: self.max_failed_attempts,
                    preserve_security_boundary: true,
                    preserve_validation_frontier: true,
                    preserve_patch_receipts: true,
                },
            )?;

            ledger.append_event(run_id, "context.compacted", &packet)?;
            Ok(PromptInputs::StatePacketPlusTail {
                packet,
                recent_events: tail.minimal_events(),
            })
        } else {
            Ok(PromptInputs::RecentLedgerTail(tail))
        }
    }
}
```

# 2. Action Cursor Protocol: token-cheap file/tool observations with expandable handles

**Why this is huge:** The logs show Quorp often pays to reread large support files or entire source files. Case 06 read challenge docs and multiple source/support files before a late first write; case 08 had 9 reads and a first write at step 9; case 10 read a large `src/lib.rs` even though the relevant fix area was narrower. ([GitHub][6])

**Core idea:** Every tool observation should be stored in the ledger as a content-addressed object, but the model should receive only a compact envelope by default: file path, range, symbol hits, hashes, short summary, and expandable handles. The model can request `expand_observation(handle, range)` when it actually needs the bytes. This is the same spirit as SWE-agent’s custom file viewer and succinct search results, but Quorp can make it ledger-native and proofable. SWE-agent explicitly found that a special file viewer, 100-line windows, succinct search, and linter feedback improved agent behavior. ([GitHub][7])

    raw: ToolOutput,
    budget: ObservationBudget,
    store: &mut ObservationStore,
    index: &IndexReader,
    ledger_seq: u64,

    })

# 3. Generalized Repair Kernel to replace benchmark-specific playbooks

**Why this matters:** Quorp’s 5/5 success appears partly due to genuinely good scaffolding—typed runtime actions, validation loops, Patch VM receipts, proof packets—and partly due to benchmark-specific repair paths. The runtime contains strict parsing/recovery logic plus benchmark playbook injection paths such as `maybe_inject_case04_playbook_patch`, `maybe_inject_case05_playbook_patch`, and exact benchmark source patch behavior; recovery metadata tracks benchmark case ledgers, expected targets, fast-loop commands, named tests, patch packets, and honored ranges. ([GitHub][9])

**The upgrade:** Preserve the winning behavior, but reframe it as a **general repair kernel** driven by evidence, not case IDs. The repair kernel should infer: target file, relevant range, failing assertion, required validation command, stale hash, bad edit pattern, and next minimal action. Benchmark playbooks should move behind an explicit eval-only feature flag and become test fixtures, not runtime cognition.

# 4. Git-trackable Rule Cards with shadow evaluation, negative memory, and trust boundaries

**Why this is one of the most important ideas:** Quorp already has durable memory and failed-attempt blocking, and RuleForge has candidate/shadow/active concepts, but the rule forge is still effectively an in-memory state machine rather than a durable, reviewable, git-trackable rules product. ```rust
pub fn load_applicable_rules(
    store: &RuleStore,
    scope: RuleScope,
    trust: TrustContext,
    budget: RulePromptBudget,
) -> anyhow::Result<Vec<RuleCard>> {
    let mut rules = store.query(scope)?;

    rules.retain(|rule| {
        rule.state.is_prompt_visible()
            && !rule.trust.can_affect_permissions
            && trust.allows_rule_source(&rule.trust.source)
            && rule.prompt.max_tokens <= budget.max_tokens_per_rule
    });

    rules.sort_by_key(|r| {
        (
            std::cmp::Reverse(r.effect_score.success_delta_milli),
            r.prompt.max_tokens,
            r.id.clone(),
        )
    });

    rules.truncate(budget.max_rules);
    Ok(rules)
}
```

pub struct FileLeaseManager {
    leases: BTreeMap<PathBuf, LeaseRecord>,
}

impl FileLeaseManager {
    pub fn acquire(
        &mut self,
        task_id: TaskId,
        path: &Path,
        mode: LeaseMode,
        ttl_ms: u64,
    ) -> anyhow::Result<FileLease> {
        if let Some(existing) = self.leases.get(path) {
            if existing.conflicts_with(mode) && !existing.expired() {
                anyhow::bail!(
                    "file lease conflict: {} held by {:?}",
                    path.display(),
                    existing.task_id
                );
            }
        }

        let lease = FileLease::new(task_id, path, mode, ttl_ms);
        self.leases.insert(path.to_path_buf(), lease.clone().into());
        Ok(lease)
    }
}

# 6. Semantic Patch VM v2: AST-aware, hash-preconditioned, minimal edits only

**Why this is high leverage:** The 5/5 run succeeded, but some writes were late or broad. Case 07 performed a whole-file write after reading a narrow range; case 10 changed a substantial chunk of `src/lib.rs`. Quorp already has a Patch VM with preconditions, SHA-256 validation, add/update/delete/move, preview IDs, rollback tokens, and receipts. The next step is to make the write contract semantic and minimal by default. ([GitHub][13])

**Core idea:** The model should not normally write whole files. It should propose semantic edits: replace this function body, insert this enum variant, add this test, update this TOML field. Full-file rewrite becomes an explicit high-risk action requiring justification.

**Semantic operation model:**

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "op", rename_all = "snake_case")]
pub enum SemanticEditOp {
    ReplaceFunctionBody {
        path: PathBuf,
        function: String,
        expected_file_hash: String,
        expected_function_hash: String,
        new_body: String,
    },
    InsertRustItemAfter {
        path: PathBuf,
        anchor_item: String,
        expected_file_hash: String,
        item_source: String,
    },
    ReplaceRange {
        path: PathBuf,
        expected_file_hash: String,
        start_line: u32,
        end_line: u32,
        replacement: String,
    },
    UpdateTomlValue {
        path: PathBuf,
        expected_file_hash: String,
        key_path: Vec<String>,
        new_value: toml::Value,
    },
}

pub struct SemanticPatchVm {
    parser_cache: ParserCache,
    policy: PatchVmPolicy,
}

impl SemanticPatchVm {
    pub fn preview_semantic(
        &mut self,
        op: SemanticEditOp,
    ) -> anyhow::Result<PatchPreview> {
        let before = self.load_and_hash(op.path())?;
        op.validate_hashes(&before)?;

        let after = match op {
            SemanticEditOp::ReplaceFunctionBody { .. } => self.replace_rust_function_body(op)?,
            SemanticEditOp::InsertRustItemAfter { .. } => self.insert_rust_item_after(op)?,
            SemanticEditOp::ReplaceRange { .. } => self.replace_range(op)?,
            SemanticEditOp::UpdateTomlValue { .. } => self.update_toml_value(op)?,
        };

        let diff = minimal_diff(&before.text, &after.text)?;
        self.policy.validate_diff(&diff)?;
        Ok(PatchPreview::from_diff(diff, before.hash, after.hash))
    }
}
```

# 7. Proof-directed verification planner that chooses the next cheapest decisive test

**Core idea:** Build a verification planner that maps patch delta → likely tests → fast command → broader command → proof-full command. It should use context index tables for tests/imports/diagnostics/proof history, Patch VM receipts, and ledger failures.

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct VerifyPlan {
    pub patch_receipt_ids: Vec<String>,
    pub stages: Vec<VerifyStage>,
    pub stop_condition: StopCondition,
    pub cache_key: String,
    pub rationale: Vec<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct VerifyStage {
    pub id: String,
    pub command: String,
    pub expected_signal: ExpectedSignal,
    pub cost: VerifyCost,
    pub required: bool,
}

pub fn plan_verification(
    delta: &PatchDelta,
    index: &IndexReader,
    proof_history: &ProofHistory,
    policy: VerifyPolicy,
) -> anyhow::Result<VerifyPlan> {
    let affected_symbols = index.symbols_for_paths(&delta.paths)?;
    let direct_tests = index.tests_referencing_symbols(&affected_symbols)?;
    let prior_failures = proof_history.failures_for_paths(&delta.paths)?;

    let mut stages = Vec::new();

    stages.extend(fastest_named_tests(direct_tests));
    stages.extend(commands_from_prior_failures(prior_failures));
    stages.push(policy.default_project_smoke.clone());

    if delta.risk >= PatchRisk::Medium {
        stages.push(policy.full_proof_command.clone());
    }

    Ok(VerifyPlan::dedup_and_order(stages, delta.cache_material()))
}
```

# 8. Capability Firewall: full-auto safety for untrusted repos without approval fatigue

**Why this is necessary:** Quorp wants high-agency full-auto behavior, but safe autonomy depends on enforced boundaries, not instructions. ```rust
#[derive(Debug, Clone, PartialEq, Eq, PartialOrd, Ord, Serialize, Deserialize)]
pub enum CapabilityToken {
    ReadWorkspace,
    WriteWorkspace,
    ShellRoutine,
    ShellMeta,
    Network,
    DependencyInstall,
    DockerOrContainer,
    GitRemoteMutation,
    GeneratedExecutable,
    SecretsRead,
    Browser,
    McpExternal,
    DeleteMany,
    PrivilegeEscalation,
}

pub fn classify_shell_command(argv_or_shell: &str) -> Vec<CapabilityToken> {
    let mut caps = BTreeSet::new();

    if contains_shell_meta(argv_or_shell) {
        caps.insert(CapabilityToken::ShellMeta);
    }
    if matches_network(argv_or_shell) {
        caps.insert(CapabilityToken::Network);
    }
    if matches_dependency_install(argv_or_shell) {
        caps.insert(CapabilityToken::DependencyInstall);
    }
    if matches_docker(argv_or_shell) {
        caps.insert(CapabilityToken::DockerOrContainer);
    }
    if matches_secret_path_or_env(argv_or_shell) {
        caps.insert(CapabilityToken::SecretsRead);
    }
    if matches_generated_executable(argv_or_shell) {
        caps.insert(CapabilityToken::GeneratedExecutable);
    }
    if caps.is_empty() {
        caps.insert(CapabilityToken::ShellRoutine);
    }

    caps.into_iter().collect()
}
```

## My final priority order

1. **Ledger-native Context OS** — highest impact on long-form reliability and token reduction.
2. **Action Cursor Protocol** — immediate token savings without capability loss.
3. **Generalized Repair Kernel** — preserves the 5/5 strengths while reducing benchmark-specific risk.
4. **Git-trackable Rule Cards** — turns mistakes into reviewable, scoped, measurable learning.
5. **Task DAG + proof-backed subagents** — unlocks longer and more complex repo work.
6. **Semantic Patch VM v2** — improves correctness, minimality, rollback, and auditability.
7. **Proof-directed verification planner** — reduces wasted validation while preventing false “done.”
8. **Capability Firewall** — makes full-auto safe enough for untrusted repos.
9. **Mission Control TUI** — makes Quorp feel like an autonomous coding IDE, not just a CLI.
10. **Replay-based Eval Factory** — prevents regressions and proves improvements beyond the top-5 cohort.

The most novel combined thesis is this: **Quorp should become the first open-source coding agent where cognition, context, edits, proofs, permissions, memory, and UI are all ledger-native.** That would make it unusually debuggable, safe, replayable, and efficient—exactly the properties needed to beat transcript-first agents on long-form autonomous coding.
