I could not complete a local `git clone` in the container because GitHub DNS resolution failed, so I inspected Quorp through the available GitHub repository interface and current web sources. The repo evidence is strong enough for a real ranking.

The single most important conclusion: **Quorp already has unusually good primitives — hash-chained ledger, Patch VM, proof packets, persistent context index, rule forge scaffolding, permission classifier, strict runtime events, and a serious renderer.** The best upgrades are not generic “add memory” features. They are upgrades that make those primitives **oracle-free, stateful, visible, compact, replayable, and policy-governed**.

Also, one uncomfortable but high-leverage finding: the 5/5 Rust SWE-bench archive is real as an execution result, but `crates/quorp_agent_core/src/runtime/benchmark_playbooks.rs` contains **benchmark-specific deterministic repair playbooks** gated under benchmark mode. That must be quarantined from capability claims and converted into a generalized repair-pattern system. It is the first thing I would fix before making Quorp’s benchmark story public-facing. The archive itself reports 5/5 successes, 28 requests, 180,546 billed tokens, and about 696 seconds wall clock; the run summary also shows delayed first writes in cases 06 and 08, parser recovery in 07/09/10, and one redundant read in 08.  

## My top 10 ranked ideas

### 1. Oracle-Free Repair Kernel: quarantine benchmark playbooks and replace them with general repair patterns

**Why this is #1:** Quorp’s most important next credibility upgrade is separating “runtime capability” from “benchmark oracle assistance.” The existing `benchmark_playbooks.rs` is explicitly labeled benchmark-specific and contains case-matched deterministic source patches for the Rust SWE-bench top-5 cohort. That is useful as a harness/oracle baseline, but it cannot sit in the normal agent runtime if the goal is to prove generalized coding capability. 

**What to build:** Move exact deterministic playbooks out of `quorp_agent_core` and into a benchmark-only oracle crate, then replace runtime-side repair with a **Repair Pattern Kernel** that is case-agnostic. The kernel should consume diagnostics, context-pack provenance, changed-file leases, validation failures, and Patch VM receipts, then emit typed repair intents such as:

```rust
pub enum RepairPattern {
    BorrowedDataEscapesOwner {
        owner_type: String,
        borrowed_field: String,
        sink_fn: String,
    },
    OverBroadConditionMerge {
        predicate_symbols: Vec<String>,
        conflicting_branch: String,
    },
    DateTimeRoundingBoundary {
        unit: String,
        negative_epoch: bool,
        boundary_test: String,
    },
    MultiFileTemplatePropagation {
        source_template: PathBuf,
        generated_targets: Vec<PathBuf>,
    },
    BuildArtifactPathContract {
        input_path: PathBuf,
        expected_intermediate_dir: PathBuf,
    },
}
```

The model should be asked to choose among repair patterns only after evidence has been collected. The pattern can constrain the edit, but it must **never include benchmark-specific target text**.

**Concrete file changes:**

* Move `crates/quorp_agent_core/src/runtime/benchmark_playbooks.rs` to something like:

  * `benchmark/challenges/rust-swebench-top5/oracles/`
* Add:

  * `crates/quorp_agent_core/src/runtime/repair_contract.rs`
* Modify:

  * `script/quorp-benchmark-regression-gate`

**Hard guardrail snippet:**


### 2. Context OS: automatic compaction with lossless state packets, provenance, and visible context pressure

Quorp already has a persistent SQLite context index under `.quorp/index/index.sqlite`, with file chunks, Rust symbols, imports, tests, proof history, and context-pack provenance.  The runtime already has `PromptCompactionPolicy` variants and rich `AgentTaskStateSnapshot` fields.  But the system needs to become **context-governed at runtime**, not just context-capable.

Current competing tools have moved context management into user-visible surfaces: Codex exposes `/status`, context stats, `/compact`, and footer/status-line controls; Claude Code documents auto-compaction, memory reinjection, path-scoped rules, and context-cost concerns. ([OpenAI Developers][1]) Quorp can beat them by making compaction **ledger-native and proof-aware**.

**What to build:** A `ContextGovernor` that continuously tracks token pressure and produces a **lossless operational state packet** before compaction. This packet is not a vague summary. It is a typed, replay-verifiable snapshot of everything the agent must not forget:

* user objective
* explicit constraints
* current task DAG
* active file leases
* changed files and hashes
* failed attempts and blocked repeats
* validation state
* proof DAG state
* permission/sandbox boundaries
* active rules
* context-pack provenance
* unresolved questions
* next legal action

**Add this data model:**

### 3. Proof-Carrying Task DAG with scoped subagents and worktree returns

Codex’s current docs explicitly describe subagents as a way to keep noisy exploration off the main thread and reduce context pollution, while warning that write-heavy parallel workflows can create conflicts. ([OpenAI Developers][2]) Quorp should go one step further: subagents should not just return prose summaries. They should return **proof-carrying task results**.

Quorp already has the right building blocks: hash-chained ledger events, subscriber cursors, snapshots, and proof DAGs.  The missing piece is a first-class task DAG.

**What to build:** A `quorp_task_dag` crate where every autonomous run is a graph of tasks. Each task has a scope, allowed files, allowed tools, context budget, validation obligations, and return contract. Subagents run in isolated worktrees or temp copies and return:

* summary

### 4. Git-trackable Rule Forge v2: tiny scoped rules with shadow promotion and malicious-rule protection

Quorp’s memory system already has six tiers, failed-attempt records, SQLite/FTS evidence, and retry blocking for exact repeated failures.  `RuleForge` already clusters failures and can promote candidate rules in memory.  The missing breakthrough is to make rules **reviewable, tiny, scoped, lifecycle-managed, and safe**.

Claude Code has persistent memory and path-scoped rules, but its docs also emphasize that memory/rules are context, not hard enforcement. ([Claude API Docs][3]) Codex has command rules with inline tests and project-local trust restrictions. ([OpenAI Developers][4]) Quorp should combine both ideas with proof-backed rule learning.

**What to build:** Use two rule stores:

```text
agent/rules/                  # git-trackable, human-reviewed project rules
  active/
    rust-borrow-owned-decode.rule.toml
  rejected/
  deprecated/
  README.md

.quorp/rules/                 # local/generated/untrusted runtime candidates
  candidates/
  shadow/
  evidence/
  promotion-log.jsonl
```

**Rule schema:**

```toml
id = "rust.borrow.de-owned-deserializer"
state = "active" # candidate | draft | shadow | active | rejected | deprecated
scope = { crate = "bincode", paths = ["src/features/serde/*.rs"] }
trigger = { failure_kind = "borrowed_data_escapes", rust_error = "lifetime" }

statement = """
When fixing owned deserialization, do not return borrowed data tied to a temporary
decoder buffer. Prefer owned allocation or an API path that preserves ownership.
"""

safety = { may_change_permissions = false, may_run_commands = false }
evidence = [
  { run_id = "2026-04-28-205401", proof = "proof-full", outcome = "prevented-repeat" }
]

match = [
  "borrowed data escapes outside of function",
  "temporary value dropped while borrowed"
]
not_match = [
  "simple formatting warning"
]
```

**Runtime rule loading:** Load only the top-N applicable active rules, ranked by scope/path/failure trigger. Never dump the whole rule corpus into the prompt.

**Concrete file changes:**

* Add:

**Hard safety rule:** repo-authored rules can **narrow** behavior but cannot grant more permission, disable verification, expand sandbox roots, approve MCP, or hide files from audit.

```rust
pub fn validate_rule_capability(rule: &RuleCard, source: RuleSource) -> anyhow::Result<()> {
    if matches!(source, RuleSource::ProjectTracked | RuleSource::GeneratedCandidate) {
        if rule.safety.may_change_permissions || rule.safety.may_run_commands {
            anyhow::bail!("project/generated rules cannot grant capabilities");

    Ok(())
}
```

**Acceptance tests:**

* Malicious project rule trying to allow network or disable tests is rejected.
* Replay can compare “with rule” vs “without rule” on historical ledger.

**Expected impact:** This is the real-time learning loop the project needs: fast, local, reviewable, and safe from prompt injection or noisy self-reinforcement.

### 6. Patch VM v2: semantic transactions, risk budgeting, and mandatory write-path convergence

Quorp’s Patch VM is already much better than raw file writes: it validates hashes, previews changes, requires matching preview IDs for high-risk patches, stores rollback tokens, and rejects stale hashes. The next step is to make every mutation a **semantic transaction** with an attached proof obligation.

**What to build:**

* Multi-file transaction object.
* Semantic diff summary: changed symbols, changed tests, changed config keys.

Quorp’s ledger already validates hash chains, supports subscriber cursors, snapshots, replay validation, and ledger append semantics.  The runtime emits detailed events, including parser recovery, verifier drains, subscriber backpressure, failed edits, checkpoints, tool starts/finishes, and model request estimates. 

The next leap is to make Quorp debug itself from its own ledger.

**What to build:** `quorp replay --debug` should reconstruct a run and produce:

* wasted token report
* delayed first-write cause
* redundant read report
* parser recovery cause
* context pressure timeline
* permission prompts
* failed edit fingerprints
* validation proof timeline
* rule hits/misses
* suggested regression tests

**Example output:**

```text
Run 2026-04-28-205401 / case 08
  success: yes
  requests: 9
  first write: turn 9  ⚠ late
  redundant reads: 1
  likely cause: missing read lease after axum/src/routing/mod.rs
  suggested gate: fail if same file is read twice without changed validation state
  generated test: runtime_replay_no_redundant_read_axum_merge
```

**Concrete file changes:**

* Add:

  * `crates/quorp_replay_debugger/`
  * `crates/quorp_cli/src/quorp/replay_debug.rs`
  * `crates/quorp_benchmark/src/trajectory_metrics.rs`
* Modify:

  * `crates/quorp_cli/src/quorp/benchmark`
  * `script/quorp-benchmark-regression-gate`
  * `justfile`

**Acceptance tests:**

* Replay of the 5/5 archive computes the same totals: 5 cases, 28 requests, 180,546 billed tokens.
* Gate flags first-write > N turns for simple localized tasks.
* Gate flags redundant reads unless validation state changed.
* Gate flags benchmark oracle usage in oracle-free mode.
* Gate emits a minimal regression fixture from a real failed/redundant trajectory.

**Expected impact:** This turns Quorp’s own history into a regression factory. That is how it can improve faster than competitors.

### 10. Token Economy Engine: cache-stable prompts, retrieval handles, and model routing by cognitive phase

The 5/5 run had high cache-read in case 06, but still consumed 52,083 billed tokens with 9 requests.  The goal is not just “shorter prompts.” The goal is **stable prompts whose expensive parts are cacheable, and variable parts are tiny handles**.

OpenAI’s current docs emphasize context management, token counting, compaction, and prompt caching in the broader developer platform. ([OpenAI Developers][8]) Claude skills also illustrate a useful pattern: large procedural content should not load until needed, and descriptions should stay compact. ([Claude API Docs][9])

**What to build:**

* Phase-aware model routing:

  * cheap/fast model for repo indexing summaries
  * stronger model for patch synthesis
  * deterministic verifier for validation/proof
* Cache-stable prompt layers:

  * system/runtime contract stable
  * project rules stable
  * task state packet small and changing
  * context handles instead of raw repeated excerpts
* Tool-output handles:

  * `ctxpack:<hash>`
  * `proof:<run>/<seq>`
  * `diff:<patch-id>`
  * `failure:<fingerprint>`

**Concrete file changes:**

* Add:

  * `crates/quorp_provider/src/routing_policy.rs`
  * `crates/quorp_context_model/src/handles.rs`
  * `crates/quorp_agent_core/src/runtime/prompt_layers.rs`
* Modify:

  * `crates/quorp_provider`
  * `crates/quorp_context/src/quorp_context.rs`
  * `crates/quorp_agent_core/src/runtime/state_messages.rs`

**Hard snippet:**

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PromptLayer {
    pub name: String,
    pub cache_key: String,
    pub stability: PromptStability,
    pub estimated_tokens: u32,
    pub body: PromptBody,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum PromptBody {
    Inline(String),
    Handle { kind: String, hash: String, summary: String },
}

#[derive(Debug, Clone, Copy, Serialize, Deserialize)]
pub enum PromptStability {
    StaticRuntime,
    ProjectStable,
    TaskMutable,
    TurnEphemeral,
}
```

**Acceptance tests:**

* Prompt assembler reports layer token costs.
* Repeated turns preserve cache keys for static/project layers.
* Repeated file excerpts are replaced by handles after first use.
* Token gate: 20–35% reduction on replayed long runs without lower pass rate.
* `/status` shows token spend by layer.

**Expected impact:** Lower cost, faster runs, and less context drift.

## The most important build order

I would implement these in this order:

1. **Quarantine benchmark playbooks** and add oracle-free gates.
2. **Context OS state packets + `/status` pressure visibility.**
3. **Read leases + high-level localization tools** to reduce delayed first writes.
4. **Rule Forge v2** with `agent/rules/` and `.quorp/rules/`.
5. **Task DAG + proof-carrying subagents.**
6. **Patch VM semantic transactions.**
7. **Permission policy reports + trust layers.**
8. **Mission-control TUI panels.**
9. **Replay debugger and trajectory regression generation.**
10. **Prompt-layer/token economy routing.**

My strongest bet: **Context OS + proof-carrying task DAG + replayable rule learning** is the unique combination. Claude/Codex have strong UX and permission surfaces, SWE-agent proved ACI matters, and newer benchmarks like Terminal-Bench and SWE-rebench emphasize realistic, fresh, executable tasks rather than narrow static cohorts. ([NeurIPS Proceedings][5]) Quorp’s differentiator should be: **every autonomous decision is visible, compactable, replayable, permissioned, and backed by proof.**
