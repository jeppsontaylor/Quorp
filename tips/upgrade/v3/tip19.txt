Note: the local `git clone` path was blocked by DNS in the execution container, so I inspected Quorp through the available GitHub/API tooling plus current public sources. The ranking below is based on the repo files and run artifacts I could read directly, not a generic wishlist.

My core diagnosis: **Quorp already has unusually strong bones**—strict structured turns, typed tools, event fanout, Patch VM receipts, verification/proof infrastructure, context indexing, permission classification, memory/rule scaffolding, and benchmark telemetry. But the biggest capability jump comes from **making those systems general, durable, visible, and replayable**, while quarantining benchmark-specific oracle code that currently inflates confidence.

The 5/5 run is real evidence, but not “done”: it used **28 requests, 180,546 billed tokens, and ~696 seconds**, with watchdog pressure in four of five cases and late first writes in the two most expensive cases. Case 06 first wrote at step 7 and spent 52,083 tokens; case 08 first wrote at step 9 and spent 62,702 tokens.    

## Ranked top 10 upgrade ideas

### 1. Quarantine benchmark-oracle repair code and replace it with a general Recovery Controller

This is the most important upgrade. The current runtime has excellent recovery concepts, but some of the 5/5 success is entangled with exact Rust SWE-bench top-5 playbooks. `runtime/turn.rs` calls benchmark-specific injection paths such as `maybe_inject_case04_playbook_patch`, `maybe_inject_case05_playbook_patch`, and exact benchmark-source patch injection; `benchmark_playbooks.rs` contains deterministic patch functions for the specific bincode, chrono, axum, cargo-dist, and cc-rs cases.  

That code should not be deleted outright; it should be moved into benchmark fixtures where it can serve as an ablation oracle. But production Quorp needs a general controller that learns from failure packets, owner maps, validation logs, target leases, and patch receipts.

**Files to change or add**

Move or quarantine:

```text
crates/quorp_agent_core/src/runtime/benchmark_playbooks.rs
    -> crates/quorp_benchmark/src/fixtures/rust_swebench_top5_playbooks.rs

crates/quorp_agent_core/src/runtime/turn.rs
    remove direct production calls to maybe_inject_case04_playbook_patch,
    maybe_inject_case05_playbook_patch, maybe_inject_exact_benchmark_source_patch
```

Add:

```text
crates/quorp_agent_core/src/runtime/recovery_controller.rs
crates/quorp_agent_core/src/runtime/failure_classifier.rs
crates/quorp_agent_core/src/runtime/target_lease.rs
crates/quorp_agent_core/src/runtime/repair_contract.rs
script/quorp-no-benchmark-oracles
```

**Hardest code shape**

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]

    pub failure: FailureClass,
    pub target_lease: TargetLease,
    pub evidence: Vec<EvidenceRequirement>,
    pub allowed_write_leases: Vec<WriteLease>,
    pub validation_plan: ValidationPlan,
    pub stale_context_guards: Vec<ContentHashGuard>,
    pub forbidden_actions: Vec<ForbiddenAction>,
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct TargetLease {
    pub path: String,
    pub confidence: f32,
    pub reason: String,
    pub source_refs: Vec<ProofRef>,
    pub expires_after_steps: u32,
}

pub trait RecoveryController {
    fn classify_failure(
        &self,
        proof: &quorp_verify_model::ProofPacket,
        state: &AgentTaskStateSnapshot,
    ) -> anyhow::Result<RecoveryPacket>;

    fn required_next_action(
        &self,
        packet: &RecoveryPacket,
        state: &AgentTaskStateSnapshot,
    ) -> anyhow::Result<ActionConstraint>;

    fn validate_repair_turn(
        &self,
        packet: &RecoveryPacket,
        turn: &AgentTurnResponse,
    ) -> anyhow::Result<RepairTurnDecision>;
}
```

The key is to make recovery **contract-driven**, not benchmark-case-driven. The model can still patch, but the controller says: “You have enough evidence; write within this lease or explain why the lease is wrong.”

**Acceptance tests**

Add a hard CI gate that fails if production runtime files contain benchmark case names, exact challenge paths, or included benchmark patches. Then run two scoreboards: `production` and `fixture-assisted`. The production scoreboard must be the one advertised externally.

**Why this beats competitors**

Codex, Claude Code, OpenHands, and mini-SWE-agent are judged on generalization, not curated top-5 repair playbooks. SWE-bench Verified exists specifically as a human-validated, broader benchmark, and mini-SWE-agent emphasizes a minimal bash-only loop without special scaffolding. ([SWE-bench][1]) ([GitHub][2])

**Expected impact**

Very high trust lift, very high benchmark realism lift, likely short-term score drop on the top-5 fixture cohort. That score drop would be healthy.

### 2. Build a Ledger-Native Context OS with lossless state packets and automatic compaction

Quorp already has pieces of this: `PromptCompactionPolicy`, `AgentTaskStateSnapshot`, `AgentRepairMemory`, context-pack provenance, event fanout, memory tiers, and checkpoint events.   

But the run reports show `compacted_prompt_est=n/a` and monotonic prompt growth in the expensive cases. Case 06 grew from a 2,127-token first prompt to 8,237 max; case 08 grew from 2,554 to 7,747.  

OpenAI’s Codex agent-loop writeup explicitly describes the repeating loop where tool outputs are appended to the prompt, and notes that history growth/context-window management becomes a central agent responsibility. ([OpenAI][3])

The upgrade: make Quorp’s primary cognition object a durable, typed **Mission State Packet**, not an ever-growing chat transcript.

### 3. Add a Delta-Turn Protocol with content-addressed observation handles

This is the most direct token-cutting idea. Today, Quorp’s transcript path still risks re-sending too much text. The context index already stores chunks, symbols, hashes, and context-pack provenance under `.quorp/index/index.sqlite`; use that as the backing store for **handles**, not raw transcript spam. 

Claude’s Skills design is relevant: reference material is loaded only when the skill is invoked, so long material has nearly zero context cost until needed. Quorp can apply the same principle to file slices, validation logs, diffs, and tool outputs. ([Claude API Docs][4])

### 6. Add a Permission Kernel with deterministic hooks, receipts, and secret-aware full-auto

Quorp’s permission classifier is already stronger than a toy model. It classifies read/write/delete/run/network/MCP/browser actions, detects shell metacharacters, compound commands, dependency installs, Docker, git remote mutation, secret reads, generated executables, and more.

### 9. Add a Fast Target Lease and First-Write Optimizer

The clearest inefficiency in the 5/5 run is delayed first writes. Case 06 first wrote at step 7; case 08 first wrote at step 9 and did nine reads for one write.  

Quorp already tracks implementation target leases and repair memory. Make that mechanism faster and stricter.

**Files to add**

```text
crates/quorp_agent_core/src/runtime/target_lease.rs
crates/quorp_agent_core/src/runtime/first_write_gate.rs
crates/quorp_rust_intel/src/failure_to_owner.rs
crates/quorp_verify/src/failure_slicer.rs
```

**Behavior**

After the first failing validation, Quorp should construct:

```text
Failure -> likely owner file -> relevant range -> write lease -> required rerun
```

Then the model gets a bounded choice:

1. patch within the lease,
2. request one specific additional evidence slice,
3. reject the lease with a concrete reason.

No generic rereading.

**Hardest code shape**

```rust
pub enum FirstWriteDecision {
    AllowInspection { reason: String },
    RequireWrite { lease: WriteLease },
    RequireLeaseChallenge { target: TargetLease },
    RunFastValidation { plan: ValidationPlan },
}

pub fn decide_first_write_pressure(
    state: &AgentTaskStateSnapshot,
    repair: &AgentRepairMemory,
    budget: &FirstWriteBudget,
) -> FirstWriteDecision {
    if state.has_mutating_change {
        return FirstWriteDecision::RunFastValidation {
            plan: state.validation_queue.front().cloned().unwrap_or_default(),
        };
    }

    if repair.context_sufficient {
        if let Some(path) = repair.implementation_target_lease.clone() {
            return FirstWriteDecision::RequireWrite {
                lease: WriteLease::owner_file(path),
            };
        }
    }

    FirstWriteDecision::AllowInspection {
        reason: "target lease not yet evidence-backed".to_string(),
    }
}
```

**Acceptance tests**

On held-out Rust tasks, median first valid write should be ≤4 model requests when the failure maps cleanly to an owner file. Add a “late first write” regression metric to `benchmark score --fail-on-regression`.

**Expected impact**

Direct reductions in requests, wall time, and token usage.

---

### 10. Add lazy Skill Capsules for procedures, not prompts

This is the most novel “capability without prompt bloat” idea. Skills should be executable and reviewable procedure capsules: Rust failure triage, Cargo dependency edit, Patch VM conflict recovery, benchmark replay triage, permission audit, etc.

Claude’s Skills model is the right external anchor: skills can contain instructions, scripts, and resources, and the body is loaded only when used. ([Claude API Docs][4]) Quorp can go further by making every skill content-addressed, scoped, sandbox-bound, and replay-tested.

**Files to add**

```text
crates/quorp_skill_model/src/quorp_skill_model.rs
crates/quorp_skill_loader/src/quorp_skill_loader.rs
crates/quorp_context/src/skill_context.rs
crates/quorp_cli/src/commands/skills.rs
agent/skills/
```

**Skill layout**

```text
agent/skills/rust-compiler-triage/
  SKILL.md
  skill.yaml
  scripts/
    explain_error.rs
  tests/
    e0308.fixture.json
```

**Skill metadata**

```yaml
schema: quorp.skill.v1
name: rust-compiler-triage
version: 0.1.0
scope:
  languages: ["rust"]
  tools: ["run_validation", "expand_context", "read_file"]
safety:
  may_execute: false
  may_write: false
  may_network: false
load:
  trigger_terms: ["E0308", "E0382", "cargo check", "rustc"]
  summary_token_budget: 300
```

Only the metadata summary is loaded initially. The full skill body is loaded through `ExpandContext` when the trigger fires.

**Acceptance tests**

A Rust compile-error fixture should trigger the skill metadata, not the full body. The full body should appear only after explicit expansion. Skill files must never grant permission or change sandbox policy.

**Expected impact**

Better specialist behavior without bloating every prompt.

## The PR order I would use

Start with the upgrades that make Quorp more honest and measurable, then ship the compounding capability systems.

1. **Oracle quarantine + CI gate**
   Move exact benchmark patch code out of production runtime, add `script/quorp-no-benchmark-oracles`, keep fixture-assisted mode for ablation only.

2. **Mission State Packet V2 + `/status` context pressure**
   Add the durable state packet, context pressure meter, and compaction invariant tests.

3. **PromptFrame + observation handles**
   Replace raw repeated tool-output stuffing with deltas and expandable handles.

4. **Rule Forge persistence + shadow rules**
   Add `agent/rules/` layout, rule YAML schema, shadow injection, and replay A/B promotion.

5. **TUI Mission Control MVP**
   Implement `/status`, `/tasks`, `/context`, `/proof`, `/permissions`, and replay timeline views using existing RuntimeEventFanout subscribers.

My strongest recommendation: **do #1 and #2 immediately**. Quorp’s biggest opportunity is not another benchmark trick; it is becoming the first terminal coding agent with a truly inspectable, replayable, proof-backed “context and cognition OS.” That would be genuinely distinctive.
