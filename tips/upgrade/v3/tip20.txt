I verified the repo through GitHub/API file reads and public sources after the sandbox could not resolve GitHub for a direct `git clone`. My strongest conclusion: **Quorp should not try to win by adding more prompt text or more ad hoc benchmark repair. It should win by becoming a ledger-native autonomous engineering control plane: compact state, proof-backed tasks, scoped learning, deterministic target routing, and a mission-control TUI.**

The 5/5 Rust SWE-bench run is real evidence, but it also exposes the next bottleneck. The archive reports **5/5 successes, 0 failures, 28 total requests, and 180,546 billed tokens**; cases 06 and 08 were especially expensive, with **9 requests / 52,083 tokens / first write turn 7** and **9 requests / 62,702 tokens / first write turn 9**, respectively.     The most important upgrade theme is therefore: **preserve the good repair behavior, but move it out of prompt bloat and benchmark-specific patch packets into reusable controller state, evidence routing, compaction, and proof-ledger mechanisms.**

External anchors agree with that direction. SWE-bench Verified is a human-filtered 500-instance set meant to make coding-agent evaluation more reliable, but newer work such as SWE-rebench highlights contamination and freshness risks, so Quorp should not optimize only for the top-5 cohort. ([SWE-bench][1]) Current tools are also moving toward explicit subagents, approval/sandbox controls, status/context visibility, and compaction; Codex documents AGENTS.md layering, subagents, `/status`, `/compact`, and sandbox/approval behavior, while Claude Code exposes hooks and pre-tool permission control. ([OpenAI Developers][2]) A 2026 Claude Code architecture analysis also emphasizes that the simple model loop is not the hard part; the surrounding systems for permissions, compaction, subagents, append-only storage, and extensibility are where most capability lives. ([arXiv][3])

# 1. Ledger-native Context OS + lossless compaction

**This is my highest-confidence upgrade.** Quorp already has the ingredients: a hash-chained run ledger, context packs with provenance, an index, runtime snapshots, proof packets, Patch VM receipts, and prompt compaction policies. The missing breakthrough is to treat context as a **managed operating system resource** rather than a prompt string.

Current runtime already exposes `PromptCompactionPolicy` variants and an `AgentTaskStateSnapshot`, and the ledger is hash-chained with snapshots and subscriber cursors.   The context compiler already supports token budgets, handles, memory recall, agent contracts, and provenance recording; the SQLite index already stores files, lexical chunks, symbols, imports, tests, diagnostics, proof history, and context-pack items.   But the 5/5 run still had prompt growth and high-token cases, especially 06 and 08.  

**The idea:** add `CompactionPacketV2`, a deterministic, ledger-replayable state packet with hard sections that cannot be lost: objective, constraints, open task DAG nodes, decisions, failed attempts, observed file hashes, proof state, permission boundaries, active rules, working set, and context handles.

# 2. Evidenceboard target router + first-write SLA

The most actionable signal from the 5/5 run is not “the model solved it.” It is: **when the controller gave narrow evidence and a target lease, the system succeeded; when it spent many turns gathering evidence, token cost exploded.**

Case 06 had first write on turn 7 and case 08 on turn 9; both used a patch packet / target lease eventually.   The runtime currently has benchmark repair packet machinery and injected reads/patch guidance; `state_messages.rs` constructs benchmark repair packets, and `turn.rs` contains benchmark-specific repair injection and redundant-inspection loop guards.   The right upgrade is to generalize that into a **deterministic Evidenceboard** for all tasks.

**The idea:** before the first model call, or immediately after a failing validation, a local controller builds a compact evidence card:

```text
objective
failure anchor
top 3 target files
owner/test/proof lanes
known file hashes
minimal read ranges
risky paths to avoid
required next action contract
```

Then the model sees the Evidenceboard, not a loose prompt asking it to explore.

### Hard snippet: trust-aware rule loader

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize, PartialEq, Eq)]
#[serde(rename_all = "snake_case")]
pub enum RuleTrust {
    User,
    ReviewedProject,
    UntrustedProject,
    GeneratedCandidate,
}

#[derive(Debug, Clone)]
pub struct LoadedRule {
    pub rule: quorp_rule_model::Rule,
    pub path: std::path::PathBuf,
    pub sha256: String,
    pub trust: RuleTrust,
}

pub fn load_rules_for_prompt(
    workspace_root: &std::path::Path,
    trust_store: &TrustStore,
    scope: &RuleQueryScope,
) -> anyhow::Result<Vec<LoadedRule>> {
    let mut loaded = Vec::new();

    for path in discover_rule_files(workspace_root.join("agent/rules"))? {
        let bytes = std::fs::read(&path)?;
        let sha256 = quorp_agent_core::ledger::sha256_hex(&bytes);
        let parsed: RuleFile = serde_json::from_slice(&bytes)?;

        let trust = if trust_store.is_user_approved(workspace_root, &sha256)? {
            RuleTrust::ReviewedProject
        } else {
            RuleTrust::UntrustedProject
        };

        if !parsed.matches(scope) {
            continue;
        }

        if trust == RuleTrust::UntrustedProject && parsed.allowed_effects.contains(&RuleEffect::PermissionGrant) {
            continue;
        }

        loaded.push(LoadedRule {
            rule: parsed.into_rule()?,
            path,
            sha256,
            trust,
        });
    }

    loaded.sort_by(|left, right| {
        right.rule.confidence
            .partial_cmp(&left.rule.confidence)
            .unwrap_or(std::cmp::Ordering::Equal)
            .then_with(|| left.rule.id.as_str().cmp(right.rule.id.as_str()))
    });

    Ok(loaded.into_iter().take(8).collect())
}
```

# 6. Capability firewall with explainable permission policy

Quorp’s permission engine is already stronger than I expected. It has five modes, yolo denied on host, capability tokens for shell meta, compound commands, filesystem writes, network, dependency installs, Docker, git remote mutation, find delete/exec, secrets reads, generated executables, MCP, and browser. 

The upgrade is to make it a **capability firewall**: every action receives an explainable verdict with trust source, sandbox surface, risk tokens, and a replayable policy decision. Claude Code’s hook docs show how valuable pre-tool allow/deny/ask/defer controls are; Quorp should implement a Rust-native version tied to ledger and TUI, not arbitrary shell scripts as the primary control path. ([Claude API Docs][7]) OpenHands also emphasizes isolated Docker/Kubernetes runtimes with auditability; Quorp should bring that discipline to local terminal full-auto. ([OpenHands][8])

### Hard snippet: explainable verdict

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum PermissionVerdict {
    Allow {
        scope: AllowScope,
        reason: String,
        tokens: Vec<CapabilityToken>,
    },
    Ask {
        reason: String,
        suggested_scopes: Vec<AllowScope>,
        tokens: Vec<CapabilityToken>,
    },
    Deny {
        reason: String,
        tokens: Vec<CapabilityToken>,
    },
    Defer {
        ticket: String,
        reason: String,
    },
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum AllowScope {
    Once,
    Session,
    ProjectReviewed { policy_hash: String },
    UserGlobal { policy_hash: String },
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct PermissionAuditRecord {
    pub ledger_ref: LedgerRef,
    pub tool_name: String,
    pub rendered_action: String,
    pub surface: ExecutionSurface,
    pub trust_state: TrustState,
    pub verdict: PermissionVerdict,
}
```

### Policy rules I would enforce

```text
Untrusted repo:
  - cannot enable network from project config
  - cannot auto-approve dependency installs
  - cannot auto-run generated executables
  - cannot grant MCP/browser access
  - cannot read secret-like paths without ask/deny policy
  - cannot execute compound shell commands without parsed-token review

Trusted repo:
  - may define reviewed project workflows
  - still cannot override user/enterprise deny rules

Full-auto:
  - only inside sandbox/worktree/tmp-copy
  - network off unless explicit reviewed policy
  - dependency installs require explicit lane and cache isolation
```

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct ToolReceipt {
    pub id: String,
    pub tool_name: String,
    pub input_hash: String,
    pub output_hash: String,
    pub summary: String,
    pub artifact_path: Option<std::path::PathBuf>,
```

# 8. Anti-overfit replay/eval scientist

The runtime currently has benchmark-specific repair behavior that helped the 5/5 run. That is acceptable in a benchmark mode only if it is quarantined, tested, and proven not to leak into general runs. `turn.rs` contains benchmark playbook injection paths, and the workplan/phase tracker already identify benchmark quarantine and broader eval lanes as gaps.   

**The idea:** build an eval scientist that treats every run as a replayable experiment and every benchmark-specific helper as suspect unless it passes generalization gates.

### Eval lanes I would add

```text
1. rust-swebench-top5 replay, current known green lane
2. rust-swebench-holdout, unseen Rust cases
3. SWE-bench Verified, official comparison lane
4. Terminal-Bench adapter, real terminal tasks with sandbox/test scripts
5. SWE-rebench-style fresh task lane, contamination resistance
6. context-compaction torture lane, 100+ turn synthetic repo task
7. rule-learning lane, repeated failure must produce useful shadow rule
8. security-full-auto lane, malicious repo config/prompt injection/network/deps
```

Terminal-Bench is useful because it evaluates real terminal tasks with a sandbox and test scripts, not just patch generation. ([Terminal-Bench][10]) SWE-rebench-style fresh task generation is useful because static SWE benchmarks can become contaminated. ([Cool Papers][11])

## The biggest “do not do this” warnings

Do **not** keep adding benchmark-specific repair packets to generic runtime. The current repair behavior is valuable, but the named playbook injections in `turn.rs` should become quarantined benchmark-only modules or generalized Evidenceboard policies. 

Do **not** dump more AGENTS.md / `.rules` text into every prompt. Recent evidence suggests broad repository context can increase cost and reduce success when it adds unnecessary requirements. ([arXiv][4]) Quorp should prefer scoped, evidence-backed, budget-capped rules.

Do **not** make full-auto safe by hiding risk. Make it safe by showing the capability firewall, sandbox surface, trust state, policy source, and replayable audit record.

Do **not** build a pretty TUI that is disconnected from the runtime. The TUI should render the ledger, context budget, task DAG, proof packets, rules, and permission queue directly.

---

## My strongest bet

The most uniquely powerful Quorp capability would be:

> **A ledger-native Context OS that compacts, routes, learns, delegates, verifies, and renders from the same append-only evidence stream.**

That is how Quorp can become more than “another coding CLI.” Codex and Claude Code already have strong model loops, approvals, subagents, context files, and compaction surfaces. Quorp’s chance to beat them is to make every important agent cognition artifact — objective, task, rule, proof, patch, permission, context atom, recovery decision — **typed, hash-linked, replayable, inspectable, and visible in the terminal.**
