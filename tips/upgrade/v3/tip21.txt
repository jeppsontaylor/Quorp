I could not successfully `git clone` in the sandbox because DNS resolution for GitHub failed, so I audited through GitHub raw/html views and the published benchmark artifacts. I’m not claiming I ran Quorp locally. The strongest evidence I found: Quorp already has a serious runtime shape—headless `quorp run`/`exec`, typed tools, git-worktree/tmp-copy sandboxing, permission modes, Patch VM receipts, proof packets, context/memory/rule scaffolding, and benchmark receipts—but the 5/5 Rust SWE-bench run is not clean capability evidence yet. The archive reports 5/5 successes, 28 requests, 180,546 billed tokens, and about 696 seconds wall time; however, two cases were rescued by deterministic benchmark source-patch injection, and several cases show delayed first writes, watchdog pressure, parser recovery, or large full-file writes. ([GitHub][1])

Below are my **top 10 ranked upgrade ideas**. My ranking optimizes for coding capability, token reduction, memory/state quality, safety, and a TUI experience that can become meaningfully better than Claude Code CLI or Codex CLI.

## 1. Quarantine benchmark-oracle repair and replace it with a no-oracle repair controller

**Why this is #1:** Quorp’s biggest near-term credibility and generalization risk is that the runtime currently contains benchmark-specific deterministic repair playbooks. `benchmark_playbooks.rs` is explicitly labeled “Benchmark-specific deterministic repair playbooks,” includes exact top-5 case variants, and even pulls a `test.patch` from the benchmark challenge tree for cargo-dist. The run logs for cases 09 and 10 show `benchmark_repair_injection` with deterministic source patch writes after parser recovery. That made 5/5 possible, but it is not a general autonomous coding capability. ([GitHub][2])

**Best version of the idea:** keep oracle playbooks only as a private benchmark debugging fixture, never as a runtime capability path. Replace them with a generalized, evidence-gated repair controller that can read owner slices, classify failures, propose target files, enforce patch contracts, and retry without any hardcoded benchmark patch knowledge.

// crates/quorp_agent_core/src/runtime/repair_controller.rs

use crate::runtime::{
    AgentTaskStateSnapshot, RuntimeEventFanout,
};
use quorp_patch_model::PatchReceipt;
use quorp_verify_model::{FailurePacket, ProofPacket};

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum RepairDirective {
    RequireEvidenceRead {
        path: String,
        start_line: Option<usize>,
        end_line: Option<usize>,
        reason: String,
    },
    RequireMinimalPatch {
        owner_path: String,
        allowed_ranges: Vec<LineRange>,
        stale_hash_policy: StaleHashPolicy,
        reason: String,
    },
    RequireValidation {
        command: String,
        reason: String,
    },
    EscalateToHuman {
        reason: String,
    },
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct LineRange {
    pub start: usize,
    pub end: usize,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum StaleHashPolicy {
    ReReadAndReplan,
    Deny,
}

pub trait RepairController: Send + Sync {
    fn name(&self) -> &'static str;

    /// Must never inject source code. It can only request evidence,
    /// constrain tools, or schedule validation.
    fn next_directive(
        &self,
        state: &AgentTaskStateSnapshot,
        failure: &FailurePacket,
        latest_proof: Option<&ProofPacket>,
    ) -> Option<RepairDirective>;

    fn may_inject_patch_bytes(&self) -> bool {
        false
    }
}

pub struct NoOracleRepairController;

impl RepairController for NoOracleRepairController {
    fn name(&self) -> &'static str {
        "no_oracle_repair"
    }

    fn next_directive(
        &self,
        state: &AgentTaskStateSnapshot,
        failure: &FailurePacket,
        latest_proof: Option<&ProofPacket>,
    ) -> Option<RepairDirective> {
        let owner = failure
            .owner_hint
            .clone()
            .or_else(|| state.last_implementation_target.clone())?;

        if !state.has_fresh_read_for(&owner) {
            return Some(RepairDirective::RequireEvidenceRead {
                path: owner,
                start_line: failure.suspect_range.map(|r| r.start),
                end_line: failure.suspect_range.map(|r| r.end),
                reason: "repair requires fresh owner-file evidence".into(),
            });
        }

        Some(RepairDirective::RequireMinimalPatch {
            owner_path: owner,
            allowed_ranges: failure
                .suspect_range
                .map(|r| vec![LineRange { start: r.start, end: r.end }])
                .unwrap_or_default(),
            stale_hash_policy: StaleHashPolicy::ReReadAndReplan,
            reason: "patch must be grounded in current failure packet and fresh read".into(),
        })
    }
}
```

## 2. Build a ledger-native Context OS with lossless compaction and visible token pressure

**Why this is #2:** Quorp already has a context compiler, SQLite index, context-pack provenance, prompt compaction policy enum, memory tiers, and runtime checkpoint structures. But the logs show expensive prompt growth and high token burn: case 06 consumed 52,083 billed tokens with first write at turn 7; case 08 consumed 62,702 billed tokens with first write at turn 9. The current runtime has `PromptCompactionPolicy` variants such as `BenchmarkStatePacket`, `Last8Ledger1024`, and `BenchmarkRepairMinimal`, which suggests compaction is present but still policy-fragmented and not yet a general “context operating system.” ([GitHub][3])

**Best version of the idea:** every session gets a durable, replayable **Context Capsule** that is smaller than a transcript but richer than a summary. It preserves objective, constraints, decisions, open tasks, failed attempts, patch/proof heads, security boundaries, known risky commands, important files, stale hashes, and context-pack provenance. `/status` and the TUI should show context pressure before it becomes a failure mode.

## 3. Make all writes delta-first through a Semantic Patch VM write governor

**Why this is #3:** Quorp already has an unusually strong foundation here: Patch VM computes SHA-256 file hashes, validates preconditions, previews changes, tracks touched paths, creates rollback tokens, rejects stale expected hashes, and has a full-file rewrite rejection test for large updates. But the benchmark logs still show risky large writes: case 07 wrote an 805-line `src/round.rs`; case 08 wrote a 705-line `axum/src/routing/mod.rs`; case 10 wrote a 4,278-line `src/lib.rs`. That is a token, safety, and regression risk. ([GitHub][4])

**Best version of the idea:** make `write_file` a privileged escape hatch, not the default. The agent should normally use `replace_range`, `replace_block`, `modify_toml`, `structural_edit_preview`, or a new semantic patch op with expected hashes. Large full-file writes should be denied unless the file is generated, allowlisted, or human-approved.

// crates/quorp_agent_core/src/runtime/write_policy.rs

use quorp_patch_vm::{hash_bytes, FileChange, FileChangeKind, PatchVmPolicy};
use std::path::Path;

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum WriteDecision {
    AllowDelta,
    AllowFullFile { reason: String },
    DenyRequireDelta { reason: String },
    PromptHuman { reason: String },
}

#[derive(Debug, Clone)]
pub struct WritePolicy {
    pub max_full_file_bytes: usize,
    pub max_full_file_lines: usize,
    pub generated_globs: Vec<String>,
    pub allow_large_rewrite_globs: Vec<String>,
}

impl Default for WritePolicy {
    fn default() -> Self {
        Self {
            max_full_file_bytes: 64 * 1024,
            max_full_file_lines: 500,
            generated_globs: vec![
                "**/Cargo.lock".into(),
                "**/target/**".into(),
                "**/*.snap".into(),
            ],
            allow_large_rewrite_globs: vec![],
        }
    }
}

pub fn classify_write(
    policy: &WritePolicy,
    path: &Path,
    before: &[u8],
    after: &[u8],
    tool_name: &str,
) -> WriteDecision {
    let before_lines = bytecount::count(before, b'\n') + 1;
    let after_lines = bytecount::count(after, b'\n') + 1;

    let is_large = after.len() > policy.max_full_file_bytes
        || after_lines > policy.max_full_file_lines;

    let is_delta_tool = matches!(
        tool_name,
        "replace_range" | "replace_block" | "apply_patch" | "modify_toml" | "structural_edit_preview"
    );

    if is_delta_tool {
        return WriteDecision::AllowDelta;
    }

    if !is_large {
        return WriteDecision::PromptHuman {
            reason: "full-file write is small but should be reviewed".into(),
        };
    }

    if matches_generated_or_allowlisted(policy, path) {
        return WriteDecision::AllowFullFile {
            reason: "large rewrite is generated or allowlisted".into(),
        };
    }

    WriteDecision::DenyRequireDelta {
        reason: format!(
            "large full-file write denied: {} lines, {} bytes; use semantic patch or replace_range with hash {}",
            after_lines,
            after.len(),
            hash_bytes(before).0
        ),
    }
}
```

## 4. Add proof-backed task DAGs and subagents in isolated worktrees

**Why this is #4:** Current Quorp behavior appears mostly linear. The successful run still spent many turns reading and recovering before first write; case 08 first wrote at turn 9, and case 06 first wrote at turn 7. Modern competitors are moving toward subagents and lifecycle hooks: Claude Code exposes hook events for task creation/completion, subagent start/stop, pre/post compaction, worktree create/remove, and permission decisions; Codex documents `/agent` for active agent threads and subagents. ([GitHub][5])

**Best version of the idea:** Quorp should not just “spawn subagents.” It should spawn **bounded proof workers** in isolated worktrees. A worker gets a small objective, restricted context, a permission envelope, and a required return contract. The parent never ingests the full child transcript. It ingests a compact proof-backed result: patch receipt, proof packets, touched files, risk classification, and a capsule ref.

## 5. Ship git-trackable rule learning with trust, shadow evaluation, and negative memory

**Why this is #5:** Quorp already has the right primitives: memory tiers include working, episodic, semantic, procedural, negative, failed-attempts, and rule memory; failed attempts can block repeated bad fixes; Rule Forge clusters failures and creates candidate rules after repeated evidence. But the handoff docs still call out durable memory/rule subscriber workers and richer lifecycle integration as future work. ([GitHub][6])

**Best version of the idea:** create tiny, reviewable, git-trackable YAML rules under `agent/rules/`, but never let repo-authored rules escalate permissions. Rules should move through `draft → shadow → active → deprecated` only with evidence. Shadow rules run on replay/eval and can influence telemetry before they influence live behavior.

## 6. Build a capability firewall for full-auto mode

**Why this is #6:** Quorp’s permission engine is already better than a toy: it has modes for read-only, ask, accept-edits, auto-safe, and yolo-sandbox; it classifies capabilities such as write, command, network, MCP, browser; and it detects shell meta, compound commands, dependency installs, Docker, git remote mutation, secret reads, and generated executables. But the policy still needs to become a composable full-auto firewall with receipts, sandbox egress controls, and untrusted-repo config isolation. Codex documents sandbox/approval controls with no-network defaults and workspace-limited writes; Claude Code exposes lifecycle hooks and permission events that users can customize. ([GitHub][8])

**Best version of the idea:** every mutating or capability-expanding action passes through a **Capability Firewall** that emits a signed audit decision and knows the execution surface: host, sandbox, worktree, browser, MCP, network, generated binary, dependency install. `auto-safe` should mean: safe in sandbox, deterministic where possible, no secrets, no network unless granted, no generated executable unless quarantined and scanned, no repo config escalation.

## 8. Make the TUI a proof-and-context mission control system, not a prettier transcript

**Why this is #8:** Quorp already has rendering primitives, permission modals, status footer, session frame, command cards, task rows, transcript rendering, no-color accessibility potential, and a slash registry with `/plan`, `/act`, `/permissions`, `/memory`, `/rules`, `/session`, `/status`, `/compact`, `/diff`, `/verify`, and more. But `quorp_render.rs` is still tiny and describes primitives rather than a fully wired product experience. ## 9. Add a no-oracle, anti-overfitting evaluation program with token/capability scorecards

**Why this is #9:** A 5-case top cohort is useful behavioral evidence, but it is not enough. SWE-bench Verified is a 500-instance human-filtered subset, and the official leaderboard includes many system types from simple loops to RAG and multi-rollout/review systems. Terminal-Bench 2.0 adds 89 terminal tasks across software engineering, ML, security, data science, and more. SWE-bench Live is explicitly designed to update real-world issue-resolving tasks monthly for contamination resistance. ([SWE-bench][10])

Capability scoreboard:

## 10. Introduce provider lanes: scout, builder, verifier, reviewer

**Why this is #10:** The run used 28 requests and 180,546 billed tokens. Several first-token latencies were around tens of seconds, and expensive long prompts were used for exploratory reads and recovery. Quorp’s provider routing is already a major architectural seam, and current competitors are increasingly using model switching, subagents, and smaller/cheaper models for exploration or parallel work. Codex docs explicitly discuss model switching and using smaller models for less reasoning-intensive subagent work in recent changelog material, while Claude/Codex both expose model controls through slash/product UI. ([GitHub][11])

**Best version of the idea:** route tasks through specialized lanes:

```text
Scout lane:     cheap/fast model, read-only, retrieves evidence, no writes.
Builder lane:   strong model, minimal context capsule + owner slices, writes only through Patch VM.
Verifier lane:  cheap deterministic/tool-heavy lane, parses failures and proof packets.
Reviewer lane:  strong or medium model, reviews diff/proof, looks for missed tests/security issues.
```

**Files to add or modify:**

```text
crates/quorp_provider/src/router.rs
crates/quorp_provider/src/lane_policy.rs
crates/quorp_agent_core/src/runtime/model_policy.rs
crates/quorp_agent_core/src/runtime/tool_lane.rs
crates/quorp_cli/src/quorp/provider_commands.rs
crates/quorp_render/src/widgets/provider_lanes.rs
```

**Core policy sketch:**

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
pub enum ProviderLane {
    Scout,
    Builder,
    Verifier,
    Reviewer,
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct LanePolicy {
    pub lane: ProviderLane,
    pub model_profile: String,
    pub max_prompt_tokens: usize,
    pub allowed_tools: Vec<String>,
    pub can_write: bool,
    pub can_request_permissions: bool,
    pub context_strategy: ContextStrategy,
}

impl LanePolicy {
    pub fn default_scout() -> Self {
        Self {
            lane: ProviderLane::Scout,
            model_profile: "fast-small".into(),
            max_prompt_tokens: 16_000,
            allowed_tools: vec![
                "read_file".into(),
                "search_text".into(),
                "search_symbols".into(),
                "cargo_diagnostics".into(),
            ],
            can_write: false,
            can_request_permissions: false,
            context_strategy: ContextStrategy::CapsulePlusEvidence,
        }
    }
}
```

**Acceptance gates:**

```text
cargo test -p quorp_provider lane_policy_blocks_scout_writes
cargo test -p quorp_agent_core builder_receives_capsule_not_full_transcript
cargo test -p quorp_benchmark token_per_success_regression_gate
```

A good initial target: **35–50% token reduction** on no-oracle replay/eval without reducing success rate, mostly by preventing full transcripts and long exploratory context from going to the builder model.

# My top 3 to build first

1. **No-oracle repair controller and benchmark quarantine.** This is existential for credibility. Keep the 5/5 run as behavioral evidence, but stop letting benchmark patch injection appear in capability runs.

2. **Ledger-native Context OS.** This is the highest-leverage token and reliability improvement. It turns long sessions from transcript soup into durable state.

3. **Delta-first Semantic Patch VM write governor.** This converts Quorp’s existing Patch VM into a hard safety and token-efficiency advantage.

After that, I would build **task DAG/subagents**, then **rule learning**, then the **TUI mission-control layer**. The endgame is not “a better chat CLI.” It is a replayable, proof-backed, permission-aware coding operating system where the agent’s state, risks, tasks, context, learned rules, patches, and validations are all visible, resumable, and testable.
