I could not complete a normal local `git clone` in the execution sandbox because GitHub DNS resolution failed, so I audited through GitHub web/raw artifacts and downloaded the available raw source files. The recommendation quality is still grounded in the repo files, run archive, and current public product/research sources, but I did not run Quorp’s test suite locally.

My strongest read: **Quorp already has unusually good primitives**: Rust-native runtime state, typed actions, bounded event fanout, Patch VM receipts, verification packets, context indexing, memory/rule scaffolding, permissions modes, slash commands, and benchmark artifacts. The opportunity is to turn those primitives into a **ledger-native autonomous coding operating system**: compact state, proof-backed task DAGs, safer full-auto, rule learning, and a terminal “mission control” UI. The 5/5 run is promising, but the archive also shows heavy token use, delayed first writes, parser recovery, watchdog pressure, and benchmark-specific deterministic patch injection in at least two cases, so I would not treat it as proof of general capability. Quorp’s repo describes the existing agent/runtime/context/Patch VM/eval scaffolding, and the batch report records 5/5 success with 28 requests, 180,546 billed tokens, and ~696 seconds wall clock. ([GitHub][1])

## Ranked best ideas

### 1. Build a **Ledger-Native Context OS** with lossless compaction and visible token pressure

**Why this is #1:** It improves coding capability, memory, long-session reliability, and token cost simultaneously. Quorp already has the ingredients: runtime snapshots, event logs, context index provenance, memory tiers, and a `PromptCompactionPolicy`. But the current system should become more than “summarize the chat.” It should compile the ledger into a structured, replayable, hash-addressed **mission state packet** that preserves objectives, constraints, task DAG, failed attempts, decisions, validation state, security boundaries, and context provenance.

The 5/5 run shows why this matters: the top-5 batch consumed 180,546 billed tokens, and cases like bincode and axum each used 9 requests and more than 50k/60k tokens. Axum had first write at turn 9 and one redundant read, which is exactly the type of behavior a context OS should compress and redirect earlier. ([GitHub][2])

**Core idea:** Every model request should be built from four layers:

1. **Stable prefix:** system/developer instructions, tool schema, safety policy, repo rules. Keep byte-identical where possible for cache efficiency.
2. **Mission state packet:** canonical JSON generated from the run ledger.
3. **Working context handles:** file/range/proof/memory references, not giant pasted blobs.
4. **Volatile tail:** only the latest user turn, newest tool outcomes, and active recovery packet.

This aligns with OpenAI’s own Codex prompt-caching guidance: exact prefix matches matter, and changing tools/config/sandbox/CWD can break cache reuse, so static content should remain stable and variable content should move later. ([OpenAI][3])

```rust
// crates/quorp_context/src/budget.rs

pub struct ContextBudgetController {
    pub warn_ratio: f32,       // e.g. 0.55
    pub compact_ratio: f32,    // e.g. 0.70
    pub hard_stop_ratio: f32,  // e.g. 0.88
}

impl ContextBudgetController {
    pub fn classify(&self, used: u32, limit: u32, reserved: u32) -> ContextPressure {
        let available = limit.saturating_sub(reserved).max(1);
        let ratio = used as f32 / available as f32;

        if ratio >= self.hard_stop_ratio {
            ContextPressure::HardStop
        } else if ratio >= self.compact_ratio {
            ContextPressure::Red
        } else if ratio >= self.warn_ratio {
            ContextPressure::Yellow
        } else {
            ContextPressure::Green
        }
    }

    pub fn maybe_compact(
        &self,
        input: CompactionInput<'_>,
    ) -> anyhow::Result<CompactionDecision> {
        let pressure = self.classify(
            input.estimated_prompt_tokens,
            input.model_context_limit,
            input.reserved_completion_tokens,
        );

        if pressure < ContextPressure::Red && !input.force {
            return Ok(CompactionDecision::Noop { pressure });
        }

        let packet = MissionStatePacketBuilder::new()
            .from_ledger(input.ledger)?
            .from_runtime_snapshot(input.runtime_snapshot)?
            .from_context_index(input.context_index)?
            .preserve_security_boundaries()
            .preserve_open_tasks()
            .preserve_failed_attempts()
            .preserve_validation_state()
            .build_canonical()?;

        Ok(CompactionDecision::ReplaceTranscriptPrefix {
            pressure,
            packet,
            dropped_transcript_range: input.compactable_span(),
        })
    }
}
```

### 2. Quarantine benchmark oracle patches and replace them with a **General Repair Planner**

**Why this is urgent:** The current 5/5 result includes evidence of deterministic benchmark-specific repair. That helped the run, but it is the biggest credibility and generalization risk. The runtime source contains named playbooks such as `CargoDistCreateRelease`, `CcRsCompileIntermediates`, `AxumFallback`, `ChronoEpochRound`, and `BincodeDeOwnedBorrow`, and the event logs for cargo-dist and cc-rs show deterministic “Case 04” and “Case 05” source patch injection after parser recovery. The cargo-dist case wrote multiple large files, and the cc-rs case wrote an entire 4,278-line `src/lib.rs`. ([GitHub][4])

**Core idea:** Keep benchmark playbooks only as an **eval-only oracle baseline**, never as product/runtime agent behavior. Replace exact case patch injection with a generic, testable **Repair Planner**:

```text
failure classification
→ target lease
→ anchored reread
→ minimal patch plan
→ Patch VM guarded write
→ validation ladder
→ no-progress detector
→ rollback or task replan
```

This lets Quorp retain the useful part of the 5/5 run — structured repair packets, write guards, validation queues, failed-edit memory — without benchmark hardcoding.

```rust
// crates/quorp_repair/src/policy.rs

pub trait RepairPolicy: Send + Sync {
    fn decide(&self, cx: &RepairContext) -> anyhow::Result<RepairDecision>;
}

#[derive(Debug, Clone)]
pub struct RepairContext {
    pub goal: String,
    pub latest_failure: FailureClassification,
    pub runtime_state: AgentTaskStateSnapshot,
    pub validation_history: Vec<ValidationObservation>,
    pub touched_files: Vec<TouchedFile>,
    pub available_context: Vec<ContextItemRef>,
    pub benchmark_metadata: Option<BenchmarkMetadata>, // visible but not oracle-authoritative
}

#[derive(Debug, Clone)]
pub enum RepairDecision {
    AskModelWithRecoveryPacket(RecoveryPacket),
    RequireAnchoredRead {
        path: String,
        reason: String,
        suggested_ranges: Vec<LineRange>,
    },
    LeasePatchTarget {
        lease: PatchLease,
        required_preconditions: Vec<HashPrecondition>,
    },
    RollBackAndReplan {
        checkpoint_id: String,
        reason: String,
    },
    StopForHuman {
        reason: String,
    },
}

// Product/runtime policy must never emit exact benchmark patches.
pub struct ProductRepairPolicy;

impl RepairPolicy for ProductRepairPolicy {
    fn decide(&self, cx: &RepairContext) -> anyhow::Result<RepairDecision> {
        match &cx.latest_failure {
            FailureClassification::ParserFailure { .. } => {
                Ok(RepairDecision::AskModelWithRecoveryPacket(
                    RecoveryPacket::strict_json_retry(cx)?
                ))
            }
            FailureClassification::TestFailure { primary_span, .. } => {
                Ok(RepairDecision::RequireAnchoredRead {
                    path: primary_span.path.clone(),
                    reason: "Primary failing span must be reread before writing".into(),
                    suggested_ranges: vec![primary_span.expand(40)],
                })
            }
            FailureClassification::NoProgress { checkpoint_id, .. } => {
                Ok(RepairDecision::RollBackAndReplan {
                    checkpoint_id: checkpoint_id.clone(),
                    reason: "No progress across repair window".into(),
                })
            }
            _ => Ok(RepairDecision::AskModelWithRecoveryPacket(
                RecoveryPacket::general(cx)?
            )),
        }
    }
}
```

### 3. Implement **Rule Forge v2**: tiny, scoped, git-trackable rules with shadow evaluation

**Why this is high leverage:** Quorp has memory and rule scaffolding already: working/episodic/semantic/procedural/negative/rule tiers, failure clustering, draft rules, and shadow-validation ambitions. The missing product breakthrough is to turn mistakes into tiny reviewable rules that can be loaded fast, scoped narrowly, evaluated on replay, and protected from malicious repo escalation.

Claude Code has visible memory, skills, hooks, tasks, permissions, and compaction surfaces; OpenAI Codex reads `AGENTS.md` and treats trusted project config differently from untrusted project config. Quorp can go further by making rules **evidence-backed, lifecycle-managed, replay-tested, and permission-safe**. ([Claude][5])

**Core idea:** Two locations:

```text
agent/rules/                         # human-reviewed, git-trackable project rules
.quorp/rules/candidates/             # generated candidates, not trusted by default
.quorp/rules/shadow-results.jsonl     # replay/eval evidence
```

Rules have lifecycle:

```text
candidate → draft → shadow → active → deprecated
                       ↘ rejected
```

Generated rules should **never** grant permissions. Untrusted repo-authored rules can give low-priority behavioral guidance only after trust classification. Permission-granting rules require explicit human approval and a separate schema.

```rust
// crates/quorp_rule_forge/src/selector.rs

pub struct RuleSelector {
    active_rules: Vec<CompiledRule>,
    shadow_rules: Vec<CompiledRule>,
}

impl RuleSelector {
    pub fn select_for_turn(
        &self,
        cx: &RuleSelectionContext<'_>,
    ) -> SelectedRules {
        let mut selected = Vec::new();
        let mut shadow = Vec::new();

        for rule in &self.active_rules {
            if rule.matches(cx) && rule.trust.allows_prompt_injection() {
                selected.push(rule.render(cx));
            }
        }

        for rule in &self.shadow_rules {
            if rule.matches(cx) {
                shadow.push(ShadowRuleObservation {
                    rule_id: rule.id.clone(),
                    would_have_injected: rule.render(cx),
                    evidence_key: cx.evidence_key(),
                });
            }
        }

        SelectedRules {
            prompt_rules: selected.into_iter().take(cx.max_rules).collect(),
            shadow_observations: shadow,
        }
    }
}
```

### 4. Add a **proof-backed task DAG** with subagents, worktrees, and resumable task leases

**Why it matters:** Long-form coding fails when the agent loses track of “what is open,” “what was already tried,” and “what proof exists.” Quorp currently has runtime state and slash command scaffolding, but it should promote tasks into first-class ledger objects.

Claude Code now exposes `/tasks` and `/batch`, with `/batch` decomposing large changes into many worktree agents. GitHub Copilot’s coding agent can work in a cloud dev environment, plan, make changes, validate, and create PRs. Quorp can beat both by making every subagent return **proof-backed, replayable, and merge-gated** rather than just “agent said done.” ([Claude][5])

**Core idea:**

```text
Task DAG node
→ has objective, scope, allowed files, permissions, context refs
→ can be leased to main agent or subagent
→ writes through Patch VM
→ validates through proof ladder
→ returns diff receipt + proof packet
→ parent accepts, merges, rolls back, or replans
```

```rust
pub enum ToolPayload {

        text: String,
        token_estimate: u32,
    },
    Handle {
        handle: ResultHandle,
        synopsis: ToolSynopsis,
        slices: Vec<ObservedSlice>,

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ResultHandle {
    pub id: String,
    pub kind: HandleKind,
    pub source_hash: String,
    pub path: Option<String>,
    pub byte_len: u64,
    pub line_count: Option<u32>,
    pub created_at_turn: u64,
}

impl ToolPayload {
    pub fn render_for_model(&self, budget: ToolRenderBudget) -> String {
        match self {
            ToolPayload::Full { text, .. } if budget.allow_full => text.clone(),
            ToolPayload::Full { text, .. } => truncate_with_expand_hint(text, budget),
```

```rust
// crates/quorp_permissions/src/risk.rs

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RiskAssessment {
    pub decision: PermissionDecision,
    pub severity: RiskSeverity,
    pub surface: ExecutionSurface,
    pub tokens: Vec<CapabilityToken>,
    pub reasons: Vec<String>,
    pub reversible: bool,
    pub touches_secrets: bool,
    pub network_possible: bool,
    pub executes_generated_code: bool,
    pub repo_trust: RepoTrustLevel,
}

pub fn assess_action(
    action: &PlannedAction,
    parsed: Option<&ParsedCommandPolicyInput>,
    repo_trust: RepoTrustLevel,
    surface: ExecutionSurface,
) -> RiskAssessment {
    let mut reasons = Vec::new();
    let mut severity = RiskSeverity::Low;

    let tokens = parsed
        .map(|p| p.capability_tokens.clone())
        .unwrap_or_default();

    if tokens.contains(&CapabilityToken::SecretsRead) {
        severity = severity.max(RiskSeverity::Critical);
        reasons.push("command may read secret material".into());
    }

    if tokens.contains(&CapabilityToken::Network) {
        severity = severity.max(RiskSeverity::High);
        reasons.push("network access is possible".into());
    }

    if tokens.contains(&CapabilityToken::GeneratedExecutable) {
        severity = severity.max(RiskSeverity::High);
        reasons.push("generated executable requires sandbox/approval".into());
    }

    if surface == ExecutionSurface::Host && severity >= RiskSeverity::High {
        return RiskAssessment {
            decision: PermissionDecision::Deny,
            severity,
            surface,
            tokens,
            reasons,
            reversible: false,
            touches_secrets: true,
            network_possible: true,
            executes_generated_code: false,
            repo_trust,
        };
    }

    // Auto-safe only for known reversible, local, non-secret, non-network workflows.
    let decision = if severity <= RiskSeverity::Low && repo_trust.is_trusted_enough() {
        PermissionDecision::Allow
    } else {
        PermissionDecision::Ask
    };

    RiskAssessment {
        decision,
        severity,
        surface,
        tokens,
        reasons,
        reversible: true,
        touches_secrets: false,
        network_possible: false,
        executes_generated_code: false,
        repo_trust,
    }
}
```

### 9. Upgrade verification into a **Proof Ladder v2** with semantic patch receipts

Each rung has:

Patch receipts should include semantic impact:

```text
which symbols changed?
which tests/proofs cover the changed symbols?
was it whole-file or minimal?
what rollback token exists?
```
