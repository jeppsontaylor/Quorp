The biggest conclusion: **Quorp already has unusually strong primitives**: sandbox-bound full-auto modes, typed permissions, Patch VM receipts, context packs, proof packets, durable ledgers, memory/rule-forge scaffolding, and custom ANSI rendering. ## 1. Ledger-Native Recovery Controller: replace benchmark playbooks with general self-repair

**This is my highest-ranked idea.** Quorp should turn every failure into a structured, replayable recovery packet and use the ledger as the source of truth for what the agent has tried, what failed, what evidence exists, what files are leased, and what next action is legal. This would directly improve long-form coding capability and reduce “going off the rails.”

The evidence is strong: Quorp’s runtime currently contains benchmark-specific repair playbooks, including deterministic patches for `cargo-dist`, `cc-rs`, `axum`, `chrono`, and `bincode`, and those playbooks can replace patch-phase actions or inject exact benchmark source patches. The run logs show deterministic “Case 04” and “Case 05” source patch injections for cases 09 and 10, including broad full-file writes. That is useful for learning, but it is not a general coding capability. ([GitHub][2])

**Build it as a general recovery controller, not a prompt patch.**

## 3. Patch-only write path with semantic leases and write-amplification gates

Quorp already has the right foundation: `quorp_patch_vm` validates preconditions, previews file changes, applies rollback tokens, rejects large full-file rewrites by default, and produces stable receipts. ([GitHub][5])

But the successful run still used risky broad writes. Case 08 wrote a 705-line `axum/src/routing/mod.rs`; case 09 wrote six broad files, including 768-line and 2,339-line files; case 10 wrote a 4,278-line `src/lib.rs`. ([GitHub][6])

The upgrade: make **diff/patch/replace-range** the default for source files, and require an explicit lease for full-file rewrites. Generated files, docs, lockfiles, and known allowlisted artifacts can be exceptions, but production source should not be casually overwritten.

Core gate:

```rust
#[derive(Debug, Clone)]
pub struct WriteAmplification {
    pub path: PathBuf,
    pub old_bytes: usize,
    pub new_bytes: usize,
    pub old_lines: usize,
    pub new_lines: usize,
    pub changed_lines_estimate: usize,
    pub operation: WriteOperation,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum WriteOperation {
    ReplaceRange,
    ApplyPatch,
    StructuralEdit,
    FullFileRewrite,
}

pub fn classify_write_amplification(w: &WriteAmplification) -> PatchRisk {
    if matches!(w.operation, WriteOperation::FullFileRewrite)
        && w.old_lines > 200
        && w.changed_lines_estimate < w.old_lines / 2
    {
        return PatchRisk::High;
    }

    if w.new_bytes > 256 * 1024 {
        return PatchRisk::High;
    }

    PatchRisk::Low
}

pub fn require_lease_for_write(
    write: &WriteAmplification,
    lease: Option<&WriteLease>,
    allowlist: &WriteAllowlist,
) -> Result<(), QuorpError> {
    if allowlist.allows(&write.path, write.operation) {
        return Ok(());
    }

    let risk = classify_write_amplification(write);
    if risk == PatchRisk::Low {
        return Ok(());
    }

    let Some(lease) = lease else {
        return Err(QuorpError::PreconditionFailed(format!(
            "high-amplification write to {} requires explicit write lease",
            write.path.display()
        )));
    };

    lease.validate_path(&write.path)?;
    lease.validate_operation(write.operation)?;
    Ok(())
}
```

The model-facing behavior should be simple:

```text

Use replace_range/apply_patch on the leased range:
  path: src/foo.rs
  range: 120..168
  expected_hash: ...
```

**Acceptance tests:**

```text

```

Expected impact: **high capability and safety, medium token reduction.** It forces the agent to make smaller, more reviewable edits and reduces accidental regression.

```text
├ Command Palette: /status /tasks /diff /verify /proof /rules /compact ─┤
└───────────────────────────────────────────────────────────────────────┘
```

Reducer-style implementation:

```rust
#[derive(Debug, Default)]
pub struct MissionModel {
    pub tasks: TaskDagView,
    pub active_turn: Option<LiveTurnView>,
    pub context: ContextBudgetTelemetry,
    pub permissions: PermissionStateView,
    pub diff: DiffStateView,
    pub proof: ProofStateView,
    pub recovery: Option<RecoveryPacketView>,
}

impl MissionModel {
    pub fn apply_event(&mut self, event: &RuntimeEvent) {
        match event {
            RuntimeEvent::TaskSnapshot(snapshot) => {
                self.tasks.update_from_snapshot(snapshot);
            }
            RuntimeEvent::ToolStarted(tool) => {
                self.active_turn = Some(LiveTurnView::from_tool(tool));
            }
            RuntimeEvent::PatchPreview(preview) => {
                self.diff.preview = Some(preview.into());
            }
            RuntimeEvent::ValidationCompleted(report) => {
                self.proof.update(report);
            }
            RuntimeEvent::ContextPressure(pressure) => {
                self.context = pressure.clone();
            }
            RuntimeEvent::RecoveryPacket(packet) => {
                self.recovery = Some(packet.into());
            }
            _ => {}
        }
    }
}
```

## 9. Verification Ladder 2.0: validation planner, proof cache, and failure-directed tests

Quorp’s verification layer already preserves command, cwd, exit code, duration, raw-log hash, cargo diagnostics, failing tests, security findings, proof DAGs, cache entries, and artifact verification. ([GitHub][15])

The next capability is a validation planner that chooses the cheapest proof lane likely to catch the bug.

Add:

```text
crates/quorp_verify/src/planner.rs

crates/quorp_verify/src/failure_classifier.rs
crates/quorp_verify/src/mutation_probe.rs
crates/quorp_cli/src/quorp/verify.rs
agent/proof-lanes.toml
agent/test-map.json
```

Planner behavior:

```text
changed Cargo.toml        -> cargo metadata + cargo check + relevant package tests
changed parser file       -> targeted unit tests + integration tests containing fixture names
changed unsafe/security   -> cargo test + cargo audit/gitleaks if available
changed public API        -> cargo test -p affected + doc tests
validation failed timeout -> narrower target + owner source read + rerun with 120s only if needed
```

This matters because cases 09 and 10 passed final evaluation despite weak internal validation signals: case 09’s report says `validation_command_fast_loop_seen=false`, and case 10 succeeded after a deterministic controller patch with a broad `src/lib.rs` write. ([GitHub][16])

**Acceptance tests:**

```text
planner selects package-local tests from changed file
cache invalidates when changed_files_hash changes
proof DAG verifies raw-log hashes
failed cargo JSON diagnostic maps to source span
timeout failure triggers narrower rerun before broad retry
```

Expected impact: **high reliability and token reduction.** Better verification reduces wasted reasoning loops and catches bad fixes earlier.

## My final priority order

1. **Ledger-Native Recovery Controller** — biggest capability/reliability jump; removes the benchmark-oracle risk.
2. **Context OS V2** — best long-session and token-pressure solution; gives Quorp a unique “stateful agent OS” identity.
3. **Patch-only write path with semantic leases** — prevents catastrophic broad writes and reduces review burden.
4. **Rule Learning V2** — fast, git-trackable learning from mistakes without unsafe repo escalation.
5. **Token-cold-start planner** — direct token/latency reduction without sacrificing capability.
6. **Task DAG with proof-backed subagents** — makes full-auto repo work scalable and resumable.
7. **Security Trust Kernel** — makes full-auto credible against untrusted repos.
8. **Terminal Mission Control** — strongest UX differentiator versus Claude/Codex.
9. **Verification Ladder 2.0** — reduces wasted loops and catches wrong fixes sooner.
10. **Replay debugger + eval autogeneration** — turns Quorp’s ledger into a compounding research advantage.

The most urgent PR sequence I would start with is: **quarantine benchmark playbooks**, then **add recovery packets**, then **add context-budget telemetry to `/status`**, then **block broad source rewrites without leases**, then **add rule-store V2 with shadow rules**. That sequence directly targets the exact weaknesses visible in the successful 5/5 run while preserving the strengths that made it succeed.
