Below are my ranked **top 8 ideas**. I’m ranking by capability impact, token reduction, uniqueness, security/eval defensibility, and feasibility as real PRs.

The biggest finding: **Quorp already has unusually strong primitives** — typed runtime actions, Patch VM receipts, verification reports, context/index infrastructure, event fanout, memory/rule scaffolding, sandbox/permission primitives, and benchmark telemetry — but the next leap is to turn those primitives into a **ledger-native autonomous engineering OS** rather than adding more one-off agent tricks. The 5/5 Rust SWE-bench run is valuable evidence, but it also exposes the key risk: some success came from benchmark-specific deterministic repair playbooks, so Quorp should generalize that behavior or quarantine it, not build on it as product logic. The run archive shows 5/5 success, 28 requests, 180,546 billed tokens, about 696 seconds wall clock, and several signs of inefficiency: delayed first writes, parser recovery, watchdog pressure, and patch packet injection. ([GitHub][1])

I could not run a local build in this environment, so I am not claiming these snippets compile as-is. The recommendations are based on repo files and benchmark artifacts inspected via the public repo and current public sources.

## 1. Ledger-native Context OS: lossless state packets, visible context pressure, and automatic compaction

**This is the highest-leverage upgrade.** Quorp should not merely “summarize the conversation.” It should maintain a durable, typed **State Packet** that is regenerated from the ledger after every important event and used as the authoritative compacted prompt substrate.

### Why this matters

The benchmark run shows token and latency pressure even on only five short-ish cases. The bincode case used 52,083 billed tokens with first write at step 7; axum used 62,702 billed tokens with first write at step 9; several cases hit watchdog pressure. ([GitHub][2]) Quorp already has pieces for this: `PromptCompactionPolicy`, token usage telemetry, runtime events, checkpoint events, and `AgentTaskStateSnapshot` with acceptance criteria, working set, verifier state, failed edits, total tokens, and repair memory. ([GitHub][3]) The context index already stores lexical chunks, symbols, diagnostics, imports, tests, proof history, and context-pack provenance in `.quorp/index/index.sqlite`. ([GitHub][4])

Current competitors point in the same direction but do not fully expose a ledger-native context OS. Codex uses sandboxed persistent workspaces and resumable file/command work; Codex also uses progressive disclosure for skills, loading only a small skill list before loading full instructions. ([OpenAI Developers][5]) Claude exposes `/compact`-style lifecycle hooks through `PreCompact` and `PostCompact`, but Quorp can go further by making compaction proof-backed and replayable. ([Claude][6])

### What to build

Add a typed state layer that lives between the runtime ledger and prompt builder.

**Add files**

```text
crates/quorp_agent_core/src/runtime/state_packet.rs

crates/quorp_render/src/context_gauge.rs
crates/quorp_slash/src/commands/status.rs
crates/quorp_slash/src/commands/compact.rs
```

**Store artifacts**

```text

.quorp/runs/<run_id>/state/state-packet.latest.json
.quorp/runs/<run_id>/state/state-packet.<step>.json
.quorp/runs/<run_id>/context/context-pack.<step>.json
.quorp/runs/<run_id>/proof/proof-dag.json
```

### Hard part: make the compacted packet auditable and loss-resistant

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ContextPressure {
    pub model_context_limit: u32,
    pub prompt_tokens_est: u32,
    pub response_budget_tokens: u32,
    pub tool_schema_tokens: u32,
    pub ledger_tail_tokens: u32,
    pub state_packet_tokens: u32,
    pub retrieved_context_tokens: u32,
    pub pressure: PressureLevel,
    pub next_compaction_at: u32,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum PressureLevel {
    Green,
    Yellow,
    Orange,
    Red,
    Critical,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct StatePacket {
    pub run_id: RunId,
    pub step: u64,

    // Never drop unless user explicitly changes the task.
    pub objective: ObjectiveRecord,
    pub user_constraints: Vec<ConstraintRecord>,
    pub security_boundary: SecurityBoundaryRecord,

    // Planning state.
    pub task_dag: TaskDagSnapshot,
    pub current_focus: Option<TaskId>,
    pub open_questions: Vec<OpenQuestion>,
    pub decisions: Vec<DecisionRecord>,

    // Code/work state.
    pub working_set: Vec<FileStateRecord>,
    pub changed_files: Vec<ChangedFileRecord>,
    pub important_symbols: Vec<SymbolRecord>,
    pub context_pack_refs: Vec<ContextPackRef>,

    // Failure and validation state.
    pub failed_attempts: Vec<FailedAttemptRecord>,
    pub validation_state: ValidationStateRecord,
    pub proof_refs: Vec<ProofRef>,

    // Memory/rule state.
    pub active_rule_refs: Vec<RuleRef>,
    pub negative_memory_refs: Vec<MemoryRef>,

    // Replay/provenance.
    pub source_event_range: EventRange,
    pub hash: ContentHash,
}

pub fn maybe_compact(
    pressure: &ContextPressure,
    packet: &StatePacket,
    ledger_tail: &[RuntimeEvent],
) -> Option<CompactionPlan> {
    let needs_compaction = matches!(
        pressure.pressure,
        PressureLevel::Orange | PressureLevel::Red | PressureLevel::Critical
    );

    if !needs_compaction {
        return None;
    }

    Some(CompactionPlan {
        preserve: PreserveSet {
            objective: true,
            user_constraints: true,
            security_boundary: true,
            task_dag: true,
            failed_attempts: true,
            validation_state: true,
            proof_refs: true,
            active_rules: true,
        },
        ledger_tail_events: select_recent_high_signal_events(ledger_tail, 32),
        retrieved_context_budget: pressure
            .model_context_limit
            .saturating_sub(pressure.response_budget_tokens)
            .saturating_mul(35)
            / 100,
        packet_ref: packet.hash.clone(),
    })
}
```

## 2. Quarantine benchmark playbooks; replace them with a generic Failure Pattern Doctor

This is the most important **trust** upgrade. Quorp should not ship benchmark-specific deterministic repair playbooks in the core autonomous runtime.

### Why this matters

The file `crates/quorp_agent_core/src/runtime/benchmark_playbooks.rs` is explicitly described as benchmark-specific deterministic repair playbooks and contains named cases such as cargo-dist, cc-rs, axum, chrono, and bincode, with exact source/patch generation functions. ([GitHub][7]) The runtime turn loop also contains benchmark repair injection paths and parser recovery paths. ([GitHub][8]) The 5/5 result is still useful, but several cases report patch packet injection or controller reads; axum and bincode especially look dependent on repair scaffolding. ([GitHub][1])

The right move is not to delete the learning. It is to **extract the general strategy** and quarantine the oracle-like patch content.

### What to build

**Move**

```text
crates/quorp_agent_core/src/runtime/benchmark_playbooks.rs
```

to:

```text
crates/quorp_benchmark/src/oracle_playbooks.rs
```

behind:

```rust
#[cfg(any(test, feature = "benchmark-oracles"))]
```

Then add:

```text
crates/quorp_repair_doctor/src/lib.rs
crates/quorp_repair_doctor/src/failure_signature.rs
crates/quorp_repair_doctor/src/repair_packet.rs
crates/quorp_repair_doctor/src/patterns/rust.rs
crates/quorp_repair_doctor/src/patterns/cargo.rs
```

### Replace exact patches with general diagnosis packets

```rust

```

The Repair Doctor may say:

> “The failing test suggests fallback route matching changed. Read routing matcher and fallback tests. Preserve existing route ordering. Do not rewrite the entire module unless a minimal diff is impossible.”

It must not say:

> “Write this exact axum source file.”

### Runtime guardrail


### Acceptance tests

1. `cargo test -p quorp_agent_core` should fail if core runtime references `BenchmarkPlaybook`.
2. `quorp run` should not load benchmark oracle code.
3. `quorp benchmark --allow-oracle-playbooks` may use quarantined playbooks only as an ablation/control, never as a leaderboard result.
4. Top-5 replay should report two numbers: **with oracles** and **general runtime only**.
5. Regression gate should fail if exact benchmark source strings appear in non-benchmark crates.

This may temporarily reduce the visible 5/5 score, but it will make Quorp’s progress real and defensible.

## 3. InspectPlan: multi-action read/search batching to cut turns before the first write

This is the fastest way to reduce tokens and wall clock without reducing capability.

### Why this matters

The axum event stream shows a very expensive pattern: one request for directory listing, another for a file read, another for tests, another for search, another for command, another for another file read, another for a targeted read, and only then a write at step 9. ([GitHub][9]) The case report confirms 9 requests, 62,702 billed tokens, 9 reads, redundant reads, and first write at step 9. ([GitHub][10])

Quorp already has typed read/search actions and context expansion hooks. ([GitHub][11]) But the model appears to be operating too serially. Competitors such as mini-SWE-agent show that simple trajectories and history processing can be very effective; the lesson is not “more complexity,” but “less unnecessary conversational round-tripping.” ([GitHub][12])

### What to build

Add one read-only compound action: `InspectPlan`.

**Modify**

```text
crates/quorp_agent_core/src/agent_protocol.rs
crates/quorp_agent_core/src/runtime/turn.rs
crates/quorp_agent_core/src/runtime/actions.rs
crates/quorp_context/src/quorp_context.rs
crates/quorp_session/src/events.rs
```

**Add**

```text
crates/quorp_agent_core/src/runtime/inspect_plan.rs
```

### Protocol shape

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct InspectPlan {
    pub reason: String,
    pub max_output_tokens: u32,
    pub reads: Vec<ReadSpec>,
    pub searches: Vec<SearchSpec>,
    pub symbol_queries: Vec<SymbolQuery>,
    pub commands: Vec<ReadOnlyCommandSpec>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct InspectDigest {
    pub plan_id: PlanId,
    pub files: Vec<FileExcerptDigest>,
    pub search_hits: Vec<SearchDigest>,
    pub symbols: Vec<SymbolDigest>,
    pub command_results: Vec<CommandDigest>,
    pub omitted: Vec<OmittedContext>,
    pub recommended_next_actions: Vec<NextActionHint>,
}
```

Runtime rule: `InspectPlan` may only contain read-only operations. It cannot write, install dependencies, access network, run generated executables, or mutate git state.

### Hard runtime logic

```rust
pub async fn execute_inspect_plan(
    plan: InspectPlan,
    ctx: &mut RuntimeContext,
) -> anyhow::Result<InspectDigest> {
    let budget = TokenBudget::new(plan.max_output_tokens.min(12_000));

    let read_futs = plan.reads.iter().map(|r| ctx.tools.read_file_excerpt(r));
    let search_futs = plan.searches.iter().map(|s| ctx.tools.search_text(s));
    let symbol_futs = plan.symbol_queries.iter().map(|q| ctx.index.search_symbols(q));

    let (reads, searches, symbols) = tokio::try_join!(
        collect_bounded(read_futs, 8),
        collect_bounded(search_futs, 8),
        collect_bounded(symbol_futs, 8),
    )?;

    let digest = InspectDigestBuilder::new(budget)
        .add_reads(reads)
        .add_searches(searches)
        .add_symbols(symbols)
        .dedupe_by_file_and_range()
        .prefer_failing_tests()
        .prefer_changed_symbols()
        .build();

    ctx.ledger.record(RuntimeEvent::InspectPlanFinished {
        plan_id: digest.plan_id,
        read_count: digest.files.len(),
        search_count: digest.search_hits.len(),
        estimated_tokens: digest.estimated_tokens(),
    })?;

    Ok(digest)
}
```

### Acceptance tests

1. In the axum replay, first write should move from step 9 to step ≤5.
2. Total billed tokens for the top-5 cohort should drop by at least 25% without lowering deterministic pass rate.
3. `InspectPlan` must be denied if it includes mutation.
4. Event logs must show which read/search outputs were omitted and why.
5. `InspectPlan` output must be replayable from ledger artifacts.

This would make Quorp feel dramatically faster in practice.

## 4. Rule Forge 2.0: tiny git-trackable rules with shadow evaluation and hostile-repo protection

Quorp should learn from mistakes during real work, but the learning must be scoped, reviewable, and safe.

### Why this matters

Quorp already has a Memory OS with working, episodic, semantic, procedural, negative, and rule tiers. ([GitHub][13]) It also has a Rule Forge concept that mines failure signals into draft rules, validates in shadow mode, and promotes rules after evidence. ([GitHub][14]) That is exactly the right direction, but it needs to become **git-trackable, inspectable, and trust-aware**.

Codex’s AGENTS.md behavior is relevant: instructions are layered, nearer files override earlier ones, and files have a size cap. ([OpenAI Developers][15]) Codex also only loads project config from trusted projects. ([OpenAI Developers][16]) Quorp should apply that same trust discipline to repo-authored rules.

### Hard part: protect against malicious repo-authored rules

Rules should have trust levels:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum RuleTrust {
    Builtin,
    UserReviewed,
    WorkspaceGenerated,
    RepoAuthoredUntrusted,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum RuleState {
    Candidate,
    Draft,
    Shadow,
    Active,
    Rejected,
    Deprecated,
}

pub fn load_rules(
    workspace: &Workspace,
    trust_policy: &TrustPolicy,
) -> anyhow::Result<Vec<LoadedRule>> {
    let mut rules = Vec::new();

    rules.extend(load_builtin_rules()?);
    rules.extend(load_user_rules(trust_policy.user_rules_dir())?);

    let repo_rules = load_repo_rules(workspace.join("agent/rules"))?;
    for rule in repo_rules {
        let trust = if trust_policy.is_workspace_trusted(workspace) {
            RuleTrust::WorkspaceGenerated
        } else {
            RuleTrust::RepoAuthoredUntrusted
        };

        rules.push(rule.with_trust(trust));
    }

    Ok(rules
        .into_iter()
        .filter(|r| r.is_schema_valid())
        .filter(|r| !r.contains_prompt_injection_markers())
        .filter(|r| trust_policy.may_load(r.trust, r.state))
        .collect())
}
```

Untrusted repo rules can be displayed and shadow-evaluated, but not allowed to grant permissions, weaken sandboxing, suppress security warnings, or override user constraints.

1. Generated rule enters `candidate` or `shadow`, never `active`.

## 5. Proof-backed Task DAG with subagents in isolated worktrees

Quorp should stop treating long autonomous work as a linear chat. It should make the task graph explicit, visible, resumable, and proof-backed.

### Why this matters

The current runtime already tracks task state, phases, checkpoints, failed edits, validation queues, and recovery packets. ([GitHub][3]) But long repository work needs a real **Task DAG**: independent subtasks, dependencies, leases on files, validation obligations, and merge rules.

This would also compete directly with modern agent systems. Codex CLI now advertises subagents, cloud tasks, approval modes, and terminal workflows. ([OpenAI Developers][17]) Claude exposes lifecycle events for `TaskCreated`, `TaskCompleted`, `SubagentStart`, `SubagentStop`, worktree creation/removal, and compaction hooks. ([Claude][6]) OpenAI’s Symphony work frames the issue tracker as a control plane for many coding agents, with the claim that it materially increased landed PR throughput in production-like workflows. ([OpenAI][18])

### What to build

**Add**

```text
crates/quorp_agent_core/src/runtime/task_graph.rs
crates/quorp_agent_core/src/runtime/subagent_runner.rs
crates/quorp_render/src/task_dag.rs
crates/quorp_slash/src/commands/tasks.rs
crates/quorp_slash/src/commands/agents.rs
```

### Task shape


### Critical design rule

A subagent should not return prose as the main artifact. It should return:

1. a patch or patch proposal,
2. proof packets,
3. validation results,
4. touched paths,
5. unresolved questions,
6. exact merge preconditions.

### Acceptance tests

1. Two subagents cannot lease the same file mutably at the same time.
2. Parent agent cannot merge subagent diff without Patch VM hash preconditions.
3. Interrupted task DAG resumes from ledger and worktree state.
4. `/tasks` shows blocked/running/failed/validated tasks.
5. A subagent result with no proof packet is marked incomplete.

This would give Quorp a serious advantage for long-form work: not just “agent mode,” but **auditable multi-agent engineering**.

## 6. Unified Tool Gateway: permissions, sandbox, hooks, and audit receipts in one path

Every mutating action should pass through a single **Tool Gateway** that classifies, authorizes, sandboxes, executes, audits, and emits hook events.

### Why this matters

Quorp already has strong primitives. The permissions crate has modes such as `ReadOnly`, `Ask`, `AcceptEdits`, `AutoSafe`, and `YoloSandbox`, and detects risky capabilities such as compound shell commands, network, dependency installs, Docker, git remote mutation, generated executables, MCP, browser, and secrets access. ([GitHub][19]) The sandbox crate supports host, git-worktree, tmp-copy, and container plans with network disabled, capabilities dropped, `no-new-privileges`, and redacted environment handling. ([GitHub][20])

But the product-level security model should be stricter: **full-auto should never mean unsupervised host execution**. Codex’s approval model uses sandboxing plus approval policy, with default network off and approval required for network or out-of-workspace work. ([OpenAI Developers][21]) Claude’s auto mode also uses background safety checks, while its permission modes separate plan/read/edit/auto behavior. ([Claude][22])

### Hard path

```rust
pub async fn authorize_execute(
    &self,
    action: ToolAction,
    ctx: ToolContext,
) -> anyhow::Result<ToolReceipt> {
    let classified = self.classifier.classify(&action, &ctx)?;

    let pre_hook = self.hooks.pre_tool_use(&classified, &ctx).await?;
    if pre_hook.denied() {
        return Ok(ToolReceipt::denied(classified, pre_hook.reason()));
    }

    let decision = self.permissions.decide(&classified, &ctx)?;
    if decision.denied() {
        return Ok(ToolReceipt::denied(classified, decision.reason()));
    }

    let lease = self.sandbox.lease_for(&classified, &ctx, &decision)?;

    let start = Instant::now();
    let result = lease.execute(action).await;

    let receipt = ToolReceipt {
        id: ToolReceiptId::new(),
        action_class: classified.action_class,
        capability_tokens: classified.capability_tokens,
        sandbox_surface: lease.surface(),
        permission_decision: decision,
        command_fingerprint: classified.command_fingerprint,
        changed_paths: result.changed_paths(),
        stdout_digest: result.stdout_digest(),
        stderr_digest: result.stderr_digest(),
        duration_ms: start.elapsed().as_millis() as u64,
        result: result.into_status(),
    };

    self.hooks.post_tool_use(&receipt, &ctx).await?;
    self.audit.write_receipt(&receipt)?;

    Ok(receipt)
}
```

### Acceptance tests

1. In `AutoSafe`, network, dependency install, generated executable, MCP, browser, and git remote mutation are denied or prompted unless sandbox policy explicitly allows them.

5. Hook decisions are recorded and replayable.

This would make Quorp safer than “YOLO mode” agents while still supporting high-agency work.

## 7. Semantic Edit Sequencer: smaller patches, stale-hash recovery, and diff-density enforcement

Quorp should make the **minimal safe edit** the default, not the model’s responsibility.

### Why this matters

The Patch VM is already a major strength: it validates preconditions, previews changes, applies with rollback tokens, and has policy controls such as `allow_full_file_rewrite = false` and max file counts. ([GitHub][23]) But the axum run still had a 705-line `write_file` to `axum/src/routing/mod.rs`, even though the actual logical change was likely much smaller. ([GitHub][9]) Large rewrites increase token cost, merge risk, and regression risk.

### What to build

### Diff-density gate

```rust
#[derive(Debug, Clone)]
pub struct PatchDensity {
    pub files_changed: usize,
    pub total_lines_before: usize,
    pub touched_lines: usize,
    pub inserted_lines: usize,
    pub deleted_lines: usize,
}

impl PatchDensity {
    pub fn rewrite_ratio(&self) -> f32 {
        if self.total_lines_before == 0 {
            return 1.0;
        }
        self.touched_lines as f32 / self.total_lines_before as f32
    }

    pub fn is_suspicious_rewrite(&self) -> bool {
        self.files_changed == 1 && self.total_lines_before > 200 && self.rewrite_ratio() > 0.20
    }
}

pub fn enforce_density_policy(
    density: &PatchDensity,
    policy: &PatchVmPolicy,
    preview: Option<PreviewId>,
) -> anyhow::Result<()> {
    if density.is_suspicious_rewrite()
        && !policy.allow_full_file_rewrite
        && preview.is_none()
    {
        anyhow::bail!(
            "large rewrite requires semantic patch plan or approved preview: touched {} / {} lines",
            density.touched_lines,
            density.total_lines_before
        );
    }

    Ok(())
}
```

### Runtime behavior

When the model proposes `write_file` on an existing large file:

1. compute diff density,
2. attempt to convert to semantic hunks,
3. require preview if still broad,
4. re-read if hash is stale,
5. run targeted formatting and validation,
6. emit proof receipt.

### Acceptance tests

1. The axum replay should not permit a 705-line full-file rewrite without explicit preview approval.
2. A stale hash should trigger auto-reread and rebase, not blind overwrite.
3. Semantic patch must preserve unrelated code byte-for-byte.
4. `/diff` should show density, risk, touched symbols, and rollback token.
5. Patch VM receipts should include density metrics.

This is a capability upgrade and a token upgrade: smaller patches are easier for the model to reason about, easier to validate, and easier for humans to trust.

Quorp should not merely imitate Codex CLI or Claude Code. It should become the best terminal interface for **watching and steering autonomous engineering**.

```rust
pub struct MissionControlFrame<'a> {
    pub status: StatusRibbon<'a>,
    pub task_dag: TaskDagPane<'a>,
    pub transcript: TranscriptPane<'a>,
    pub inspector: InspectorPane<'a>,
    pub permission_bar: PermissionBar<'a>,
    pub composer: ComposerPane<'a>,
}

pub enum InspectorPane<'a> {
    Diff(DiffView<'a>),
    Proof(ProofView<'a>),
    Context(ContextView<'a>),
    Rules(RuleView<'a>),
    Memory(MemoryView<'a>),
    Sandbox(SandboxView<'a>),
    Replay(ReplayScrubber<'a>),
}

pub fn render_mission_control(frame: &MissionControlFrame<'_>, width: u16) -> Vec<Line> {
    if width < 100 {
        return render_compact_single_column(frame, width);
    }

    let layout = MissionLayout::new(width)
        .left(28)
        .right(34)
        .min_center(48);

    let mut out = Vec::new();
    out.extend(render_status_ribbon(&frame.status, width));
    out.extend(render_three_pane(
        &frame.task_dag,
        &frame.transcript,
        &frame.inspector,
        layout,
    ));
    out.extend(render_permission_bar(&frame.permission_bar, width));
    out.extend(render_composer(&frame.composer, width));
    out
}
```

# The most novel breakthrough

The unique opportunity is to make Quorp a **ledger-native coding-agent OS**:

1. every model thought is less important than every typed event;
2. every context pack has provenance;
3. every compaction is loss-audited;
4. every rule is scoped and replay-evaluated;
5. every edit has hash preconditions and rollback;
6. every subagent return is proof-backed;
7. every permission decision has a receipt;
8. every benchmark result can be replayed, ablated, and inspected in the TUI.

That combination would be genuinely differentiated. Codex and Claude are strong terminal agents; Quorp can become the first open-source Rust coding agent where **state, proof, memory, permissions, replay, and UI are all first-class parts of the same runtime**.
