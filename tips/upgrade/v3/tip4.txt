## 1. Replace benchmark playbooks with a generic Failure Contract Kernel

This is the highest-leverage idea because Quorp’s biggest risk is that the 5/5 result mixes real agent capability with benchmark-specific repair machinery. The repo contains a file explicitly titled **“Benchmark-specific deterministic repair playbooks”**, with enum variants for the five top-5 cases and exact source patch transforms for bincode, chrono, axum, cargo-dist, and cc-rs. That needs to be quarantined from the production runtime and replaced by general recovery logic. ([GitHub][2])

The useful capability is not the hardcoded patches. The useful capability is the pattern: classify failure, lease a target file/range, restrict bad actions, inject evidence, require hash preconditions, rerun the exact failing verifier, and stop only with proof. Case 06 shows why this matters: the agent first detoured into `Cargo.toml` dependency edits, validation failed, then the controller injected a narrower source-read/repair path and the final source patch succeeded. ([GitHub][3])

```rust
pub fn next_repair_policy(
    state: &AgentTaskStateSnapshot,
    contract: &FailureContract,
    candidate: &AgentTurnResponse,
) -> RepairPolicyDecision {
    if candidate.touches_path_outside_lease(&contract.target_lease) {
        return RepairPolicyDecision::RejectAndExplain {
            reason: "candidate edit leaves active target lease".into(),
            suggested_actions: vec![ActionKind::ReadFile, ActionKind::ReplaceRange],
        };
    }

    if contract.class == FailureClass::TestAssertionFailure
        && candidate.modifies_manifest()
        && !state.last_failing_verifier_contains("unresolved import")
    {
        return RepairPolicyDecision::RejectAndExplain {
            reason: "manifest edit blocked: failure is behavioral, not missing dependency".into(),
            suggested_actions: vec![ActionKind::ReadFile, ActionKind::ReplaceRange],
        };
    }

    RepairPolicyDecision::Allow
}
```

## 2. Build a Ledger-Native Context OS with automatic lossless compaction

Quorp already has the beginning of this: `PromptCompactionPolicy` includes variants like `BenchmarkStatePacket`, `Last8Ledger1024`, and `BenchmarkRepairMinimal`, and the runtime snapshot already tracks mode, criteria, working set, last verifier, checkpoint, parse/stall counts, total tokens, and repair memory. ([GitHub][4])

But the run archive shows prompt growth and no real context-pressure control: case 06 prompt estimates climb past 8k tokens, case 08 climbs to ~7.7k, and first writes were delayed to request 7 and 9 respectively. ([GitHub][5])

The upgrade is to stop treating compaction as “summarize the transcript” and instead make it a **ledger-derived state packet** with mandatory fields that cannot be lost: objective, constraints, open tasks, decisions, failed attempts, validation state, touched files, security boundary, and rule/memory state.

This aligns with the public Codex lesson: prompt caching depends on stable prefixes, and automatic compaction is needed when context exceeds thresholds. Codex’s own engineering writeup emphasizes exact-prefix caching and automatic compaction after a configured limit. ([OpenAI][6])

**Hard part: invariant-preserving compaction**

```rust
pub fn compact_for_next_turn(
    ledger: &RunLedger,
    transcript: &[TranscriptItem],
    budget: &ContextBudgetReport,
    invariants: &CompactionInvariants,
) -> Result<CompactedPrompt> {
    let packet = LosslessStatePacket::from_ledger(ledger)?;

    packet.validate_invariants(invariants).map_err(|err| {
        anyhow::anyhow!("refusing lossy compaction: {err}")
    })?;

    let mut builder = PromptBuilder::new();

    builder.push_static_system_prefix()?;        // cacheable
    builder.push_tools_static_manifest()?;       // cacheable
    builder.push_state_packet(packet.clone())?;  // compact variable state
    builder.push_recent_high_signal_events(
        ledger.high_signal_tail(HighSignalTailConfig {
            max_events: 16,
            include_failures: true,
            include_edits: true,
            include_validation: true,
            include_permissions: true,
        })?,
    )?;

    let estimated = builder.estimated_tokens();
    if estimated > budget.target_input_tokens {
        return Err(anyhow::anyhow!(
            "compacted prompt still exceeds target: {estimated} > {}",
            budget.target_input_tokens
        ));
    }

    Ok(CompactedPrompt {
        messages: builder.finish(),
        receipt: CompactionReceipt::new(packet, estimated),
    })
}
```

## 3. Add action batching + compressed tool-result deltas

Case 08 is the clearest token/request smell: the agent spent 9 model requests before the first write, including separate read/search/command turns, then wrote a full 705-line file. ([GitHub][7])

Quorp should let the model issue a **read/search bundle** in a single turn, execute independent actions together, and return compressed tool-result deltas with expandable handles. pub fn compress_tool_result(
```rust
    action: &AgentAction,
    raw: &ToolArtifact,
    budget: ToolResultBudget,
) -> Result<ToolResultEnvelope> {
    let structured = StructuredToolResult::try_from_raw(action, raw)?;
    let brief = match &structured {
        StructuredToolResult::SourceFile(file) => {
            summarize_source_file(file, budget.max_brief_tokens)?
        }
        StructuredToolResult::SearchHits(hits) => {
            summarize_search_hits(hits, budget.max_brief_tokens)?
        }
        StructuredToolResult::CommandOutput(output) => {
            summarize_command_output(output, budget.max_brief_tokens)?
        }
    };

    Ok(ToolResultEnvelope {
        action_id: raw.action_id,
        status: raw.status,
        brief,
        artifact_ref: raw.persist()?,
        content_hash: raw.sha256.clone(),
        estimated_tokens_saved: raw.estimated_tokens().saturating_sub(estimate_tokens(&brief)),
        expandable: true,
        structured: Some(structured),
    })
}
```

```rust
pub fn evaluate_rules_for_candidate_action(
    state: &AgentTaskStateSnapshot,
    action: &AgentAction,
    rule_index: &RuleIndex,
) -> RuleDecision {
    let mut decision = RuleDecision::default();

    for rule in rule_index.matching_rules(state, action) {
        match rule.trust_level {
            TrustLevel::TrustedActive => {
                decision.apply(rule.effect.clone(), RuleMode::Enforced);
            }
            TrustLevel::RepoCandidate | TrustLevel::Shadow => {
                decision.record_shadow_match(rule.id.clone());
            }
            TrustLevel::Rejected | TrustLevel::Deprecated => {}
        }
    }

    decision
}
```

## 6. Harden full-auto with a trust-policy sandbox matrix

Quorp already has a permission engine with read-only, ask, accept-edits, auto-safe, and yolo-sandbox modes, plus command classification for shell meta, compound commands, dependency installs, network, Docker, git remote mutations, secrets, generated executables, MCP, and browser actions. The upgrade is to make this policy visible, stricter, and tied to sandbox enforcement. Codex’s official docs frame sandbox and approval as two separate controls: sandbox defines what the agent can technically do, approval policy defines when it must ask; local Codex defaults include no network and workspace-limited writes, and `--full-auto` is lower-risk than full access. ([OpenAI Developers][12])

Claude Code hooks also show a useful pattern: pre-tool hooks can inspect command input and return JSON decisions such as deny with a reason. ([Claude][13]) Quorp can go beyond that by making every decision a ledger receipt and by refusing unsafe host execution in full-auto.

```rust
pub fn decide(action: &Action, ctx: &PermissionContext) -> PermissionDecision {
    if ctx.surface == ExecutionSurface::Host && ctx.mode.is_full_auto() {
        return deny(action, "full-auto host execution is forbidden");
    }

    if action.tokens.contains(&CapabilityToken::SecretsRead) {
        return deny(action, "secret-like path read blocked");
    }

    if action.tokens.contains(&CapabilityToken::GeneratedExecutable)
        && !ctx.user_approved_generated_executables
    {
        return ask(action, "generated executable requires approval");
    }

    if action.tokens.contains(&CapabilityToken::Network)
        && !ctx.network_policy.allows(&action.command_repr)
    {
        return ask(action, "network access is outside current policy");
    }

    if action.tokens.contains(&CapabilityToken::DependencyInstall)
        && !ctx.sandbox.is_ephemeral()
    {
        return ask(action, "dependency install requires ephemeral sandbox or approval");
    }

    allow(action, "allowed by sandboxed policy")
}
```

## 9. Upgrade Patch VM into an Edit Intent VM

Patch VM already validates preconditions, previews changes, applies with rollback tokens, and preserves receipts. ([GitHub][16]) But case 08 shows the next problem: even with a good write contract, the model can choose an overly broad write shape. It wrote an entire 705-line file when a narrower edit should have been possible. ([GitHub][7])

The upgrade is to add an **Edit Intent VM** above Patch VM. The agent submits intent; the VM proposes the narrowest safe edit shape.

**Broad-write guard**

```rust
pub fn plan_edit(intent: EditIntent, file: &ObservedFile) -> Result<EditPlan> {
    let candidate = intent.proposed_content.as_ref()
        .ok_or_else(|| anyhow::anyhow!("edit intent missing proposed content"))?;

    let diff = Diff::between(&file.text, candidate);

    if diff.changed_line_count() > 300 && !intent.constraints.contains(&EditConstraint::MaxChangedLines(300)) {
        bail!("broad edit exceeds default threshold");
    }

    if let Some(range) = diff.single_contiguous_range() {
        return Ok(EditPlan::ReplaceRange {
            path: intent.path,
            range: range.to_line_range(),
            expected_hash: file.hash_for_range(range)?,
            replacement: diff.replacement_text(range),
        });
    }

    if let Some(ast_plan) = AstRangePlanner::try_plan(&file.text, candidate, &intent.anchors)? {
        return Ok(EditPlan::StructuralEdit {
            path: intent.path,
            anchor: ast_plan.anchor,
            operation: ast_plan.operation,
        });
    }

    Ok(EditPlan::FullFileWrite {
        path: intent.path,
        expected_hash: file.full_hash.clone(),
        content: candidate.clone(),
        justification: FullFileJustification::new(diff.summary()),
    })
}
```

## What I would build first

The best first PR sequence is:

1. **Quarantine benchmark playbooks** and add the no-oracle audit gate.
2. **Add Failure Contract Kernel v0** for Rust verifier failures.
3. **Add ContextBudgetReport + `/status` context pressure**.
4. **Add ToolResultEnvelope compression** for large reads/searches.
5. **Add two rule candidates from the 5/5 run**: no manifest detour, no broad full-file write.
6. **Add replay findings for case 06 and case 08**.
7. **Add Mission Control right rail**: tasks, context, permissions, validation.

The single most important principle: **preserve the successful control-loop behavior, but remove benchmark-specific source knowledge from production runtime**. That gives Quorp the best path toward becoming a serious open-source coding-agent runtime rather than a top-5 benchmark specialist.
