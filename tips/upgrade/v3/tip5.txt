I’ll be direct: the **highest-leverage path** is not “make the model smarter.” Quorp already has several unusually strong primitives: strict tool contracts, Patch VM, proof packets, persistent context index, memory tiers, permissions, slash registry, and benchmark telemetry. The best upgrades are to make those primitives **ledger-native, visible, general, and impossible to bypass**.

A note on evidence: direct `git clone` was blocked in my execution environment by network/DNS, so I audited through GitHub-rendered/raw source and the checked-in run archive. The run archive confirms the 5/5 result, 28 requests, 180,546 billed tokens, and about 696 seconds wall clock. It also shows the important inefficiencies: case 06 first write at step 7, case 08 first write at step 9, parser recovery in 07/09/10, and no compacted prompt estimate despite `benchmark-state-packet` being active. ([GitHub][1])

# 1. Ledger-Native Context OS

**Build this first.** It is the biggest capability and token-efficiency multiplier.

Quorp already has a real Context Compiler: token-budgeted context packs, memory recall, handles, indexed lexical/symbol retrieval, agent contracts, context-pack provenance, and a `.quorp/index/index.sqlite` schema with `context_pack_items`. ([GitHub][2]) But the benchmark reports show `compacted_prompt_est=n/a`, rising prompt estimates, and watchdog pressure in important cases. Case 06 reached 52,083 billed tokens with first write at step 7; case 08 reached 62,702 billed tokens with first write at step 9 and one redundant read. ([GitHub][3])

External research agrees this is central: Anthropic describes context as a finite resource requiring ongoing curation across long agent loops, and a 2026 Claude Code architecture analysis specifically calls out compaction, permissions, hooks, subagents, and append-oriented storage as core surrounding systems around the model loop. ([Anthropic][4])

## What to add

Add a **durable mission-state capsule** that is continuously derived from the ledger and used as the root of every prompt after compaction. The capsule should preserve:

* objective and user constraints
* open task DAG nodes
* decisions already made
* files read and their hashes
* owner leases
* failed attempts and blocked retry fingerprints
* active rules and rule provenance
* validation status and proof packets
* sandbox/trust/permission boundaries
* current diff/patch receipts
* unresolved uncertainties

This should be exposed through `/status`, `/context`, and `/compact`.

# 2. Action ROI / First-Write Controller

This is the best way to improve actual coding capability.

The 5/5 succeeded partly because Quorp supplied strong repair scaffolding: target leases, patch packets, fast-loop commands, and strict action contracts. But the run shows delayed writes in the two most expensive cases. Case 06 had 8 reads before 1 write, repair submode for 6 turns, patch packet injected, first write step 7. Case 08 had 9 reads, one redundant read, patch packet injected, first write step 9. ([GitHub][3])

More importantly, current `benchmark_playbooks.rs` contains deterministic benchmark repair playbooks and exact patch actions for specific top-5 cases. The file explicitly names `BenchmarkPlaybook`, gates on `BenchmarkAutonomous`, and includes exact source patch injection paths such as `axum/src/routing/mod.rs`, `src/round.rs`, and `src/features/serde/de_owned.rs`. ([GitHub][6]) That is useful evidence, but it must be generalized or quarantined.

## What to add

Add a general **Progress Controller** that decides what kind of action is allowed next:

* read/search
* targeted read
* preview edit
* apply edit
* validate
* recover parser
* rollback
* ask human
* compact context
* stop

It should score each possible next action by **expected information gain**, **token cost**, **risk**, and **readiness to patch**.

Once Quorp has:

* failing test or issue anchor
* implementation target lease
* relevant source slice
* base hash
* expected validation command

…it should stop allowing generic exploration and force a write contract.

```rust
// crates/quorp_agent_core/src/runtime/progress.rs

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum NextActionContract {
    Explore {
        allowed: Vec<ActionKind>,
        reason: String,
    },
    TargetedRead {
        path: String,
        range_hint: Option<LineRange>,
        reason: String,
    },
    MustPreviewEdit {
        path: String,
        expected_hash: Option<String>,
        allowed_edit_kinds: Vec<EditKind>,
        reason: String,
    },
    MustValidate {
        command: String,
        reason: String,
    },
    MustCompact {
        reason: String,
    },
    StopSuccess,
    StopFailure {
        reason: String,
    },
}

pub fn choose_next_action_contract(
    state: &AgentTaskState,
    telemetry: &TurnTelemetry,
    context: &ContextBudgetSnapshot,
) -> NextActionContract {
    if context.pressure.must_compact_before_model_turn() {
        return NextActionContract::MustCompact {
            reason: "context pressure requires state-capsule compaction before another model turn".into(),
        };
    }

    if state.validation.green && state.patch_receipts.has_user_visible_change() {
        return NextActionContract::StopSuccess;
    }

    if telemetry.parser_recovery_streak >= 2 {
        return NextActionContract::Explore {
            allowed: vec![ActionKind::EmitStrictJsonOnly],
            reason: "parser recovery streak; require minimal strict-json repair turn".into(),
        };
    }

    if let Some(target) = state.implementation_target_lease.as_ref() {
        let has_failure = state.primary_failure.is_some();
        let has_slice = state.agent_repair_memory
            .observed_slices
            .iter()
            .any(|s| canonical_path(&s.path) == canonical_path(&target.path));

        let has_hash = state.agent_repair_memory
            .observed_slices
            .iter()
            .any(|s| {
                canonical_path(&s.path) == canonical_path(&target.path)
                    && s.content_fingerprint.is_some()
            });

        if has_failure && has_slice && has_hash && telemetry.new_fact_count == 0 {
            return NextActionContract::MustPreviewEdit {
                path: target.path.clone(),
                expected_hash: state.expected_hash_for(&target.path),
                allowed_edit_kinds: vec![
                    EditKind::ReplaceRange,
                    EditKind::ReplaceFunctionBody,
                    EditKind::ModifyToml,
                ],
                reason: "target, failure, source slice, and hash are known; more reads are lower ROI than a preview edit".into(),
            };
        }
    }

    if telemetry.same_action_repeated >= 2 {
        return NextActionContract::Explore {
            allowed: vec![ActionKind::DifferentEvidence, ActionKind::Validate, ActionKind::Compact],
            reason: "no-progress loop detected; repeated action is temporarily disallowed".into(),
        };
    }

    NextActionContract::Explore {
        allowed: ActionKind::standard_safe_set(),
        reason: "insufficient evidence for write".into(),
    }
}
```

# 3. Tool Result Distillation + Proof Cache

This is the most direct token-reduction upgrade.

Quorp already has proof packets, raw artifacts, Patch VM receipts, and validation reports. The Patch VM has strong preconditions, preview IDs, rollback tokens, high-risk preview requirements, stale hash rejection, and rollback tests. ([GitHub][7]) The missing piece is to make every command/tool result prompt-visible only as a compact **receipt**, with raw logs stored out-of-context unless specifically expanded.

## What to add

For every command:

1. Store raw stdout/stderr as an artifact with SHA-256.
2. Extract structured failure spans:

   * compiler error code
   * file/range
   * failing test
   * assertion excerpt
   * command exit code
   * duration
   * likely owner file
3. Feed only the compact receipt into the model.
4. Allow expansion by handle only when needed.
5. Cache validation by `(git_tree_hash, command, env_digest, toolchain_digest)`.

This would reduce prompt growth from repeated command output and make replay/debugging more deterministic.

# 7. Capability-Token Permission OS

Quorp already has a good start: five modes, `Permissions::check`, command parsing, shell wrapper detection, and capability tokens such as `ShellMeta`, `CompoundCommand`, `FilesystemWrite`, `Network`, `DependencyInstall`, `Docker`, `GitRemoteMutation`, `SecretsRead`, `GeneratedExecutable`, `Mcp`, and `Browser`. ([GitHub][14])

Codex documents approval mode and sandbox mode as the two key knobs for controlling when commands require permission and what filesystem access the agent has. ([OpenAI Developers][15]) Claude Code hooks can make JSON permission decisions, including denying destructive commands before tool use. ([Claude][16]) Quorp can beat both by making permissions **ledger-visible, typed, replayable, and workflow-aware**.

## Hardest core snippet

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum PermissionDecision {
    Allow {
        receipt: PermissionReceipt,
    },
    Ask {
        prompt: PermissionPrompt,
        proposed_receipt: PermissionReceipt,
    },
    Deny {
        reason: DenyReason,
        tokens: Vec<CapabilityToken>,
    },
}

pub fn evaluate_permission(
    action: &Action,
    trust: &TrustContext,
    allowlist: &AllowList,
) -> PermissionDecision {
    let tokens = classify_action(action);

    if !trust.repo_trusted && tokens.iter().any(CapabilityToken::can_escape_sandbox) {
        return PermissionDecision::Deny {
            reason: DenyReason::UntrustedRepoSandboxEscape,
            tokens,
        };
    }

    if tokens.contains(&CapabilityToken::SecretsRead) {
        return PermissionDecision::Ask {
            prompt: PermissionPrompt::SecretsAccess { action: action.clone() },
            proposed_receipt: PermissionReceipt::new(action, &tokens),
        };
    }

    if tokens.contains(&CapabilityToken::Network) && !trust.network_allowed {
        return PermissionDecision::Ask {
            prompt: PermissionPrompt::NetworkAccess { action: action.clone() },
            proposed_receipt: PermissionReceipt::new(action, &tokens),
        };
    }

    if allowlist.matches(action, &tokens) {
        return PermissionDecision::Allow {
            receipt: PermissionReceipt::new(action, &tokens),
        };
    }

    PermissionDecision::Ask {
        prompt: PermissionPrompt::General { action: action.clone(), tokens },
        proposed_receipt: PermissionReceipt::new(action, &tokens),
    }
}
```

# 9. Replay-First Evaluation + Anti-Overfit Gates

The 5/5 top-5 run is valuable, but it is not enough. SWE-bench Verified is a human-filtered 500-instance subset and its leaderboard includes many agent scaffold styles. ([SWE-bench][18]) SWE-rebench explicitly warns that static benchmarks can become contaminated and proposes continuously extracted fresh interactive tasks. ([arXiv][19]) SWE-Bench Pro targets long-horizon, enterprise-style tasks requiring hours to days for humans, multiple files, and substantial modifications. ([arXiv][20]) Terminal-Bench 2.0 evaluates terminal agents on 89 hard, realistic tasks, including containerized environments, tests, oracle solutions, and long chains of interdependent actions. ([arXiv][21])

## What to add

Evaluation should track not just pass/fail, but the exact behaviors Quorp needs to protect:

* tokens to first write
* reads before first write
* parser recovery count
* no-progress loops
* context pressure and compaction loss
* proof completeness
* permission violations
* Patch VM bypasses
* rule-helped vs rule-hurt
* replay determinism
* benchmark-playbook quarantine

## The implementation order I would actually use

**PR 1: Context pressure visibility.**
Add `ContextBudgetSnapshot`, emit ledger events, wire `/status`, update `status_footer.rs`. This is small and immediately valuable.

**PR 2: Progress Controller.**
Add `choose_next_action_contract`, block repeated low-ROI reads, create first-write metrics. Replay 06/08 and prove earlier writes.

**PR 3: Benchmark playbook quarantine.**
Keep top-5 eval support, but feature-gate exact patch injection and prove normal runtime cannot call it.

**PR 4: Tool result receipts.**
Distill command output into proof receipts; store raw logs by hash; add validation cache.

**PR 5: Rule cards.**
Persist candidate/shadow/active rules as `.quorp/rules/**/*.yaml`; wire `/rules`; prevent permission escalation.

**PR 6: Mission-control TUI.**
Start with `/status`, `/context`, `/proof`, `/diff`, `/permissions`, `/tasks` panes. Do not wait for subagents.

**PR 7: Task DAG + worktree subagents.**
Add file leases, proof-backed returns, and patch tournament only after Patch VM and proof receipts are universal.

## The most novel combination

The breakthrough version of Quorp is:

> **A ledger-native coding agent where every thought-relevant fact, context item, permission decision, edit, validation, rule, and task transition is replayable; the TUI is a mission-control debugger over that ledger; compaction is a verified projection of durable state rather than lossy summarization; and learning happens through tiny shadow-evaluated rule cards.**

That would make Quorp meaningfully different from Codex CLI, Claude Code, OpenHands, and SWE-agent-style harnesses—not because it has more commands, but because it can run long autonomous coding missions with **visible state, provable safety, recoverable context, testable learning, and replayable cognition**.
