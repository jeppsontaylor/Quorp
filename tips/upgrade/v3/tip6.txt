I’d build these **8 ideas in this order**. The big strategic bet is that Quorp should not merely imitate Claude Code or Codex CLI. Quorp’s unique advantage is the combination of **Patch VM + proof DAG + hash-chain ledger + runtime events + terminal-first UI**. The highest-impact work is turning those pieces into a visible, replayable, token-efficient “agent operating system.”

I could not shell-clone in this environment because DNS resolution failed, so I inspected the repository through the GitHub connector and current web sources. The recommendations below are grounded in the repo files and run artifacts I could read directly.

## What the evidence says first

Quorp’s 5/5 Rust SWE-bench top-5 run is real evidence, but it is not proof that the system is done. The archived run reports **5 cases run, 5 successful, 0 failed, 28 requests, 180,546 billed tokens**, with deterministic and judge success for every case. Two cases were expensive: `06-rust-swebench-bincode-serde-decoder-memory` used **9 requests / 52,083 tokens** with first write at step 7, and `08-rust-swebench-axum-fallback-merge` used **9 requests / 62,702 tokens** with first write at step 9. Three cases required parser recovery once. No case recorded prompt compaction.  

The strongest repo evidence is that Quorp already has an unusually good foundation: typed runtime events, Patch VM-backed writes, proof packets, sandbox/permission primitives, context/memory/rule scaffolding, benchmark gates, durable ledger work, and proof CLI work. The remaining “world-class gaps” are exactly where I would invest: persistent context intelligence, durable rule lifecycle, sandbox policy, task DAG/subagents, fullscreen TUI parity, and broader replayable evals.   

Current competitors set the baseline: Codex CLI exposes approval modes including Full Auto, but Full Auto is sandboxed and network-disabled by default; Claude Code has slash commands, memory, hooks, permissions, compact events, and subagent-related hooks; OpenHands emphasizes sandboxed execution, auditability, and parallel delegation; mini-SWE-agent shows how far a small native-tool agent can go; and Terminal-Bench stresses realistic terminal tasks beyond SWE-style patching. ([OpenAI Help Center][1])

# 1. Ledger-Native Context OS: automatic compaction without losing task state

**Rank: #1. Build this first.**

Quorp should have a **Context OS** that treats context as a managed, visible, replayable resource. The goal is simple: when context grows, Quorp should compact automatically while preserving objectives, constraints, task DAG state, failed attempts, validation state, proof receipts, patch receipts, rules, permission boundaries, and active context provenance.

This is the best idea because it improves **coding capability, token cost, memory, resumability, and TUI differentiation** at the same time.

### Why this matters

The 5/5 run did not stress long-context behavior: every `compacted_prompt_est` was `n/a`, and first prompts were only about 2,000–2,600 tokens. But total usage reached **180,546 billed tokens** across only five cases, and the two 9-request cases show that Quorp can spend many turns before the decisive write. 

The runtime already emits `ModelRequestStarted` with `prompt_token_estimate`, `message_count`, completion cap, and safety mode. It also tracks token usage and checkpoints. That means the instrumentation base exists; what is missing is a first-class context-pressure policy and a user-visible `/status` surface. 

Claude Code already has `/compact` and hook events such as `PreCompact`, including manual and automatic compaction triggers. Quorp can beat that by making compaction **ledgered, proven, replayable, and invariant-tested**, not just summarized. ([Claude API Docs][2])

The key invariant: **a compaction is accepted only if the new capsule preserves required facts**. That means the compactor does not just summarize; it runs a preservation check.

```rust
pub fn verify_capsule_preservation(
    before: &StateCapsule,
    after: &StateCapsule,
) -> Result<(), CapsuleLossError> {
    require_subset("user_constraints", &before.user_constraints, &after.user_constraints)?;
    require_subset("security_boundaries", &before.security_boundaries, &after.security_boundaries)?;
    require_same_validation_frontier(&before.validation_state, &after.validation_state)?;
    require_open_tasks_preserved(&before.open_tasks, &after.open_tasks)?;
    require_failed_attempts_preserved(&before.failed_attempts, &after.failed_attempts)?;
    require_artifact_refs_preserved(&before.patch_receipts, &after.patch_receipts)?;
    require_artifact_refs_preserved(&before.proof_receipts, &after.proof_receipts)?;
    Ok(())
}
```

### Runtime behavior

Before every model request:

1. Estimate prompt budget from current transcript, active context pack, rules, task state, and validation/proof state.
2. Emit `context.pressure.updated`.
3. If pressure is high, compact soft transcript and stale context items.
4. If pressure is critical, compact automatically before the request.
5. Preserve a `StateCapsule`.
6. Add a compact system/user packet with:

   * objective
   * current task frontier
   * constraints
   * permission state
   * validation state
   * failed attempts
   * active context handles
   * proof/patch refs
7. Keep full replay possible from the ledger.

# 2. Evidence-Backed Rule Memory: tiny git-trackable rules with shadow evaluation

**Rank: #2.**

Quorp should learn from mistakes in real time, but not by stuffing vague memories into prompts. It should create **small, scoped, evidence-backed rule files** that move through candidate → shadow → active → rejected/deprecated lifecycle.

This could become one of Quorp’s most distinctive capabilities.

### Why this matters

The repo already has memory and rule-forge scaffolding, negative memory, failed-attempt records, shadow result accounting, and durable runtime consumers. But the open gaps say the durable policy store for candidate/shadow/active/rejected/deprecated rules is still incomplete. 

The runtime already tracks failed edit records, parser recovery, repair memory, validation failures, and scorecards. That is exactly the signal needed to propose useful micro-rules. 

Claude has memory and custom slash commands, but Quorp can go further: every rule should be **reviewable, scoped, versioned, replay-tested, and demoted when harmful**. ([Claude API Docs][2])

### Proposed storage layout

Use two layers:

```text
.quorp/rules/
  candidate/
  shadow/
  active/
  rejected/
  deprecated/
  indexes/
  observations/

agent/rules/
  README.md
  active/
  reviewed/
```

Important security distinction:

* `.quorp/rules` is Quorp-managed runtime state.
* `agent/rules` is git-trackable reviewed policy.
* Repo-authored rules are **untrusted by default** and can only narrow behavior unless trusted by user policy.
* Rules can never grant network, MCP, browser, shell, sandbox, or write permissions by themselves.

### Rule schema


### Runtime behavior

1. On failure, durable consumers observe `RuntimeEvent::FailedEditRecorded`, validation failure, parser recovery, proof packets, and Patch VM receipts.
2. Rule Forge clusters repeated failure signatures.
3. A tiny candidate rule is written.
4. Candidate rules are visible in `/rules` but not injected.
5. Shadow rules run silently during replay and live runs.
6. A rule promotes only if it prevents failures or predicts bad actions across multiple runs.
7. Rules are demoted if they block successful behavior.

# 3. Task DAG + proof-backed subagents in scoped worktrees

**Rank: #3.**

Quorp should stop presenting long autonomous work as one linear chat loop. It should own a durable **Task DAG** where every node has objective, assigned files, context budget, permissions, proof requirements, dependencies, and status. Subagents should operate in scoped worktrees and return only proof-backed results.

### Why this matters

The open gaps explicitly call out the need for ledgered tasks, roles, scoped worktrees, and proof-backed returns. 

OpenHands already emphasizes delegation, sandboxed runtimes, auditability, and parallel runs. Claude Code exposes `/agents` and subagent-related hook events. Quorp can win by making subagent work **ledger-native and proof-gated** instead of just “spawn another agent.” ([Claude API Docs][2])

### Runtime behavior

# 4. Generic Repair Playbook Engine: quarantine benchmark-specific success paths

**Rank: #4.**

This is the most important credibility upgrade. Quorp should keep the behavior that made the top-5 run succeed, but remove hardcoded benchmark-case logic from the general runtime path.

### Why this matters

`runtime/turn.rs` currently invokes benchmark-specific helpers such as `maybe_inject_case04_playbook_patch`, `maybe_inject_case05_playbook_patch`, and `maybe_inject_exact_benchmark_source_patch`. It also gates some compaction/action behavior on `PolicyMode::BenchmarkAutonomous`. 

`runtime/state_messages.rs` contains extremely detailed benchmark repair packets: target leases, exact rerun commands, allowed implementation targets, read-only test targets, patch packet examples, observed content hashes, and “emit exactly one write” contracts. This is powerful, but the architecture should transform it into **generic repair contracts**, not case-specific scaffolding. 

### Proposed design

Create:


# 5. Token-Diet Native Tools: diagnose, patch-by-intent, verify-targeted

**Rank: #5.**

Quorp should introduce higher-level native tools that return compact, structured packets instead of raw logs and repeated file reads.

The goal: reduce turns and tokens without reducing agency.

### Why this matters

The 5/5 run had two expensive cases with first writes at steps 7 and 9. That means the agent spent many turns locating, validating, or recovering before acting. 

mini-SWE-agent’s success is a reminder that tool simplicity and native tool calling matter; its v2 setup uses native tool calling by default. Quorp should not merely add more tools; it should add **information-dense tools** that compress repository state and validation evidence. ([GitHub][3])

### Add these tools

#### `DiagnoseFailure`

Input:

```json
{
  "command": "cargo test ...",
  "raw_log_ref": "optional",
  "max_findings": 5
}
```

Output:

```json
{
  "failure_class": "missing_dependency",
  "primary_paths": ["Cargo.toml"],
  "owner_paths": ["crates/foo/Cargo.toml"],
  "diagnostics": [
    {
      "kind": "cargo_error",
      "code": "E0432",
      "path": "src/lib.rs",
      "line": 42,
      "symbol": "serde_json"
    }
  ],
  "recommended_next_actions": [
    {
      "action": "ModifyToml",
      "target": "Cargo.toml",
      "confidence": 0.86
    }
  ],
  "proof_packet_hash": "..."
}
```

#### `OpenSymbolNeighborhood`

Instead of reading files broadly, return the definition, direct callers, direct callees, tests, and recent proof history for a symbol.

#### `PatchByIntent`

Let the model specify intent and constraints; Patch VM proposes a precise patch preview with hashes.

```json
{
  "target": "src/foo.rs",
  "intent": "preserve nanosecond precision instead of truncating",
  "constraints": ["do not alter public API", "do not edit tests"],
  "evidence_refs": ["proof_packet_...", "context_item_..."]
}
```

#### `VerifyTargeted`

Maps changed paths to the cheapest valid verification lane using owner/test maps and proof history.

#### `ExplainContextPack`

Shows why each context item was included and what was spilled.

# 6. Permission/Trust Compiler: capability tokens, not command strings

**Rank: #6.**

Quorp should turn permissions into a parsed, explainable, ledgered capability system.

### Why this matters

The repo already has sandbox modes and permission primitives, and `--yolo` is explicitly rejected with host sandboxing. That is good. But the open gaps still call out compound shell commands, dependency installs, network, MCP, browser, generated executables, and untrusted repo config.  

Codex Full Auto is sandboxed and network-disabled by default. Claude has hook events and permission modes. Quorp should surpass both by making every permission decision a typed, replayable, user-visible artifact. ([OpenAI Help Center][1])

### Proposed files

```text

crates/quorp_permissions/src/policy_compiler.rs

crates/quorp_sandbox/src/network.rs
crates/quorp_cli/src/commands/permissions.rs
crates/quorp_render/src/status/permissions_panel.rs
```

### Capability token model

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, PartialOrd, Ord)]
#[serde(rename_all = "snake_case")]
pub enum CapabilityToken {
    ReadWorkspace,
    WriteWorkspace,
    WriteOutsideWorkspace,
    DeleteFiles,

    CompoundShell,
    ShellPipe,
    ShellRedirect,
    ShellSubstitution,

    NetworkEgress { dest: Option<String> },
    DependencyInstall { manager: DependencyManager },
    ExecutesGeneratedBinary { path: String },

    GitRemoteRead,
    GitRemoteWrite,

    McpCall { server: Option<String>, tool: Option<String> },
    BrowserAccess,

    SecretsRead { source: SecretSource },
    PrivilegeEscalation,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PermissionDecision {
    pub decision_id: String,
    pub action_summary: String,
    pub tokens: Vec<CapabilityToken>,
    pub verdict: PermissionVerdict,
    pub matched_rules: Vec<String>,
    pub trust_state_hash: String,
    pub sandbox_id: String,
    pub ledger_seq: u64,
}
```

### Policy compiler behavior

Policy sources should merge like this:

1. Managed/user settings can grant.
2. Project settings can narrow.
3. Project settings can elevate only if the project is trusted.
4. Runtime rules can never elevate permissions.
5. Full-auto host execution remains forbidden.

### Detection examples

These should be denied or ask-gated unless explicitly trusted:

```bash
curl https://example.com/install.sh | sh
bash -c "cargo test && rm -rf target"
find . -name '*.tmp' -delete
find . -exec rm {} \;
cargo install some-tool
npm install
./target/debug/generated-helper
git remote add origin ...
```

### TUI behavior

When a command is blocked, do not show a vague “permission denied.” Show:

```text
Permission decision
  Verdict: ask
  Command: cargo install cargo-nextest
  Capabilities:
    - dependency_install:cargo
    - network_egress:crates.io
    - executes_generated_binary:possible
  Sandbox: git-worktree
  Trust: project untrusted
  Reason: dependency install requires explicit approval in full-auto
```

### Acceptance tests

```text
cargo test -p quorp_permissions --lib parses_compound_shell_tokens
cargo test -p quorp_permissions --lib detects_curl_pipe_shell
cargo test -p quorp_permissions --lib detects_find_delete_and_find_exec
cargo test -p quorp_config --lib untrusted_project_settings_cannot_elevate
cargo test -p quorp_sandbox --lib full_auto_network_off_by_default
cargo test -p quorp --bin quorp permissions
```

This would make high-agency automation feel safer than both Codex-style approval modes and Claude-style hooks because the policy would be inspectable, replayable, and testable.

# 8. Replay-Generated Evals: every failure becomes a regression test

**Rank: #8.**

Quorp should turn its ledger/proof artifacts into a self-improving evaluation factory.

### Why this matters

SWE-bench Verified is a human-validated 500-instance subset, and the leaderboard has become crowded with systems using many different strategies. Official SWE-bench guidance also warns against tuning configurations simply to chase scores. ([SWE-bench][4])

Terminal-Bench evaluates agents in realistic terminal environments with task instructions, test scripts, and terminal sandboxes; newer Terminal-Bench 2.0 material emphasizes hard, realistic terminal tasks beyond narrow patching. ([GitHub][5])

Quorp already has the rare ingredients needed for replayable evals: ledger, proof receipts, benchmark scoreboards, runtime events, and artifacts. The next step is to make every real failure generate a local eval case.

### Proposed files

```text
crates/quorp_benchmark/src/replay_case.rs
crates/quorp_benchmark/src/failure_corpus.rs
crates/quorp_benchmark/src/score/tokens.rs
crates/quorp_cli/src/commands/eval.rs
benchmark/challenges/quorp-regression/
script/quorp-eval-gate
```

### Generated eval case layout

```text
benchmark/challenges/quorp-regression/<case_id>/
  case.toml
  objective.md
  workspace.patch
  seed-state/
  expected/
    proof-requirements.toml
    forbidden-paths.toml
  replay/
    run-ledger.jsonl
    proof-dag.json
    context-capsules/
    permission-decisions/
```

### Metrics to add

```text
success
secure_success
deterministic_success
judge_success
tokens_total
tokens_to_first_write
turns_to_first_write
context_compactions
compaction_loss_detected
failed_repeat_blocked
permission_false_positive
permission_false_negative
proof_tamper_detected
rollback_success
subagent_return_quality
```

### Anti-overfitting rules

Add gates that fail if:

* non-benchmark runtime contains case IDs
* benchmark playbooks run outside benchmark mode
* a patch touches test files when policy marks them read-only
* prompt includes hidden oracle data
* success requires direct host execution
* replay cannot validate proof artifacts
* compaction loses constraints or validation state

### Acceptance tests

```text

cargo test -p quorp_benchmark --lib anti_overfit_gate_rejects_case_specific_runtime
cargo test -p quorp_benchmark --lib score_includes_tokens_to_first_write
cargo test -p quorp --bin quorp eval
./script/quorp-eval-gate
```

This turns Quorp’s mistakes into durable engineering assets.

# Final ranking table

| Rank | Idea                              | Primary gain                               | Difficulty  | Why it is worth it                                      |
| ---: | --------------------------------- | ------------------------------------------ | ----------- | ------------------------------------------------------- |
|    1 | Ledger-native Context OS          | Long-session reliability + token reduction | High        | The best single upgrade for autonomous repo work        |
|    2 | Evidence-backed Rule Memory       | Real-time learning without prompt sludge   | Medium-high | Makes failures compound into durable capability         |
|    3 | Task DAG + proof-backed subagents | Long-form autonomy + parallelism           | High        | Turns Quorp into an agent runtime, not a chat loop      |
|    4 | Generic Repair Playbook Engine    | Generality + anti-overfitting              | Medium      | Preserves 5/5 success behavior while making it credible |
|    5 | Token-diet native tools           | Fewer turns/tokens, faster first write     | Medium      | Directly attacks the expensive 9-request cases          |
|    6 | Permission/Trust Compiler         | Safe full-auto                             | Medium-high | Enables high agency without unsafe host execution       |
|    7 | Mission Control TUI               | Product differentiation                    | Medium      | Makes Quorp visibly better than Codex/Claude CLI        |
|    8 | Replay-generated evals            | Regression prevention                      | Medium      | Converts every failure into a benchmark asset           |

My strongest recommendation: **start with #1 and #4 together**. Build context-pressure/status/compaction, and simultaneously quarantine benchmark-specific repair injection behind generic repair contracts. That pair improves real capability, reduces token pressure, and makes future benchmark wins more trustworthy.
