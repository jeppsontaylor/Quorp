## My top 10 ranked upgrade ideas for Quorp

The biggest finding: **Quorp’s primitives are already strong, but the runtime is still spending expensive model turns rediscovering facts that the system already has in structured form.** The 5/5 Rust SWE-bench run is encouraging, but it also exposes the exact next frontier: fewer blind turns, better state compression, stricter edit contracts, proof-backed task structure, and a TUI that makes the agent’s cognition inspectable.

I would prioritize these in this order.

---

## 1. Replace benchmark-specific repair playbooks with a general **Evidence-Gated Action Governor**

**Why this is my #1 idea:** this is the highest leverage improvement for real coding capability and token reduction. Quorp already has proof packets, validation summaries, parser recovery, failed-edit records, context packs, and repair state. The missing layer is a deterministic controller that turns those facts into the next safe action before asking the model again.

The current runtime has explicit benchmark-repair machinery and even exact benchmark patch injection in `crates/quorp_agent_core/src/runtime/benchmark_playbooks.rs`; that file names deterministic repair playbooks and includes exact patch-action generation for specific Rust SWE-bench cases. The turn loop also includes repair-read and exact benchmark patch injection hooks in `crates/quorp_agent_core/src/runtime/turn.rs`. Those mechanisms helped the 5/5 run, but they should be quarantined as benchmark harness code, not core agent capability.  

The logs show why a general governor is needed. Case 06 succeeded, but it took 9 requests and 52,083 billed tokens; first write was turn 7, and controller recovery injected a required read after an invalid repair action. Case 08 also took 9 requests and 62,702 billed tokens, with first write only at turn 9. The batch-level run was 5/5, but 28 requests and 180,546 billed tokens for five small Rust cases is the signal to optimize. ([GitHub][1])

**Core design:** add a small controller that observes `ProofPacket`, `AgentTaskStateSnapshot`, `ContextPack`, parser failures, edit failures, validation failures, and working-set history, then emits a bounded `ActionPrescription`.

The hard part is not “forcing” the model. It is giving the runtime a narrow, auditable right to say: “The next useful action is objectively a read of this range,” or “You are repeating a failed path,” or “You may not write until you have leased an implementation target.”

## 2. Build a **Context OS**: lossless state packets, automatic compaction, and visible context pressure

Quorp already has a token-budgeted `ContextCompiler`, SQLite index, context-pack provenance, memory recall, and progressive handles. The current index stores `.quorp/index/index.sqlite`, symbol rows, lexical chunks, and `context_pack_items`, while the compiler can load agent contracts from `agent/owner-map.json`, `agent/test-map.json`, `agent/proof-lanes.toml`, and `agent/generated-zones.toml`.  

What is missing is a **first-class state packet** that survives compaction and resume. Current compaction policies exist in the runtime, but they are policy labels like `Last8Ledger1024`, `BenchmarkRepairMinimal`, and `BenchmarkStatePacket`, not a general, verified state model with invariants. 

This is where Quorp can beat Claude Code and Codex CLI. Claude Code exposes memory and compaction-like concepts; its docs describe CLAUDE.md files, auto memory, `.claude/rules/`, and limited auto-memory loading. ([Claude API Docs][2]) Codex now also has CLI/app docs around memories, rules, subagents, worktrees, skills, and compaction-like product surfaces. ([OpenAI Help Center][3]) Quorp should go beyond both by making context pressure and compaction **provable, replayable, and inspectable**.

## 3. Add **Patch VM 2.0**: minimal-edit enforcement, AST-aware diff risk, and broad-write quarantine

Hard code snippet:

```rust
// crates/quorp_patch_vm/src/diff_risk.rs

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum EditRiskClass {
    Low,
    Medium,
    High,
    ForbiddenWithoutPreview,
}

#[derive(Debug, Clone)]
pub struct DiffRisk {
    pub changed_lines: usize,
    pub total_lines_before: usize,
    pub touched_hunks: usize,
    pub unchanged_prefix_lines: usize,
    pub unchanged_suffix_lines: usize,
    pub class: EditRiskClass,
    pub reason: String,
}

pub fn classify_text_update(before: &str, after: &str, policy: &WriteContractPolicy) -> DiffRisk {
    let before_lines: Vec<&str> = before.lines().collect();
    let after_lines: Vec<&str> = after.lines().collect();

    let prefix = before_lines
        .iter()
        .zip(after_lines.iter())
        .take_while(|(a, b)| a == b)
        .count();

    let suffix = before_lines[prefix..]
        .iter()
        .rev()
        .zip(after_lines[prefix..].iter().rev())
        .take_while(|(a, b)| a == b)
        .count();

    let changed_before = before_lines.len().saturating_sub(prefix + suffix);
    let changed_after = after_lines.len().saturating_sub(prefix + suffix);
    let changed_lines = changed_before.max(changed_after);

    let rewrite_ratio = if before_lines.is_empty() {
        1.0
    } else {
        changed_lines as f32 / before_lines.len() as f32
    };

    let class = if before_lines.len() > policy.max_full_rewrite_lines
        && rewrite_ratio > policy.max_rewrite_ratio_without_preview
    {
        EditRiskClass::ForbiddenWithoutPreview
    } else if changed_lines > policy.high_risk_changed_lines {
        EditRiskClass::High
    } else if changed_lines > policy.medium_risk_changed_lines {
        EditRiskClass::Medium
    } else {
        EditRiskClass::Low
    };

    DiffRisk {
        changed_lines,
        total_lines_before: before_lines.len(),
        touched_hunks: estimate_hunks(&before_lines, &after_lines),
        unchanged_prefix_lines: prefix,
        unchanged_suffix_lines: suffix,
        class,
        reason: format!(
            "changed_lines={changed_lines} rewrite_ratio={rewrite_ratio:.2}"
        ),
    }
}
```

## 8. Create a **Security Policy Kernel** for full-auto against untrusted repos

Quorp already has a good start: five permission modes, capability tokens, shell meta detection, dependency-install detection, generated-executable detection, network/MCP/browser classification, and host-vs-sandbox decisions.  Codex’s Rust CLI exposes sandbox policies such as read-only, workspace-write, and danger-full-access, and warns conceptually that full access should only be used in isolated environments. ([GitHub][6])

The upgrade is to make full-auto safe enough for hostile repos.

Add:

```text
crates/quorp_security_policy/

  src/capability_ledger.rs
  src/shell_ast.rs
  src/secret_guard.rs
  src/untrusted_config.rs
```

Core policies:

```text
1. Repo-authored config is untrusted until accepted.
2. Repo rules cannot grant capabilities.
3. Network is denied by default in full-auto.
4. Dependency installs require explicit policy or sandbox lockfile mode.
5. Generated executables are quarantined unless built from leased paths.
6. Secret reads are blocked or redacted before model context.
7. MCP/browser access requires separate capability tokens.
8. Compound shell commands are decomposed or prompted.
9. Every mutating action gets a receipt.
```

## 10. Build an eval program around **anti-overfitting, token efficiency, and long-horizon reliability**

The 5/5 top-5 run is useful evidence, but it is not enough. SWE-bench Verified is a 500-instance human-filtered benchmark intended for reliable coding-agent evaluation, and it explicitly includes a range from simple LM loops to RAG and multi-rollout systems. ([SWE-bench][8]) Terminal-Bench evaluates agents in real terminal environments with tasks, sandboxed harnesses, verification scripts, and oracle solutions. ([GitHub][9]) SWE-rebench-style work highlights the risk of stale/static benchmark contamination and argues for fresh, decontaminated interactive SWE tasks. ([Cool Papers][10])

Add:

```text
benchmark/suites/
  rust-swebench-random-50/
  rust-crates-regression/
  terminal-bench-adapter/
  context-compaction/
  rule-learning/
  sandbox-security/
  broad-write-quarantine/
  replay-derived/

crates/quorp_benchmark/src/gates/
  token_efficiency.rs
  no_benchmark_playbooks.rs
  broad_write.rs
  context_compaction.rs
  replay_equivalence.rs
```

Release gates:

```text
1. No benchmark-specific source patch injection in normal runtime.
2. Top-5 cohort still passes with benchmark playbooks disabled.
3. Median first-write turn decreases.
4. Median billed tokens per solved case decreases.
5. No full-file rewrites for small semantic changes.
6. Context compaction preserves state invariants.
7. Rule candidates do not become active without evidence.
8. Security eval blocks network, secrets, generated executable, risky MCP.
9. Replay fixtures pass deterministically.
10. Terminal-Bench adapter runs at least a smoke suite.
```

**Success metric:** Quorp is not “optimized for five cases”; it is measurably improving across fresh, replayable, adversarial, and long-context tasks.

# Final ranking by expected capability gain

1. **Evidence-Gated Action Governor** — biggest reduction in wasted turns; generalizes benchmark recovery.
2. **Context OS with lossless state packets** — unlocks long sessions and token reduction.
3. **Patch VM 2.0 minimal-edit enforcement** — improves correctness, safety, reviewability.
4. **Git-trackable Rule Forge files** — real-time learning without unsafe hidden memory.
5. **Proof-backed task DAG/subagents** — long-horizon repo work and parallelism.
6. **Mission Control TUI** — makes Quorp feel obviously more powerful than chat-style CLIs.
7. **Validation Intelligence Plane** — converts failures into targeted repair actions.
8. **Security Policy Kernel** — makes full-auto credible on untrusted repos.
9. **Replayable Agent Debugger** — turns every failure into a regression/eval asset.
10. **Anti-overfitting evaluation program** — proves Quorp is genuinely improving.

My strongest opinion: **do not add more model prompts first.** Quorp’s next breakthrough is not another instruction block; it is a runtime that treats agent cognition as a ledgered, proof-backed, compactable, inspectable state machine. That is the path to fewer tokens, fewer wrong turns, safer autonomy, and a terminal UX that feels ahead of Claude Code and Codex CLI rather than adjacent to them.
